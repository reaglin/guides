Developing and Documenting Mobile App Prototypes

📱 Course Module Overview

This guide will help you develop and document a mobile app prototype that other students can effectively evaluate. You'll learn about different prototyping approaches, tools, and how to create documentation that supports Cognitive Walkthroughs and Heuristic Evaluations.

Key Goal: Create a prototype and documentation package that allows peer reviewers to thoroughly evaluate your design without your presence.

Why Prototype?

Prototyping is the process of creating an early version of your application to test ideas, gather feedback, and refine your design before investing in full development. For mobile apps, prototyping is especially critical because:

💰 Cost-Effective

Finding and fixing design problems in a prototype costs a fraction of what it costs after development.

⚡ Fast Iteration

Quickly test multiple design approaches without writing production code.

👥 User Feedback

Get real user input on your design before committing resources to development.

🎯 Validate Assumptions

Test whether your design actually solves the user's problem.

The Prototype Spectrum

Prototypes exist on a spectrum from low-fidelity (quick sketches) to high-fidelity (looks and behaves like the final product):

Fidelity Dimensions

  • Visual Fidelity: How closely does it match the final visual design?
  • Functional Fidelity: How much of the functionality is actually working?
  • Content Fidelity: Is it using real content or placeholder text/images?

🎯 For This Course

  • You'll create a medium to high-fidelity prototype of a mobile app
  • It should be interactive enough for others to walk through key tasks
  • Documentation is as important as the prototype itself
  • Your prototype must support evaluation by peers who haven't seen it before

Low-Fidelity Prototypes

📝 Paper Prototypes

What: Hand-drawn screens on paper, index cards, or sticky notes

Best For: Early ideation, rapid iteration, team brainstorming

Advantages:

  • Extremely fast to create
  • No tools required
  • Easy to throw away and start over
  • Encourages creativity

Disadvantages:

  • Difficult to share remotely
  • Can't simulate mobile interactions well
  • May feel too crude for stakeholders
  • Hard to test visual design
For This Course: Paper prototypes alone are not recommended for your final submission, as they're difficult for peers to evaluate independently. However, they're great for initial design work.

🖼️ Digital Wireframes

What: Simple, grayscale layouts showing structure and content without visual design

Best For: Defining information architecture, layout, and navigation

Advantages:

  • Focus on functionality over aesthetics
  • Quick to create with tools
  • Easy to share digitally
  • Can be made clickable

Disadvantages:

  • Lacks visual appeal
  • May be hard for non-designers to visualize
  • Doesn't test visual hierarchy
  • Limited interaction testing

Medium-Fidelity Prototypes

🎨 Interactive Mockups

What: Clickable designs with some visual styling and realistic content

Best For: User testing, stakeholder presentations, design validation

Advantages:

  • Looks more like final product
  • Can test visual design and interaction
  • Good balance of speed and fidelity
  • Works well for remote evaluation

Disadvantages:

  • Takes more time than wireframes
  • May still have limited interactions
  • Can create false expectations
  • Requires design tool proficiency
For This Course: This is the recommended approach. Interactive mockups provide enough fidelity for meaningful evaluation while being achievable within course timeframes.

High-Fidelity Prototypes

📱 High-Fidelity Interactive Prototypes

What: Fully designed, highly interactive prototypes that closely mimic the final app

Best For: Final validation, investor presentations, complex interaction testing

Advantages:

  • Realistic user experience
  • Tests complex interactions
  • Impressive to stakeholders
  • Can include animations and transitions

Disadvantages:

  • Time-intensive to create
  • Harder to make changes
  • Requires advanced tool knowledge
  • May over-commit to a design direction

💻 Coded Prototypes

What: Functional prototypes built with actual code

Best For: Testing technical feasibility, performance testing, developer handoff

For This Course: Coded prototypes are not required and generally not recommended unless you have strong development skills. The focus should be on design and usability, not implementation.

The Prototype Development Process

Step-by-Step Workflow

PHASE 1: PLANNING (Before You Design)

1. Define Core User Tasks
- List 3-5 primary tasks users will complete
- Write brief use cases for each
- Identify the screens needed for each task

2. Create a Screen Flow Diagram
- Map out navigation between screens
- Show decision points
- Identify entry and exit points

3. Identify Key Screens
- Onboarding/login
- Home/dashboard
- Main task screens
- Forms and inputs
- Confirmation/success states
- Error states

PHASE 2: DESIGN (Creating Screens)

4. Start with Low-Fi Sketches
- Sketch 2-3 versions of key screens
- Get feedback from classmates
- Select the best direction

5. Create Screen Designs
- Set up artboards in your tool
- Use a UI kit or design system
- Design all screens for your main tasks
- Maintain visual consistency

6. Add Real Content
- Replace Lorem Ipsum with realistic text
- Use appropriate images
- Include actual data examples
- Make it feel authentic

PHASE 3: PROTOTYPING (Making it Interactive)

7. Create Interactive Flows
- Link screens together with hotspots
- Add navigation interactions
- Define transitions (if supported)
- Test the flow yourself

8. Add Interaction States
- Button hover/pressed states
- Form field focus states
- Loading states
- Success/error feedback

9. Refine and Polish
- Check all links work
- Verify mobile gestures (tap targets)
- Test on actual device size
- Fix any broken interactions

PHASE 4: DOCUMENTATION (Critical for Peer Review)

10. Create Documentation
- Write overview of the app
- List user tasks with instructions
- Provide access link/files
- Include any special notes

Design Best Practices for Mobile

Touch Targets

  • Minimum size: 44x44 pixels (iOS) or 48x48 dp (Android)
  • Larger for primary actions
  • Adequate spacing between targets
  • Easy to reach with thumb

Typography

  • Body text: 16px minimum
  • Line height: 1.4-1.6 for readability
  • Limit line length on larger screens
  • Sufficient contrast (4.5:1 minimum)

Navigation

  • Use standard mobile patterns (tab bar, hamburger menu)
  • Make navigation always accessible
  • Show current location clearly
  • Provide back navigation

Forms

  • One column layout
  • Appropriate input types (email, number, date)
  • Clear field labels
  • Inline validation

What to Include in Your Prototype

✓ Required Screens and States

  • All screens for 3-5 core user tasks (must be complete flows)
  • Welcome/onboarding screen (if applicable to your app)
  • Home screen or main dashboard
  • Navigation UI (tab bar, menu, back buttons)
  • Form screens (if users input data)
  • At least one confirmation/success screen
  • At least one error state (show how errors are handled)
  • Empty states (what shows when there's no content)

⚠️ Common Mistakes to Avoid

  • Incomplete flows: Don't leave dead ends—every interactive element should go somewhere
  • Inconsistent navigation: Use the same navigation pattern throughout
  • Too many screens: Focus on core tasks; don't try to design the entire app
  • Ignoring mobile constraints: Design for thumbs and small screens
  • Lorem Ipsum everywhere: Use real content so evaluators understand context
  • No feedback: Show loading, success, and error states

Testing Your Prototype

Self-Testing Checklist

Before sharing with peers, test your prototype yourself:

  1. Walk through each task
    • Can you complete it from start to finish?
    • Are there any broken links or dead ends?
    • Does the flow make logical sense?
  2. Check on an actual mobile device
    • Text readable? Touch targets large enough?
    • Does it feel like a real mobile app?
    • Any layout issues on the actual screen size?
  3. Test with a friend
    • Give them a task without explaining
    • Can they figure out what to do?
    • Watch where they get confused
  4. Review visual consistency
    • Same fonts and sizes throughout?
    • Consistent colors and buttons?
    • Aligned elements?

Why Documentation Matters

Your prototype will be evaluated by other students who:

  • Have never seen your app before
  • Don't know what it's supposed to do
  • Need to understand the context to provide meaningful feedback
  • Will be conducting structured evaluations (Cognitive Walkthrough, Heuristic Evaluation)
Critical Point: Without good documentation, evaluators can't do their job. A brilliant prototype with poor documentation will receive poor evaluations. Treat documentation as 50% of your deliverable.

What is a Cognitive Walkthrough?

A Cognitive Walkthrough is a usability evaluation method where evaluators step through specific tasks in your interface, asking themselves questions about whether users will understand what to do at each step.

What Evaluators Will Ask

For each action in a task, evaluators consider:

  1. Will users know what to do? Is the correct action obvious?
  2. Will users see how to do it? Is the control/element visible and understandable?
  3. Will users understand the feedback? After acting, will they know if it worked?
  4. Will users know they're making progress? Can they tell they're moving toward their goal?

WALKTHROUGH TIPS:
- Step through each action as if you're the user
- At each step, ask: Would the user know what to do?
- Note any confusion, hesitation, or unclear paths
- Don't assume knowledge the user wouldn't have

Design Considerations for Cognitive Walkthroughs

✓ Make Actions Obvious

  • Use clear, action-oriented labels
  • Place buttons where users expect them
  • Use familiar icons with labels
  • Make tappable elements look tappable

✓ Provide Clear Feedback

  • Show when buttons are pressed
  • Display loading states
  • Confirm successful actions
  • Explain errors clearly

✓ Show Progress

  • Use progress indicators for multi-step tasks
  • Highlight current location in navigation
  • Show what's completed vs. remaining
  • Provide breadcrumbs when appropriate

✓ Match Mental Models

  • Use familiar patterns and conventions
  • Group related items logically
  • Label things as users would name them
  • Follow platform guidelines (iOS/Android)

Example Task Write-Up for Cognitive Walkthrough

TASK: Book a Hair Appointment

User Profile:
Jennifer is a 28-year-old teacher who has used the salon's services before
but this is her first time using their new mobile app. She's comfortable
with smartphone apps and regularly uses apps like OpenTable and ClassPass.

Context/Scenario:
It's Sunday evening and Jennifer realizes she needs a haircut before next
weekend. She wants to book an appointment with her regular stylist, Maria,
for sometime next Wednesday after 3pm. She opens the salon's app on her
iPhone.

Starting Point:
App Home Screen (user is already logged in)

Goal:
Successfully book a haircut appointment with Maria for next Wednesday
after 3pm.

Success Criteria:
User sees a confirmation screen showing:
- Appointment date and time
- Stylist name (Maria)
- Service (Haircut)
- Confirmation number

Expected Path:
1. Home Screen → Tap "Book Appointment"
2. Service Selection → Select "Haircut"
3. Stylist Selection → Select "Maria"
4. Date/Time Selection → Choose next Wednesday, time after 3pm
5. Confirmation → Review and confirm booking
6. Success Screen → See confirmation details

Things to Watch For:
- Can user easily find the booking feature?
- Is it clear how to select a specific stylist?
- Can user see only available time slots?
- Does the app confirm the booking clearly?

⚠️ Common Issues Found in Cognitive Walkthroughs

  • Unclear labels: Buttons that don't clearly indicate what they do
  • Hidden features: Important actions buried in menus
  • Weak feedback: User taps but nothing appears to happen
  • No error recovery: User makes mistake but can't go back
  • Inconsistent patterns: Same action works differently in different places

What is a Heuristic Evaluation?

A Heuristic Evaluation is where evaluators examine your interface against established usability principles (heuristics). Unlike Cognitive Walkthrough which focuses on specific tasks, Heuristic Evaluation looks at the overall design quality.

Nielsen's 10 Usability Heuristics

Evaluators will check your design against these principles:

  1. Visibility of system status - Keep users informed
  2. Match between system and real world - Use familiar language
  3. User control and freedom - Provide undo/escape
  4. Consistency and standards - Follow conventions
  5. Error prevention - Design to prevent problems
  6. Recognition rather than recall - Make options visible
  7. Flexibility and efficiency - Support different skill levels
  8. Aesthetic and minimalist design - Remove unnecessary elements
  9. Help users recognize, diagnose, and recover from errors - Clear error messages
  10. Help and documentation - Provide support when needed

Design Checklist by Heuristic

Self-Evaluation Before Submission

Check your design against each heuristic:

1. VISIBILITY OF SYSTEM STATUS
□ Loading indicators for any wait times
□ Progress indicators for multi-step processes
□ Selected/active state clearly shown in navigation
□ Success messages after actions complete
□ Current location obvious in the app

2. MATCH BETWEEN SYSTEM AND REAL WORLD
□ Language users would actually use (not technical jargon)
□ Icons that match user expectations
□ Familiar metaphors and concepts
□ Information presented in logical order

3. USER CONTROL AND FREEDOM
□ Back button on every screen (except home)
□ Way to cancel actions in progress
□ Undo for destructive actions
□ Easy way to return to main navigation

4. CONSISTENCY AND STANDARDS
□ Same action button style throughout
□ Navigation works the same way on all screens
□ Same placement for common elements (e.g., titles, buttons)
□ Follows iOS or Android design guidelines

5. ERROR PREVENTION
□ Confirmation before destructive actions (delete, cancel)
□ Input validation (e.g., correct keyboard type)
□ Constraints on inputs (date pickers, not free text)
□ Disable invalid actions rather than showing errors

6. RECOGNITION RATHER THAN RECALL
□ Options visible, not hidden in menus
□ Previous entries/history shown
□ Labels on all icons
□ Context always clear

7. FLEXIBILITY AND EFFICIENCY OF USE
□ Quick actions for common tasks
□ Shortcuts where appropriate
□ Recently used items accessible
□ Favorites or bookmarks for frequent items

8. AESTHETIC AND MINIMALIST DESIGN
□ Only essential information on each screen
□ Clear visual hierarchy
□ White space used effectively
□ No unnecessary decorative elements

9. ERROR RECOGNITION, DIAGNOSIS, AND RECOVERY
□ Clear error messages (not codes)
□ Explain what went wrong
□ Suggest how to fix it
□ Error visible where relevant (e.g., next to form field)

10. HELP AND DOCUMENTATION
□ Onboarding for new users
□ Tooltips or hints where needed
□ Help accessible but not intrusive
□ FAQ or support easy to find

Mobile-Specific Heuristics

Additional Considerations for Mobile

Beyond Nielsen's heuristics, mobile apps have unique requirements:

  • Touch-friendly: Adequate touch target sizes (44x44+ pixels)
  • Thumb-reachable: Important actions within easy thumb reach
  • Readable: Text size appropriate for mobile viewing
  • Responsive: Fast interactions, no lag
  • Offline-capable: Graceful degradation without network
  • Battery-conscious: No unnecessary processing or GPS
  • Contextual: Adapts to user's context and environment

Heuristic Compliance Notes:
H1 (Visibility): Progress shown, current exercise highlighted, timer visible
H2 (Real world): Clear labels, workout jargon users know
H3 (Control): Back button, ability to pause or end
H4 (Consistency): Matches patterns from other workout apps
H5 (Error prevention): Next exercise disabled during rest
H6 (Recognition): Visual progress indicators, don't need to remember
H8 (Minimalist): Only relevant info for current context

⚠️ Common Heuristic Violations in Student Projects

  • No feedback after actions: User taps button, nothing indicates it worked
  • Inconsistent navigation: Back button works differently on different screens
  • Cryptic error messages: "Error 404" instead of "We couldn't find that workout"
  • No way to undo: Especially for delete or cancel actions
  • Hidden critical features: Important actions in overflow menus
  • Cluttered screens: Trying to show too much at once
  • Unlabeled icons: Users have to guess what icons mean

4. USER TASK SCENARIOS
─────────────────────────────────────────────────

TASK 1: Find a Vegetarian Recipe for Dinner

User Profile:
Sarah is a 20-year-old sophomore living in a dorm with a shared kitchen.
She recently became vegetarian and is looking for easy recipes she can
make with limited equipment. She's comfortable using recipe apps like
AllRecipes but finds them overwhelming with too many options.

Scenario:
It's 5pm on a Tuesday and Sarah wants to find something quick to make
for dinner. She has about 30 minutes and wants a vegetarian recipe that's
easy for a beginner. She opens MealMatch on her iPhone.

Starting Point: Home Screen

Goal:
Find a vegetarian recipe that takes 30 minutes or less and is rated
"Easy" difficulty.

Success Criteria:
User has found a recipe that meets the criteria and can view the full
recipe details including ingredients and instructions.

Expected Path:
1. Home Screen → Tap "Browse Recipes"
2. Browse Recipes → Tap "Filters" button
3. Filters → Select "Vegetarian" dietary filter
4. Filters → Select "Under 30 min" time filter
5. Filters → Select "Easy" difficulty
6. Filters → Tap "Apply Filters"
7. Browse Recipes (filtered) → Tap on recipe card
8. Recipe Detail → View full recipe

What to Look For:
- Can user easily find the filter option?
- Are filter options clearly labeled?
- Is it obvious how to apply filters?
- Do results clearly show they're filtered?
- Is the recipe detail screen clear and complete?

Sarah found a recipe she likes and wants to save it so she can easily

6. DESIGN DECISIONS & NOTES
─────────────────────────────────────────────────

Color Scheme:
Warm, food-friendly colors (orange, cream) to feel inviting and
appetizing without being childish.

Navigation:
Used iOS tab bar pattern because users are familiar with it and it
makes main features always accessible.

Filters:
Decided on a filter panel rather than inline chips because there are
many filter options and panel allows better organization by category.

Recipe Cards:
Show photo, title, time, difficulty, and rating at a glance so users
can quickly scan options.

Form Design:
Add Recipe form uses one column layout with clear sections. Multiple
ingredients use a dynamic list that grows as you add more.

Known Limitations:
- Search is visual only (not interactive)
- Filter combinations are limited to what's shown
- Recipe detail is simplified (no reviews shown, just rating)
- Profile screen is a placeholder

Common Submission Mistakes

⚠️ Avoid These Pitfalls

  • Broken prototype link: Link requires sign-in or doesn't work in incognito mode
  • Wrong sharing permissions: Link set to private or requires request access
  • Missing documentation: Just submitting prototype without written documentation
  • Vague task descriptions: Tasks lack context, starting points, or success criteria
  • Incomplete flows: Tasks can't actually be completed in the prototype
  • No scope definition: Evaluators don't know what to evaluate vs. what's incomplete
  • Poor formatting: Documentation is hard to read or navigate
  • Last-minute rush: Not enough time to test or refine before submission
