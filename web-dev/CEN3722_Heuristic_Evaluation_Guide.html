CEN3722: Human-Computer Interaction

Guide to Performing Heuristic Evaluations

📊 Module Overview

This guide will teach you how to perform a Heuristic Evaluation—a systematic method for examining an interface against established usability principles. You'll learn what it is, understand the ten heuristics, see complete examples, and master how to document your findings to provide valuable feedback.

Definition

A Heuristic Evaluation (HE) is a usability inspection method where evaluators examine an interface and judge its compliance with recognized usability principles called "heuristics."

Unlike Cognitive Walkthrough which focuses on specific tasks, Heuristic Evaluation looks at the overall design quality of the interface.

Core Concept

You systematically review the interface, checking whether it follows established usability principles. When you find a violation of a heuristic, you document it as a usability problem.

Nielsen's 10 Usability Heuristics

The most widely used heuristics were developed by Jakob Nielsen. These are the principles you'll use to evaluate the interface:

1Visibility of System Status

The system should always keep users informed about what is going on, through appropriate feedback within reasonable time.

Examples: Loading indicators, progress bars, confirmation messages, highlighting current page in navigation

2Match Between System and Real World

The system should speak the users' language, using words, phrases, and concepts familiar to the user rather than system-oriented terms.

Examples: Using "Shopping Cart" instead of "SKU Repository," calendar icons for dates, familiar metaphors

3User Control and Freedom

Users often choose system functions by mistake and need a clearly marked "emergency exit" to leave the unwanted state.

Examples: Undo/redo, cancel buttons, back navigation, ability to exit processes

4Consistency and Standards

Users should not have to wonder whether different words, situations, or actions mean the same thing. Follow platform conventions.

Examples: Same button style throughout, consistent terminology, following iOS/Android guidelines

5Error Prevention

Even better than good error messages is careful design which prevents problems from occurring in the first place.

Examples: Confirmation dialogs, input validation, disabling invalid options, constraints on inputs

6Recognition Rather Than Recall

Minimize the user's memory load by making objects, actions, and options visible. The user should not have to remember information from one part of the dialogue to another.

Examples: Visible menu options, autocomplete, recently used items, persistent navigation

7Flexibility and Efficiency of Use

Accelerators—unseen by the novice user—may often speed up the interaction for the expert user such that the system can cater to both inexperienced and experienced users.

Examples: Keyboard shortcuts, quick actions, favorites, search, advanced filters

8Aesthetic and Minimalist Design

Dialogues should not contain information which is irrelevant or rarely needed. Every extra unit of information competes with relevant units of information.

Examples: Clean layouts, appropriate white space, hiding advanced options, progressive disclosure

9Help Users Recognize, Diagnose, and Recover from Errors

Error messages should be expressed in plain language (no codes), precisely indicate the problem, and constructively suggest a solution.

Examples: "Email address is required" instead of "Error 403," specific guidance on fixing errors

10Help and Documentation

Even though it's better if the system can be used without documentation, it may be necessary to provide help and documentation.

Examples: Tooltips, help sections, tutorials, FAQs, contextual help

🎯 Key Characteristics of Heuristic Evaluation

  • Evaluates against established principles, not personal opinion
  • Looks at overall design, not just specific tasks
  • Can be done by experts without users present
  • Finds both specific issues and general design problems
  • Fast and cost-effective evaluation method

HE vs. Cognitive Walkthrough

Aspect Heuristic Evaluation Cognitive Walkthrough
Focus Overall design quality Task completion and learnability
Method Check against usability principles Step through tasks asking questions
Scope Entire interface or major sections Specific user tasks
Finds Broad usability issues, inconsistencies Task-specific problems, unclear paths
Best For General usability assessment Evaluating specific workflows

Why Use Heuristic Evaluation?

  • Efficient: Can evaluate entire interface quickly
  • Comprehensive: Catches diverse types of usability issues
  • Early application: Works with prototypes and designs
  • No users needed: Expert evaluation method
  • Proven principles: Based on well-established heuristics

Example Scenario: Fitness Class Booking App

Let's walk through a complete Heuristic Evaluation example to see exactly how it works.

📱 App Context

App: "FitBook" - Mobile app for booking fitness classes

Target Users: Adults who attend gym classes
Platform: iOS mobile app

Screens Being Evaluated:
- Home/Dashboard
- Browse Classes
- Class Detail
- Booking Confirmation

Evaluation Approach:
Systematically review each screen, checking for violations of each
of the 10 heuristics

Heuristic-by-Heuristic Evaluation

Let's examine the app against each heuristic and document findings:

1Visibility of System Status

EVALUATION:

Screen: Browse Classes
Checking: Do users know what the system is doing?

✓ GOOD: When filtering classes, a subtle "Filtering..." indicator appears
✓ GOOD: Current tab in bottom navigation is highlighted
✓ GOOD: Selected date in calendar is clearly marked

🔴 ISSUE FOUND:
When user taps "Book Class" button, there's a 2-3 second delay before
the confirmation screen appears. During this time, there's NO loading
indicator or feedback. User might think the button didn't work and tap
again.

Severity: MAJOR
Heuristic Violated: #1 - Visibility of System Status
Location: Class Detail screen → Booking transition
Problem: No loading indicator during booking process
Impact: Users are left uncertain if their action worked. May tap
multiple times, potentially creating duplicate bookings.
Recommendation: Add loading spinner or progress indicator immediately
after "Book Class" is tapped, with message "Processing your booking..."

───────────────────────────────────────────────

Screen: Home Dashboard

✓ GOOD: "Next Class" widget shows upcoming class clearly
✓ GOOD: "Classes Today" count is prominently displayed

No additional issues found for Heuristic #1 on this screen.

2Match Between System and Real World

EVALUATION:

Screen: Browse Classes
Checking: Does the app use familiar language and concepts?

✓ GOOD: Uses "Classes" instead of "Sessions" or "Events"
✓ GOOD: Time slots use 12-hour format (9:00 AM) not 24-hour
✓ GOOD: Difficulty levels use "Beginner/Intermediate/Advanced" not numbers

🟡 MINOR ISSUE FOUND:
Filter section uses "Duration" as a label. While technically correct,
"Class Length" or "Time" would be more natural language for average users.

Severity: MINOR
Heuristic Violated: #2 - Match Between System and Real World
Location: Browse Classes screen, filter panel
Problem: "Duration" is slightly formal/technical term
Impact: Minor - most users will understand, but could be more natural
Recommendation: Change label to "Class Length" or "Time"

───────────────────────────────────────────────

Screen: Class Detail

🔴 ISSUE FOUND:
Class capacity shows "14/20 slots occupied." The phrase "slots occupied"
is awkward and slightly technical. Users typically think in terms of
"spots available" or "spaces left."

Severity: MODERATE
Heuristic Violated: #2 - Match Between System and Real World
Location: Class Detail screen, capacity indicator
Problem: "Slots occupied" is not natural language
Impact: Users have to mentally convert to understand "how many
spots are left"
Recommendation: Change to "6 spots available" or "14 of 20 spots taken"

3User Control and Freedom

EVALUATION:

Screen: All screens
Checking: Can users undo actions and exit unwanted states?

✓ GOOD: Back button on every screen (except home)
✓ GOOD: Can cancel out of booking process at any step

🔴 CRITICAL ISSUE FOUND:
After booking a class, there is NO way to cancel or undo the booking
from within the app. The confirmation screen only shows "View My Schedule"
and "Book Another Class" buttons. Users who accidentally book the wrong
class or wrong time are stuck.

Severity: CRITICAL
Heuristic Violated: #3 - User Control and Freedom
Location: Booking Confirmation screen
Problem: No way to cancel a booking that was just made
Impact: Users who make mistakes have no recourse. This will cause
frustration and support calls. Users may stop using app out of fear of
making irreversible mistakes.
Recommendation: Add "Cancel This Booking" button on confirmation screen.
Also add ability to cancel bookings from "My Schedule" screen.

───────────────────────────────────────────────

Screen: Browse Classes with Filters Applied

🟡 MODERATE ISSUE FOUND:
When filters are applied (e.g., "Yoga only"), there's no quick way to
clear all filters at once. User must open filter panel and deselect
each filter individually.

Severity: MODERATE
Heuristic Violated: #3 - User Control and Freedom
Location: Browse Classes screen with active filters
Problem: Cannot quickly clear all filters
Impact: Inefficient when users want to start over with browsing
Recommendation: Add "Clear All Filters" button or "X" to remove
individual active filter tags

4Consistency and Standards

EVALUATION:

Across all screens
Checking: Is the design consistent? Does it follow platform conventions?

✓ GOOD: Primary action buttons all use same blue color
✓ GOOD: Navigation follows iOS tab bar pattern
✓ GOOD: Swipe gestures work as expected (back navigation)

🟡 MODERATE ISSUE FOUND:
The app uses "Book" button on some screens and "Reserve" button on others
for the same action. On Class Detail screen, button says "Book This Class."
On Quick Book widget, it says "Reserve Spot."

Severity: MODERATE
Heuristic Violated: #4 - Consistency and Standards
Location: Multiple screens (Class Detail, Quick Book widget)
Problem: Inconsistent terminology for same action
Impact: Creates confusion about whether these are different actions
Recommendation: Standardize on one term throughout app. "Book" is
recommended as it's more common in this context.

───────────────────────────────────────────────

🔴 ISSUE FOUND:
Cancel buttons appear on the RIGHT side on some screens (booking flow)
and LEFT side on other screens (profile edit). iOS convention is typically
cancel on left, confirm on right.

Severity: MAJOR
Heuristic Violated: #4 - Consistency and Standards
Location: Multiple dialog boxes
Problem: Inconsistent button placement violates iOS conventions
Impact: Users might tap wrong button accidentally, especially when
acting quickly
Recommendation: Follow iOS convention: Cancel left, Primary action right

💡 Note on the Example

In this example, we only showed detailed evaluation for 4 of the 10 heuristics. In your actual evaluation, you would systematically check ALL 10 heuristics across ALL screens. This example demonstrates the methodology—you would apply the same approach to the remaining heuristics (#5-10).

🎯 Key Takeaways from This Example

  • Go through each heuristic systematically across all screens
  • Note both violations AND good examples of compliance
  • Severity is based on impact, not just principle violation
  • Be specific about location and exact problem
  • Provide actionable recommendations for each issue
  • Look for patterns (e.g., consistency issues across multiple screens)

Documentation Principles

Effective documentation helps designers understand and fix usability problems. Your report should be clear, organized, and actionable.

What Makes Good HE Documentation?

  • Systematic: Organized by heuristic or by severity
  • Specific: Exact location and clear description
  • Justified: Explains which heuristic is violated and why
  • Actionable: Provides concrete recommendations
  • Balanced: Notes both problems and good design

Severity Ratings

Rate issues based on their impact on usability:

Severity Definition Examples
CRITICAL Prevents users from completing important tasks or causes data loss No way to undo critical action; app crashes; data lost; core feature broken
MAJOR Causes significant problems for most users Poor error messages; inconsistent navigation; missing important feedback
MODERATE Causes annoyance or inefficiency; affects some users Unclear labels; minor inconsistencies; missing shortcuts; cluttered screens
MINOR Cosmetic issues or very minor problems Small text inconsistency; slight alignment issue; minor wording improvement

💡 Determining Severity

Consider three factors:

  • Frequency: How often will users encounter this?
  • Impact: How badly does it affect users?
  • Persistence: Will users learn to work around it or is it always a problem?

High frequency + High impact + Persistent = Higher severity

Examples: Good vs. Bad Documentation

✗ Bad Example

Issue: Error messages are bad

Heuristic: #9

Severity: Bad

Fix: Make them better

Why this is bad:

  • No specific location
  • Doesn't explain what's wrong
  • Doesn't explain the violation
  • Not actionable
  • "Bad" is not a severity level

✓ Good Example

ISSUE #5: Error Message Uses Code
Instead of Plain Language

Heuristic: #9 - Help Users Recognize,
Diagnose, and Recover from Errors

Location: Login screen, after
failed login attempt

Severity: MAJOR

Problem: When login fails, the error
message displays "Error 401:
Authentication failed." This is a
technical error code that doesn't help
users understand what went wrong.

Violation: Heuristic #9 requires error
messages in plain language that explain
the problem and suggest solutions. This
message uses a code (401) and technical
term (Authentication) instead of
user-friendly language.

Impact: Users don't understand if they
entered wrong password, wrong email, or
if their account has an issue. They
can't diagnose the problem.

Recommendation: Change message to:
"The email or password you entered is
incorrect. Please try again or tap
'Forgot Password' if you need help."

Documentation Best Practices

✓ Do This

  • Evaluate ALL 10 heuristics across ALL screens
  • Be systematic - use a checklist to ensure coverage
  • Document as you go, don't rely on memory
  • Take screenshots of every issue
  • Clearly tie each issue to a specific heuristic
  • Include both problems and positive examples
  • Provide specific, actionable recommendations
  • Use consistent severity ratings
  • Proofread and organize before submitting

⚠️ Common Documentation Mistakes

  • Skipping heuristics: Not checking all 10 systematically
  • Personal preference: "I don't like the color" (not a heuristic violation)
  • Vague issues: Not specifying exact location or problem
  • No justification: Not explaining which heuristic or why it's violated
  • Inconsistent severity: Rating everything major or being too lenient
  • Only negative: Not noting what works well
  • No recommendations: Identifying problems without solutions
  • Too brief: Not providing enough detail to be actionable

Evaluation Process Tips

During Evaluation

  • Go through the app multiple times
  • First pass: General exploration
  • Subsequent passes: Systematic heuristic checks
  • Take notes as you find issues
  • Screenshot everything

For Each Screen

  • Check against all 10 heuristics
  • Look at navigation, content, interactions
  • Try different paths through the screen
  • Consider edge cases and errors
  • Note what works well too

Time Management

  • Allow 15-30 minutes per screen
  • Budget time for documentation
  • Take breaks between screens
  • Review and organize findings at the end

Quality Checks

  • Did I check all 10 heuristics?
  • Is each issue clearly documented?
  • Are severity ratings consistent?
  • Are recommendations specific?
  • Did I include positive findings?

The 10 Heuristics (Quick Reference)

H1: VISIBILITY OF SYSTEM STATUS
Keep users informed about what's happening
→ Look for: Loading indicators, progress, feedback, current location

H2: MATCH BETWEEN SYSTEM AND REAL WORLD
Use familiar language and concepts
→ Look for: Jargon, unclear terms, unusual metaphors

H3: USER CONTROL AND FREEDOM
Provide undo and ways to exit unwanted states
→ Look for: Undo/cancel options, back buttons, escape routes

H4: CONSISTENCY AND STANDARDS
Follow conventions and be consistent
→ Look for: Inconsistent terminology, different patterns, platform violations

H5: ERROR PREVENTION
Prevent problems before they occur
→ Look for: Confirmation dialogs, validation, constraints, defaults

H6: RECOGNITION RATHER THAN RECALL
Make options visible, minimize memory load
→ Look for: Hidden features, remembering info across screens

H7: FLEXIBILITY AND EFFICIENCY OF USE
Provide shortcuts and options for power users
→ Look for: Lack of shortcuts, no customization, inefficient workflows

H8: AESTHETIC AND MINIMALIST DESIGN
Remove unnecessary information
→ Look for: Clutter, irrelevant info, poor visual hierarchy

H9: HELP USERS RECOGNIZE, DIAGNOSE, AND RECOVER FROM ERRORS
Error messages should be helpful and in plain language
→ Look for: Cryptic errors, no guidance, error codes

H10: HELP AND DOCUMENTATION
Provide assistance when needed
→ Look for: No help available, hard to find, not contextual

Severity Quick Guide

Severity When to Use Quick Test
Critical Prevents completion of important tasks "Can users accomplish their main goals?"
Major Causes significant problems for most users "Will this frustrate or confuse many users?"
Moderate Creates inefficiency or confusion "Will users notice and be annoyed?"
Minor Small cosmetic or polish issues "Is this a nitpick or minor improvement?"

Common Pitfalls to Avoid

  • ❌ Only looking for problems, ignoring what works well
  • ❌ Basing evaluation on personal taste instead of heuristics
  • ❌ Giving vague feedback like "navigation is confusing"
  • ❌ Not connecting issues to specific heuristics
  • ❌ Rating everything as major or minor (inconsistent severity)
  • ❌ Providing no recommendations, just identifying problems
  • ❌ Skipping heuristics that seem less relevant
  • ❌ Rushing through the evaluation
  • ❌ Focusing only on visual design, ignoring interaction issues
