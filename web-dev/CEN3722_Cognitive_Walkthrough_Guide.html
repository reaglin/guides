CEN3722: Human-Computer Interaction

Guide to Performing Cognitive Walkthroughs

🔍 Module Overview

This guide will teach you how to perform a Cognitive Walkthrough—a systematic method for evaluating the usability of an interface by stepping through tasks from a user's perspective. You'll learn what it is, see a complete example, and understand how to document your findings to provide actionable feedback.

Definition

A Cognitive Walkthrough (CW) is a usability evaluation method where you systematically walk through an interface, asking yourself at each step: "Would a user know what to do here?"

You act as a "surrogate user," examining whether the interface makes it clear how to accomplish specific tasks, especially for first-time users.

Core Purpose

The Cognitive Walkthrough focuses on evaluating learnability—how easy it is for users to learn to use the interface without extensive training or documentation.

The Four Key Questions

At every action point in a task, you ask these four questions:

Q1: Will users know what to do?

Will users understand what action they need to take next to move toward their goal?

Q2: Will users see how to do it?

Is the control or element they need visible and recognizable?

Q3: Will users understand the connection?

Will users recognize that this action will achieve their goal?

Q4: Will users see that progress is being made?

After taking the action, will users get feedback that they're moving toward their goal?

What Makes CW Different?

Task-Focused

You evaluate specific user tasks, not the entire interface. This makes the evaluation concrete and actionable.

Question-Driven

The four questions provide a systematic structure, ensuring you don't miss important usability issues.

User-Centered

You think from the target user's perspective (their skills, knowledge, context), not your own.

Action-by-Action

You examine each individual action in the task sequence, not just the overall impression.

Why Use Cognitive Walkthrough?

  • Finds issues early: Can be done with prototypes before coding
  • No users needed: Expert evaluation, don't need to recruit participants
  • Systematic: Structured approach catches issues consistently
  • Focuses on first-time use: Reveals learnability problems
  • Actionable results: Identifies specific problems at specific locations

When to Use CW

Good For Not Ideal For
• Evaluating task flows
• Finding learnability issues
• Early design stages
• Prototype evaluation
• Identifying unclear paths 		• Overall aesthetic evaluation
• Testing performance/speed
• Understanding emotions
• Evaluating visual design
• Gathering user preferences

Example Scenario: Recipe App

Let's walk through a complete example to see exactly how a Cognitive Walkthrough works.

📱 App Context

App: "QuickMeal" - A recipe discovery app for college students

Target User:
Sarah, 20-year-old college student
- Lives in dorm with shared kitchen
- Basic smartphone user (uses social media, email)
- Never used a recipe app before
- Wants simple, quick meals

Task: Save a recipe to favorites for later

Scenario:
Sarah is browsing recipes and found "15-Minute Pasta" that looks good.
She wants to save it so she can make it later this week.

Starting Point: Recipe Detail screen (viewing "15-Minute Pasta")

Goal: Save the recipe and confirm it's saved

Step-by-Step Walkthrough

1 ACTION 1: Save the Recipe

Current State:
Sarah is viewing the "15-Minute Pasta" recipe detail screen.
Screen shows: recipe photo, title, ingredients list, cooking instructions.

What Sarah needs to do:
Find and tap a way to save this recipe for later.

Correct action: Tap the bookmark icon in the top-right corner

───────────────────────────────────────────────

Q1: Will Sarah know what to do?

Answer: YES

Reasoning: Sarah wants to save the recipe. She understands she needs to
find some kind of "save" or "favorite" function. The concept of saving
content is familiar from social media and other apps.

───────────────────────────────────────────────

Q2: Will Sarah see how to do it?

Answer: MAYBE ⚠️

Reasoning: There's a bookmark icon in the top-right corner of the screen.
However:
- The icon is small (about 24x24 pixels)
- It's the same dark color as the header, making it blend in
- There's no text label saying "Save" or "Favorite"
- Sarah might not immediately notice it, especially if she's focused on
reading the recipe content

🔴 POTENTIAL ISSUE: Icon may not be noticeable enough

───────────────────────────────────────────────

Q3: Will Sarah understand the connection?

Answer: PROBABLY, but with some uncertainty ⚠️

Reasoning:
- Bookmark icons typically mean "save for later" in many apps
- However, some apps use hearts, stars, or plus signs for saving
- Without a label, Sarah might wonder: "Does this bookmark it? Save it?
Add it to a list?"
- She'll probably try it, but there's slight uncertainty

🟡 MINOR ISSUE: Lack of label creates ambiguity

───────────────────────────────────────────────

Q4: Will Sarah see that progress is being made?

Answer: YES ✓

Reasoning: After Sarah taps the bookmark icon:
- The icon changes from outline to filled (visual feedback)
- A small message appears at bottom: "Recipe saved!"
- The change is clear and immediate

Good feedback provided.

───────────────────────────────────────────────

SUMMARY OF ACTION 1:

✓ User knows they need to save (Q1)
⚠️ Icon might not be noticed immediately (Q2)
⚠️ Icon meaning could be clearer (Q3)
✓ Good feedback after action (Q4)

Issues Found: 2 (both moderate severity)

2 ACTION 2: Verify Recipe Was Saved

Current State:
Still on Recipe Detail screen. Bookmark icon is now filled in. Toast
message "Recipe saved!" has appeared and faded away.

What Sarah needs to do:
Confirm the recipe is actually saved somewhere she can find it later.

Correct action: Navigate to "Saved" section via bottom tab bar

───────────────────────────────────────────────

Q1: Will Sarah know what to do?

Answer: YES

Reasoning: Sarah saw the confirmation message. Now she wants to verify
where the recipe was saved. She knows she needs to navigate somewhere to
view her saved recipes.

───────────────────────────────────────────────

Q2: Will Sarah see how to do it?

Answer: YES ✓

Reasoning: Bottom tab bar is clearly visible with 4 tabs:
- Home
- Search
- Saved
- Profile

The "Saved" tab with a bookmark icon is easy to spot.

───────────────────────────────────────────────

Q3: Will Sarah understand the connection?

Answer: YES ✓

Reasoning: The tab is labeled "Saved" which matches her mental model.
The bookmark icon reinforces this. She'll understand that tapping "Saved"
will show her saved recipes.

───────────────────────────────────────────────

Q4: Will Sarah see that progress is being made?

Answer: YES ✓

Reasoning: After tapping "Saved":
- Screen transitions to Saved Recipes screen
- She sees a list of saved recipes
- "15-Minute Pasta" appears at the top with its photo
- Clear confirmation that her recipe is there

Excellent feedback and verification.

───────────────────────────────────────────────

SUMMARY OF ACTION 2:

✓ All four questions answered positively
✓ No issues found
✓ Clear navigation and feedback

Overall Task Evaluation

Task Completion Summary

═══════════════════════════════════════════════
TASK: Save a recipe to favorites
═══════════════════════════════════════════════

Result: ✓ Task can be completed

Total Actions Evaluated: 2
Issues Found: 2

Issue #1: Save icon not prominent enough
Severity: MODERATE
Location: Recipe Detail screen, top-right corner
Problem: Bookmark icon is small (24px) and blends with header color.
First-time users might not notice it immediately.
Impact: Users may spend time looking for save function or may give up.
Recommendation: Increase icon size to 32-40px, use contrasting color,
or add text label "Save"

Issue #2: Save icon lacks text label
Severity: MINOR
Location: Recipe Detail screen, top-right corner
Problem: Bookmark icon has no accompanying text label. Users familiar
with this convention will understand, but first-time users might be
briefly uncertain about what it does.
Impact: Slight hesitation before using feature.
Recommendation: Add "Save" label below or next to icon

Positive Aspects:
+ Clear feedback when recipe is saved (icon change + toast message)
+ Easy to find saved recipes (clear "Saved" tab)
+ Recipe appears immediately in saved list (good confirmation)
+ Overall task flow is logical and straightforward

Overall Assessment:
The save functionality works and is discoverable, but could be made more
obvious for first-time users. The issues are not severe but addressing
them would improve initial learnability.

🎯 Key Takeaways from This Example

  • Every action is evaluated systematically with all four questions
  • You document both problems AND positive aspects
  • Severity is based on impact on users, not your personal opinion
  • Findings include specific recommendations for improvement
  • You think from the target user's perspective (Sarah, not yourself)

Documentation Principles

Good documentation is the key to providing actionable feedback. Your findings should be clear, specific, and constructive.

What Makes Good Documentation?

  • Specific: Exact location, screen, and context
  • Objective: Based on CW questions, not personal preference
  • Actionable: Designer knows what to fix and how
  • Complete: Includes severity, impact, and recommendations
  • Constructive: Helps improve the design, not just criticize

Severity Ratings

Rate each issue by its impact on users:

Severity Definition Example
CRITICAL Prevents task completion. User cannot proceed. Button required for next step doesn't work; essential feature is missing; no way to complete the task
MAJOR Significant difficulty. Most users struggle or fail. Critical feature very hard to find; confusing navigation causes errors; no feedback after important action
MODERATE Causes confusion or inefficiency. Users can work around it. Feature not obvious; weak feedback; ambiguous labels; requires extra steps
MINOR Small annoyance. Minimal impact on usability. Slightly unclear wording; small touch target; minor inconsistency

Template for Documenting Each Issue

Action/Step: [Which specific action in the task?]

[Which of the 4 questions had a NO or MAYBE answer? Why?]

[What is the issue? Be specific about what you observed.]

[How does this affect the target user? What happens because of this?]

[Specific suggestion for how to fix this issue]

[Screenshot if available]

Examples: Good vs. Bad Documentation

✗ Bad Example

Issue: Button is confusing

Severity: Bad

Fix: Make it better

Why this is bad:

  • Not specific (which button?)
  • No context (which screen? which task?)
  • No explanation (why confusing?)
  • Not actionable (how to "make it better"?)

✓ Good Example

ISSUE #3: Submit Button Label Unclear

Task: Book a fitness class
Action: Confirming the booking
Screen: Booking Summary
Severity: MODERATE

CW Question: Q3 (Will users
understand the connection?)
Answer: NO - Button says "Submit" but
users don't know what they're submitting
or what happens next

Problem: After selecting a class and
time, the confirmation button is labeled
"Submit." This is vague—users are unsure
if this books the class, sends a request,
or moves to another screen.

Impact: Users hesitate at a critical
decision point. They want to be certain
they understand what clicking will do.

Recommendation: Change button label to
"Confirm Booking" or "Book This Class"
to clearly communicate the action

Documentation Best Practices

✓ Do This

  • Document as you go, not after finishing all tasks
  • Take screenshots of every issue you find
  • Quote the specific CW question(s) that revealed each issue
  • Include both problems and positive findings
  • Be specific about locations (name the screen, button, element)
  • Provide constructive recommendations, not just criticism
  • Think from the target user's perspective in your reasoning
  • Use consistent severity ratings based on user impact

⚠️ Common Documentation Mistakes

  • Too vague: "The app is confusing" (where? why? for whom?)
  • Personal opinion: "I don't like the colors" (not a CW finding)
  • No context: Listing issues without saying which task or screen
  • No recommendations: Identifying problems without suggesting solutions
  • Inconsistent severity: Rating all issues as "major" or being too lenient
  • Only negative: Not noting what works well
  • Not tied to CW questions: Reporting issues without showing which question failed

Tips for Effective Evaluation

Before You Start

  • Read all documentation thoroughly
  • Understand the target user profile
  • Review all task scenarios
  • Test that the prototype works
  • Set up your documentation template

During the Walkthrough

  • Ask all 4 questions at every step
  • Think aloud to clarify your reasoning
  • Be thorough, not rushed
  • Screenshot every issue
  • Note positive aspects too

After Completing Tasks

  • Review all findings for completeness
  • Ensure severity ratings are consistent
  • Check that recommendations are specific
  • Proofread for clarity
  • Organize by priority

Time Management

  • Allow 30-45 minutes per task
  • Don't rush through questions
  • Take breaks between tasks
  • Budget time for documentation
  • Review before submitting

Step-by-Step Process

Cognitive Walkthrough Procedure

  1. Read the task scenario - Understand user, context, and goal
  2. Go to starting point - Navigate to where the task begins
  3. Identify first action - What does user need to do?
  4. Ask all 4 questions - Systematically evaluate this action
  5. Document findings - Record any issues (or note if all good)
  6. Perform the action - Move to next state
  7. Repeat steps 3-6 - Continue until task is complete (or fails)
  8. Summarize task - Overall completion status and key findings
  9. Move to next task - Repeat entire process

Severity Quick Guide

Level Can task be completed? Example
Critical NO - Task cannot be completed Required button missing/broken
Major BARELY - Most users fail or struggle significantly Feature very hard to find
Moderate YES, but with confusion/extra effort Ambiguous labels, weak feedback
Minor YES, minimal impact Small inconsistencies

Common Pitfalls to Avoid

  • ❌ Evaluating based on YOUR knowledge, not target user's
  • ❌ Skipping questions because you think you know the answer
  • ❌ Being too lenient ("users will figure it out")
  • ❌ Documenting vague issues without specifics
  • ❌ Only noting problems, not positive aspects
  • ❌ Confusing visual design preferences with usability issues
  • ❌ Rushing through tasks without proper analysis
