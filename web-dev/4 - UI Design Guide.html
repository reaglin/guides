Why This Process Matters

Jumping straight into coding without design leads to:

  • ❌ Constant rework as requirements become clear
  • ❌ Poor user experience due to lack of planning
  • ❌ Inconsistent interfaces across pages
  • ❌ Missed requirements and features
  • ❌ Higher development costs and longer timelines

Good design upfront saves 10x the time later!

Phase 1: Discovery & Requirements

1 Understand Your Users

Key Questions:

  • Who will use this application? (students, administrators, customers, etc.)
  • What is their technical proficiency?
  • What devices will they use? (desktop, mobile, tablet)
  • What are their goals and pain points?
  • What is their workflow/process?

Example: User Profile

Name: Sarah, College Student

Goal: Register for courses quickly during busy registration period

Technical Level: Moderate - uses smartphone daily, familiar with web apps

Pain Points: Current system is slow, confusing navigation, hard to see availability

Device: Primarily laptop, sometimes mobile phone

2 Define Core Features

List out what the application must do:

  • Must Have: Critical features (MVP - Minimum Viable Product)
  • Should Have: Important but not critical
  • Nice to Have: Future enhancements
Priority Feature User Story
Must Have Add Product As an admin, I need to add new products to the catalog
Must Have Edit Product As an admin, I need to update product information
Should Have Bulk Upload As an admin, I want to import products from CSV
Nice to Have Image Gallery As an admin, I want to add multiple product images
3 Identify Data Requirements

For each feature, determine what data is needed:

Product Entity: - ID (auto-generated) - Name (required, max 100 chars) - Description (optional, max 500 chars) - Price (required, decimal, > 0) - Category (required, dropdown) - SKU (required, unique, alphanumeric) - Stock Quantity (required, integer, >= 0) - Image URL (optional) - Created Date (auto-generated) - Modified Date (auto-generated) - Status (Active/Inactive, default: Active)
4 Choose CRUD Patterns

Decide how users will interact with data (refer to CRUD guide for detailed patterns):

Operation UI Pattern Rationale
List/Read Data table with search/filter Many products to browse
Create Modal dialog Simple form, keeps user in context
Edit Same modal as Create Consistent UX, reuse components
Delete Confirmation dialog Prevent accidental deletion
Quick Edit Inline editing for price Common task, needs speed

💡 Pro Tip: Create a Feature Matrix

Document your decisions in a simple table before designing:

  • Feature: What functionality
  • UI Pattern: How user interacts
  • Data Fields: What information is needed
  • Validation Rules: What constraints apply
  • User Role: Who can access

Organizing Your Application

📋 What is Information Architecture?

Information Architecture (IA) is the practice of organizing and structuring content in a way that helps users find information and complete tasks efficiently.

1 Create a Site Map

Map out all pages and their relationships:

E-Commerce Admin Panel - Site Map Home (Dashboard) ├── Products │ ├── Product List │ ├── Add Product (Modal) │ ├── Edit Product (Modal) │ ├── Product Categories │ └── Bulk Import ├── Orders │ ├── Order List │ ├── Order Details │ └── Order Status Management ├── Customers │ ├── Customer List │ └── Customer Details ├── Reports │ ├── Sales Report │ ├── Inventory Report │ └── Customer Analytics └── Settings ├── User Profile ├── Store Settings └── Payment Configuration
2 Define Navigation Structure

Primary Navigation: Main sections (always visible)

  • Dashboard
  • Products
  • Orders
  • Customers
  • Reports

Secondary Navigation: Sub-sections within a page

  • Tabs for different views (All Products, Active, Archived)
  • Breadcrumbs to show location
  • Contextual menus for actions
⚠️ Navigation Rules:
  • Keep primary navigation to 5-7 items max
  • Use clear, action-oriented labels
  • Highlight current page/section
  • Keep navigation consistent across all pages

User Flows

Map out how users accomplish key tasks:

3 Document User Flows

Example: Adding a New Product

Flow: Admin adds a new product 1. User clicks "Products" in main navigation └─→ Products list page loads 2. User clicks "+ Add Product" button └─→ Modal dialog opens with empty form 3. User fills in required fields: - Product Name: "Wireless Mouse" - Price: "29.99" - Category: "Electronics" (dropdown) - SKU: "MOUSE-001" - Stock: "50" 4. User clicks "Upload Image" (optional) └─→ File picker opens └─→ User selects image └─→ Preview shows in form 5. User clicks "Save Product" └─→ Validation runs ✓ If valid: └─→ Product saved to database └─→ Success message shows └─→ Modal closes └─→ Product appears in list ✗ If invalid: └─→ Error messages show next to fields └─→ User corrects errors └─→ User clicks "Save Product" again

Alternative Flow: Cancel Action

At step 5, user clicks "Cancel" instead: └─→ "Are you sure?" confirmation appears If "Yes": └─→ Modal closes └─→ No data saved └─→ User returns to product list If "No": └─→ User stays in form └─→ Can continue editing

💡 User Flow Best Practices:

  • Keep flows as short as possible (3-5 steps ideal)
  • Document all decision points (if/else scenarios)
  • Include error states and recovery paths
  • Show what happens on success and failure
  • Consider edge cases (empty states, max limits, etc.)
4 Define Page States

Every page can have multiple states. Plan for all of them:

State Description UI Treatment
Loading Data is being fetched Show spinner or skeleton screen
Empty No data exists yet Show helpful message + "Add" button
Data Normal state with content Display list/table with data
Error Something went wrong Show error message + retry option
No Results Search/filter returned nothing Show "No matches" + clear filters button

What Are Wireframes?

Wireframes are low-fidelity sketches or diagrams that show the structure and layout of a page without visual design details (colors, fonts, images).

Purpose:

  • Focus on layout and functionality, not aesthetics
  • Quick to create and modify
  • Easy to get feedback and iterate
  • Help developers understand structure
  • Prevent costly changes later in development

Wireframing Levels

Low-Fidelity (Lo-Fi) Wireframes

What: Simple sketches, boxes and lines

When: Early stages, brainstorming, quick iterations

Tools: Paper and pencil, whiteboard, simple drawing tools

Wireframing Best Practices

✅ DO

  • Start with pen and paper for speed
  • Focus on layout and structure
  • Use real content when possible
  • Show all interactive elements
  • Label everything clearly
  • Include annotations and notes
  • Design mobile and desktop views
  • Show different states (loading, error, empty)

❌ DON'T

  • Add colors, images, or fonts yet
  • Spend time on visual polish
  • Use "Lorem ipsum" if you know real content
  • Skip error states and edge cases
  • Wireframe every single page (focus on unique layouts)
  • Get attached to your first version
  • Forget to consider mobile users
  • Design in isolation - get feedback!

  3. Sketch header with navigation
  4. Add main content area with boxes for content
  5. Label each element
  6. Add notes about interactions
  7. Take a photo and share with team

Time: 5-10 minutes per screen!

Core Principles for Effective UIs

1. Clarity

Users should immediately understand what they can do

  • Use clear, descriptive labels (not "Submit", use "Create Product")
  • Make clickable elements obvious (buttons look like buttons)
  • Provide context for actions (show what will happen)
  • Use familiar patterns (don't reinvent navigation)
✅ Clear: "Delete Product" button in red with trash icon
❌ Unclear: "Remove" link in tiny gray text

2. Consistency

Similar elements should look and behave the same way

  • Use same button styles for similar actions
  • Keep navigation in same place on every page
  • Use consistent terminology (don't mix "Remove" and "Delete")
  • Match platform conventions (e.g., iOS vs Android patterns)

Example: Consistent Button Styles

3. Feedback

Always acknowledge user actions

  • Show loading indicators during waits
  • Display success messages after actions complete
  • Highlight active/selected items
  • Show errors clearly with guidance to fix
  • Disable buttons during processing to prevent double-clicks
✅ Good Feedback Examples:
  • "Product saved successfully!"
  • "Saving..." (with spinner)
  • Form field turns green when valid
  • "5 items selected" when bulk selecting

4. Forgiveness

Help users recover from mistakes

  • Confirm destructive actions (deletes, overwrites)
  • Allow undo when possible
  • Save drafts automatically
  • Warn before navigating away from unsaved changes
  • Make it easy to go back and correct errors

5. Efficiency

Help users accomplish tasks quickly

  • Minimize required clicks/steps
  • Provide keyboard shortcuts for power users
  • Remember user preferences
  • Auto-fill when possible
  • Support bulk operations for repeated tasks

6. Accessibility

Everyone should be able to use your application

  • Sufficient color contrast (4.5:1 minimum for text)
  • Don't rely on color alone to convey meaning
  • Provide text alternatives for images
  • Support keyboard navigation
  • Use semantic HTML elements
  • Label form fields properly
⚠️ Common Accessibility Mistakes:
  • Low contrast text (light gray on white)
  • Clickable elements too small (minimum 44x44 pixels)
  • Missing alt text on images
  • Forms without proper labels
  • Color as only indicator (red/green for status)

Visual Hierarchy

📐 Guide Users' Eyes:

Use size, color, spacing, and position to show importance:

  • Size: Larger = more important
  • Color: Bright colors attract attention
  • Position: Top-left gets noticed first (in Western cultures)
  • Contrast: High contrast stands out
  • White Space: Breathing room around important elements
  • Alignment: Creates clean, organized appearance

The F-Pattern and Z-Pattern

💡 How Users Scan Pages:

F-Pattern: For text-heavy content

  • Users scan horizontally across the top
  • Then move down and scan another horizontal line (shorter)
  • Then scan vertically down the left side

Z-Pattern: For simple layouts

  • Eyes start top-left (logo)
  • Move to top-right (navigation/CTA)
  • Diagonal down to bottom-left
  • Across to bottom-right (call-to-action)

Design Implication: Place important elements along these paths!

Form Design Fundamentals

Forms are the primary way users interact with CRUD applications. Good form design makes data entry fast, accurate, and painless.

Form Layout Best Practices

1 Single Column Layouts

Always prefer single column forms

  • Natural top-to-bottom flow
  • Works on mobile without changes
  • Users complete forms 15-20% faster
  • Exception: Short, related fields (First Name | Last Name)

✅ DO: Single Column

Name          [___________]
Email         [___________]
Phone         [___________]
Address       [___________]
City          [___________]
State         [___________]

❌ DON'T: Multi-Column

Name  [_____]  Email [_____]
Phone [_____]  City  [_____]
Addr  [_____]  State [_____]
(confusing flow!)
2 Label Placement

Labels above fields (not to the left)

  • Easier to scan
  • Works better on mobile
  • Allows longer labels
  • Creates clear visual grouping
3 Field Sizing

Input width should match expected content

  • Short fields for short data (ZIP code: 5 chars wide)
  • Long fields for long data (email: full width)
  • This gives users a hint about expected length

Field Types Selection

Data Type Best Field Type Why
Short text (name, title) Text input Simple and familiar
Long text (description) Textarea Shows user they can write more
Email Email input Triggers email keyboard on mobile
Number Number input Shows +/- buttons, prevents letters
Date Date picker Prevents format errors
2-5 choices Radio buttons All options visible at once
6+ choices Dropdown/Select Saves space
Multiple selections Checkboxes Clear that multiple allowed
True/False Checkbox or Toggle Visual on/off state

Form Validation Design

4 When to Validate

Inline Validation (Real-time)

  • Validate as user types or on blur (leaving field)
  • Show success indicators (green checkmark)
  • Show errors immediately when field is invalid
  • Don't show errors while user is still typing

Submit Validation

  • Catch any missed errors on submit
  • Scroll to first error
  • Highlight all error fields
  • Keep valid data (don't make user re-enter)
5 Error Message Design

Good Error Messages

✗ Please enter a valid email address (e.g., [email protected])
✗ Password must be at least 8 characters long
✓ Looks good!
✅ Good Error Messages:
  • Specific: Say exactly what's wrong
  • Helpful: Suggest how to fix it
  • Positive tone: "Please enter" not "You forgot"
  • Next to field: Not just at top of form
  • Include example: Show correct format
❌ Bad Error Messages:
  • "Invalid input" (which field? what's wrong?)
  • "Error 402" (meaningless to users)
  • "This field is required" (why? what goes here?)
  • Errors only at top of long form

Helper Text & Placeholders

💡 When to Use Each:

Helper Text (below field):

  • Always visible instructions
  • Format requirements: "Use format: MM/DD/YYYY"
  • Character limits: "Maximum 500 characters"
  • Important context: "This will be visible to customers"

Placeholder Text (inside empty field):

  • Examples: "e.g., [email protected]"
  • Optional hints (disappears when typing)
  • DON'T use for critical instructions (might be missed)
  • DON'T use as labels (accessibility issue)

Form Design Checklist

Before Developer Handoff:

  • All required fields marked with * or "required"
  • Field types specified (text, email, number, etc.)
  • Validation rules documented for each field
  • Error messages written for each validation
  • Success state designed and documented
  • Loading state shown (disabled buttons, spinners)
  • Mobile layout considered and designed
  • Tab order makes logical sense
  • Helper text provided where needed
  • Button labels are action-oriented

Designing for Multiple Devices

Responsive Design means your interface adapts to different screen sizes—from large desktop monitors to small mobile phones.

Common Breakpoints

Device Width Range Design Considerations
Mobile 320px - 767px Single column, larger touch targets, hamburger menu
Tablet 768px - 1023px 2 columns possible, some sidebar content
Desktop 1024px+ Full layout, sidebars, multi-column content
Large Desktop 1440px+ Max width containers, extra white space

Mobile-First Design Approach

1 Start with Mobile

Design for smallest screen first, then scale up

  • Forces you to prioritize content
  • Ensures core functionality works everywhere
  • Easier to add features than remove them
  • Most users now browse on mobile devices

Responsive Pattern Examples

Navigation Patterns

Desktop: Full horizontal navigation bar

┌────────────────────────────────────────────────┐
│ [Logo]  Dashboard  Products  Orders  Reports  │
└────────────────────────────────────────────────┘

Mobile: Hamburger menu with drawer

┌─────────────────────┐
│ [☰]  Logo           │
└─────────────────────┘

Tap ☰ to open menu drawer

Data Table Patterns

Desktop: Full table with all columns

Tablet: Hide less important columns

Mobile: Card-based layout instead of table

Mobile Card Example: ┌─────────────────────────┐ │ Wireless Mouse │ │ Price: $29.99 │ │ Stock: 50 units │ │ [Edit] [Delete] │ └─────────────────────────┘ ┌─────────────────────────┐ │ USB Cable │ │ Price: $9.99 │ │ Stock: 120 units │ │ [Edit] [Delete] │ └─────────────────────────┘

Touch-Friendly Design

⚠️ Critical Mobile Requirements:

  • Touch Targets: Minimum 44x44 pixels (48x48 better)
  • Spacing: 8px minimum between clickable elements
  • Font Size: Minimum 16px (prevents zoom on iOS)
  • Forms: Use appropriate input types for mobile keyboards
  • Scrolling: Avoid horizontal scroll, enable vertical scroll

Responsive Form Design

💡 Mobile Form Best Practices:

  • Stack all fields vertically (no side-by-side)
  • Make buttons full-width on mobile
  • Use large, tappable form controls
  • Minimize typing with dropdowns and pickers
  • Auto-capitalize where appropriate
  • Use appropriate input modes (email, tel, number)
  • Place labels above fields (not floating or inline)

Testing Responsive Design

Responsive Testing Checklist:

  • Test on actual devices (not just browser resize)
  • Check all breakpoints (mobile, tablet, desktop)
  • Test in portrait and landscape orientations
  • Verify touch targets are large enough
  • Ensure text is readable without zooming
  • Check that all content is accessible
  • Test forms on mobile keyboards
  • Verify images scale properly
  • Check navigation works on all sizes

Creating Developer-Ready Specifications

The goal is to provide developers with everything they need to build your design accurately and efficiently, without constant back-and-forth questions.

What Developers Need

1 Design Files

Provide:

  • High-fidelity mockups (Figma, XD, Sketch)
  • All screen states (loading, error, empty, success)
  • Mobile and desktop versions
  • Interactive prototypes showing flows
  • Component library if available
2 Specifications Document

Include:

A. Component Specifications

Product Form Component FIELDS: 1. Product Name - Type: Text input - Required: Yes - Max length: 100 characters - Validation: Required, alphanumeric + spaces/hyphens - Error: "Product name is required (max 100 characters)" 2. Price - Type: Number input - Required: Yes - Min: 0.01 - Max: 999999.99 - Decimal places: 2 - Validation: Required, must be > 0 - Error: "Price must be greater than $0.00" 3. Category - Type: Dropdown - Required: Yes - Options: Load from /api/categories endpoint - Default: First option - Error: "Please select a category" 4. Description - Type: Textarea - Required: No - Max length: 500 characters - Rows: 4 - Helper text: "Optional: Describe product features" 5. Image - Type: File upload - Required: No - Accept: .jpg, .png, .gif - Max size: 5MB - Preview: Show thumbnail after upload

B. Behavior Specifications

Product Form Behavior VALIDATION: - Real-time validation on blur (leaving field) - Show error messages inline below each field - Submit button disabled until all required fields valid - Success checkmark appears when field validates LOADING STATES: - Show "Saving..." on submit button - Disable all form fields during save - Show spinner next to button SUCCESS STATE: - Show toast notification: "Product saved successfully!" - Close modal after 1 second - Refresh product list - Clear form if creating new product ERROR STATE: - If server error, show message at top of form - Keep form open with data intact - Re-enable submit button - Highlight any server-side validation errors
3 API Integration Points
API Endpoints Needed GET /api/products - Purpose: Fetch product list - Response: Array of product objects - Pagination: page, pageSize parameters - Sorting: sortBy, sortOrder parameters GET /api/products/:id - Purpose: Fetch single product for editing - Response: Product object POST /api/products - Purpose: Create new product - Body: Product object (JSON) - Response: Created product with ID PUT /api/products/:id - Purpose: Update existing product - Body: Updated product object (JSON) - Response: Updated product DELETE /api/products/:id - Purpose: Delete product - Response: Success message GET /api/categories - Purpose: Fetch categories for dropdown - Response: Array of category objects

If using a design system, provide:

5 Interaction Details

Document all interactive behaviors:

  • Hover states: Buttons change color, cursor becomes pointer
  • Focus states: Outline appears on keyboard navigation
  • Active states: Button appears pressed
  • Disabled states: Grayed out, not clickable
  • Animations: Transitions (300ms ease-in-out)
  • Modal behavior: Click outside to close? ESC key closes?

End-to-End Project: Student Grade Management

Let's walk through designing a complete CRUD interface for managing student grades, from initial idea to developer-ready specs.

1 Discovery Phase

User Profile

Primary User: Professor Sarah

  • Goal: Quickly enter and update student grades
  • Pain Points: Current system requires too many clicks, hard to find students, no bulk entry
  • Frequency: Uses system weekly during semester, heavily during finals
  • Device: Primarily desktop in office, occasionally laptop at home

Core Features (MoSCoW)

Must Have:

  • View list of students in course
  • Add/edit individual grades
  • Filter by assignment
  • Calculate average grades

Should Have:

  • Bulk grade entry (paste from Excel)
  • Grade distribution chart
  • Export grades to CSV

Could Have:

  • Email students about grades
  • Curve grade calculation
  • Historical grade comparison
2 Data Model
Student Grade Entry: - Student ID (auto, hidden) - Student Name (read-only, from enrollment) - Assignment Name (dropdown) - Points Earned (number, 0-100) - Points Possible (number, read-only, from assignment) - Percentage (calculated) - Letter Grade (calculated) - Notes (optional text, max 200 chars) - Date Submitted (auto) - Graded By (auto, current user)

Add Grade Inline row or modal Single grade: inline. Multiple grades: modal with form

5 Specifications
Grade Entry - Inline Editing Specification BEHAVIOR: 1. Click on score field (shows as [85]/100) 2. Field becomes editable input 3. User types new score 4. On blur or Enter key: - Validate: 0 <= score <= points possible - If valid: Save via PUT /api/grades/:id - Auto-calculate percentage and letter grade - Show brief success indicator (green flash) - Field returns to display mode 5. On Escape key: - Discard changes - Return to display mode VALIDATION: - Required: Yes - Type: Number - Min: 0 - Max: Points Possible value - Decimals: Up to 2 places - Error (inline): "Score must be between 0 and [max]" LOADING STATE: - Show small spinner while saving - Disable field during save ERROR STATE: - If save fails: Show error, keep in edit mode - Error message: "Could not save grade. Try again." - Preserve user's input KEYBOARD SHORTCUTS: - Enter: Save and move to next row - Escape: Cancel edit - Tab: Save and move to next field
