Client-Server Security in Full-Stack Development

📚 Module Overview

Security is not a feature you add at the end — it is a discipline woven into every layer of a full-stack application from the first day of design. This module walks through every surface of a client-server system that requires protection: the browser, the network, the API, the server, the database, and the data itself. Understanding what needs to be secured and why is the foundation for building applications that can be trusted in production.

Think Like an Attacker

📖 Core Principle

Defense in Depth means layering multiple independent security controls so that if one layer fails, the attacker still has to breach several more before reaching your data. No single control is sufficient on its own.

The Full-Stack Attack Surface

Every full-stack application has multiple surfaces where an attacker can attempt to gain access or cause harm:

🌐Browser / Client
📡Network / Transport
🔌API Layer
⚙️Application Server
🗄️Database
📁Data & Files

Each layer must be independently secured. Relying on a single outer wall — such as a firewall — leaves you completely exposed if that wall is breached.

Security Principles Every Developer Must Know

🔐 Least Privilege

Grant the minimum access needed to perform a task — nothing more.

🛡️ Fail Securely

When something goes wrong, deny access by default rather than allowing it.

🔍 Never Trust Input

All data arriving from outside the server must be treated as hostile until proven otherwise.

🏗️ Security by Design

Build security in from the start. Retrofitting it is far more expensive and error-prone.

🔄 Defense in Depth

Multiple overlapping controls. One failure should not compromise the system.

📋 Separation of Concerns

Keep authentication, authorization, business logic, and data access clearly separated.

🧩 Reduce Attack Surface

Disable unused features, ports, and services. Less exposure means fewer entry points.

📣 Assume Breach

Design systems expecting a component may be compromised. Limit blast radius.

⚠️ The #1 Developer Security Mistake

Treating security as someone else's job — the network team's, the DBA's, the DevOps team's. In full-stack development, every developer is responsible for the security of the code they write. Security decisions happen at the keyboard, not just in the firewall configuration.

📖 Definition

Authentication is the process of verifying that a user or system is who they claim to be. It answers the question: "Are you really you?"

Authentication Factors

Something You Know — Knowledge Factors

  • Passwords and passphrases
  • Security questions (considered weak — avoid where possible)
  • PINs

Something You Have — Possession Factors

  • One-time passwords (OTP) via authenticator app (TOTP)
  • SMS codes (weak — vulnerable to SIM swapping)
  • Hardware security keys (FIDO2 / WebAuthn)
  • Email magic links

Something You Are — Inherence Factors

  • Fingerprint
  • Face recognition
  • Voice recognition

Multi-Factor Authentication (MFA)

✅ MFA is One of the Single Most Effective Controls Available

Requiring two or more independent factors makes credential theft far less impactful. Even if an attacker steals a password, they still cannot authenticate without the second factor. MFA should be mandatory for administrative accounts and strongly encouraged for all users.

Password Security Requirements

✅ DO

  • Hash passwords with a strong, slow algorithm (bcrypt, Argon2, scrypt)
  • Add a unique salt per password before hashing
  • Enforce a minimum length (12+ characters recommended)
  • Check passwords against known breach lists (Have I Been Pwned API)
  • Allow passphrases and spaces
  • Allow paste in password fields
  • Offer a "show password" toggle

❌ DON'T

  • Store passwords in plain text — ever
  • Use MD5 or SHA-1 for password storage
  • Enforce arbitrary complexity rules that reduce passphrase length
  • Force frequent password rotation without cause
  • Display passwords in emails or logs
  • Limit password length unnecessarily
  • Build your own authentication system when battle-tested libraries exist

Account Enumeration

⚠️ Don't Reveal Whether an Account Exists

Login and password-reset error messages must be generic. Saying "No account found with that email" allows attackers to build a list of valid usernames. Instead always say something like: "If an account exists with that email, you will receive a reset link."

Account Lockout and Brute-Force Protection

  • Rate limiting: Limit login attempts per IP per time window
  • Progressive delays: Slow down responses after repeated failures
  • Temporary lockout: Lock accounts after N failed attempts with admin-reset or timed unlock
  • CAPTCHA: Challenge suspicious traffic, particularly after multiple failures
  • Alerting: Notify users of login attempts and new device logins

Third-Party Authentication (OAuth / OIDC)

💡 Using "Sign in with Google / Microsoft / GitHub"

Delegating authentication to a trusted Identity Provider (IdP) can be a strong security choice — the IdP has world-class security teams protecting those credentials. However, you still need to validate tokens correctly, restrict allowed providers, and manage your own authorization layer after authentication succeeds.

📖 Definition

Authorization determines what an authenticated user is allowed to do. It answers: "You are who you say you are — but are you allowed to do this?" Authentication and authorization are separate concerns and must both be enforced.

Authorization Models

Role-Based Access Control (RBAC)

Users are assigned to roles, and roles are granted permissions.

  • Example roles: Admin, Manager, Employee, Guest
  • Simple to implement and reason about
  • Works well for most business applications
  • Roles must be assigned carefully — role sprawl is a real risk

Attribute-Based Access Control (ABAC)

Access decisions based on attributes of the user, resource, and environment.

  • Example: A user in the "Finance" department can access budget reports only during business hours
  • More flexible than RBAC, but more complex to manage
  • Suited to complex enterprise environments

Row-Level Security

Users can only access rows in a table that belong to them.

  • Example: A customer can only view their own orders, not all orders in the database
  • Must be enforced at the server/API layer, not the client
  • Can also be enforced at the database layer for extra depth

The Golden Rule of Authorization

🚨 Never Enforce Authorization on the Client Alone

Hiding a button in the browser does not prevent an attacker from calling the underlying API directly. Every API endpoint must validate that the current user has permission to perform that action — every single time, on the server. Client-side checks are UI niceties only.

Insecure Direct Object Reference (IDOR)

What is IDOR?

When an API exposes internal IDs and does not verify ownership, users can manipulate IDs to access other users' data.

  • Example: /api/orders/1042 — if the server doesn't check that order 1042 belongs to the logged-in user, any authenticated user can fetch it by changing the number.
  • Fix: Always filter queries by the current user's identity in addition to the requested ID. The query should include both the record ID and the user's ID.

Privilege Escalation

⚠️ Vertical and Horizontal Escalation

  • Vertical: A regular user accesses admin-only functionality
  • Horizontal: A user accesses another user's data at the same privilege level (IDOR)

Both must be guarded against at every API endpoint.

Authorization Checklist

Every API Endpoint Must:

  • Verify the user is authenticated before anything else
  • Verify the user has the required role or permission
  • Verify the user owns or has rights to the specific resource requested
  • Return a generic 403 Forbidden — not detailed error info — on failure
  • Log unauthorized access attempts for monitoring

📖 Why Transport Security Matters

Data traveling between a browser and server passes through many intermediate network nodes — routers, ISPs, proxies. Without encryption, anyone on the network path can read or alter that data. This is called a Man-in-the-Middle (MitM) attack.

HTTPS and TLS

HTTPS (HTTP over TLS) — Non-Negotiable

  • All production applications must use HTTPS — no exceptions
  • TLS (Transport Layer Security) encrypts all data in transit
  • Use TLS 1.2 minimum; TLS 1.3 preferred
  • Disable deprecated versions: SSL 2/3, TLS 1.0, TLS 1.1
  • Certificates can be obtained free from Let's Encrypt
  • Redirect all HTTP traffic permanently to HTTPS (301 redirect)

HTTP Strict Transport Security (HSTS)

✅ Tell Browsers to Always Use HTTPS

The HSTS response header instructs browsers to only connect to your domain over HTTPS for a defined period — even if the user types http://. This prevents protocol downgrade attacks. Include subdomains in the policy and consider HSTS preloading.

Certificate Management

  • Monitor certificate expiry — expired certificates break the site and destroy trust
  • Use automated renewal (e.g., Let's Encrypt with Certbot, cloud-managed certificates)
  • Use strong key lengths (RSA 2048-bit minimum, or ECDSA P-256)
  • Keep private keys private — never commit them to version control
  • Consider Certificate Pinning for high-security mobile applications

Protecting APIs and Internal Traffic

  • Enforce TLS on internal service-to-service communication as well — not just public-facing endpoints
  • Validate TLS certificates when making server-side HTTP requests to other services — don't disable certificate verification
  • Use mutual TLS (mTLS) for machine-to-machine authentication in microservices

Content Security and Integrity

Subresource Integrity (SRI)

When loading scripts or stylesheets from a Content Delivery Network (CDN), use SRI hashes. This ensures that if the CDN is compromised and the file is altered, the browser will refuse to load it. SRI hashes are added as an attribute on <script> and <link> tags.

🚨 The Root Cause of Most Attacks

The majority of web application vulnerabilities — SQL Injection, XSS, Command Injection, Path Traversal — all share a single root cause: trusting data supplied by the user without validating or sanitizing it first. This section is foundational to everything else.

Validation vs. Sanitization

ConceptWhat It DoesExample
Validation Checks that input meets expected rules — rejects it if not Email field must match email format; price must be a positive number
Sanitization Transforms input to remove or neutralize dangerous content Strip HTML tags from a comment field; encode special characters
Encoding (Output) Encodes data appropriately for the context it is being rendered in HTML-encode user content before inserting into a web page

Where to Validate

✅ Server-Side Validation (Required)

All validation must happen on the server. This is the only validation that can be trusted because the server is under your control.

  • Validates data type, format, length, range
  • Applies business rules
  • Cannot be bypassed by the user

✅ Client-Side Validation (UX Enhancement)

Client-side validation improves user experience with immediate feedback, but must never be the only layer of defense.

  • HTML5 input attributes (required, type, pattern)
  • JavaScript validation
  • Can always be bypassed — treat it as cosmetic

What to Validate on Every Input Field

  • Data type: Is it actually a number, date, email, etc.?
  • Length: Is it within acceptable min/max character limits?
  • Format: Does it match the expected pattern?
  • Range: For numbers/dates, is the value in an acceptable range?
  • Allowed characters: Does it contain unexpected characters?
  • Required fields: Are mandatory fields present and non-empty?
  • Business rules: End date must be after start date; quantity must be positive

File Upload Security

File Uploads Are a High-Risk Area

  • Validate file type using magic bytes (file signature), not just the extension or Content-Type header — both can be spoofed
  • Enforce maximum file size limits
  • Restrict allowed file types to only what is needed
  • Store uploaded files outside the web root so they cannot be executed directly
  • Rename files on upload — never use the user-supplied filename
  • Scan uploaded files for malware where possible
  • Serve files through a handler that sets correct Content-Type and Content-Disposition headers

SQL Injection — Deep Dive

How It Works

When user input is concatenated directly into SQL queries, an attacker can inject SQL syntax to change the meaning of the query — extracting all rows, bypassing login checks, deleting tables, or even reading files from the server.

The classic example: entering ' OR '1'='1 into a username field can make a poorly written login query return all users, bypassing authentication entirely.

Prevention: Parameterized Queries / Prepared Statements

The only reliable defense is to never concatenate user input into SQL. Use parameterized queries (also called prepared statements) where the query structure is defined separately from the data. The database engine then treats the user's input as pure data — it can never be interpreted as SQL syntax.

Stored procedures also help, provided they themselves use parameters internally rather than dynamic SQL concatenation.

Cross-Site Scripting (XSS) — Deep Dive

Three Types of XSS

  • Stored XSS: Malicious script saved to the database, then rendered to all visitors (e.g., a comment field). Most dangerous.
  • Reflected XSS: Malicious script in a URL parameter, reflected in the response immediately. Delivered via phishing links.
  • DOM-Based XSS: JavaScript in the page itself processes attacker-controlled data unsafely.

Impact: Steal session cookies, capture keystrokes, redirect users, deface pages, deliver malware.

Prevention: Output Encoding + Content Security Policy

  • HTML-encode all user-supplied content before inserting it into HTML — this turns <script> into harmless text
  • Use your framework's built-in escaping (React, Angular, Razor all do this by default)
  • Never use innerHTML, document.write, or eval() with user data
  • Implement a Content Security Policy (CSP) header to restrict which scripts the browser is allowed to execute

Cross-Site Request Forgery (CSRF) — Deep Dive

How It Works

If a user is logged into your site and visits a malicious page, that page can cause the user's browser to send a request to your site using the user's existing session cookie — without the user's knowledge. The browser automatically attaches cookies to requests matching the domain.

Prevention

  • CSRF tokens: A unique, secret, per-session token embedded in forms and validated on every state-changing request
  • SameSite cookie attribute: Setting cookies to SameSite=Strict or SameSite=Lax prevents them from being sent in cross-site requests
  • Verify the Origin/Referer header on sensitive endpoints
  • For APIs using tokens in headers (not cookies), CSRF is typically not a concern since cross-origin requests cannot set custom headers

📖 Why Sessions Need Protecting

HTTP is stateless. Sessions and tokens are the mechanism that remembers who a user is between requests. If an attacker can steal or forge a session, they can impersonate that user completely — without ever knowing their password.

Session Cookies

Secure Cookie Attributes

AttributePurpose
HttpOnlyPrevents JavaScript from reading the cookie — protects against XSS-based session theft
SecureCookie is only sent over HTTPS — never over HTTP
SameSite=StrictCookie never sent in cross-site requests — strongest CSRF protection
SameSite=LaxCookie sent on top-level navigations only — good balance of protection and usability
DomainRestrict which subdomains receive the cookie
PathRestrict which paths on the server receive the cookie
Expires / Max-AgeSession should expire — never persist indefinitely

Session ID Requirements

  • Session IDs must be long and cryptographically random — not predictable or sequential
  • Generate a new session ID after login (session fixation prevention)
  • Invalidate the session completely on logout — don't just clear the client cookie
  • Implement server-side session expiry for idle and absolute timeouts
  • Never put session IDs in URLs — they end up in server logs and browser history

JSON Web Tokens (JWT)

JWTs — Common in Modern APIs

JWTs are self-contained tokens used especially in stateless API authentication. They contain signed claims about the user.

  • Always verify the token's signature on every request — never skip this step
  • Explicitly specify the expected signing algorithm — reject the "none" algorithm
  • Validate expiry (exp), issuer (iss), and audience (aud) claims
  • Use short expiry times for access tokens (15 minutes is common) and longer-lived refresh tokens
  • Store JWTs in HttpOnly cookies — not in localStorage (vulnerable to XSS)
  • Implement token revocation for refresh tokens and high-security scenarios

⚠️ localStorage vs HttpOnly Cookie for Token Storage

Storing tokens in localStorage or sessionStorage exposes them to any JavaScript running on the page. A single XSS vulnerability can immediately steal all tokens. HttpOnly cookies are the safer choice because JavaScript cannot read them, even if XSS occurs.

Logout and Session Termination

  • Provide a clear logout option and make it work completely
  • On logout, invalidate the server-side session record
  • Clear the session cookie on the client
  • For JWTs: revoke the refresh token and implement short-lived access tokens
  • Offer "log out from all devices" for user account security
  • Automatically expire inactive sessions after a reasonable idle period

📖 APIs Are a Primary Attack Target

In modern full-stack applications, the API is the backbone connecting client and server. It must be treated as a first-class security boundary — not a trusted internal channel.

Authentication and Authorization at the API Layer

  • Every endpoint (except intentional public ones) must require authentication
  • Authorization must be checked on every request for every endpoint — no exceptions
  • Use token-based auth (JWT, API keys) for stateless APIs
  • API keys must be kept secret — never embedded in client-side code or source repositories

Rate Limiting and Throttling

Preventing API Abuse

  • Rate limiting: Restrict the number of requests a client can make per time window (e.g., 100 requests per minute)
  • Throttling: Slow down responses rather than blocking, for softer limiting
  • Apply limits per user, per IP, and globally
  • Return appropriate 429 Too Many Requests responses with a Retry-After header
  • Rate limiting protects against brute-force, credential stuffing, scraping, and denial-of-service

API Versioning and Deprecation

  • Version your API (e.g., /api/v1/) so old versions can be deprecated and removed
  • Remove or restrict old versions that may have unpatched vulnerabilities
  • Don't expose functionality that is no longer actively maintained

Input Validation at the API Layer

  • Validate all incoming request data: body, query parameters, path parameters, and headers
  • Define and enforce an explicit schema for every endpoint's expected input
  • Reject requests with unexpected or extra fields (prevent mass assignment)
  • Limit request body size to prevent resource exhaustion attacks

Error Responses

✅ Secure Error Handling

  • Return generic error messages to clients
  • Log detailed errors server-side for debugging
  • Use standard HTTP status codes consistently
  • Return 401 for unauthenticated, 403 for unauthorized

❌ Information Leakage

  • Don't expose stack traces to clients
  • Don't reveal database names, query details, or internal paths
  • Don't expose framework version numbers
  • Don't confirm whether a username exists in error messages

CORS — Cross-Origin Resource Sharing

Configuring CORS Correctly

CORS controls which domains are allowed to make browser-based requests to your API. Misconfigured CORS is a common vulnerability.

  • Define an explicit allowlist of permitted origins — never use a wildcard (*) for authenticated APIs
  • Never reflect the request's Origin header back without validation — this grants any site access
  • Only allow the HTTP methods your API actually uses
  • Only expose headers that are needed
  • Credentials (cookies) require Access-Control-Allow-Credentials: true and a non-wildcard origin — be very deliberate with this

GraphQL-Specific Concerns

  • Disable introspection in production — it reveals your entire schema to attackers
  • Implement query depth and complexity limits to prevent Denial of Service via deeply nested queries
  • Apply authorization checks at the resolver level — not just at the entry point
  • Disable detailed error messages that reveal schema information

Principle of Least Privilege for Database Accounts

Database Users Should Only Do What They Need To

  • The application's database user should have only SELECT, INSERT, UPDATE, DELETE on the tables it needs
  • It should not have DROP TABLE, CREATE, or schema modification rights in production
  • Administrative database operations should use a separate, tightly controlled account
  • Migration scripts should run under a separate account with schema-change permissions, not the application account

Preventing SQL Injection

  • Use parameterized queries (prepared statements) — this is the primary defense
  • Use an ORM (Object-Relational Mapper) correctly — ORMs parameterize by default, but raw query methods can bypass this
  • Never construct SQL queries by concatenating user input, regardless of sanitization attempts
  • Apply stored procedures with parameterization where appropriate

Protecting Sensitive Data at Rest

Sensitive Data Categories

The following data requires special handling and must never be stored as plain text:

  • Passwords: One-way hashed with bcrypt, Argon2, or scrypt — never plain text or reversibly encrypted
  • Payment card data: Use a PCI-DSS compliant payment provider (Stripe, Braintree) — never store raw card numbers
  • Social Security / National ID numbers: Encrypt at rest with strong encryption
  • Health records: Subject to HIPAA; encryption and access logging required
  • API keys and secrets: Never stored in code or plain text — use encrypted secret stores

Database Encryption

  • Encryption at rest: Encrypt the entire database volume or use transparent data encryption (TDE)
  • Column-level encryption: Encrypt specific sensitive columns so even database administrators cannot read them without the key
  • Key management: Encryption keys must be stored separately from the data they protect — a separate key management service (KMS) is ideal

Database Access Controls

  • Do not expose the database port to the public internet — restrict to the application server's IP
  • Use a VPC or private network for database communication
  • Enable database audit logging for sensitive operations
  • Disable or remove unused database features, stored procedures, and user accounts
  • Change all default credentials immediately on installation
  • Regularly patch the database engine

Backups

✅ Backups Are Part of Security

  • Encrypt all database backups
  • Test restore procedures regularly — an untested backup is not a backup
  • Store backups in a separate location from the primary database
  • Control who can access and delete backup files
  • Define and document your Recovery Time Objective (RTO) and Recovery Point Objective (RPO)

Server Hardening

1 Minimize the Attack Surface
  • Disable all services, ports, and features that are not actively needed
  • Remove unused software and packages from the server
  • Use a minimal base OS image — smaller surface means fewer vulnerabilities
2 Keep Everything Patched
  • Apply OS security patches promptly — many breaches exploit known, patched vulnerabilities
  • Keep web server software (Nginx, Apache, IIS) updated
  • Keep runtime environments (Node.js, .NET, Java) updated
  • Subscribe to security advisories for your technology stack
3 Network Controls
  • Use a firewall to allow only necessary inbound traffic (typically 80, 443, and SSH from restricted IPs)
  • Place databases and internal services in private subnets — never publicly accessible
  • Use network segmentation so a compromised web server cannot directly reach the database
  • Consider a Web Application Firewall (WAF) for additional protection against common attacks
4 SSH and Remote Access
  • Disable password-based SSH — use SSH keys only
  • Disable root login over SSH
  • Restrict SSH access to specific, known IP addresses
  • Change the default SSH port (minor obscurity benefit, not a substitute for other controls)
  • Use a bastion host / jump server for production access
5 Secrets Management
  • Store all secrets (database passwords, API keys, encryption keys) in a secrets manager — not in environment files or source code
  • Use tools like AWS Secrets Manager, Azure Key Vault, HashiCorp Vault
  • Never commit secrets to version control — use pre-commit hooks to scan for accidental secrets
  • Rotate secrets regularly and immediately when a person with access leaves

Dependency Security

🚨 Your Dependencies Are Part of Your Attack Surface

Modern applications depend on hundreds of third-party packages. Each one is a potential source of vulnerabilities.

  • Use dependency scanning tools (npm audit, Dependabot, Snyk, OWASP Dependency-Check)
  • Keep dependencies updated and patch known vulnerabilities promptly
  • Audit new dependencies before adding them — prefer packages with active maintenance and large communities
  • Use a lock file to pin dependency versions and prevent unexpected updates
  • Be aware of supply chain attacks — malicious packages that mimic legitimate ones (typosquatting)

Container Security

  • Run containers as a non-root user
  • Use minimal base images (Alpine Linux, distroless images)
  • Scan container images for known vulnerabilities before deployment
  • Do not store secrets in Docker images or environment variables in Dockerfiles
  • Use read-only file systems where possible
  • Apply resource limits to containers to prevent resource exhaustion

📖 The Browser as a Security Boundary

The browser runs code in an environment you do not control. Users may have malicious extensions, run on compromised machines, or be tricked by other sites. Client-side security is about making the browser itself an ally against attacks.

Security Headers

HTTP response headers can instruct the browser to enforce security policies. These are set by the server.

Header Purpose Priority
Content-Security-Policy (CSP) Restricts which scripts, styles, images, and connections are allowed — the primary XSS defense Critical
Strict-Transport-Security (HSTS) Forces browsers to always use HTTPS Critical
X-Frame-Options Prevents your site being loaded in an iframe — protects against clickjacking High
X-Content-Type-Options Prevents browsers from MIME-sniffing responses — stops content type confusion attacks High
Referrer-Policy Controls how much referrer information is shared with other sites Medium
Permissions-Policy Restricts browser features (camera, microphone, geolocation) the page can use Medium
Cross-Origin-Opener-Policy Isolates the browsing context to protect against cross-origin attacks Medium

💡 Test Your Security Headers

Use securityheaders.com to scan your site and get a grade with actionable recommendations. This is a free tool every deployed application should be run through.

Content Security Policy (CSP) — Key Directives

  • default-src: Fallback policy for all resource types not explicitly listed
  • script-src: Where scripts can be loaded from — restrict to your own domain and trusted CDNs
  • style-src: Where stylesheets can be loaded from
  • img-src: Where images can be loaded from
  • connect-src: Which URLs the page can make requests to (fetch, XHR, WebSocket)
  • frame-ancestors: Which sites can embed this page in an iframe (replaces X-Frame-Options)
  • Start with a report-only mode to test CSP without breaking functionality, then enforce it

Sensitive Data in the Browser

✅ Safe Practices

  • Store tokens in HttpOnly cookies
  • Clear sensitive data from memory when no longer needed
  • Use HTTPS to prevent interception
  • Be deliberate about what you put in URLs (logs capture them)

❌ Don't Store These in the Browser

  • Auth tokens in localStorage (XSS risk)
  • Passwords in any form
  • Credit card numbers
  • Social security numbers
  • Sensitive PII in session or local storage

Third-Party Scripts

⚠️ Every Third-Party Script Is a Trust Decision

Analytics scripts, chat widgets, A/B testing tools, and advertising scripts all run with the same privileges as your own code. A compromised third-party script can steal everything on the page. Evaluate every third-party script carefully, use SRI hashes where possible, and restrict them with CSP to limit what they can access.

Data Minimization

Collect Only What You Need

The most secure data is data you never collect. Every field you store is a liability — it must be protected, maintained, and disclosed in a breach. Before adding any data field, ask: "Do we actually need this?"

  • Define a clear purpose for each piece of data collected
  • Set retention periods and delete data when it is no longer needed
  • Avoid collecting unnecessary PII (Personally Identifiable Information)

Sensitive Data Classification

ClassificationExamplesRequired Controls
Public Product names, marketing content Basic integrity controls
Internal Employee directories, internal documents Authentication, access logging
Confidential Customer PII, financial data, business strategy Encryption at rest and in transit, strict access controls, audit logging
Restricted Passwords, payment data, health records, government IDs Strongest encryption, minimal access, regulatory compliance, dedicated storage

Regulatory Compliance

Key Regulations Developers Must Be Aware Of

  • GDPR (EU): Requires consent for data collection, right to access, right to erasure ("right to be forgotten"), breach notification within 72 hours. Applies to any app serving EU users.
  • CCPA (California): California Consumer Privacy Act — similar rights to GDPR for California residents. Applies broadly to businesses handling California residents' data.
  • HIPAA (US): Governs health information. Requires encryption, audit logging, minimum necessary access, and Business Associate Agreements.
  • PCI-DSS: Payment Card Industry standard. If handling card data, requires a formal compliance program. Use a payment processor (Stripe, Square) to avoid direct card handling.
  • COPPA (US): Governs data collection from children under 13. Parental consent required.

Data in Transit and at Rest

  • In transit: All data must be encrypted using TLS. No plain HTTP for any data exchange.
  • At rest: Encrypt sensitive database columns, files, and backups. Use a managed encryption service to handle key management.
  • In logs: Never log passwords, tokens, full credit card numbers, or other sensitive data. Mask or truncate where necessary.
  • In URLs: Never put sensitive values in query strings — they appear in server logs, browser history, and referrer headers.
  • In error messages: Errors must never include sensitive data or internal system details.

Breach Notification

🚨 Know Your Obligations Before a Breach Happens

Most jurisdictions require notification of affected users and/or regulators within a specified time window after discovering a breach. GDPR requires notification within 72 hours. Define your incident response plan before you need it — not after a breach has occurred.

📖 You Cannot Defend What You Cannot See

Logging and monitoring are the difference between discovering a breach in minutes and discovering it months later — or never at all. Many organizations only learn they were breached from external reports, not their own systems.

What to Log

✅ Security-Relevant Events to Log

  • All authentication events (successes and failures)
  • Authorization failures (403 responses)
  • Password changes and resets
  • Account creation and deletion
  • Privilege changes (role assignments)
  • API errors and unusual request patterns
  • Large data exports or bulk operations
  • Admin actions
  • System errors and exceptions

❌ Never Log These

  • Passwords (even hashed)
  • Full session tokens or JWTs
  • Credit card numbers
  • Social security numbers
  • API keys or secrets
  • Health information
  • Full request bodies that may contain sensitive data

Log Quality Requirements

  • Each log entry should include: timestamp (UTC), user/session ID, action, resource, outcome, source IP
  • Use structured logging (JSON format) rather than plain text — easier to query and analyze
  • Ensure log timestamps are synchronized (NTP) across all servers
  • Logs must be tamper-evident — store them where application servers cannot modify or delete them
  • Retain logs for a sufficient period to support investigations (90 days minimum; regulatory requirements may demand more)

Monitoring and Alerting

Set Up Alerts For

  • Multiple failed login attempts in a short window (brute-force indicator)
  • Login from a new country or unusual location
  • Spike in 403/500 error rates (attack or system failure)
  • Sudden increase in database query volume (potential data exfiltration)
  • Admin account logins, especially outside business hours
  • Changes to security configuration
  • Certificate expiry approaching
  • Dependency vulnerability alerts from scanning tools

Incident Response Plan

1 Preparation

Document the plan, assign roles, test it. Know who to call and what to do before an incident occurs. A plan written during a breach is too late.

2 Detection and Analysis

Monitoring alerts fire or suspicious activity is reported. Investigate to confirm and understand the scope. Preserve evidence — do not modify systems until logs are captured.

3 Containment

Stop the bleeding — isolate affected systems, revoke compromised credentials, block attacker IPs. Choose between short-term containment (keep system alive for investigation) and long-term containment (take offline).

4 Eradication

Remove the threat — close the vulnerability exploited, remove any installed malware or backdoors, reset all compromised credentials.

5 Recovery

Restore systems from clean backups, verify integrity, monitor closely for re-attack. Notify affected users and regulators as required.

6 Lessons Learned

Conduct a post-incident review. Document what happened, what worked, what failed, and what changes are needed. Update security controls and the incident response plan.

✅ Use This Before Every Production Deployment

Work through each category before launching any application. This is a starting point — your specific application may require additional controls.

🔐 Authentication

  • Passwords are hashed with bcrypt, Argon2, or scrypt
  • MFA is available (required for admin accounts)
  • Account lockout / rate limiting is implemented on login
  • Error messages do not reveal whether an account exists
  • Password reset tokens expire and can only be used once
  • New session ID is issued on login

🛡️ Authorization

  • Every API endpoint checks authentication
  • Every API endpoint checks the user's permission for that action
  • Resource ownership is verified (IDOR prevention)
  • Client-side-only authorization checks have been identified and eliminated
  • Admin functionality is fully separated from user functionality

📡 Transport Security

  • HTTPS is enforced everywhere — HTTP redirects to HTTPS
  • TLS 1.2+ only — older versions disabled
  • HSTS header is set with a long max-age
  • Certificate is valid and auto-renewal is configured
  • Sensitive data never sent via HTTP

🧹 Input and Output

  • All user input is validated server-side
  • Parameterized queries are used everywhere — no SQL concatenation
  • All user content is output-encoded for the context (HTML, JS, URL)
  • File uploads are validated for type, size, and stored outside web root
  • No stack traces or internal details returned in error responses

🌐 Browser Security

  • Content Security Policy (CSP) header is set and tested
  • X-Frame-Options or CSP frame-ancestors is set
  • X-Content-Type-Options: nosniff is set
  • Referrer-Policy is set appropriately
  • Cookies have HttpOnly, Secure, and SameSite attributes
  • No sensitive data in localStorage

🔌 API

  • Rate limiting is in place on all endpoints
  • CORS is configured with an explicit origin allowlist
  • API keys and secrets are not embedded in client code
  • Request body size limits are enforced
  • Unexpected fields are rejected (mass assignment protection)

🗄️ Database

  • Database account has minimum required permissions
  • Database port is not exposed to the internet
  • Default credentials have been changed
  • Sensitive columns are encrypted
  • Backups are encrypted and tested

🖥️ Server / Infrastructure

  • OS and all packages are patched to current
  • Unused ports and services are disabled
  • Secrets are stored in a secrets manager — not in code or .env files in repos
  • Dependencies have been scanned for known vulnerabilities
  • SSH password login is disabled
  • Firewall rules follow least-privilege

📋 Logging and Monitoring

  • Authentication events are logged
  • Authorization failures are logged
  • Logs do not contain sensitive data
  • Alerts are configured for suspicious patterns
  • Log retention meets regulatory requirements
  • An incident response plan exists and has been reviewed

🔏 Data Protection

  • Only necessary data is collected
  • Data retention and deletion policies are defined
  • Applicable regulations (GDPR, HIPAA, PCI) have been reviewed
  • Privacy policy is accurate and accessible to users
  • Users can request deletion of their data

💡 Further Security Testing

  • OWASP ZAP: Free automated web application security scanner
  • Burp Suite Community: Web security testing platform
  • Nmap: Network port and service scanner
  • SSL Labs (ssllabs.com/ssltest): Grade your HTTPS/TLS configuration
  • Security Headers (securityheaders.com): Grade your HTTP security headers
  • Penetration Testing: Periodic manual security testing by a qualified professional
