CEN3722: Human-Computer Interaction

Designing Good User Login & Authentication

🔐 Module Overview

Login and authentication is the front door of every application. From a HCI perspective it must strike a careful balance: it must be secure enough to protect users and their data, but easy enough to use that users aren't frustrated before they even reach the application. Poor authentication UX causes users to abandon apps, choose weak passwords, write credentials on sticky notes, or disable security features entirely—all of which defeat the purpose of having security at all.

Core Principle: Security that users work around is no security. Good authentication design makes the secure choice the easy choice.

The Tension Between Security and Usability

Authentication sits at a fundamental tension in interface design. Making it more secure often makes it harder to use—and making it easier to use often reduces security. Your job as a designer is to resolve that tension intelligently, not ignore one side of it.

When Security Wins Over UX

Users respond to overly complex authentication by:

  • Using simple, reused passwords
  • Writing credentials on paper
  • Sharing accounts with others
  • Disabling security features
  • Abandoning the app entirely

When UX Wins Over Security

Overly simplified authentication leads to:

  • Easily guessed or stolen credentials
  • No protection against breaches
  • Account takeover and fraud
  • Legal and regulatory liability
  • Loss of user trust

The Three Factors of Authentication

All authentication methods are based on one or more of these three factors. Combining multiple factors (Multi-Factor Authentication) dramatically increases security:

  • Something you know — Password, PIN, security question answer
  • Something you have — Phone, hardware token, smart card, email access
  • Something you are — Fingerprint, face, voice, iris (biometrics)

A single factor can be compromised. Two independent factors working together resist most common attack types.

What Good Authentication UX Looks Like

✓ Principles of Well-Designed Authentication

  • Minimum friction for the most common action (logging in)
  • Friction that scales with risk (step-up authentication for sensitive actions)
  • Clear error messages that help users recover without aiding attackers
  • Transparent explanation of why security requirements exist
  • Multiple paths for recovery (not one single point of failure)
  • Mobile-first design — most users will authenticate on phones
  • Accessibility — works for users with disabilities

The Cost of Bad Authentication Design

Design Failure User Behavior Result
Complex password requirements with no guidance Users choose "Password1!" to satisfy rules Weak password despite complexity
Frequent forced re-authentication Users choose shorter, simpler passwords Reduced security from the friction intended to improve it
No "show password" option on mobile Users choose short passwords they can type correctly Weak credentials driven by input difficulty
Obscure account lockout with no path forward Users call support or abandon account High support costs; user churn
Confusing MFA enrollment Users skip MFA setup Security feature adoption near zero

Overview of Authentication Methods

There is no single "best" authentication method. Each has trade-offs across security strength, user experience, cost, and implementation complexity. Understanding all options allows you to choose the right combination for your application's needs and audience.

🔑

Username + Password

The traditional credential pair — an identifier and a secret

The most familiar authentication method. Users provide a username or email address and a password that only they should know. Despite being the oldest digital authentication method, it remains the baseline that most other methods build upon or replace.

✓ Advantages
  • Universal familiarity — no user learning curve
  • Works on any device and platform
  • No dependency on third-party services
  • Works offline or without network access
  • Low implementation complexity
  • User controls their own credential
✗ Disadvantages
  • Users reuse passwords across sites
  • Weak passwords chosen for memorability
  • Vulnerable to phishing and credential stuffing
  • Password reset is a major support burden
  • Passwords can be leaked in data breaches
  • Must be combined with MFA for adequate security
UX Best Practice: Never block password managers. Design fields with standard HTML types (type="email", type="password", autocomplete="current-password") so browsers and password managers can autofill. Password managers dramatically improve security by enabling long, unique passwords.
🔢

PIN (Personal Identification Number)

A short numeric or alphanumeric code for quick re-authentication

A PIN is a short (typically 4–6 digit) code used as a lower-friction alternative to passwords, usually for re-authentication after a full login rather than as a standalone authentication. PINs are tied to a specific device, which adds implicit security — knowing someone's PIN without their device is useless.

✓ Advantages
  • Fast and easy on mobile
  • Device-bound — stolen PIN without device is useless
  • Easy to change if compromised
  • Works for users with limited dexterity
  • No typing errors from autocorrect
✗ Disadvantages
  • Small keyspace — many PINs are easily guessable (1234, 0000)
  • Vulnerable to shoulder surfing
  • Must be combined with account lockout after failures
  • Not suitable as sole authentication for sensitive apps
  • Users tend to choose predictable patterns
👆

Biometrics (Fingerprint, Face, Voice)

Authenticating using unique physical or behavioral characteristics

Biometric authentication uses physical traits unique to a person. On mobile devices, this most commonly means fingerprint recognition (Touch ID) or facial recognition (Face ID). These are typically implemented through the device's secure enclave—the biometric data never leaves the device.

✓ Advantages
  • Extremely fast (under 1 second)
  • Nothing to remember or type
  • High user satisfaction
  • Difficult to steal or duplicate
  • Strong accessibility for motor-impaired users
  • Biometric data stored on-device securely
✗ Disadvantages
  • Cannot be changed if compromised (unlike passwords)
  • Privacy concerns — users wary of biometric collection
  • Requires modern device hardware
  • Can fail with injuries, dirt, masks, poor lighting
  • Accessibility issues for some users
  • Must always provide fallback (PIN/password)
UX Best Practice: Always offer biometrics as an opt-in enhancement after initial login, not as the only path. Provide a clear PIN/password fallback. Explain during setup what is stored and where.
🔗

Single Sign-On (SSO) — Social & Enterprise

"Continue with Google / Apple / Microsoft / GitHub"

SSO allows users to authenticate using credentials from a trusted third-party identity provider (IdP) such as Google, Apple, or Microsoft. The user logs into the IdP once, and your application accepts that identity without storing passwords itself. OAuth 2.0 and OpenID Connect are the standards that power this.

✓ Advantages
  • No new password to create or remember
  • Significantly reduces registration friction
  • Your app never handles passwords
  • Inherits MFA from the provider
  • Account recovery is the provider's problem
  • Familiar and trusted by users
✗ Disadvantages
  • Dependency on third-party availability
  • If provider account is compromised, your app is too
  • Users may not want to link accounts
  • Privacy concerns about data sharing with provider
  • Some providers track logins across apps
  • Must handle account merging carefully
Important: Always offer at least one SSO option and a traditional email/password option. Never force SSO. Some users don't have Google accounts; others don't want to link them. Apple SSO is required by App Store guidelines if you offer any other SSO option.
✉️

Magic Link (Passwordless Email)

One-time login link sent to the user's email address

The user enters their email address and receives a unique, time-limited link that logs them in immediately when clicked. No password is ever set or stored. Authentication is based entirely on access to the email account. This approach is increasingly popular for low-frequency-use applications where users would otherwise forget their password every time.

✓ Advantages
  • No password to remember, forget, or mistype
  • Eliminates credential stuffing attacks entirely
  • Email address is already verified
  • Very low friction for infrequent users
  • Simple to implement
✗ Disadvantages
  • Requires access to email to log in
  • Email inbox compromise = account compromise
  • Frustrating for frequent use (check email each time)
  • Email delays can break the experience
  • Links can be forwarded or misused
  • Doesn't work if email is unavailable
Best Use Case: Applications with infrequent logins (monthly billing portals, booking confirmations, event apps). Poor choice for daily-use apps where checking email for every login creates unacceptable friction.
📱

One-Time Password (OTP) — SMS or TOTP App

Time-limited codes sent by text or generated by an authenticator app

OTP adds a second factor by requiring a code that changes every 30 seconds. This code is either sent via SMS/text message or generated by an authenticator app (Google Authenticator, Authy, Microsoft Authenticator). TOTP (Time-based One-Time Password) apps are significantly more secure than SMS because they cannot be intercepted by SIM-swapping attacks.

✓ Advantages
  • Strong second factor — codes expire quickly
  • SMS is familiar to almost all users
  • TOTP apps work offline
  • Widely supported standard (TOTP/RFC 6238)
  • Protects against password-only breaches
✗ Disadvantages
  • SMS vulnerable to SIM-swap and interception
  • SMS OTP adds 30–90 second wait time
  • App-based TOTP requires initial setup effort
  • Phone loss can lock users out
  • SMS delivery unreliable internationally
  • Cost of SMS at scale can be significant
Security Note: NIST guidelines (SP 800-63B) have deprecated SMS OTP as a sole second factor for high-security applications due to SIM-swapping vulnerabilities. For higher-risk apps, prefer authenticator apps (TOTP) or hardware keys. SMS OTP remains acceptable for moderate-risk applications.
🛡️

Passkeys (FIDO2 / WebAuthn)

The emerging standard — cryptographic keys replacing passwords entirely

Passkeys use public-key cryptography to replace passwords entirely. The user authenticates using their device's biometrics or PIN, and a cryptographic challenge is solved using a private key stored on the device. The server only stores a public key — there is nothing to steal in a data breach. Supported by Apple, Google, and Microsoft since 2022.

✓ Advantages
  • Phishing-resistant by design
  • No password stored anywhere
  • Extremely fast user experience
  • Combines authentication factors automatically
  • Syncs across devices via cloud keychain
  • Industry direction — growing platform support
✗ Disadvantages
  • Requires modern device and OS (2022+)
  • Users unfamiliar with the concept yet
  • Recovery path still being standardized
  • Implementation more complex than passwords
  • Must maintain password fallback for now
The Future: Passkeys represent the industry direction for authentication. Major platforms (iOS, Android, Windows) support them natively. For new applications targeting modern devices, designing passkey support now future-proofs your authentication.
🔐

Hardware Security Keys (FIDO U2F / FIDO2)

Physical keys (e.g., YubiKey) providing the strongest second factor

Physical USB, NFC, or Bluetooth devices that the user taps or inserts to authenticate. They use the same cryptographic principles as passkeys but are stored on a dedicated hardware device rather than the user's phone or computer. Used in high-security enterprise and government environments.

✓ Advantages
  • Strongest possible second factor
  • Completely phishing-resistant
  • Works without battery or connectivity
  • Physical possession required to authenticate
  • Preferred for privileged/admin accounts
✗ Disadvantages
  • Cost (typically $25–$70 per key)
  • Can be lost or forgotten
  • Requires backup key strategy
  • Poor UX for non-technical users
  • Not practical for consumer applications

Design Decisions Annotated

✓ Email first, not username — Users remember their email; usernames are often forgotten
✓ "Show" toggle on password — Critical on mobile; prevents typos without compromising security
✓ "Forgot password?" near the password field — Where users look when they fail to log in

Password Field UX Best Practices

✓ Do This

  • Show/hide toggle — Allows verification on mobile
  • Support paste — Never block paste; it enables password managers
  • Use autocomplete attributes — autocomplete="current-password" enables autofill
  • Standard type="password" — Triggers native keyboard and manager support
  • Allow long passwords — Support at least 64 characters
  • Preserve entered email on failed login so user doesn't retype it

✗ Don't Do This

  • Never block paste in password fields — forces weak passwords
  • Don't disable autocomplete — breaks password managers
  • Don't clear both fields on wrong password — only clear the password
  • Don't impose arbitrary max lengths — truncating long passwords silently is dangerous
  • Don't show password requirements only after submission — show them during entry
  • Don't require special characters in confusing combinations — use strength meters instead

Error Messages: A UX and Security Balance

Error messages at login must balance two competing goals: giving users enough information to fix their problem, while not giving attackers information about whether an email address exists in the system.

Error Message Examples

SCENARIO: User enters wrong password for a valid account

✗ Too specific (reveals account exists):
"The password you entered is incorrect."

✗ Too vague (unhelpful):
"An error occurred."

✓ Correctly balanced:
"The email or password you entered doesn't match our records.
Please check both and try again."

─────────────────────────────────────────────

SCENARIO: User enters an email not in the system

✗ Too revealing (confirms email not registered — aids account enumeration):
"No account found with that email address."

✓ Correct (same message for both wrong email and wrong password):
"The email or password you entered doesn't match our records."

─────────────────────────────────────────────

SCENARIO: Account is locked out

✗ No path forward:
"Account locked."

✓ Explains and provides a path:
"Your account has been temporarily locked after multiple failed
attempts. Please wait 15 minutes, or reset your password to
regain access immediately."

─────────────────────────────────────────────

SCENARIO: MFA code is incorrect

✓ Helpful with context:
"That code doesn't match. Make sure you're using the most
recent code from your authenticator app — codes expire after
30 seconds. You have 2 attempts remaining."

Password Creation UX

The registration form's password field is where you shape the security of your entire user base. Good UX here means users end up with strong passwords they can actually use.

Password Strength Feedback

Show strength feedback in real time during password creation, not as error messages after submission:

Create password

sunshine
Strength: Weak

Try adding numbers and symbols

Create password

Blue-Mountain$47!
Strength: Strong

Great password!

Modern Best Practice (NIST SP 800-63B): Focus on password length rather than complexity rules. A 15-character passphrase like "Blue-Mountain-Rain" is stronger and more memorable than "P@ssw0rd!1" and easier to use. Check passwords against lists of known breached passwords rather than requiring arbitrary symbols.

Account Recovery UX

Account recovery is one of the most important and most poorly designed flows in most applications. If users can't get back in, they're gone. If recovery is too easy, attackers use it to bypass security.

Designing the Password Reset Flow

GOOD PASSWORD RESET FLOW:

Step 1: User clicks "Forgot password?" on login screen

Step 2: Show simple, clean form
- Ask for email address only
- Display: "We'll send a reset link to this email if an account exists."
- IMPORTANT: Show same message whether email exists or not
(prevents account enumeration)

Step 3: Send reset email immediately
- Email arrives within 60 seconds
- Link expires after 15-60 minutes
- Link is single-use
- Invalidate other pending reset links when new one is requested

Step 4: Reset page (opened from link)
- New password field with strength indicator
- Confirm password field
- After reset: force log out of all other sessions
- Confirmation email sent: "Your password was changed. If this wasn't
you, contact us immediately."

WHAT TO AVOID:

✗ Security questions ("What was your first pet's name?") — easily guessed
or researched from social media; no longer recommended by NIST

✗ Long-lived reset links (24+ hours) — too much window for exploitation

✗ Sending passwords by email — if you can email a password, you stored
it in plain text, which means your database is compromised

✗ Requiring a current password to reset — if user forgot the password,
they can't provide it; circular design failure

Choosing the Right Security Level

Not every application needs the same authentication. Over-engineering login for a recipe sharing app frustrates users unnecessarily. Under-engineering it for a banking app is negligent. The right security level is determined by the sensitivity of the data and the potential harm from a breach.

Questions to Determine Required Security Level

  • What happens if an account is compromised? Loss of convenience, financial loss, safety risk, legal liability?
  • What data is stored? Public content, personal preferences, personal information, health data, financial data?
  • Are financial transactions involved? Any payment or money movement significantly raises the risk profile.
  • Are there regulatory requirements? HIPAA (health), PCI-DSS (payments), FERPA (education), GDPR (EU personal data) all impose requirements.
  • Who are the users? Children require additional protections (COPPA). Professional users may tolerate more friction.
  • Is there a privacy expectation? Does the user expect confidentiality? (Medical, legal, financial apps)

The Four Security Tiers

🟢
Tier 1 — Low Risk
Public content, preferences, non-sensitive personal data

Examples: Recipe app, game leaderboard, news preference settings, public forum profile. Harm from compromise: loss of preferences or embarrassment — no financial, health, or safety risk.

✓
Email + Password login Required
Standard credential-based login with proper storage (bcrypt/Argon2 hashing).
✓
HTTPS everywhere Required
All authentication traffic must be encrypted in transit.
✓
Account lockout after repeated failures Required
Lock or rate-limit after 5–10 failed attempts. Prevents brute force attacks.
✓
Secure password reset via email link Required
Time-limited, single-use tokens. No security questions.
✓
SSO option (Google / Apple) Recommended
Reduces friction; inherits provider security including MFA.
Password strength meter at registration Recommended
Encourages stronger passwords through real-time guidance.
Biometric login option (mobile) Optional
Improves UX on mobile; appropriate for low-risk data.
MFA option (user-initiated) Optional
Offer but do not require; some users will want the extra protection.
🟡
Tier 2 — Moderate Risk
Personal identifiable information, transaction history, private communications

Examples: E-commerce (purchase history, saved addresses), social platforms, project management tools, workplace productivity apps. Harm from compromise: personal data exposure, reputation damage, minor financial loss.

Includes all Tier 1 requirements, plus:

✓
MFA available and actively encouraged Required
Prompt users to enable MFA at registration; highlight security benefit. Do not bury it in settings.
✓
Session management with timeout Required
Inactive sessions expire (30–60 minutes of inactivity). Persistent login ("remember me") is appropriate but should be revocable.
✓
Notification of login from new device/location Required
Email user when login occurs from an unrecognized device. Provides out-of-band detection.
✓
Active session management Required
Allow users to view and revoke active sessions. "Sign out of all devices."
✓
Breach detection (HaveIBeenPwned integration) Recommended
Check newly created or changed passwords against known breach databases. Alert users if their credential has been exposed.
Re-authentication for sensitive changes Recommended
Require password confirmation when changing email, password, or payment methods even if already logged in.
Account activity log Recommended
Show users their recent login history: time, device, location. Empowers users to spot unauthorized access.
🔴
Tier 3 — High Risk
Financial data, health records, legal documents, significant personal data

Examples: Online banking, investment platforms, insurance apps, patient portals, legal document platforms. Harm from compromise: significant financial loss, identity theft, health privacy violations, legal liability. Likely subject to PCI-DSS, HIPAA, or similar regulations.

Includes all Tier 1 and Tier 2 requirements, plus:

✓
MFA mandatory for all users Required
No option to skip. At minimum, TOTP app (preferred over SMS). Enrollment required at registration.
✓
Re-authentication before every sensitive action Required
Require fresh authentication (not just valid session) before viewing full account numbers, initiating transfers, downloading statements.
✓
Step-up authentication for high-value transactions Required
Transfers above a threshold or to new recipients require additional verification beyond standard login.
✓
Short session timeouts Required
Sessions expire after 10–15 minutes of inactivity. Warn users before timeout with option to extend.
✓
Device registration and trusted device management Required
Track and manage registered devices. Require additional verification when logging in from an unregistered device.
✓
Fraud and anomaly detection Required
Flag and challenge logins from unusual locations, times, or after impossible travel (logged in from NYC then London 1 hour later).
✓
Comprehensive audit logging Required
Log all authentication events, session activity, and sensitive actions with timestamps and IP addresses. Retained for regulatory compliance period.
Passkey or hardware key support Recommended
Offer as a more secure alternative to password+MFA. Particularly important for admin accounts.
⚫
Tier 4 — Critical Risk
Admin access, infrastructure control, multi-user platform administration

Examples: Application admin panels, database administration tools, content management systems with publish access, DevOps dashboards, government systems. Harm from compromise: system-wide data breach, service disruption, potential harm to all users of the platform.

Includes all lower-tier requirements, plus:

✓
Hardware security key or passkey mandatory Required
Software-based MFA is not sufficient for admin access. Physical key possession required.
✓
Privileged Access Management (PAM) Required
Admin credentials managed separately from regular user accounts. Just-in-time access provisioning.
✓
Network-level access controls Required
Admin interfaces accessible only from specific IP addresses or via VPN. Not exposed on public internet.
✓
Admin accounts never shared Required
Every individual has their own admin credential. No shared "admin/admin123" accounts. Every action is attributable.
✓
Real-time alerting on admin authentication Required
Security team notified immediately of every admin login. Unusual patterns trigger incident response.
✓
Regular credential rotation and access review Required
Admin credentials rotated on a schedule. Access rights reviewed quarterly and revoked when roles change.

⚠️ A Note on Regulatory Compliance

For any application operating under regulatory frameworks, the framework's authentication requirements supersede these guidelines:

  • HIPAA (health data) — Requires unique user identification, automatic logoff, encryption, and audit controls
  • PCI-DSS (payment card data) — Requires MFA for all admin access, 90-day password changes, specific complexity requirements
  • GDPR (EU personal data) — Requires "appropriate technical measures" to protect personal data; breach notification within 72 hours
  • COPPA (children under 13) — Parental consent required; stricter data handling
  • FERPA (education records) — Strict access controls on student data

When in doubt, consult the specific standard. These tiers are guidelines for design decision-making, not legal compliance advice.

MFA UX — The Biggest Adoption Challenge

MFA is one of the most effective security controls available, reducing account takeover by over 99% in most studies. Yet adoption rates in consumer apps remain low. The primary reason is poor UX during setup and use. Designing MFA well dramatically increases adoption.

MFA Enrollment Design

Principles for MFA Enrollment

1. TIME THE PROMPT CORRECTLY
✓ Prompt after first successful login, not during registration
✓ Frame as "secure your account" — positive, not scary
✓ Show concrete benefit: "Protect your account even if your password is stolen"
✗ Never pop up MFA prompts while user is mid-task

2. MAKE SETUP FEEL SHORT
✓ Show progress: "Step 2 of 3"
✓ Estimate time: "Takes about 2 minutes"
✓ Explain each step clearly with visuals
✗ Don't front-load with warnings and legal text

3. OFFER MULTIPLE OPTIONS
✓ Authenticator app (most secure, offer first)
✓ SMS code (most accessible, offer as fallback)
✓ Email code (last resort, weakest but universal)
✗ Don't offer only one method — if it fails, users are locked out

4. MAKE RECOVERY EXPLICIT DURING SETUP
✓ Provide backup codes (8–10 one-time-use codes)
✓ Clearly instruct: "Save these somewhere safe — you'll need them
if you lose your phone"
✓ Allow download, copy, or print
✗ Don't proceed without confirming user has saved backup codes

5. LET USERS TRUST DEVICES
✓ "Trust this device for 30 days" reduces daily friction
✓ Show which devices are trusted; allow revocation
✗ Never trust a device permanently without an expiry
✗ Never offer "trust device" on shared or public devices

Method Selection Quick Guide

If Your App Is... Use This Authentication Avoid
A consumer mobile app (recipes, games, news) Email/password + SSO (Google/Apple) + biometric on mobile + optional MFA Forced MFA at Tier 1; hardware keys; magic-link only
An e-commerce or marketplace Email/password + SSO + strongly promoted MFA + session notifications Security questions; SMS-only MFA for high-value accounts
A healthcare or financial app Email/password + mandatory TOTP/app MFA + biometric + device registration SMS-only MFA; magic links; optional MFA
An admin / internal tool SSO via enterprise IdP + mandatory hardware key or passkey + VPN/IP restriction Password-only; SMS MFA; shared credentials
An infrequent-use service (booking, billing portal) Magic link or SSO; no-password experience Requiring users to remember passwords they use twice a year

✓ Core Principles to Remember

  • Security that users work around is no security — design for adoption
  • The secure choice should always be the easy choice
  • Never block password managers — they are a security feature
  • Always provide a fallback path — never a single point of failure
  • Error messages protect users, not your data — be helpful without being exploitable
  • Match your authentication strength to the actual risk, not the maximum possible risk
  • Test your login flow with real users — friction you don't notice is friction that costs you users
