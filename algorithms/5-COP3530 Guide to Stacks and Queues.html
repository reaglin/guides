Stacks & Queues

LIFO · FIFO · The Order of Operations
📚 Stack: LIFO - Last In, First Out 🚶 Queue: FIFO - First In, First Out ⚡ Operations: Push, Pop, Enqueue, Dequeue 🎯 Level: Fundamental
The Power of Ordered Access

Stacks and Queues are fundamental linear data structures that control the order in which elements are accessed. Unlike arrays or lists where you can access any element at any time, these structures enforce specific insertion and removal patterns that mirror real-world scenarios.

A Stack follows LIFO (Last In, First Out) — like a stack of plates where you can only add or remove from the top. A Queue follows FIFO (First In, First Out) — like a line at a coffee shop where the first person in line is the first to be served.

These simple concepts are incredibly powerful. Function calls use stacks. Operating systems use queues for process scheduling. Web browsers use stacks for the back button. Understanding when to apply LIFO vs FIFO thinking is essential for solving algorithmic problems and building efficient systems.

⚡
Why Order Matters The restriction of access patterns isn't a limitation — it's a feature. By constraining how elements can be added and removed, stacks and queues provide predictable behavior that algorithms can depend on. This predictability enables elegant solutions to complex problems.
The Stack — LIFO in Action
📚
Stack Fundamentals & Operations

A stack is a collection where all insertions and deletions happen at one end, called the top. The most recently added element is always the first to be removed — hence LIFO (Last In, First Out).

STACK VISUALIZATION
Item 3 ← TOP
Item 2
Item 1 ← BOTTOM
↑ Push adds here   |   Pop removes from here ↑
Push Operation

Add an element to the top of the stack. Always O(1) time complexity regardless of implementation. The new element becomes the new top.

push(item) O(1)
Pop Operation

Remove and return the top element. Fails if stack is empty. Always O(1) time complexity. The element below becomes the new top.

pop() O(1)
Peek/Top Operation

View the top element without removing it. Used to inspect what will be popped next. Essential for algorithms that need to look ahead.

peek() / top() O(1)
isEmpty Check

Check if the stack contains no elements. Critical for preventing errors when popping from an empty stack. Used in loop termination conditions.

isEmpty() O(1)
// Stack Implementation using Array (C++) class Stack { private: int* arr; // Dynamic array to store elements int capacity; // Maximum size of stack int top; // Index of top element (-1 if empty) public: Stack(int size) { arr = new int[size]; capacity = size; top = -1; // Empty stack } // O(1) - Push operation void push(int item) { if (top >= capacity - 1) { throw std::overflow_error("Stack overflow"); } arr[++top] = item; // Increment top, then assign } // O(1) - Pop operation int pop() { if (top == -1) { throw std::underflow_error("Stack underflow"); } return arr[top--]; // Return value, then decrement top } // O(1) - Peek operation int peek() { if (top == -1) { throw std::underflow_error("Empty stack"); } return arr[top]; // Just return, don't modify } // O(1) - Check if empty bool isEmpty() { return top == -1; } };
💡
Implementation Choice: Array vs Linked List Array-based stacks offer better cache performance and lower memory overhead. Linked list-based stacks provide dynamic resizing but have pointer overhead. For most applications, array-based stacks with dynamic resizing (like std::stack) provide the best balance.
The Queue — FIFO in Action
🚶
Queue Fundamentals & Operations

A queue is a collection where insertions happen at one end (called the rear or back) and deletions happen at the other end (called the front). The first element added is the first to be removed — hence FIFO (First In, First Out).

QUEUE VISUALIZATION
Item 1
→
Item 2
→
Item 3
↑ FRONT (dequeue here)                  REAR (enqueue here) ↑
Enqueue Operation

Add an element to the rear (back) of the queue. New elements join the end of the line. Always O(1) with proper implementation.

enqueue(item) O(1)
Dequeue Operation

Remove and return the front element. The element that has been waiting longest is served first. Fails if queue is empty.

dequeue() O(1)
Front Operation

View the front element without removing it. Shows who is next in line without serving them. Essential for priority-based algorithms.

front() / peek() O(1)
isEmpty Check

Check if the queue has no elements. Prevents dequeue operations on empty queues. Used in processing loops and algorithms.

isEmpty() O(1)
// Queue Implementation using Circular Array (C++) class Queue { private: int* arr; // Circular array to store elements int capacity; // Maximum size of queue int front; // Index of front element int rear; // Index of rear element int size; // Current number of elements public: Queue(int cap) { arr = new int[cap]; capacity = cap; front = rear = 0; size = 0; } // O(1) - Enqueue operation void enqueue(int item) { if (size == capacity) { throw std::overflow_error("Queue overflow"); } arr[rear] = item; rear = (rear + 1) % capacity; // Circular increment size++; } // O(1) - Dequeue operation int dequeue() { if (size == 0) { throw std::underflow_error("Queue underflow"); } int item = arr[front]; front = (front + 1) % capacity; // Circular increment size--; return item; } // O(1) - Front operation int getFront() { if (size == 0) { throw std::underflow_error("Empty queue"); } return arr[front]; } // O(1) - Check if empty bool isEmpty() { return size == 0; } };
⚠️
Circular Array Implementation The circular array technique prevents wasted space that would occur with naive array implementation. Without circularity, repeatedly enqueuing and dequeuing would shift the queue to the right, eventually running out of space even with empty slots at the beginning.

Array-Based Advantages

Better cache locality leads to faster access. Lower memory overhead (no pointers). Random access to elements (though not used in stack/queue interface). Generally preferred in performance-critical applications.

Linked List Advantages

No size limitations (bounded only by available memory). No waste of allocated space. Easier to implement without complex resizing logic. Natural choice when maximum size is unknown.

📊
Standard Library Choices C++ std::stack and std::queue use deque (double-ended queue) by default, which provides the benefits of both arrays and linked lists. Java's Stack class uses Vector (dynamic array), while ArrayDeque is preferred for queue operations.
Where LIFO & FIFO Matter in the Real World
🌍
Production Systems & Applications

Stacks and queues aren't academic concepts — they're fundamental building blocks that power the software you use every day. Understanding these applications helps you recognize when to apply LIFO vs FIFO thinking in your own programs.

🧠
Key Insight LIFO (Stacks) are perfect when you need to reverse order or handle nested structures. FIFO (Queues) are ideal when fairness, ordering, or buffering is important.
🌐
Web Browser Back Button

Each page you visit gets pushed onto a stack. Clicking "Back" pops the most recent page. This LIFO behavior lets you retrace your navigation path in reverse order.

Implementation: std::stack<URL> in browser engines
⚡
Function Call Stack

Every function call pushes a frame onto the call stack. When a function returns, its frame is popped. This enables recursion and proper variable scoping in every programming language.

Implementation: Hardware/OS-level stack management
✏️
Undo/Redo Systems

Text editors, image editors, and IDEs use stacks to track operations. Each action is pushed onto the undo stack. Undo pops operations and pushes them to the redo stack.

Implementation: Command pattern with dual stacks
🧮
Expression Evaluation

Calculators and programming language interpreters use stacks to evaluate mathematical expressions, handle operator precedence, and convert between infix, postfix, and prefix notation.

Implementation: Shunting-yard algorithm, recursive descent parsers
🗂️
Balanced Parentheses Checking

Compilers, IDEs, and text editors use stacks to check if parentheses, brackets, and braces are properly matched. Each opening symbol pushes, each closing symbol pops and validates.

Implementation: Stack-based parsing in compiler front-ends
💻
Operating System Process Scheduling

OS schedulers use queues to manage processes waiting for CPU time. FIFO ensures fairness — processes are served in the order they arrive, preventing starvation.

Implementation: Multi-level feedback queues in Linux/Windows
🖨️
Print Spoolers

Print jobs are queued in FIFO order. First document sent to printer is first printed. This prevents chaos when multiple users share a printer and ensures fair access.

Implementation: CUPS (Common Unix Printing System) uses priority queues
🔍
Breadth-First Search (BFS)

Graph and tree traversal algorithms use queues to explore nodes level by level. This FIFO approach ensures you visit all nodes at distance d before any at distance d+1.

Implementation: std::queue in graph libraries
🌐
Web Server Request Handling

HTTP servers queue incoming requests. FIFO processing ensures that requests are handled in order of arrival, providing predictable response times and fair service.

Implementation: Thread pools with work queues in Apache/Nginx
📡
Network Packet Buffers

Network interfaces queue packets for transmission. FIFO ensures packets are sent in order, maintaining network protocols and preventing reordering issues.

Implementation: Ring buffers in network driver code
🎮
Game Event Processing

Game engines queue player inputs and game events. Processing events in FIFO order ensures consistent gameplay and prevents input lag from affecting game state.

Implementation: Event queues in Unity, Unreal Engine
Decision Guide: Stack vs Queue
🎯
Choosing the Right Data Structure
Use a Stack When...

• You need to reverse the order of processing
• Handling nested or recursive structures
• Implementing undo functionality
• Processing most recent items first
• Managing temporary storage during algorithms
• Parsing expressions or syntax

Use a Queue When...

• Fairness is important (first come, first served)
• Processing items in arrival order
• Buffering between fast producers and slow consumers
• Implementing breadth-first algorithms
• Scheduling tasks or processes
• Handling streaming data

💡
The Ultimate Test Ask yourself: "If I were physically handling these items, would I naturally pick from a pile (stack) or wait in a line (queue)?" This intuitive test works remarkably well for choosing the right data structure.
Essential Algorithms & Patterns
🔧
Classic Problems & Solutions

Certain algorithmic patterns appear repeatedly when working with stacks and queues. Mastering these fundamental patterns prepares you for technical interviews and practical programming challenges.

Balanced Parentheses (Stack)

Check if parentheses, brackets, and braces are properly matched. Push opening symbols, pop and validate closing symbols. Stack empty at end means balanced.

O(n) time O(n) space
Postfix Expression Evaluation (Stack)

Evaluate mathematical expressions in postfix notation. Push operands, pop two operands when encountering operators, push result back. Classic stack application.

O(n) time O(n) space
Next Greater Element (Stack)

For each element in array, find the next element that is greater. Use a monotonic decreasing stack. Essential pattern for many optimization problems.

O(n) time O(n) space
Level Order Traversal (Queue)

Visit tree nodes level by level using BFS. Enqueue children of current level, dequeue nodes to process next level. Foundation of many tree algorithms.

O(n) time O(w) space
Sliding Window Maximum (Deque)

Find maximum in every window of size k. Use double-ended queue (deque) to maintain candidates. Combines benefits of both stack and queue operations.

O(n) time O(k) space
Generate Binary Numbers (Queue)

Generate binary representations of numbers 1 to n using queue. Start with "1", generate next by appending "0" and "1" to front of queue. Elegant BFS application.

O(n) time O(n) space
// Balanced Parentheses - Classic Stack Problem bool isBalanced(string s) { stack<char> st; for (char c : s) { // Push opening brackets if (c == '(' || c == '[' || c == '{') { st.push(c); } // Check closing brackets else if (c == ')' || c == ']' || c == '}') { if (st.empty()) return false; char top = st.top(); st.pop(); // Check if brackets match if ((c == ')' && top != '(') || (c == ']' && top != '[') || (c == '}' && top != '{')) { return false; } } } return st.empty(); // Balanced if stack is empty } // Level Order Traversal - Classic Queue Problem void levelOrder(TreeNode* root) { if (!root) return; queue<TreeNode*> q; q.push(root); while (!q.empty()) { int levelSize = q.size(); // Process one level at a time for (int i = 0; i < levelSize; i++) { TreeNode* node = q.front(); q.pop(); cout << node->val << " "; // Process current node // Add children for next level if (node->left) q.push(node->left); if (node->right) q.push(node->right); } cout << endl; // New line after each level } }
🎓
Interview Success Pattern When you see problems involving nested structures, expression parsing, or backtracking — think Stack. When you see problems involving level-by-level processing, fair scheduling, or buffering — think Queue. These patterns appear in 80%+ of technical interviews.
