What Is This Course?

Data Structures is the study of how information is organized, stored, and accessed in computer memory — and how that organization affects the speed and efficiency of every operation performed on that data. It is one of the most foundational courses in computer science because every program, from a simple to-do list to a search engine indexing billions of web pages, depends on choosing the right structure for the right problem.

A data structure is a specific way of arranging data so that it can be used efficiently. Paired with algorithms — the step-by-step procedures for processing that data — data structures form the building blocks of all software engineering. The famous maxim from computer scientist Niklaus Wirth captures it precisely: "Algorithms + Data Structures = Programs."

This course builds your ability to select, implement, and analyze the right tool for the job. By the end, you will think in terms of time complexity, space trade-offs, and structural properties rather than reaching for the first available solution.

Fundamental Data Structures

These structures form the bedrock of all computer science. Every programmer must have deep, fluent knowledge of these before advancing.

📦
Arrays & Dynamic Arrays Fundamental
Static Array
Fundamental

A contiguous block of memory holding elements of the same type, indexed from 0. Fixed in size at creation. The simplest and most memory-efficient structure, with instant O(1) random access by index. The foundation for understanding memory addressing.

Full Coverage
Dynamic Array (Vector)
Fundamental

An array that automatically resizes by doubling capacity when full. Amortized O(1) appends with O(1) random access. Used behind C++ std::vector and Java's ArrayList. Understanding amortized analysis begins here.

Full Coverage
Multi-Dimensional Arrays
Fundamental

Arrays of arrays used to represent matrices, grids, and tabular data. Critical for dynamic programming, image processing, and graph adjacency matrix representations. Row-major vs. column-major memory layouts affect cache performance.

Partial Coverage
🔗
Linked Lists Fundamental
💡
Linked lists trade away random access (O(n) lookup) in exchange for efficient insertions and deletions at any position without shifting elements. They are the first pointer-based structure most students encounter, making them essential for building intuition about heap memory and references.
Singly Linked List
Fundamental

Each node holds data and a pointer to the next node. The list ends at a null pointer. Traversal is one-directional. Used as the basis for stacks and queues. Foundational for understanding pointer manipulation, node creation, and dynamic memory.

Full Coverage
Doubly Linked List
Fundamental

Each node holds data plus pointers to both the next and previous nodes. Enables O(1) insertion and deletion at both ends, and supports reverse traversal. Used in browser history, undo/redo systems, and as the backing structure of std::list.

Full Coverage
Circular Linked List
Fundamental

The tail node's next pointer points back to the head, forming a loop. Used in round-robin schedulers, circular buffers, and Josephus-problem type algorithms. Requires careful traversal logic to avoid infinite loops.

Partial Coverage
📚
Stacks & Queues Fundamental
Stack (LIFO)
Fundamental

Last-In, First-Out structure. Core operations: push (add to top), pop (remove top), peek (view top). All O(1). Implemented with arrays or linked lists. Used for function call frames, expression evaluation, undo systems, and depth-first search.

Full Coverage
Queue (FIFO)
Fundamental

First-In, First-Out structure. Core operations: enqueue (add to rear), dequeue (remove from front). Used in breadth-first search, print spoolers, task scheduling, and message queues. Circular array implementation avoids wasted space.

Full Coverage
Deque (Double-Ended Queue)
Fundamental

Supports O(1) insertion and deletion at both ends. Generalizes both stacks and queues. Used in sliding window problems, undo/redo with front/back access, and as the backing structure for many language standard library data structures.

Full Coverage
Priority Queue
Intermediate

Elements are dequeued in order of priority rather than insertion order. Usually implemented with a heap. Critical for Dijkstra's algorithm, A* pathfinding, job scheduling, and Huffman encoding. Bridge between fundamental and intermediate structures.

Full Coverage
🔑
Hash Tables & Hash Maps Fundamental

Hash tables achieve average O(1) lookup, insertion, and deletion — the gold standard for key-value storage. Understanding how they work internally is essential for avoiding worst-case scenarios and writing cache-efficient code.

Hash Table
Fundamental

Maps keys to array indices using a hash function. Supports average O(1) get, put, and delete. The foundation of std::unordered_map, Python dictionaries, and Java's HashMap. Every modern language's most-used container depends on this structure.

Full Coverage
Hash Functions
Fundamental

The function that transforms a key into a valid array index. Good hash functions distribute keys uniformly to minimize collisions. Common approaches: division method, multiplication method, polynomial rolling hash, FNV hash. Directly affects performance.

Full Coverage
Collision Resolution
Fundamental

Strategies for handling two keys mapping to the same index. Chaining: store a linked list at each slot. Open Addressing: probe to find the next available slot (linear probing, quadratic probing, double hashing). Trade-offs involve memory use and cache behavior.

Full Coverage
Hash Set
Fundamental

A hash table that stores only keys (no associated values). Used to track membership, remove duplicates, and check containment in O(1) average time. The back-end of std::unordered_set and Python's set.

Full Coverage
Intermediate Data Structures

These structures introduce hierarchical organization, ordered access, and more complex performance guarantees. Mastery here separates competent programmers from strong ones.

🌳
Trees — Hierarchical Structures Intermediate
ℹ️
A tree is a connected, acyclic graph where one node is designated as the root. Every other node has exactly one parent. Trees model hierarchies naturally: file systems, organization charts, HTML DOM, and decision processes are all trees. Most tree variants in this section share O(log n) average operation time when balanced.
Binary Tree
Intermediate

Each node has at most two children (left and right). Forms the base concept for all binary tree variants. Core traversals — in-order, pre-order, post-order, and level-order — apply to all binary trees and must be mastered here.

Full Coverage
Binary Search Tree (BST)
Intermediate

A binary tree with the ordering property: left subtree values < node < right subtree values. Enables O(log n) average search, insert, and delete. Degrades to O(n) on sorted input, motivating balanced variants. Foundation for understanding tree-based search.

Full Coverage
AVL Tree
Intermediate

A self-balancing BST that maintains a height-balance condition: for every node, the heights of left and right subtrees differ by at most 1. Rotations (single and double) restore balance after insertions and deletions. Guarantees O(log n) worst-case performance.

Full Coverage
Heap (Binary Heap)
Intermediate

A complete binary tree stored as an array, satisfying the heap property: each node is greater (max-heap) or smaller (min-heap) than its children. Supports O(1) peek at min/max, O(log n) insert and extract. The natural structure for priority queues and heap sort.

Full Coverage
Trie (Prefix Tree)
Intermediate

A tree where each path from root to a leaf spells a key, character by character. O(m) search, insert, and delete where m is key length. Used in autocomplete, spell checkers, IP routing, and dictionary lookups. Nodes are shared for common prefixes, saving space.

Full Coverage
B-Tree
Intermediate

A self-balancing tree where each node can have many children and keys. Designed for disk-based storage: minimizes disk reads by keeping wide, shallow trees. The underlying structure of most relational database indexes and file systems.

Concept Overview
🕸️
Graphs — Relational Structures Intermediate
💡
Graphs are the most general structure in computer science. Any relationship between entities can be modeled as a graph. Social networks, road maps, dependency systems, the internet itself — all are graphs. Algorithms on graphs (BFS, DFS, shortest path, spanning trees) are the most broadly applicable algorithms in software engineering.
Graph Fundamentals
Intermediate

Vertices (nodes) connected by edges. May be directed or undirected, weighted or unweighted, cyclic or acyclic. Understanding the vocabulary — adjacency, degree, path, connected components, cycle — is prerequisite to all graph algorithms.

Full Coverage
Graph Representation
Intermediate

Adjacency Matrix: 2D array; O(1) edge lookup, O(V²) space. Adjacency List: array of lists; space-efficient for sparse graphs, O(V+E) space. Choice of representation significantly impacts algorithm efficiency and code simplicity.

Full Coverage
Directed Acyclic Graph (DAG)
Intermediate

A directed graph with no cycles. Used to represent dependency orders, task scheduling, and compilation dependencies. Topological sort — ordering vertices by dependency — is the essential DAG algorithm and is computable in O(V+E).

Full Coverage
Weighted Graph
Intermediate

Each edge carries a numeric weight representing cost, distance, or capacity. Used in routing, network flow, and minimum spanning tree problems. Enables meaningful shortest-path computation. Dijkstra's and Bellman-Ford algorithms require weighted edges.

Full Coverage
🧩
Disjoint Set (Union-Find) Intermediate

A structure that efficiently tracks a collection of non-overlapping sets. Supports two core operations: Union (merge two sets) and Find (identify which set an element belongs to). With path compression and union by rank, both operations approach O(1) amortized — nearly constant time.

Union-Find
Intermediate

Used to determine connected components, detect cycles in undirected graphs, and as the core data structure in Kruskal's minimum spanning tree algorithm. The "path compression" optimization makes this one of the most elegantly efficient structures in CS.

Full Coverage
Advanced Data Structures

These structures require strong fundamentals and introduce sophisticated balancing mechanisms, range query capabilities, and string-processing power. Mastery here opens the door to systems programming and competitive algorithms.

🔴
Red-Black Trees & Balanced BSTs Advanced
Red-Black Tree
Advanced

A self-balancing BST that colors nodes red or black and enforces rules to keep the tree approximately balanced. O(log n) guaranteed for all operations. The back-end of std::map, std::set in C++, and Java's TreeMap. Less strictly balanced than AVL but faster in practice for insert-heavy workloads.

Full Coverage
2-3 Trees
Advanced

Every internal node has 2 or 3 children, and all leaves are at the same depth. Conceptually simpler than red-black trees and studied as the theoretical basis for understanding B-Trees and red-black tree invariants. Guarantees perfectly balanced structure.

Concept Overview
Skip List
Advanced

A probabilistic data structure using multiple layers of linked lists to achieve O(log n) average search without tree rotations. Used in Redis, LevelDB, and some real-time databases. Interesting case study in randomized algorithms achieving balanced-tree performance with simpler implementation.

Concept Overview
📊
Segment Trees & Fenwick Trees Advanced
ℹ️
Both structures solve the range query problem: given an array, answer aggregate queries (sum, min, max) over any sub-range efficiently, and support point updates. Naively, this takes O(n) per query. These structures reduce it to O(log n), which is transformative for applications processing millions of range queries.
Segment Tree
Advanced

A binary tree where each node stores an aggregate over a sub-range of the array. Supports O(log n) range queries and O(log n) point updates. Can be extended for range updates with lazy propagation. Highly versatile — applies to sum, min, max, GCD, and custom aggregation functions.

Concept Overview
Fenwick Tree (BIT)
Advanced

A compact array structure using binary index relationships to compute prefix sums in O(log n) with O(log n) updates. Significantly simpler to implement than a segment tree for sum-specific queries. Named after Peter Fenwick; also called Binary Indexed Tree (BIT). Beloved in competitive programming.

Concept Overview
🔤
Advanced String Structures Advanced
Suffix Array
Advanced

A sorted array of all suffixes of a string. Enables O(m log n) substring search where m is pattern length. Used in data compression, bioinformatics (DNA sequence alignment), and full-text search indexing. More space-efficient than suffix trees.

Beyond Course Scope
Suffix Tree
Advanced

A compressed trie of all suffixes of a string. Supports O(m) pattern matching for any pattern of length m. Used in string algorithms, text compression, and bioinformatics. Ukkonen's algorithm builds one in O(n) time and is considered one of the most elegant algorithms in CS.

Beyond Course Scope
Aho-Corasick Automaton
Advanced

A finite-state machine built from a trie of multiple patterns. Enables simultaneous multi-pattern string search in O(n + m + z) where z is the number of matches. Used in network intrusion detection, bioinformatics, and antivirus software.

Beyond Course Scope
Expert / Specialized Data Structures

These structures appear in systems programming, competitive programming, database internals, and specialized algorithmic domains. COP 3530 introduces awareness of these; deep coverage belongs to graduate-level courses.

🧠
Expert-Level Structures Overview Expert
Fibonacci Heap
Expert

A collection of heap-ordered trees with amortized O(1) insert, decrease-key, and O(log n) extract-min. Improves Dijkstra's algorithm from O((V+E) log V) to O(V log V + E). Primarily theoretical importance; complex to implement correctly in practice.

Beyond Course Scope
Van Emde Boas Tree
Expert

Achieves O(log log U) operations for integer keys in a universe of size U. Bridges the gap between comparison-based O(log n) and RAM-model O(1) operations. Important in theoretical CS for understanding integer data structure lower bounds.

Beyond Course Scope
Persistent Data Structures
Expert

Immutable structures where modifications produce new versions rather than mutating in place, preserving all historical states. Central to functional programming (Clojure, Haskell) and version-controlled systems. Path-copying and fat-node techniques enable efficient persistence.

Beyond Course Scope
Bloom Filter
Expert

A probabilistic space-efficient structure for set membership testing. Guarantees no false negatives but allows configurable false positive rates. Used in database query optimization, web caches, and spell checkers where approximate membership is acceptable.

Concept Overview
Treap
Expert

A randomized BST that simultaneously satisfies BST property on keys and heap property on randomly-assigned priorities. Expected O(log n) height with high probability. Simpler to implement than AVL or red-black trees while maintaining expected-case balance guarantees.

Beyond Course Scope
KD-Tree
Expert

A space-partitioning binary tree for organizing multi-dimensional points. Enables efficient k-nearest neighbor search and range queries in k-dimensional space. Used in graphics, machine learning (kNN classifier), and geographic information systems.

Beyond Course Scope

Algorithm Fundamentals

Data structures are meaningless without the algorithms that operate on them. COP 3530 covers both, with a strong emphasis on understanding why an algorithm works and how to analyze its efficiency rigorously.

⏱️
Algorithm Analysis — Big-O Notation Fundamental

Big-O notation describes the upper bound on an algorithm's resource usage (time or space) as input size n grows toward infinity. It is the universal language for comparing algorithmic efficiency and is the single most important analytical tool in this course.

Complexity Class Name Example n = 1,000
O(1) Constant Array index access, hash table lookup 1 op
O(log n) Logarithmic Binary search, balanced BST operations ~10 ops
O(n) Linear Linear search, traversal, array scan 1,000 ops
O(n log n) Linearithmic Merge sort, heap sort, efficient sorts ~10,000 ops
O(n²) Quadratic Bubble/insertion/selection sort (naive) 1,000,000 ops
O(2ⁿ) Exponential Brute-force subset enumeration 2¹⁰⁰⁰ ops ⚠️
💡
Beyond Big-O, this course covers Big-Ω (lower bound), Big-Θ (tight bound), best/average/worst-case analysis, amortized analysis, and recurrence relations (Master Theorem). Understanding when a bound applies — and which case it describes — is as important as knowing the formula.

Searching Algorithms Fundamental → Intermediate
Linear Search

Scan each element in sequence until the target is found. O(n) time, O(1) space. Works on unsorted data. The baseline for all search algorithm analysis. Optimal when data is unsorted and searches are infrequent.

Full Coverage
Binary Search

Repeatedly halve the search space by comparing the target to the midpoint. O(log n) time, requires sorted input. One of the most elegant algorithms in CS. Applicable to sorted arrays, monotonic functions, and as a meta-technique (binary search on answer).

Full Coverage
BFS (Breadth-First Search)

Graph traversal that explores all neighbors at depth d before any at depth d+1. Uses a queue. Finds shortest paths in unweighted graphs. O(V+E) time and space. Foundation for network analysis, shortest path, and level-order tree traversal.

Full Coverage
DFS (Depth-First Search)

Graph traversal that goes as deep as possible before backtracking. Uses a stack (or recursion). O(V+E) time and space. Used for topological sort, cycle detection, connected components, maze solving, and as the basis for many advanced graph algorithms.

Full Coverage
⚙️
Algorithm Design Paradigms Intermediate → Advanced

Beyond individual algorithms, COP 3530 teaches general problem-solving strategies that apply across many problems. Recognizing which paradigm a problem fits is a high-level skill that separates strong CS graduates from the rest.

Divide & Conquer
Intermediate

Break the problem into independent sub-problems, solve each recursively, then combine results. Analyzed using recurrence relations and the Master Theorem. Powers merge sort, quick sort, binary search, and Strassen's matrix multiplication.

Full Coverage
Dynamic Programming
Intermediate

Solve overlapping sub-problems once, storing results in a table (memoization or tabulation). Transforms exponential brute-force into polynomial time. Applies when a problem has optimal substructure and overlapping subproblems. Fibonacci, 0/1 Knapsack, LCS, Dijkstra are all DP.

Full Coverage
Greedy Algorithms
Intermediate

Make the locally optimal choice at each step with the hope of finding a global optimum. Faster than DP but only correct when a greedy choice property holds. Powers Huffman encoding, Kruskal's MST, Prim's MST, and activity selection problems.

Full Coverage
Backtracking
Intermediate

Systematic exploration of all possibilities via incremental candidate construction with early abandonment (pruning) of candidates that cannot lead to valid solutions. Used for constraint satisfaction: N-Queens, Sudoku, permutation generation, and graph coloring.

Full Coverage
Two Pointers / Sliding Window
Intermediate

Use two index pointers moving through an array to solve problems in O(n) that would naively be O(n²). Sliding window maintains a variable-size subarray satisfying a condition. Used for substring problems, subarray sums, and sorted array pair searches.

Partial Coverage
Graph Algorithms (Weighted)
Advanced

Dijkstra's (single-source shortest path, non-negative weights), Bellman-Ford (handles negative weights), Floyd-Warshall (all-pairs shortest path), Kruskal's and Prim's (minimum spanning tree). Each leverages specific data structures for efficiency.

Full Coverage
Careers That Require Data Structures Knowledge

Data structures aren't academic exercises — they appear directly in technical interviews and daily work across virtually every software discipline. The following roles demand fluency at varying levels of depth.

💻
Software Engineer / Developer

The broadest role in tech. Engineers design, build, and maintain software systems across web, mobile, desktop, and embedded platforms. Data structures are assessed directly in technical interviews at virtually every major company (FAANG, Microsoft, Stripe, etc.).

Core DS used: Arrays, hash maps, linked lists, trees, graphs, stacks/queues for everyday code; heaps and balanced BSTs for performance-critical paths.
⚡
Systems Programmer

Builds operating systems, compilers, virtual machines, embedded firmware, and performance-critical libraries. Works at the boundary between hardware and software. Requires deep understanding of memory layout, cache behavior, and algorithmic complexity at the machine level.

Core DS used: Memory allocators (free lists), page tables (tries), schedulers (heaps/queues), symbol tables (hash tables), IR representation (DAGs).
🗄️
Database Engineer

Designs and optimizes database engines, query planners, and storage systems. Builds or tunes indexes, caching layers, and transaction managers. Companies like Oracle, PostgreSQL, MySQL, MongoDB, and Redis are built on sophisticated data structure choices.

Core DS used: B-Trees and B+-Trees (disk indexes), hash indexes, skip lists (Redis), LSM trees (NoSQL engines), query execution trees.
🤖
Machine Learning / AI Engineer

Builds and deploys ML models and the infrastructure that runs them. Efficient data preprocessing, feature engineering, and model serving all rely heavily on knowing the right structure for each task at scale. Performance at billion-record scale requires algorithmic sophistication.

Core DS used: Matrices (multi-dim arrays), KD-Trees (kNN), priority queues (beam search), graphs (neural network DAGs), hash maps (feature stores).
🌐
Network / Infrastructure Engineer

Designs routing protocols, content delivery networks, load balancers, and distributed systems. Routing tables, packet forwarding, and network topology problems are directly graph and trie problems at their core.

Core DS used: Tries (IP routing / CIDR), graphs (network topology), priority queues (routing protocols like OSPF), hash tables (ARP caches).
🔐
Cybersecurity Engineer

Builds systems to detect, prevent, and respond to security threats. Intrusion detection, malware scanning, and network packet analysis all involve real-time string and graph matching at high throughput.

Core DS used: Hash tables (password/signature storage), Bloom filters (fast lookup), Aho-Corasick automata (multi-pattern malware detection), graphs (attack path analysis).
🎮
Game Developer

Builds interactive simulations, real-time rendering engines, physics systems, and AI agents for games. Pathfinding, spatial indexing, animation blending, and game state management all depend directly on data structure choices that affect frame rates.

Core DS used: KD-Trees / Octrees (spatial partitioning), priority queues (A* pathfinding), graphs (world/level graphs), hash maps (asset caching).
📊
Data Engineer / Analytics Engineer

Builds data pipelines, ETL systems, warehouses, and the infrastructure that feeds analysts and data scientists. Processing billions of rows efficiently requires knowing when to reach for sorted structures, hash joins, and columnar representations.

Core DS used: Hash tables (hash joins in query engines), priority queues (top-k queries), sorted arrays (range scans), trees (query plan optimization).
🔬
Computational Scientist / Bioinformatician

Applies algorithms to scientific problems in genomics, computational chemistry, physics simulations, and climate modeling. Sequence alignment, protein structure prediction, and genome assembly are among the most computationally demanding problems in science.

Core DS used: Suffix arrays and trees (DNA alignment), dynamic programming tables (sequence alignment), graphs (metabolic networks), hash tables (k-mer indexing).
🎯
