Introduction

User-defined functions (UDFs) in T-SQL are reusable code modules that accept parameters, perform calculations or operations, and return a value or table. Unlike stored procedures, functions can be used directly in SQL queries, making them powerful tools for encapsulating business logic.

What Are Stored Functions?

Stored functions are database objects that:

  • Accept Input Parameters: Can receive zero or more input parameters
  • Return Values: Must return a single value (scalar) or a table
  • Are Deterministic or Non-Deterministic: Can produce consistent or varying results
  • Can Be Used in Queries: Can be called from SELECT, WHERE, JOIN clauses
  • Are Read-Only: Cannot modify database state (with rare exceptions)

When to Use Functions

  • Calculations that need to be reused across queries
  • Data transformations within SELECT statements
  • Complex filtering logic in WHERE clauses
  • Encapsulating business rules for data retrieval
  • Creating computed columns or indexed views
💡 Key Concept: Functions are designed for queries and calculations, while procedures are designed for actions and transactions. Functions return data, procedures perform operations.

Types of Functions

T-SQL provides three types of user-defined functions, each with distinct characteristics and use cases.

1. Scalar Functions

Returns: A single scalar value (INT, VARCHAR, DECIMAL, etc.)

Use Case: Calculations, conversions, single value lookups

Example: Calculate age from birthdate, format phone numbers, compute discounts

2. Inline Table-Valued Functions (TVFs)

Returns: A table (result of a single SELECT statement)

Use Case: Parameterized views, filtered result sets

Example: Get customer orders by date range, filter products by category

3. Multi-Statement Table-Valued Functions (MSTVFs)

Returns: A table (defined structure with multiple operations)

Use Case: Complex data processing requiring multiple steps

Example: Calculate hierarchical data, build summary tables

Comparison of Function Types

Example 1: Simple Calculation

Calculate Age from Birthdate

CREATE FUNCTION dbo.fn_CalculateAge
(
    @BirthDate DATE
)
RETURNS INT
AS
BEGIN
    DECLARE @Age INT;
    
    SET @Age = DATEDIFF(YEAR, @BirthDate, GETDATE()) -
        CASE 
            WHEN MONTH(@BirthDate) > MONTH(GETDATE()) THEN 1
            WHEN MONTH(@BirthDate) = MONTH(GETDATE()) 
                AND DAY(@BirthDate) > DAY(GETDATE()) THEN 1
            ELSE 0
        END;
    
    RETURN @Age;
END;
GO

-- Usage
SELECT 
    FirstName,
    LastName,
    dbo.fn_CalculateAge(BirthDate) AS Age
FROM Customers;

Example 2: Business Logic Calculation

Calculate Order Discount Based on Customer Tier

CREATE FUNCTION dbo.fn_GetCustomerDiscount
(
    @CustomerID INT,
    @OrderAmount DECIMAL(10,2)
)
RETURNS DECIMAL(10,2)
AS
BEGIN
    DECLARE @DiscountPercent DECIMAL(5,2) = 0;
    DECLARE @TotalOrders INT;
    DECLARE @Discount DECIMAL(10,2);
    
    -- Get customer order count
    SELECT @TotalOrders = COUNT(*)
    FROM Orders
    WHERE CustomerID = @CustomerID;
    
    -- Determine discount tier
    SET @DiscountPercent = CASE
        WHEN @TotalOrders >= 50 THEN 15.00  -- VIP: 15%
        WHEN @TotalOrders >= 20 THEN 10.00  -- Gold: 10%
        WHEN @TotalOrders >= 10 THEN 5.00   -- Silver: 5%
        ELSE 0.00                          -- Regular: 0%
    END;
    
    -- Calculate discount amount
    SET @Discount = @OrderAmount * (@DiscountPercent / 100);
    
    RETURN @Discount;
END;
GO

-- Usage in query
SELECT 
    o.OrderID,
    o.CustomerID,
    o.TotalAmount,
    dbo.fn_GetCustomerDiscount(o.CustomerID, o.TotalAmount) AS Discount,
    o.TotalAmount - dbo.fn_GetCustomerDiscount(o.CustomerID, o.TotalAmount) AS FinalAmount
FROM Orders o
WHERE o.OrderDate >= DATEADD(MONTH, -1, GETDATE());

Example 4: String Formatting Function

Format Phone Number

CREATE FUNCTION dbo.fn_FormatPhoneNumber
(
    @PhoneNumber VARCHAR(20)
)
RETURNS VARCHAR(20)
AS
BEGIN
    DECLARE @Formatted VARCHAR(20);
    DECLARE @CleanNumber VARCHAR(20);
    
    -- Remove non-numeric characters
    SET @CleanNumber = @PhoneNumber;
    SET @CleanNumber = REPLACE(REPLACE(REPLACE(@CleanNumber, '(', ''), ')', ''), '-', '');
    SET @CleanNumber = REPLACE(REPLACE(@CleanNumber, ' ', ''), '.', '');
    
    -- Format as (XXX) XXX-XXXX
    IF LEN(@CleanNumber) = 10
        SET @Formatted = '(' + SUBSTRING(@CleanNumber, 1, 3) + ') ' + 
                          SUBSTRING(@CleanNumber, 4, 3) + '-' + 
                          SUBSTRING(@CleanNumber, 7, 4);
    ELSE
        SET @Formatted = @PhoneNumber; -- Return original if not 10 digits
    
    RETURN @Formatted;
END;
GO

-- Usage
SELECT 
    CustomerID,
    FirstName,
    LastName,
    dbo.fn_FormatPhoneNumber(PhoneNumber) AS FormattedPhone
FROM Customers;

Example 1: Parameterized View

Get Customer Orders by Date Range

CREATE FUNCTION dbo.fn_GetCustomerOrdersByDate
(
    @CustomerID INT,
    @StartDate DATE,
    @EndDate DATE
)
RETURNS TABLE
AS
RETURN
(
    SELECT 
        o.OrderID,
        o.OrderDate,
        o.TotalAmount,
        o.Status,
        COUNT(oi.OrderItemID) AS ItemCount
    FROM Orders o
    INNER JOIN OrderItems oi ON o.OrderID = oi.OrderID
    WHERE o.CustomerID = @CustomerID
        AND o.OrderDate >= @StartDate
        AND o.OrderDate < DATEADD(DAY, 1, @EndDate)
    GROUP BY o.OrderID, o.OrderDate, o.TotalAmount, o.Status
);
GO

-- Usage
SELECT *
FROM dbo.fn_GetCustomerOrdersByDate(1, '2024-01-01', '2024-12-31');

-- Can join with other tables
SELECT 
    c.FirstName,
    c.LastName,
    o.OrderID,
    o.OrderDate,
    o.TotalAmount
FROM Customers c
CROSS APPLY dbo.fn_GetCustomerOrdersByDate(c.CustomerID, '2024-01-01', '2024-12-31') o
WHERE c.CustomerID IN (1, 2, 3);

Get Products by Category with Stock Filter

CREATE FUNCTION dbo.fn_GetProductsByCategory
(
    @CategoryID INT,
    @MinStock INT = 0,
    @InStockOnly BIT = 0
)
RETURNS TABLE
AS

(

Example 1: Hierarchical Data Processing

Get Employee Hierarchy

-- Sample Employees table structure
CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    FirstName NVARCHAR(50),
    LastName NVARCHAR(50),
    ManagerID INT,
    JobTitle NVARCHAR(100)
);

CREATE FUNCTION dbo.fn_GetEmployeeHierarchy
(
    @ManagerID INT
)
RETURNS @Hierarchy TABLE
(
    EmployeeID INT,
    FullName NVARCHAR(101),
    JobTitle NVARCHAR(100),
    Level INT,
    Path NVARCHAR(500)
)
AS
BEGIN
    -- Insert the manager
    INSERT INTO @Hierarchy
    SELECT 
        EmployeeID,
        FirstName + ' ' + LastName,
        JobTitle,
        0 AS Level,
        CAST(EmployeeID AS NVARCHAR(500))
    FROM Employees
    WHERE EmployeeID = @ManagerID;
    
    -- Recursively get subordinates
    DECLARE @CurrentLevel INT = 0;
    
    WHILE EXISTS (SELECT 1 FROM @Hierarchy WHERE Level = @CurrentLevel)
    BEGIN
        INSERT INTO @Hierarchy
        SELECT 
            e.EmployeeID,
            e.FirstName + ' ' + e.LastName,
            e.JobTitle,
            @CurrentLevel + 1,
            h.Path + ' > ' + CAST(e.EmployeeID AS NVARCHAR(500))
        FROM Employees e
        INNER JOIN @Hierarchy h ON e.ManagerID = h.EmployeeID
        WHERE h.Level = @CurrentLevel;
        
        SET @CurrentLevel = @CurrentLevel + 1;
    END
    
    RETURN;
END;
GO

-- Usage
SELECT 
    REPLICATE('  ', Level) + FullName AS EmployeeName,
    JobTitle,
    Level
FROM dbo.fn_GetEmployeeHierarchy(1)
ORDER BY Path;

Example 2: Summary Table Generation

Calculate Customer Statistics

CREATE FUNCTION dbo.fn_GetCustomerStatistics
(
    @Year INT
)
RETURNS @Stats TABLE
(
    CustomerID INT,
    CustomerName NVARCHAR(101),
    TotalOrders INT,
    TotalSpent DECIMAL(10,2),
    AverageOrderValue DECIMAL(10,2),
    LastOrderDate DATE,
    CustomerTier NVARCHAR(20)
)
AS
BEGIN
    -- Insert basic statistics
    INSERT INTO @Stats
    SELECT 
        c.CustomerID,
        c.FirstName + ' ' + c.LastName,
        COUNT(o.OrderID),
        SUM(o.TotalAmount),
        AVG(o.TotalAmount),
        MAX(o.OrderDate),
        NULL -- CustomerTier filled in next step
    FROM Customers c
    LEFT JOIN Orders o ON c.CustomerID = o.CustomerID 
        AND YEAR(o.OrderDate) = @Year
    GROUP BY c.CustomerID, c.FirstName, c.LastName;
    
    -- Update customer tier based on spending
    UPDATE @Stats
    SET CustomerTier = CASE
        WHEN TotalSpent >= 10000 THEN 'VIP'
        WHEN TotalSpent >= 5000 THEN 'Gold'
        WHEN TotalSpent >= 1000 THEN 'Silver'
        WHEN TotalSpent > 0 THEN 'Bronze'
        ELSE 'No Orders'
    END;
    
    RETURN;
END;
GO

-- Usage
SELECT *
FROM dbo.fn_GetCustomerStatistics(2024)
WHERE CustomerTier IN ('VIP', 'Gold')
ORDER BY TotalSpent DESC;

Example 3: Complex Data Transformation

Generate Sales Report with Running Totals

CREATE FUNCTION dbo.fn_GetSalesReport
(
    @StartDate DATE,
    @EndDate DATE
)
RETURNS @Report TABLE
(
    OrderDate DATE,
    DailyOrders INT,
    DailySales DECIMAL(10,2),
    RunningTotal DECIMAL(10,2),
    MovingAverage DECIMAL(10,2)
)
AS
BEGIN
    -- Get daily sales
    INSERT INTO @Report (OrderDate, DailyOrders, DailySales)
    SELECT 
        CAST(OrderDate AS DATE),
        COUNT(*),
        SUM(TotalAmount)
    FROM Orders
    WHERE OrderDate >= @StartDate 
        AND OrderDate < DATEADD(DAY, 1, @EndDate)
    GROUP BY CAST(OrderDate AS DATE);
    
    -- Calculate running total
    DECLARE @CurrentTotal DECIMAL(10,2) = 0;
    DECLARE @CurrentDate DATE;
    
    DECLARE date_cursor CURSOR FOR
        SELECT OrderDate FROM @Report ORDER BY OrderDate;
    
    OPEN date_cursor;
    FETCH NEXT FROM date_cursor INTO @CurrentDate;
    
    WHILE @@FETCH_STATUS = 0
    BEGIN
        UPDATE @Report
        SET @CurrentTotal = RunningTotal = @CurrentTotal + DailySales
        WHERE OrderDate = @CurrentDate;
        
        FETCH NEXT FROM date_cursor INTO @CurrentDate;
    END
    
    CLOSE date_cursor;
    DEALLOCATE date_cursor;
    
    -- Calculate 7-day moving average
    UPDATE r1
    SET MovingAverage = (
        SELECT AVG(r2.DailySales)
        FROM @Report r2
        WHERE r2.OrderDate <= r1.OrderDate
            AND r2.OrderDate >= DATEADD(DAY, -6, r1.OrderDate)
    )
    FROM @Report r1;
    
    RETURN;
END;
GO

-- Usage
SELECT *
FROM dbo.fn_GetSalesReport('2024-01-01', '2024-01-31')
ORDER BY OrderDate;
⚠️ Performance Warning: Multi-statement TVFs often perform poorly because SQL Server cannot optimize them with the outer query. The table variable used internally can't use statistics. Consider rewriting as inline TVFs or stored procedures when possible.

Functions vs Stored Procedures

Understanding the differences between functions and procedures is crucial for choosing the right tool for each task.

Key Differences

Feature Functions Stored Procedures
Return Type Must return a value (scalar or table) Optional return status (integer only)
Use in Queries ✅ Can use in SELECT, WHERE, JOIN ❌ Cannot use in queries
DML Operations ❌ Cannot modify database state ✅ Can INSERT, UPDATE, DELETE
Transactions ❌ Cannot use BEGIN TRAN ✅ Full transaction support
Error Handling ❌ No TRY...CATCH support ✅ Full TRY...CATCH support
OUTPUT Parameters ❌ Not supported ✅ Multiple OUTPUT parameters
Temp Tables ❌ Cannot create temp tables ✅ Can create and use temp tables
EXEC Calls ❌ Cannot call procedures ✅ Can call other procedures
Side Effects Should be side-effect free Can have side effects
Purpose Calculations and data retrieval Actions and data modification

When to Use Functions

✅ Use Functions When:
  • You need to use the result in a SELECT statement
  • Performing calculations that don't modify data
  • Creating reusable filtering or transformation logic
  • Need to use results in WHERE or JOIN clauses
  • Creating computed columns or indexed views
  • Logic is purely read-only
  • Need to return a table that can be joined

When to Use Stored Procedures

✅ Use Stored Procedures When:
  • Performing INSERT, UPDATE, DELETE operations
  • Need transaction management
  • Require complex error handling
  • Need to call other procedures
  • Working with temp tables
  • Need multiple OUTPUT parameters
  • Performing administrative tasks
  • Logic modifies database state

Practical Example: Function vs Procedure

Same Task - Different Approaches

-- FUNCTION: Get total order amount (read-only)
CREATE FUNCTION dbo.fn_GetOrderTotal
(
    @OrderID INT
)
RETURNS DECIMAL(10,2)
AS
BEGIN
    DECLARE @Total DECIMAL(10,2);
    
    SELECT @Total = SUM(LineTotal)
    FROM OrderItems
    WHERE OrderID = @OrderID;
    
    RETURN ISNULL(@Total, 0);
END;
GO

-- Usage in query
SELECT 
    OrderID,
    CustomerID,
    dbo.fn_GetOrderTotal(OrderID) AS CalculatedTotal,
    TotalAmount AS StoredTotal
FROM Orders
WHERE dbo.fn_GetOrderTotal(OrderID) > 1000;

-- PROCEDURE: Update order total (modifies data)
CREATE PROCEDURE usp_RecalculateOrderTotal
    @OrderID INT,
    @NewTotal DECIMAL(10,2) OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Calculate new total
        SELECT @NewTotal = SUM(LineTotal)
        FROM OrderItems
        WHERE OrderID = @OrderID;
        
        SET @NewTotal = ISNULL(@NewTotal, 0);
        
        -- Update order
        UPDATE Orders
        SET TotalAmount = @NewTotal
        WHERE OrderID = @OrderID;
        
        COMMIT TRANSACTION;
        RETURN 0;
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;
GO

-- Usage
DECLARE @Total DECIMAL(10,2);
EXEC usp_RecalculateOrderTotal @OrderID = 1, @NewTotal = @Total OUTPUT;
SELECT @Total AS NewTotal;

Function Limitations

Functions have important restrictions that developers must understand to use them effectively.

Cannot Modify Database State

⚠️ Prohibited Operations in Functions:
  • INSERT, UPDATE, DELETE on permanent tables
  • Cannot call stored procedures
  • Cannot create or modify database objects (tables, views, etc.)
  • Cannot use EXEC to execute dynamic SQL (exception: with SCHEMABINDING)
  • Cannot use RAISERROR or THROW
  • Cannot use BEGIN TRANSACTION
  • Cannot create temp tables (#temp)
  • Cannot use cursors that UPDATE or DELETE

What Functions CAN Do

✅ Allowed Operations:
  • SELECT statements (read data)
  • Declare and use table variables (@table)
  • Use cursors for read-only operations
  • Call other functions
  • Perform calculations and transformations
  • Return scalar values or tables
  • Use all built-in functions

Error Handling Limitations

-- Functions cannot use TRY...CATCH
CREATE FUNCTION dbo.fn_SafeDivide
(
    @Numerator DECIMAL(10,2),
    @Denominator DECIMAL(10,2)
)
RETURNS DECIMAL(10,2)
AS
BEGIN
    DECLARE @Result DECIMAL(10,2);
    
    -- Must handle errors with conditional logic
    IF @Denominator = 0
        SET @Result = NULL; -- or 0, or some default
    ELSE
        SET @Result = @Numerator / @Denominator;
    
    RETURN @Result;
END;
GO

-- Usage
SELECT 
    ProductID,
    TotalSales,
    UnitsSold,
    dbo.fn_SafeDivide(TotalSales, UnitsSold) AS AveragePrice
FROM ProductSummary;

Deterministic vs Non-Deterministic

Type Description Examples Can Use in Indexes
Deterministic Always returns same result for same inputs Mathematical calculations, string operations ✅ Yes
Non-Deterministic May return different results for same inputs GETDATE(), NEWID(), RAND() ❌ No

Schema Binding

-- SCHEMABINDING prevents changes to underlying objects
CREATE FUNCTION dbo.fn_GetProductPrice
(
    @ProductID INT
)
RETURNS DECIMAL(10,2)
WITH SCHEMABINDING  -- Locks schema of referenced objects
AS
BEGIN
    DECLARE @Price DECIMAL(10,2);
    
    -- Must use two-part names with SCHEMABINDING
    SELECT @Price = UnitPrice
    FROM dbo.Products  -- Must include schema
    WHERE ProductID = @ProductID;
    
    RETURN @Price;
END;
GO

-- Now Products table cannot be dropped or altered
-- without first dropping the function
💡 Schema Binding Benefits:
  • Makes functions deterministic (if they otherwise would be)
  • Required for indexed views and persisted computed columns
  • Prevents accidental schema changes
  • Can improve performance

Using Functions in Queries

Functions can be integrated into SQL queries in various ways, offering flexibility in how you work with data.

Scalar Functions in SELECT

-- Use in column list
SELECT 
    CustomerID,
    FirstName,
    LastName,
    BirthDate,
    dbo.fn_CalculateAge(BirthDate) AS Age
FROM Customers;

-- Use in calculations
SELECT 
    OrderID,
    TotalAmount,
    dbo.fn_GetCustomerDiscount(CustomerID, TotalAmount) AS Discount,
    TotalAmount - dbo.fn_GetCustomerDiscount(CustomerID, TotalAmount) AS FinalAmount
FROM Orders;

Scalar Functions in WHERE

-- Filter using function results
SELECT CustomerID, FirstName, LastName, BirthDate
FROM Customers
WHERE dbo.fn_CalculateAge(BirthDate) >= 18
    AND dbo.fn_CalculateAge(BirthDate) < 65;

-- Complex filtering
SELECT *
FROM Orders
WHERE dbo.fn_GetCustomerDiscount(CustomerID, TotalAmount) > 50.00;

Table-Valued Functions with CROSS APPLY

-- CROSS APPLY: Returns results only when function returns rows
SELECT 
    c.CustomerID,
    c.FirstName,
    c.LastName,
    o.OrderID,
    o.OrderDate,
    o.TotalAmount
FROM Customers c
CROSS APPLY dbo.fn_GetCustomerOrdersByDate(
    c.CustomerID, 
    '2024-01-01', 
    '2024-12-31'
) o
WHERE c.CustomerID <= 10;

Table-Valued Functions with OUTER APPLY

-- OUTER APPLY: Returns all left rows even if function returns no rows
SELECT 
    c.CustomerID,
    c.FirstName,
    c.LastName,
    o.OrderID,
    o.OrderDate,
    o.TotalAmount
FROM Customers c
OUTER APPLY dbo.fn_GetCustomerOrdersByDate(
    c.CustomerID, 
    '2024-01-01', 
    '2024-12-31'
) o
WHERE c.CustomerID <= 10;

-- Shows customers with no orders as NULL values

Table-Valued Functions in JOIN

-- JOIN with table-valued function
SELECT 
    p.ProductID,
    p.ProductName,
    prod.StockStatus
FROM Products p
INNER JOIN dbo.fn_GetProductsByCategory(1, 0, 1) prod
    ON p.ProductID = prod.ProductID;

Functions in Computed Columns

-- Add computed column using function
ALTER TABLE Customers
ADD Age AS dbo.fn_CalculateAge(BirthDate);

-- Persisted computed column (stored physically)
ALTER TABLE Customers
ADD AgeGroup AS (
    CASE 
        WHEN dbo.fn_CalculateAge(BirthDate) < 18 THEN 'Minor'
        WHEN dbo.fn_CalculateAge(BirthDate) < 65 THEN 'Adult'

Functions in ORDER BY and GROUP BY

-- ORDER BY using function
SELECT FirstName, LastName, BirthDate
FROM Customers
ORDER BY dbo.fn_CalculateAge(BirthDate) DESC;

-- GROUP BY using function
SELECT 
    dbo.fn_CalculateAge(BirthDate) AS Age,
    COUNT(*) AS CustomerCount
FROM Customers

ORDER BY Age;

Performance Considerations

Understanding function performance characteristics is critical for building efficient database applications.

Performance by Function Type

Function Type Performance Why Recommendation
Inline TVF ⭐⭐⭐⭐⭐ Excellent Query optimizer treats like a view Preferred for table returns
Scalar UDF (simple) ⭐⭐⭐ Moderate Executes once per row OK for small datasets
Scalar UDF (complex) ⭐⭐ Poor Prevents parallelism, no optimization Avoid in WHERE/large datasets
Multi-Statement TVF ⭐⭐ Poor Table variable, no statistics Use only when necessary

Scalar Function Performance Issues

⚠️ Common Performance Problems:
  • Row-by-Row Execution: Function executes once for each row
  • Prevents Parallelism: Queries using UDFs can't use parallel execution
  • No Index Usage: Functions in WHERE clause prevent index seeks
  • Statistics Ignored: Optimizer can't estimate cardinality accurately
  • No Inlining: Function body not merged with outer query (pre-SQL 2019)

Performance Example: Scalar Function Problem

Slow Query with Scalar Function

-- SLOW: Function executes for EVERY row
SELECT 
    OrderID,
    CustomerID,
    TotalAmount,
    dbo.fn_GetCustomerDiscount(CustomerID, TotalAmount) AS Discount
FROM Orders  -- 1 million rows
WHERE dbo.fn_GetCustomerDiscount(CustomerID, TotalAmount) > 100;

-- Function executes 1 million times in SELECT
-- Function executes 1 million times in WHERE
-- Total: 2 million function calls!

-- BETTER: Calculate once using CROSS APPLY or CTE
WITH OrdersWithDiscount AS (
    SELECT 
        OrderID,
        CustomerID,
        TotalAmount,
        dbo.fn_GetCustomerDiscount(CustomerID, TotalAmount) AS Discount
    FROM Orders
)
SELECT *
FROM OrdersWithDiscount
WHERE Discount > 100;

-- Function executes 1 million times total (not 2 million)

Optimization Techniques

1. Convert Scalar to Inline TVF

-- BEFORE: Scalar function
CREATE FUNCTION dbo.fn_GetProductPrice_Scalar(@ProductID INT)
RETURNS DECIMAL(10,2)
AS
BEGIN
    RETURN (SELECT UnitPrice FROM Products WHERE ProductID = @ProductID);
END;

-- AFTER: Inline TVF (much faster)
CREATE FUNCTION dbo.fn_GetProductPrice_Inline(@ProductID INT)
RETURNS TABLE
AS
RETURN
(
    SELECT UnitPrice
    FROM Products
    WHERE ProductID = @ProductID
);

Advantages of Functions

✅ Key Advantages

  • Reusability: Write once, use in multiple queries and applications
  • Query Integration: Use directly in SELECT, WHERE, JOIN clauses
  • Code Encapsulation: Hide complex logic behind simple interfaces
  • Maintainability: Change logic in one place, affects all uses
  • Readability: Makes queries more readable with descriptive names
  • Parameterization: Create flexible, parameterized views
  • Business Logic: Encode business rules in database layer
  • Security: Grant EXECUTE without exposing table structure
  • Computed Columns: Can use in persisted computed columns
  • Indexed Views: Schema-bound deterministic functions work in indexed views
  • Consistency: Ensures same calculation logic everywhere

Specific Benefits by Type

Function Type Primary Benefits Best Use Cases
Scalar Simple calculations, data transformations Age calculations, formatting, conversions
Inline TVF Excellent performance, parameterized views Complex filtering, joining with parameters
Multi-Statement TVF Complex multi-step logic, hierarchies Hierarchical data, complex transformations

Disadvantages & When Not to Use

❌ Key Disadvantages

  • Performance: Scalar functions can severely degrade query performance
  • Cannot Modify Data: Read-only, can't INSERT/UPDATE/DELETE
  • No Error Handling: Can't use TRY...CATCH
  • No Transactions: Can't use BEGIN TRAN
  • Limited Debugging: Harder to debug than procedures
  • Table Variable Limitations: MSTVFs suffer from lack of statistics
  • Optimization Barriers: Prevent parallelism and optimization
  • Context Switches: Row-by-row execution causes overhead
  • No Temp Tables: Can't use #temp tables
  • Network Overhead: Complex functions increase execution time

When NOT to Use Functions

❌ Avoid Functions When:
  • You need to modify data (INSERT, UPDATE, DELETE)
  • You need transaction management
  • You need complex error handling with TRY...CATCH
  • Performance is critical and dataset is large
  • You need to call stored procedures
  • You need temp tables for intermediate results
  • You need multiple output values (use procedure with OUTPUT params)
  • Logic requires side effects or state changes
  • You need to execute dynamic SQL
  • Debugging and testing complex logic is important

Common Anti-Patterns

❌ Anti-Pattern: Scalar Function in WHERE on Large Table

-- BAD: Executes function for every row
SELECT * 
FROM Orders -- 10 million rows
WHERE dbo.fn_ComplexCalculation(OrderID, CustomerID) > 100;

-- BETTER: Use inline logic or indexed computed column
SELECT *
FROM Orders
WHERE ComplexCalculationColumn > 100; -- Persisted computed column

❌ Anti-Pattern: Using Function for Data Modification

-- BAD: Can't modify data in function
CREATE FUNCTION dbo.fn_UpdateInventory(@ProductID INT, @Qty INT)
RETURNS INT
AS
BEGIN
    -- This will fail - functions can't UPDATE
    UPDATE Products 
    SET StockQuantity = StockQuantity - @Qty
    WHERE ProductID = @ProductID;
    
    RETURN 0;
END;

-- CORRECT: Use stored procedure
CREATE PROCEDURE usp_UpdateInventory
    @ProductID INT,
    @Qty INT
AS
BEGIN
    UPDATE Products 
    SET StockQuantity = StockQuantity - @Qty
    WHERE ProductID = @ProductID;
END;

-- Instead of calling function in SELECT, use computed column
ALTER TABLE Customers
ADD Age AS (
    DATEDIFF(YEAR, BirthDate, GETDATE()) -
    CASE 
        WHEN MONTH(BirthDate) > MONTH(GETDATE()) THEN 1
        WHEN MONTH(BirthDate) = MONTH(GETDATE())

-- Instead of function, use CTE for one-time calculation
WITH CustomerDiscounts AS (
    SELECT 
        CustomerID,
        CASE
            WHEN TotalOrders >= 50 THEN 15.00
            WHEN TotalOrders >= 20 THEN 10.00
            WHEN TotalOrders >= 10 THEN 5.00

        SELECT 
            CustomerID,
            COUNT(*) AS TotalOrders
        FROM Orders
        GROUP BY CustomerID

    o.OrderID,
    o.TotalAmount,
    cd.DiscountPercent,
    o.TotalAmount * (cd.DiscountPercent / 100) AS Discount
FROM Orders o
INNER JOIN CustomerDiscounts cd ON o.CustomerID = cd.CustomerID;

Introduction

When working with relational databases, managing data insertion across tables with foreign key relationships is a common challenge. SQL Server's stored procedures provide a powerful way to encapsulate this logic, ensuring data integrity and simplifying application code.

The Foreign Key Challenge

Consider a typical scenario: inserting an order and its line items. The order must be inserted first to generate its primary key (OrderID), which is then used as a foreign key in the order line items table. Stored procedures can handle this entire workflow atomically and efficiently.

Why Identity Columns?

  • Automatic Generation: SQL Server automatically generates unique values
  • Performance: Optimized for high-volume inserts
  • Simplicity: No need for application-level ID generation
  • Consistency: Guaranteed uniqueness within the table
  • Concurrency: Handles multiple simultaneous inserts safely
💡 Key Concept: Identity columns automatically generate sequential numeric values, but you need special functions to retrieve the generated value after an INSERT statement.

Identity Columns Overview

Defining Identity Columns

-- Basic identity column definition
CREATE TABLE Orders (
    OrderID INT IDENTITY(1,1) PRIMARY KEY,
    OrderDate DATETIME NOT NULL,
    CustomerID INT NOT NULL
);

-- Identity with custom seed and increment
CREATE TABLE Products (
    ProductID INT IDENTITY(1000,1) PRIMARY KEY,  -- Starts at 1000
    ProductName NVARCHAR(100) NOT NULL
);

-- Identity with increment by 10
CREATE TABLE BatchJobs (
    BatchID INT IDENTITY(1,10) PRIMARY KEY,  -- 1, 11, 21, 31...
    BatchName NVARCHAR(100) NOT NULL
);

Identity Syntax

Syntax Description Example
IDENTITY(seed, increment) Seed is starting value, increment is step size IDENTITY(1,1) = 1,2,3,4...
IDENTITY(1000,1) Start at 1000, increment by 1 1000,1001,1002...
IDENTITY(1,10) Start at 1, increment by 10 1,11,21,31...
⚠️ Important: Identity values are not reused even if a transaction is rolled back. Gaps in identity sequences are normal and expected.

Retrieving Identity Values

SQL Server provides several methods to retrieve the most recently generated identity value. Choosing the right method is crucial for correctness in concurrent environments.

1. SCOPE_IDENTITY() - Recommended

-- Returns the last identity value inserted in the CURRENT scope
INSERT INTO Orders (OrderDate, CustomerID)
VALUES (GETDATE(), 100);

DECLARE @NewOrderID INT = SCOPE_IDENTITY();
SELECT @NewOrderID AS NewOrderID;

2. @@IDENTITY - Use with Caution

-- Returns the last identity value in the SESSION (any scope)
-- Can return incorrect value if triggers insert into other tables
INSERT INTO Orders (OrderDate, CustomerID)
VALUES (GETDATE(), 100);

DECLARE @NewOrderID INT = @@IDENTITY;
SELECT @NewOrderID AS NewOrderID;

3. IDENT_CURRENT() - Table-Specific

-- Returns the last identity value for a specific table across all sessions
-- NOT safe for concurrent inserts
SELECT IDENT_CURRENT('Orders') AS LastOrderID;

4. OUTPUT Clause - Most Flexible

-- Returns the inserted values directly
DECLARE @InsertedOrders TABLE (OrderID INT);

INSERT INTO Orders (OrderDate, CustomerID)
OUTPUT inserted.OrderID INTO @InsertedOrders
VALUES (GETDATE(), 100);

SELECT OrderID FROM @InsertedOrders;

Comparison of Methods

Method Scope Trigger-Safe Concurrent-Safe Best Use Case
SCOPE_IDENTITY() Current scope ✅ Yes ✅ Yes Single row inserts in stored procedures
@@IDENTITY Session (all scopes) ❌ No ✅ Yes Legacy code (not recommended)
IDENT_CURRENT() Table (all sessions) N/A ❌ No Monitoring/reporting only
OUTPUT Clause Statement ✅ Yes ✅ Yes Multiple row inserts, returning multiple columns
✅ Best Practice: Use SCOPE_IDENTITY() for single-row inserts and the OUTPUT clause for multiple-row inserts. Avoid @@IDENTITY and never use IDENT_CURRENT() for retrieving just-inserted IDs.

2. Insert with One-Level Foreign Key

Example: Insert Order for Existing Customer

CREATE PROCEDURE usp_InsertOrder
    @CustomerID INT,
    @TotalAmount DECIMAL(10,2),
    @Status NVARCHAR(20),
    @OrderID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    -- Validate customer exists
    IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
    BEGIN
        RAISERROR('Customer does not exist', 16, 1);
        RETURN -1;
    END
    
    -- Insert order
    INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
    VALUES (@CustomerID, GETDATE(), @TotalAmount, @Status);
    
    -- Get the generated OrderID
    SET @OrderID = SCOPE_IDENTITY();
    
    RETURN 0;
END;
GO

DECLARE @NewOrderID INT;
EXEC usp_InsertOrder

    @Status = 'Pending',

3. Insert with Multi-Level Foreign Keys

Example: Insert Order Item

CREATE PROCEDURE usp_InsertOrderItem
    @OrderID INT,
    @ProductID INT,
    @Quantity INT,
    @OrderItemID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @UnitPrice DECIMAL(10,2);
    
    -- Validate order exists
    IF NOT EXISTS (SELECT 1 FROM Orders WHERE OrderID = @OrderID)
    BEGIN
        RAISERROR('Order does not exist', 16, 1);
        RETURN -1;
    END
    
    -- Get product price and validate product exists
    SELECT @UnitPrice = UnitPrice
    FROM Products
    WHERE ProductID = @ProductID;
    
    IF @UnitPrice IS NULL
    BEGIN
        RAISERROR('Product does not exist', 16, 1);
        RETURN -1;
    END
    
    -- Insert order item
    INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
    VALUES (@OrderID, @ProductID, @Quantity, @UnitPrice);
    
    -- Get the generated OrderItemID
    SET @OrderItemID = SCOPE_IDENTITY();
    
    RETURN 0;
END;
GO

-- Usage example
DECLARE @NewOrderItemID INT;
EXEC usp_InsertOrderItem 
    @OrderID = 1,
    @ProductID = 5,
    @Quantity = 2,
    @OrderItemID = @NewOrderItemID OUTPUT;

        BEGIN TRANSACTION;

Transaction Management

Transactions ensure that all related inserts succeed or fail together, maintaining data integrity across foreign key relationships.

Transaction Best Practices

✅ Transaction Guidelines:
  • Always use explicit transactions when inserting related data
  • Keep transactions as short as possible
  • Handle errors with TRY...CATCH blocks
  • Always check @@TRANCOUNT before ROLLBACK
  • Use appropriate isolation levels
  • Set XACT_ABORT ON for automatic rollback on errors

Comprehensive Transaction Example

Example: Order with Line Items (Full Transaction)

CREATE PROCEDURE usp_CreateOrderWithItems
    @CustomerID INT,
    @Status NVARCHAR(20),
    @OrderID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT ON; -- Auto-rollback on errors
    
    DECLARE @TotalAmount DECIMAL(10,2) = 0;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Validate customer
        IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
        BEGIN
            RAISERROR('Customer ID %d does not exist', 16, 1, @CustomerID);
            RETURN -1;
        END
        
        -- Step 1: Create order (with temporary total)
        INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
        VALUES (@CustomerID, GETDATE(), 0, @Status);
        
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Step 2: Add order items (example: products 1, 2, 3)
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
        SELECT 
            @OrderID,
            p.ProductID,
            CASE p.ProductID 
                WHEN 1 THEN 2
                WHEN 2 THEN 1
                WHEN 3 THEN 3
            END AS Quantity,
            p.UnitPrice
        FROM Products p
        WHERE p.ProductID IN (1, 2, 3);
        
        -- Step 3: Calculate and update order total
        SELECT @TotalAmount = SUM(LineTotal)
        FROM OrderItems
        WHERE OrderID = @OrderID;
        
        UPDATE Orders
        SET TotalAmount = @TotalAmount
        WHERE OrderID = @OrderID;
        
        COMMIT TRANSACTION;
        
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        -- Rollback if transaction is active
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        -- Log error details
        DECLARE @ErrorMessage NVARCHAR(4000);
        DECLARE @ErrorSeverity INT;
        DECLARE @ErrorState INT;
        
        SELECT 
            @ErrorMessage = ERROR_MESSAGE(),
            @ErrorSeverity = ERROR_SEVERITY(),
            @ErrorState = ERROR_STATE();
        
        -- Re-throw the error
        RAISERROR(@ErrorMessage, @ErrorSeverity, @ErrorState);
        
        RETURN -1;
    END CATCH
END;
GO

Error Handling

Robust error handling is crucial for stored procedures that manage complex foreign key relationships.

Comprehensive Error Handling Pattern

Example: Complete Error Handling

CREATE PROCEDURE usp_InsertOrderWithErrorHandling
    @CustomerID INT,
    @ProductID INT,
    @Quantity INT,
    @OrderID INT OUTPUT,
    @ErrorMessage NVARCHAR(4000) OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT OFF; -- We'll handle errors explicitly
    
    DECLARE @UnitPrice DECIMAL(10,2);
    DECLARE @TotalAmount DECIMAL(10,2);
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Validation 1: Customer exists
        IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
        BEGIN
            SET @ErrorMessage = 'Customer ID ' + CAST(@CustomerID AS NVARCHAR) + ' not found';
            RAISERROR(@ErrorMessage, 16, 1);
        END
        
        -- Validation 2: Product exists and get price
        SELECT @UnitPrice = UnitPrice
        FROM Products
        WHERE ProductID = @ProductID;
        
        IF @UnitPrice IS NULL
        BEGIN
            SET @ErrorMessage = 'Product ID ' + CAST(@ProductID AS NVARCHAR) + ' not found';
            RAISERROR(@ErrorMessage, 16, 1);
        END
        
        -- Validation 3: Quantity is positive
        IF @Quantity <= 0
        BEGIN
            SET @ErrorMessage = 'Quantity must be greater than zero';
            RAISERROR(@ErrorMessage, 16, 1);
        END
        
        -- Validation 4: Check stock availability
        IF EXISTS (
            SELECT 1 FROM Products 
            WHERE ProductID = @ProductID 
            AND StockQuantity < @Quantity
        )
        BEGIN
            SET @ErrorMessage = 'Insufficient stock for Product ID ' + 
                              CAST(@ProductID AS NVARCHAR);
            RAISERROR(@ErrorMessage, 16, 1);
        END
        
        -- Calculate total
        SET @TotalAmount = @UnitPrice * @Quantity;
        
        -- Insert order
        INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
        VALUES (@CustomerID, GETDATE(), @TotalAmount, 'Pending');
        
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Insert order item
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
        VALUES (@OrderID, @ProductID, @Quantity, @UnitPrice);
        
        -- Update stock
        UPDATE Products
        SET StockQuantity = StockQuantity - @Quantity
        WHERE ProductID = @ProductID;
        
        COMMIT TRANSACTION;
        
        SET @ErrorMessage = NULL; -- Success
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        -- Capture comprehensive error information
        SET @ErrorMessage = 
            'Error ' + CAST(ERROR_NUMBER() AS NVARCHAR) + ': ' +
            ERROR_MESSAGE() + 
            ' (Line ' + CAST(ERROR_LINE() AS NVARCHAR) + ')';
        
        SET @OrderID = NULL;
        RETURN -1;
    END CATCH
END;
GO

CREATE PROCEDURE usp_CreateOrderWithItemsTVP
    @CustomerID INT,
    @Status NVARCHAR(20),
    @OrderItems OrderItemType READONLY, -- TVP parameter
    @OrderID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @TotalAmount DECIMAL(10,2);
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Validate customer exists
        IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
        BEGIN
            RAISERROR('Customer does not exist', 16, 1);
        END
        
        -- Validate all products exist
        IF EXISTS (
            SELECT 1
            FROM @OrderItems oi
            LEFT JOIN Products p ON oi.ProductID = p.ProductID
            WHERE p.ProductID IS NULL
        )
        BEGIN
            RAISERROR('One or more products do not exist', 16, 1);
        END
        
        -- Calculate total amount
        SELECT @TotalAmount = SUM(oi.Quantity * p.UnitPrice)
        FROM @OrderItems oi
        INNER JOIN Products p ON oi.ProductID = p.ProductID;
        
        -- Create order
        INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
        VALUES (@CustomerID, GETDATE(), @TotalAmount, @Status);
        
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Insert all order items
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
        SELECT 
            @OrderID,
            oi.ProductID,
            oi.Quantity,
            p.UnitPrice
        FROM @OrderItems oi
        INNER JOIN Products p ON oi.ProductID = p.ProductID;
        
        COMMIT TRANSACTION;
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;
GO

Advantages of Stored Procedures

✅ Advantages

  • Data Integrity: Encapsulate complex foreign key logic, ensuring consistency
  • Performance: Compiled execution plans, reduced network traffic
  • Security: Parameterized queries prevent SQL injection
  • Maintainability: Centralized business logic, easier to update
  • Reusability: Call from multiple applications
  • Atomicity: Built-in transaction support
  • Abstraction: Hide database complexity from application
  • Version Control: Database changes through migrations
  • Testing: Can unit test database logic

❌ Considerations

  • Vendor Lock-in: T-SQL is SQL Server specific
  • Version Control: Requires database migration tools
  • Development: Requires SQL expertise
  • Testing: Need database for integration tests
  • Debugging: Limited debugging tools compared to application code
  • Deployment: Database deployments can be complex
  • Flexibility: Less flexible than ORM dynamic queries

Best Practices

1. Naming Conventions

✅ Recommended Naming:
  • Stored Procedures: usp_VerbNoun (e.g., usp_InsertOrder, usp_GetCustomerOrders)
  • Parameters: @PascalCase (e.g., @CustomerID, @OrderDate)
  • Table Types: NounType (e.g., OrderItemType, AddressType)
  • Local Variables: @PascalCase with descriptive names

2. Performance Optimization

✅ Performance Tips:
  • Always use SET NOCOUNT ON at procedure start
  • Use SCOPE_IDENTITY() instead of @@IDENTITY
  • Minimize transaction scope - only wrap necessary operations
  • Use table-valued parameters for batch inserts
  • Consider using OUTPUT clause instead of separate SELECT statements
  • Index foreign key columns for better join performance
  • Use appropriate isolation levels (READ COMMITTED is usually sufficient)

3. Error Handling Standards

✅ Error Handling Guidelines:
  • Always use TRY...CATCH blocks for error handling
  • Check @@TRANCOUNT before ROLLBACK
  • Use meaningful error messages with RAISERROR or THROW
  • Return status codes (0 for success, negative for errors)
  • Consider OUTPUT parameters for error messages
  • Log errors to an error table for diagnostics
  • Validate inputs before beginning transactions

4. Security Best Practices

✅ Security Recommendations:
  • Grant EXECUTE permissions on procedures, not direct table access
  • Use parameterized procedures to prevent SQL injection
  • Validate all input parameters
  • Use separate service accounts with minimal permissions
  • Avoid dynamic SQL when possible; if needed, use sp_executesql
  • Don't expose sensitive data in error messages
  • Audit procedure executions for sensitive operations
