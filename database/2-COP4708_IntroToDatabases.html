What Is a Database?

🗄️

Outcome 1Definition & Core Concept

Definition

A database is an organized, structured collection of related data stored electronically so that it can be easily accessed, managed, and updated by authorized users and applications.

Think of a database as a highly organized digital filing system. Unlike a simple spreadsheet or a pile of documents, a database is designed with rules, structure, and relationships so that data can be retrieved quickly and reliably—even when millions of records are stored.

Everyday Analogies

📚
Library Card Catalog Every book has a record: title, author, call number, availability. You can look up by any field. A database works the same way—structured records you can query.
🏦
Bank Records Your account, transactions, and balances are stored in a database. When you check your balance online, a query is sent to retrieve your specific records instantly.
📦

Outcome 1 continuedWhat Is Stored Inside a Database?

A modern database contains much more than raw rows of data. The following components work together to store and organize information reliably.

Core Objects Found in a Database

Tables

The primary structure. Each table stores data about a single subject (e.g., Customers, Orders). Organized into rows (records) and columns (fields).

Rows / Records

A single entry in a table. One row = one instance of the subject (e.g., one customer, one order).

Columns / Fields

An attribute of the subject. Each column has a name and a defined data type (e.g., CustomerName TEXT, OrderDate DATE).

Keys

Special columns that uniquely identify rows (Primary Key) or link tables together (Foreign Key).

Indexes

Structures that speed up data retrieval—similar to the index in the back of a book.

Views

Virtual tables built from stored queries. Simplify complex data access without duplicating data.

Stored Procedures

Pre-written blocks of SQL code stored in the database and executed on demand.

Constraints

Rules that enforce data integrity (e.g., NOT NULL, UNIQUE, CHECK, FOREIGN KEY).

Triggers

Automated actions that execute in response to INSERT, UPDATE, or DELETE events on a table.

ℹ️
Data vs. a Spreadsheet A spreadsheet stores data but lacks built-in relationships between sheets, strict data types, multi-user access controls, or transaction support. Databases are engineered to handle all of these at scale.

Metadata

🏷️

Outcome 2Definition, Purpose & Examples

Definition

Metadata is "data about data." It describes the structure, format, constraints, and context of the actual data stored in a database—without being the data itself.

If you think of the database as a library, the books are your data. Metadata is everything on the card catalog entry: author, genre, page count, language, shelf location. It tells you about the book, not what's inside it.

Where Metadata Lives in a Database

In most relational database systems (like Microsoft SQL Server, MySQL, or PostgreSQL), metadata is stored in a special area called the data dictionary or system catalog. This is queried automatically whenever you or the DBMS needs to understand the structure of the data.

Categories of Database Metadata with Examples

Category Example What It Describes
Table Metadata Table name: Orders The name and existence of a table in the database
Column Metadata Column: OrderDate, Type: DATE Column name, data type, and position in the table
Constraint Metadata OrderID is NOT NULL and PRIMARY KEY Rules that enforce data integrity on columns
Relationship Metadata Orders.CustomerID → Customers.CustomerID Foreign key links between tables
Index Metadata Index idx_last_name on Customers(LastName) Which columns are indexed for faster lookups
Size/Storage Metadata VARCHAR(100) — max 100 characters Storage allocation and field size limits
Audit Metadata CreatedAt: 2024-09-01 08:32:17 When a record was created or last modified
Permission Metadata User jsmith has SELECT on Employees Access rights and security settings

Metadata in Action — SQL Example

In SQL Server you can query the system catalog to see metadata about your tables:

-- Query metadata: see all columns in the 'Customers' table SELECT COLUMN_NAME, DATA_TYPE, CHARACTER_MAXIMUM_LENGTH, IS_NULLABLE FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_NAME = 'Customers' ORDER BY ORDINAL_POSITION;

This query returns metadata—information describing the columns in the Customers table, not the customer records themselves.

🔑
Key Takeaway Metadata is essential for database management. It tells the DBMS how to store, validate, retrieve, and secure your data. Without metadata, the database has no way to interpret raw bytes.

The Relational Database

🔗

Outcome 3Purpose & Structure of Relational Databases

Definition

A relational database organizes data into two-dimensional tables (relations) and uses defined relationships between those tables to eliminate redundancy and maintain data integrity. It is based on E.F. Codd's Relational Model (1970).

The Core Purpose

The relational model was created to solve two major problems in early data storage:

  • Data Redundancy — storing the same data in multiple places, leading to inconsistencies when updates occur
  • Data Anomalies — update, insertion, and deletion errors that arise from poor data organization

How Relationships Work: Keys

🔑
Primary Key (PK) A column (or combination) whose value uniquely identifies each row in a table. No two rows can share the same PK value, and it cannot be NULL.

Example: CustomerID in the Customers table.
🔗
Foreign Key (FK) A column in one table that references the Primary Key of another table. This link is the "relation" in relational database.

Example: Orders.CustomerID references Customers.CustomerID.

Key Benefits of the Relational Model

BenefitExplanation
Reduces RedundancyData is stored once and referenced, not duplicated across many records
Enforces IntegrityForeign key constraints prevent orphaned records (e.g., an order for a non-existent customer)
Enables Complex QueriesSQL JOIN operations let you combine data from multiple tables on demand
Scales CleanlyAdding new related entities only requires new tables and key references
Standardized LanguageSQL is the universal language for all relational databases
⚠️
Relational vs. Non-Relational (NoSQL) Not all databases are relational. NoSQL databases (like MongoDB or Cassandra) store data as documents, key-value pairs, or graphs. They trade strict structure for flexibility and horizontal scalability. The relational model remains the dominant paradigm for structured business data.

Data, Information & Knowledge

🧠

Outcome 4The DIKW Hierarchy: From Raw Facts to Wisdom

Databases store data—but the ultimate goal is to enable information and knowledge. Understanding the difference between these three concepts (and wisdom) is fundamental to database design and use.

The DIKW Model

The DIKW Pyramid (Data → Information → Knowledge → Wisdom) describes how raw facts are transformed into insight and action.

🌟 Wisdom
Wisdom: Knowing when and why to act. "Our store's revenue dropped 20% in October. We should run a promotion before the holiday season." — Judgment built on experience and knowledge.
💡 Knowledge
Knowledge: Patterns and insights drawn from information. "October sales are consistently lower than November. Holiday promotions consistently boost revenue by 30%." — Understanding why and predicting outcomes.
📊 Information
Information: Data that has been processed and given context. "Total sales in October 2024 = $42,000. Total sales in September 2024 = $52,000. October was 19% lower." — Data with meaning.
🔢 Data
Data: Raw, unprocessed facts and figures. 42000, 52000, 10-2024, 09-2024 — Numbers with no context, no interpretation, no meaning on their own.

Through queries, reports, and analysis, that data becomes information and eventually knowledge that drives business decisions. A poorly designed database produces unreliable information and, consequently, flawed decisions.

The Importance of Managing the Database

⚙️

Outcome 5Why Database Management Matters

A database without proper management quickly becomes unreliable, insecure, and unusable. Database management is the ongoing process of maintaining, securing, tuning, and evolving a database to ensure it continues to serve its users accurately and efficiently.

Core Pillars of Database Management

🛡️ Data Integrity

Ensuring data is accurate, consistent, and trustworthy. Enforced through constraints, validation rules, and transactions (ACID properties).

🔒 Security

Controlling who can read, write, or modify data. Includes user roles, permissions, encryption, and auditing access logs.

💾 Backup & Recovery

Protecting against data loss from hardware failure, human error, or disasters. Regular backups and tested recovery plans are essential.

⚡ Performance

Monitoring and tuning query speed, index efficiency, and resource usage so the database responds quickly as it grows.

👥 Concurrency

Allowing many users to read/write simultaneously without conflict. Handled through transactions, locking, and isolation levels.

📐 Schema Evolution

Safely adding new tables, columns, and relationships as business needs change—without breaking existing data or applications.

ACID Properties — The Foundation of Data Reliability

Every transaction in a well-managed relational database must satisfy the four ACID properties:

PropertyMeaningExample
Atomicity A transaction is all-or-nothing. If any step fails, the entire transaction is rolled back. Bank transfer: debit and credit must both succeed—or neither happens.
Consistency Every transaction brings the database from one valid state to another. No rules are broken. After any transaction, all foreign key and constraint rules still hold.
Isolation Concurrent transactions are executed as if they were sequential. One transaction's changes are invisible to others until committed. Two users booking the last available seat both see it available—only one succeeds.
Durability Once committed, a transaction's changes persist even if the system crashes immediately after. Your submitted order is saved even if the server crashes one second later.

What Happens Without Good Management?

⚠️
Real Consequences of Poor Database Management
  • Data Loss — No backup means a hardware failure loses everything permanently
  • Security Breaches — Unmanaged permissions can expose sensitive data to unauthorized users
  • Performance Degradation — Unindexed tables slow to a crawl as millions of records accumulate
  • Corrupt Data — Missing integrity constraints allow logically impossible values to be stored
  • Downtime — Poor capacity planning can cause system failures during peak usage
  • Compliance Violations — Regulations like HIPAA, PCI-DSS, and GDPR require specific data handling; non-compliance has legal consequences

The Role of the DBA (Database Administrator)

A Database Administrator (DBA) is the professional responsible for managing the database environment. Their responsibilities span the entire lifecycle of the database:

  • Installing, configuring, and upgrading the DBMS software
  • Designing and maintaining the database schema
  • Implementing and testing backup and recovery procedures
  • Managing user accounts, roles, and permissions
  • Monitoring performance and applying tuning recommendations
  • Ensuring compliance with data governance and security policies

Commonly Used Database Management Systems

🖥️

Outcome 6What Is a DBMS & Major Platforms

Definition

A Database Management System (DBMS) is the software layer that sits between users/applications and the database itself. It handles creating, reading, updating, deleting, securing, and maintaining data on behalf of the users.

The DBMS is what you actually install and interact with. When we say "SQL Server" or "MySQL," we are referring to a specific DBMS product. Without a DBMS, raw database files are inaccessible and meaningless.

Functions Performed by a DBMS

  • Interpreting SQL queries and returning results
  • Enforcing constraints and referential integrity
  • Managing concurrent multi-user access with locking and transactions
  • Maintaining the data dictionary (metadata storage)
  • Providing security controls (authentication, authorization)
  • Performing backup, recovery, and replication operations
  • Optimizing query execution plans for performance
