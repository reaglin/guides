DDL Overview — The Five Commands

Foundation
🗂️

Foundation · SQL Sub-languagesWhat DDL Is and Where It Fits in SQL

Definition — Data Definition Language (DDL)

DDL is the subset of SQL used to define, modify, and remove database structures — tables, columns, constraints, indexes, views, and schemas. DDL commands change the shape of the database, not its data. In SQL Server, DDL statements are auto-committed — they take effect immediately and cannot be rolled back once the statement completes.

The Four SQL Sub-Languages

Sub-LanguagePurposeKey Commands
DDL — Data DefinitionDefine and modify structuresCREATE, ALTER, DROP, TRUNCATE
DML — Data ManipulationInsert, update, delete rowsINSERT, UPDATE, DELETE
DQL — Data QueryRetrieve data from tablesSELECT
DCL — Data ControlManage access permissionsGRANT, REVOKE, DENY

The Five DDL Commands

CREATE

Builds a new object — table, view, index, schema, database, or stored procedure.

ALTER

Modifies an existing object — adds, changes, or removes columns and constraints.

DROP

Permanently removes a database object and all its data. Cannot be undone without a backup.

TRUNCATE

Removes all rows instantly without logging individual deletes. Faster than DELETE.

RENAME

Renames an object. In SQL Server, use the sp_rename system stored procedure.

⚠️
DDL Is Auto-Committed — No Undo In SQL Server, DDL statements commit immediately. Unlike DML inside a transaction, a DROP TABLE or ALTER TABLE that goes wrong cannot be rolled back once it completes. Always test DDL on a copy of the database and take a full backup before applying structural changes to production.

CREATE TABLE — Full Syntax & Working Schema

Outcome 1
🏗️

Outcome 1 · SyntaxCREATE TABLE — Annotated Syntax and Column Options

Full Syntax Template

T-SQL · CREATE TABLE — Full Syntax
CREATE TABLE SchemaName.TableName
(
    -- Column definitions
    ColumnName  DataType[(size)]
                [IDENTITY(seed, increment)]
                [NULL | NOT NULL]
                [CONSTRAINT Name DEFAULT value]
                [CONSTRAINT Name CHECK (expression)],

    -- Table-level constraints (after all columns)
    CONSTRAINT Name PRIMARY KEY (col1, col2),
    CONSTRAINT Name UNIQUE (col),
    CONSTRAINT Name FOREIGN KEY (col)
        REFERENCES ParentTable(ParentCol)
        [ON DELETE {NO ACTION | CASCADE | SET NULL | SET DEFAULT}]
        [ON UPDATE {NO ACTION | CASCADE | SET NULL | SET DEFAULT}],
    CONSTRAINT Name CHECK (expression)
);
💡
Always Name Your Constraints Explicitly Never rely on SQL Server's auto-generated constraint names like PK__Student__3A812345. They are unreadable and vary between environments. Use the naming pattern: PK_TableName, FK_Child_Parent, UQ_Table_Column, CK_Table_Column, DF_Table_Column.

Inline vs. Table-Level Constraint Placement

PlacementLocation in StatementRequired ForWorks For
InlineSame line as the columnNOT NULL (always inline)Single-column PK, UNIQUE, CHECK, DEFAULT
Table-levelAfter all column definitionsComposite (multi-column) PKs and FKsAll constraint types

Outcome 1 · Full ExampleComplete Student Registration Schema — Six Tables in Dependency Order

Tables must be created in dependency order — parent tables before the child tables that reference them. Attempting to create a table with a FK to a table that does not yet exist will fail with a reference error.

T-SQL · Table 1 of 6 — Department (no dependencies)
-- ═══════════════════════════════════════════════════════════
-- STUDENT REGISTRATION SCHEMA
-- Create tables in dependency order: parents before children
-- ═══════════════════════════════════════════════════════════

-- 1. Department — no FK dependencies, created first
CREATE TABLE Department
(
    DeptID      CHAR(5)      NOT NULL,
    DeptName    VARCHAR(80)  NOT NULL,
    Building    VARCHAR(60)  NULL,
    Phone       VARCHAR(15)  NULL,

    CONSTRAINT PK_Department   PRIMARY KEY (DeptID),
    CONSTRAINT UQ_Dept_Name    UNIQUE      (DeptName)
);
T-SQL · Tables 2 & 3 — Instructor & Course
-- 2. Instructor — depends on Department
CREATE TABLE Instructor
(
    InstructorID  INT          IDENTITY(1,1)  NOT NULL,
    DeptID        CHAR(5)                    NOT NULL,
    LastName      VARCHAR(60)                NOT NULL,
    FirstName     VARCHAR(50)                NOT NULL,
    Email         VARCHAR(100)               NOT NULL,
    Rank          VARCHAR(30)                NOT NULL,
    HireDate      DATE                        NOT NULL
                      CONSTRAINT DF_Inst_HireDate DEFAULT GETDATE(),

    CONSTRAINT PK_Instructor        PRIMARY KEY (InstructorID),
    CONSTRAINT UQ_Instructor_Email  UNIQUE      (Email),
    CONSTRAINT CK_Instructor_Rank
        CHECK (Rank IN ('Adjunct', 'Assistant', 'Associate', 'Professor')),
    CONSTRAINT FK_Instructor_Dept
        FOREIGN KEY (DeptID)
        REFERENCES Department (DeptID)
        ON DELETE NO ACTION
        ON UPDATE CASCADE
);

-- 3. Course — depends on Department
CREATE TABLE Course
(
    CourseID     CHAR(10)     NOT NULL,  -- natural key e.g. 'COP4708'
    DeptID       CHAR(5)      NOT NULL,
    CourseName   VARCHAR(100) NOT NULL,
    Credits      TINYINT      NOT NULL,
    Description  VARCHAR(500) NULL,

    CONSTRAINT PK_Course         PRIMARY KEY (CourseID),
    CONSTRAINT CK_Course_Credits CHECK       (Credits BETWEEN 1 AND 6),
    CONSTRAINT FK_Course_Dept
        FOREIGN KEY (DeptID) REFERENCES Department (DeptID)
        ON DELETE NO ACTION
);
T-SQL · Tables 4, 5 & 6 — Section, Student, Enrollment
-- 4. Section — depends on Course and Instructor
CREATE TABLE Section
(
    SectionID      INT          IDENTITY(1,1)  NOT NULL,
    CourseID       CHAR(10)                   NOT NULL,
    InstructorID   INT                          NOT NULL,
    Semester       VARCHAR(10)                 NOT NULL,
    YearOffered    SMALLINT                     NOT NULL,
    Room           VARCHAR(20)                 NULL,
    MaxEnrollment  TINYINT                      NOT NULL,

    CONSTRAINT PK_Section          PRIMARY KEY (SectionID),
    CONSTRAINT CK_Section_Sem      CHECK (Semester IN ('Spring','Summer','Fall')),
    CONSTRAINT CK_Section_Year     CHECK (YearOffered BETWEEN 2000 AND 2100),
    CONSTRAINT CK_Section_MaxEnr   CHECK (MaxEnrollment BETWEEN 1 AND 200),
    CONSTRAINT FK_Section_Course
        FOREIGN KEY (CourseID) REFERENCES Course (CourseID) ON DELETE NO ACTION,
    CONSTRAINT FK_Section_Instructor
        FOREIGN KEY (InstructorID) REFERENCES Instructor (InstructorID) ON DELETE NO ACTION
);

-- 5. Student
CREATE TABLE Student
(
    StudentID      INT          IDENTITY(1,1)  NOT NULL,
    LastName       VARCHAR(60)                NOT NULL,
    FirstName      VARCHAR(50)                NOT NULL,
    Email          VARCHAR(100)               NOT NULL,
    DateOfBirth    DATE                        NULL,
    DeclaredMajor  CHAR(10)                   NULL,
    IsEnrolled     BIT                         NOT NULL
                       CONSTRAINT DF_Student_IsEnrolled  DEFAULT 1,
    CreatedDate    DATETIME2                   NOT NULL
                       CONSTRAINT DF_Student_Created     DEFAULT GETDATE(),

    CONSTRAINT PK_Student        PRIMARY KEY (StudentID),
    CONSTRAINT UQ_Student_Email  UNIQUE      (Email)
);

-- 6. Enrollment — junction table resolving M:N; depends on Student and Section
--    Created last because it references both parent tables
CREATE TABLE Enrollment
(
    StudentID   INT      NOT NULL,
    SectionID   INT      NOT NULL,
    EnrollDate  DATE     NOT NULL
                    CONSTRAINT DF_Enroll_Date DEFAULT GETDATE(),
    Grade       CHAR(2) NULL,

    CONSTRAINT PK_Enrollment
        PRIMARY KEY (StudentID, SectionID),      -- composite PK
    CONSTRAINT CK_Enrollment_Grade
        CHECK (Grade IS NULL OR
               Grade IN ('A','A-','B+','B','B-','C+','C','C-','D','F','W','I')),
    CONSTRAINT FK_Enrollment_Student
        FOREIGN KEY (StudentID)
        REFERENCES Student (StudentID)
        ON DELETE CASCADE,          -- delete enrollments when student is removed
    CONSTRAINT FK_Enrollment_Section
        FOREIGN KEY (SectionID)
        REFERENCES Section (SectionID)
        ON DELETE NO ACTION         -- cannot drop a section with active enrollments
);

All SQL Constraint Types — Complete Reference

Outcome 3
🔒

Outcome 3 · ConstraintsPRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, CHECK, DEFAULT

PRIMARY KEY

Uniquely identifies every row. Implicitly NOT NULL. Creates a clustered index by default. Each table may have only one primary key, but it can span multiple columns (composite).

T-SQL · PRIMARY KEY
-- Simple PK — single column, table-level (recommended)
CONSTRAINT PK_Student  PRIMARY KEY (StudentID)

-- Composite PK — MUST be table-level (cannot be inline)
CONSTRAINT PK_Enrollment  PRIMARY KEY (StudentID, SectionID)

-- Inline shorthand — auto-named, NOT recommended for production
StudentID  INT  IDENTITY(1,1)  PRIMARY KEY

UNIQUE

Ensures all non-NULL values in a column (or column combination) are distinct. Implements alternate keys. SQL Server allows one NULL per unique column.

T-SQL · UNIQUE
-- Single-column unique — alternate key
CONSTRAINT UQ_Student_Email   UNIQUE (Email)

-- Multi-column unique — the combination must be unique
CONSTRAINT UQ_Section_Offering
    UNIQUE (CourseID, Semester, YearOffered)
-- Prevents scheduling the same course twice in the same semester and year

NOT NULL

Prevents a column from holding a NULL value. Always written inline on the column definition — there is no table-level NOT NULL syntax. Implements mandatory (required) attributes from the data model.

T-SQL · NOT NULL vs NULL
LastName    VARCHAR(60)  NOT NULL,  -- required: INSERT fails if value not supplied
MiddleName  VARCHAR(40)  NULL,     -- optional: NULL stored if omitted
Phone       VARCHAR(20),           -- NULL is the default when neither specified

CHECK

Validates column values against a logical expression before storing. Any INSERT or UPDATE failing the expression is rejected. NULL values pass a CHECK constraint (UNKNOWN evaluates as passing).

T-SQL · CHECK — Common Expression Patterns
-- Range check
CONSTRAINT CK_Course_Credits
    CHECK (Credits BETWEEN 1 AND 6)

-- Allowed-values list
CONSTRAINT CK_Instructor_Rank
    CHECK (Rank IN ('Adjunct', 'Assistant', 'Associate', 'Professor'))

-- Positive value
CONSTRAINT CK_OrderLine_Qty
    CHECK (Quantity > 0)

-- String length check
CONSTRAINT CK_Dept_ID
    CHECK (LEN(DeptID) = 5)

-- Cross-column date logic (must be table-level)
CONSTRAINT CK_Project_Dates
    CHECK (EndDate > StartDate)

-- NULL-aware check: NULL is acceptable until grading is complete
CONSTRAINT CK_Enrollment_Grade
    CHECK (Grade IS NULL OR
           Grade IN ('A','A-','B+','B','B-','C+','C','C-','D','F','W','I'))

DEFAULT

Automatically provides a value when an INSERT omits the column. Does not prevent NULL if NULL is explicitly supplied. Can use literal values or server functions.

T-SQL · DEFAULT — Literals and Functions
-- Literal defaults
IsActive    BIT        NOT NULL  CONSTRAINT DF_Stu_Active  DEFAULT 1,
Status      VARCHAR(20) NOT NULL CONSTRAINT DF_Ord_Status  DEFAULT 'Pending',
Quantity    SMALLINT   NOT NULL  CONSTRAINT DF_Line_Qty    DEFAULT 1,

-- Function defaults — evaluated by the server at INSERT time
CreatedAt   DATETIME2  NOT NULL  CONSTRAINT DF_Stu_Created DEFAULT GETDATE(),
RecordGUID  UNIQUEIDENTIFIER NOT NULL CONSTRAINT DF_GUID  DEFAULT NEWID(),

Constraint Behavior Summary

ConstraintEnforced OnNULL BehaviorIndex Created?
PRIMARY KEYINSERT, UPDATENULLs never allowed✅ Clustered (default)
FOREIGN KEYINSERT, UPDATE, DELETE*NULL FK = no parent required⚠️ Not automatic — add manually
NOT NULLINSERT, UPDATERejects NULLNo
UNIQUEINSERT, UPDATEOne NULL allowed per column✅ Non-clustered
CHECKINSERT, UPDATENULL passes (UNKNOWN = pass)No
DEFAULTINSERT onlyFills value; NULL still insertable explicitlyNo
ℹ️
FK Index Best Practice SQL Server does not automatically create an index on a FK column. Always add a non-clustered index manually on FK columns — join performance on unindexed FK columns degrades sharply as table size grows: CREATE INDEX IX_Order_CustomerID ON [Order] (CustomerID);

Referential Integrity Actions in SQL

Outcome 2
🔗

Outcome 2 · ON DELETE / ON UPDATEAll Four Referential Actions with Behavior and Use Cases

Every FOREIGN KEY constraint must specify what happens to child rows when the parent row is deleted or when the parent's PK is updated. If unspecified, SQL Server defaults to NO ACTION.

T-SQL · ON DELETE NO ACTION (Default)
-- NO ACTION: blocks the DELETE if any child rows reference the parent.
-- The parent row cannot be deleted while children exist.
-- Child rows must be manually removed first.

CONSTRAINT FK_Order_Customer
    FOREIGN KEY (CustomerID)
    REFERENCES Customer (CustomerID)
    ON DELETE NO ACTION
    ON UPDATE NO ACTION;

-- Effect:
--   DELETE FROM Customer WHERE CustomerID = 5;
--   → ERROR: FK violation if any Order.CustomerID = 5 exists
--   → Must DELETE the customer's orders first, then delete the customer
T-SQL · ON DELETE CASCADE
-- CASCADE: automatically deletes all matching child rows when the parent is deleted.
-- Use when children have no meaning without the parent (order lines, log entries).

CONSTRAINT FK_Enrollment_Student
    FOREIGN KEY (StudentID)
    REFERENCES Student (StudentID)
    ON DELETE CASCADE;

-- Effect:
--   DELETE FROM Student WHERE StudentID = 42;
--   → Automatically deletes all Enrollment rows where StudentID = 42
--   → No error; parent and children removed in one operation

-- ⚠️  SQL Server blocks CASCADE if it creates multiple cascade paths
--     to the same table (ambiguous cascade — use triggers instead)
T-SQL · ON DELETE SET NULL
-- SET NULL: sets the FK column to NULL in all child rows when parent is deleted.
-- The child row survives; its FK column becomes NULL.
-- The FK column MUST be defined as nullable (NULL) for this to work.

CREATE TABLE Project
(
    ProjectID    INT          IDENTITY(1,1)  NOT NULL,
    ProjectName  VARCHAR(100)               NOT NULL,
    ManagerID    INT                          NULL,    -- must be nullable

    CONSTRAINT PK_Project  PRIMARY KEY (ProjectID),
    CONSTRAINT FK_Project_Manager
        FOREIGN KEY (ManagerID)
        REFERENCES Employee (EmployeeID)
        ON DELETE SET NULL   -- project survives; ManagerID = NULL
);

-- Effect:
--   DELETE FROM Employee WHERE EmployeeID = 10;
--   → Project.ManagerID becomes NULL for all projects managed by employee 10
--   → Projects remain; they simply have no assigned manager
T-SQL · ON DELETE SET DEFAULT & ON UPDATE CASCADE
-- SET DEFAULT: sets the FK column to its DEFAULT value when parent is deleted.
-- Rarely used. The default value must reference an existing, valid parent row.

CONSTRAINT FK_Employee_Dept
    FOREIGN KEY (DeptID)
    REFERENCES Department (DeptID)
    ON DELETE SET DEFAULT;   -- reassigns to default dept ('GENRL') if dept deleted

-- ON UPDATE CASCADE: propagates a PK change to all FK columns automatically.
-- Useful when natural keys (not surrogates) may be updated.

CONSTRAINT FK_Instructor_Dept
    FOREIGN KEY (DeptID)
    REFERENCES Department (DeptID)
    ON DELETE NO ACTION
    ON UPDATE CASCADE;      -- if DeptID 'CS' changes to 'CSCI', all FKs update

Choosing the Right Referential Action

ScenarioBest ActionWhy
Order Lines → OrderCASCADELines have no meaning without their order
Order → CustomerNO ACTIONHistorical orders must not vanish if customer is removed
Employee → Department (optional)SET NULLEmployee can temporarily exist without a department
Audit Log → UserNO ACTIONAudit records must persist even after user deletion
Enrollment → StudentCASCADEWhen a student is fully purged, enrollments go too
Natural PK updatedON UPDATE CASCADEPropagates the change to all child FK columns automatically

ALTER TABLE — Modifying Existing Tables

Outcome 4
🔧

Outcome 4 · ALTER TABLEADD, ALTER COLUMN, DROP, and RENAME — All Operations with Syntax

ADD Column or Constraint

Adds new column(s) or applies a new constraint to an existing table. New NOT NULL columns require a DEFAULT.

ALTER COLUMN

Changes a column's data type, size, or nullability. Restricted when data exists — type compatibility must be verified.

DROP Column or Constraint

Removes a column and its data permanently, or removes a constraint definition. Must drop constraints before dropping their columns.

RENAME via sp_rename

Renames tables or columns. SQL Server has no native RENAME DDL command — use the sp_rename system procedure.

ADD Column

T-SQL · ALTER TABLE ADD COLUMN
-- Add a single nullable column (always safe on tables with existing data)
ALTER TABLE Student
ADD Phone  VARCHAR(20)  NULL;

-- Add multiple columns in one statement
ALTER TABLE Student
ADD
    Phone        VARCHAR(20)   NULL,
    LinkedInURL  VARCHAR(200)  NULL,
    IsVerified   BIT           NOT NULL
                     CONSTRAINT DF_Student_Verified  DEFAULT 0;
-- NOT NULL column is safe here because DEFAULT 0 fills all existing rows

ADD Constraint to an Existing Table

T-SQL · ALTER TABLE ADD CONSTRAINT
-- Add a UNIQUE constraint
ALTER TABLE Instructor
ADD CONSTRAINT UQ_Instructor_Email  UNIQUE (Email);

-- Add a CHECK constraint
ALTER TABLE Student
ADD CONSTRAINT CK_Student_DOB
    CHECK (DateOfBirth < GETDATE());

-- Add a FOREIGN KEY constraint
ALTER TABLE Course
ADD CONSTRAINT FK_Course_Dept
    FOREIGN KEY (DeptID)
    REFERENCES Department (DeptID)
    ON DELETE NO ACTION;

-- Add a DEFAULT constraint — uses FOR syntax, not inline
ALTER TABLE Student
ADD CONSTRAINT DF_Student_IsEnrolled
    DEFAULT 1  FOR IsEnrolled;

ALTER COLUMN — Change Type, Size, or Nullability

T-SQL · ALTER TABLE ALTER COLUMN
-- Increase column size (safe — larger type accommodates all existing values)
ALTER TABLE Student
ALTER COLUMN LastName  VARCHAR(80)  NOT NULL;

-- Change nullability from NULL to NOT NULL
-- ⚠️  FAILS if any existing row has NULL in this column
ALTER TABLE Student
ALTER COLUMN Phone  VARCHAR(20)  NOT NULL;

-- Expand numeric type (safe implicit conversion)
ALTER TABLE Product
ALTER COLUMN StockQty  INT  NOT NULL;  -- was SMALLINT

-- ⚠️  CANNOT ALTER COLUMN directly when it:
--     - Is part of a PRIMARY KEY or UNIQUE constraint
--     - Has a FOREIGN KEY or CHECK constraint referencing it
--     - Is referenced by an index
--     Solution: DROP the constraint → ALTER COLUMN → re-ADD the constraint

DROP COLUMN and DROP CONSTRAINT

T-SQL · ALTER TABLE DROP
-- Drop a single constraint by name
ALTER TABLE Student
DROP CONSTRAINT CK_Student_DOB;

-- Drop a column (all data in that column is permanently deleted)
ALTER TABLE Student
DROP COLUMN LinkedInURL;

-- Drop multiple columns in one statement
ALTER TABLE Student
DROP COLUMN LinkedInURL, Phone;

-- ⚠️  Cannot drop a column while any constraint references it.
--     Drop the constraint FIRST, then drop the column.
--     These must be two separate ALTER TABLE statements.
ALTER TABLE Student  DROP CONSTRAINT UQ_Student_Email;
ALTER TABLE Student  DROP COLUMN     Email;

RENAME — Using sp_rename

T-SQL · sp_rename — Tables, Columns, Constraints
-- Rename a table
EXEC sp_rename 'OldTableName', 'NewTableName';

-- Rename a column (third argument 'COLUMN' is required)
EXEC sp_rename 'Student.DateOfBirth', 'BirthDate', 'COLUMN';

-- Rename a constraint
EXEC sp_rename 'PK_OldName', 'PK_NewName';

-- ⚠️  sp_rename updates the catalog but does NOT update stored procedures,
--     views, or application code referencing the old name.
--     Search for all usages before renaming anything in production.

Altering Tables That Already Contain Data

Practical Guide
⚡

Practical Guide · Risk MatrixWhich Operations Are Safe, Cautious, or Dangerous?

🚨
The Core Problem with Live Tables When a table contains rows, every schema change must account for existing data. Adding a NOT NULL column without a default, shrinking a column smaller than its longest current value, or adding a constraint that existing data violates — all of these will fail or silently corrupt data. Always inspect the data before executing any structural change.
OperationRiskWhat Goes WrongSafe Approach
ADD NULL column✅ SafeNone — existing rows get NULL automaticallyExecute as written
ADD NOT NULL + DEFAULT column✅ SafeNone — existing rows receive the DEFAULT valueAlways pair NOT NULL with DEFAULT
INCREASE column size✅ SafeNone — larger size accommodates all existing valuesExecute as written
DROP unused NULL column✅ Safe*No structural risk — but data is permanently lostArchive data first; drop constraints first
DECREASE column size⚠️ CautionFails if any existing value exceeds the new sizeQuery MAX(LEN(col)) first; clean data if needed
NULL → NOT NULL⚠️ CautionFails if any row has NULL in that columnUPDATE to fill NULLs first, then ALTER COLUMN
ADD CHECK constraint⚠️ CautionFails if any existing row violates the expressionQuery violations; clean data; then add (or use WITH NOCHECK)
ADD FOREIGN KEY⚠️ CautionFails if any FK value has no matching parent PKQuery orphaned rows; delete or fix before adding FK
ADD NOT NULL (no default)⚠️ CautionFails if any rows exist — no value can be supplied for themAdd as NULL → UPDATE → ALTER to NOT NULL
Incompatible type change🚨 DangerousConversion failure or silent truncation/data lossAdd new column → migrate → verify → drop old column
DROP column with data🚨 DangerousAll column data permanently deleted — no undoBack up to archive table; confirm truly unneeded
DROP TABLE🚨 DangerousStructure and all data permanently removedFull backup; verify dependencies; consider RENAME first
🛡️

Practical Guide · Safe PatternsStep-by-Step Techniques for Every Common Risky Operation

Pattern 1 — Adding a NOT NULL Column to a Table with Existing Rows

T-SQL · Safe Pattern 1: Add NOT NULL Column
-- GOAL: Add 'Phone VARCHAR(20) NOT NULL' to Student which already has rows.
-- ─────────────────────────────────────────────────────────────────────────

-- Step 1: Add the column as nullable. Existing rows receive NULL — no error.
ALTER TABLE Student
ADD Phone  VARCHAR(20)  NULL;

-- Step 2: Populate existing rows with a valid placeholder or real value.
UPDATE Student
SET    Phone = '000-000-0000'
WHERE  Phone IS NULL;

-- Step 3: Verify no NULLs remain before proceeding.
SELECT COUNT(*) AS NullCount
FROM   Student
WHERE  Phone IS NULL;
-- Must return 0 before continuing

-- Step 4: Tighten the column to NOT NULL.
ALTER TABLE Student
ALTER COLUMN Phone  VARCHAR(20)  NOT NULL;

Pattern 2 — Changing a Column to an Incompatible Data Type

T-SQL · Safe Pattern 2: Change Data Type
-- GOAL: Change Student.GradePointAverage from FLOAT to DECIMAL(3,2).
--        Incompatible types — cannot ALTER in-place safely.
-- ─────────────────────────────────────────────────────────────────────────

-- Step 1: Add a new column with the target type.
ALTER TABLE Student
ADD GPA_New  DECIMAL(3,2)  NULL;

-- Step 2: Migrate data with an explicit CAST.
UPDATE Student
SET    GPA_New = CAST(GradePointAverage AS DECIMAL(3,2));

-- Step 3: Verify the converted data is accurate.
SELECT StudentID, GradePointAverage AS OldGPA, GPA_New AS NewGPA
FROM   Student
WHERE  ABS(GradePointAverage - CAST(GPA_New AS FLOAT)) > 0.01;
-- Should return 0 rows if conversion was lossless

-- Step 4: Drop the old column.
ALTER TABLE Student  DROP COLUMN  GradePointAverage;

-- Step 5: Rename the new column to the official name.
EXEC sp_rename 'Student.GPA_New', 'GPA', 'COLUMN';

-- Step 6: Apply constraints to the renamed column.
ALTER TABLE Student
ADD CONSTRAINT CK_Student_GPA
    CHECK (GPA BETWEEN 0.00 AND 4.00);

Pattern 3 — Adding a CHECK or FK Constraint When Existing Data May Violate It

T-SQL · Safe Pattern 3: Add Constraint to Existing Data
-- Step 1: Identify rows that would violate the new constraint BEFORE adding it.
SELECT StudentID, GPA
FROM   Student
WHERE  GPA NOT BETWEEN 0.00 AND 4.00
   OR  GPA IS NULL;
-- Fix or remove any rows returned before continuing

-- Step 2A: Data is clean — add constraint and validate existing rows.
ALTER TABLE Student
ADD CONSTRAINT CK_Student_GPA
    CHECK (GPA BETWEEN 0.00 AND 4.00);

-- Step 2B: Legacy data must remain but future rows must comply —
--           WITH NOCHECK skips validation of existing rows.
--           ⚠️  Constraint is "untrusted" — optimizer cannot rely on it.
ALTER TABLE Student
WITH NOCHECK
ADD CONSTRAINT CK_Student_GPA
    CHECK (GPA BETWEEN 0.00 AND 4.00);

-- After cleaning legacy data, re-validate to make the constraint trusted:
ALTER TABLE Student
WITH CHECK  CHECK CONSTRAINT CK_Student_GPA;

Pattern 4 — Decreasing a Column's Size Safely

T-SQL · Safe Pattern 4: Shrink Column Size
-- GOAL: Reduce LastName from VARCHAR(200) to VARCHAR(60).
-- ─────────────────────────────────────────────────────────────────────────

-- Step 1: Check current maximum length and rows that would be truncated.
SELECT
    MAX(LEN(LastName))  AS MaxCurrentLength,
    COUNT(*)            AS RowsThatWouldBeTruncated
FROM  Student
WHERE LEN(LastName) > 60;

-- Only proceed if RowsThatWouldBeTruncated = 0

-- Step 2: Perform the size reduction.
ALTER TABLE Student
ALTER COLUMN LastName  VARCHAR(60)  NOT NULL;

Pattern 5 — Finding Constraint Names Before Dropping Them

T-SQL · Query Constraint Names from System Catalog
-- All constraints on a specific table
SELECT
    tc.CONSTRAINT_NAME,
    tc.CONSTRAINT_TYPE,
    kcu.COLUMN_NAME
FROM  INFORMATION_SCHEMA.TABLE_CONSTRAINTS      tc
JOIN  INFORMATION_SCHEMA.KEY_COLUMN_USAGE        kcu
    ON  tc.CONSTRAINT_NAME = kcu.CONSTRAINT_NAME
   AND  tc.TABLE_NAME      = kcu.TABLE_NAME
WHERE tc.TABLE_NAME = 'Student'
ORDER BY tc.CONSTRAINT_TYPE;

-- CHECK constraints specifically
SELECT
    cc.CONSTRAINT_NAME,
    cc.CHECK_CLAUSE
FROM  INFORMATION_SCHEMA.CHECK_CONSTRAINTS   cc
JOIN  INFORMATION_SCHEMA.TABLE_CONSTRAINTS   tc
    ON  tc.CONSTRAINT_NAME = cc.CONSTRAINT_NAME
WHERE tc.TABLE_NAME = 'Student';

-- DEFAULT constraints (not in INFORMATION_SCHEMA — use sys catalog)
SELECT
    dc.name          AS ConstraintName,
    col.name         AS ColumnName,
    dc.definition    AS DefaultValue
FROM  sys.default_constraints dc
JOIN  sys.columns             col
    ON  col.object_id  = dc.parent_object_id
   AND  col.column_id  = dc.parent_column_id
WHERE dc.parent_object_id = OBJECT_ID('Student');
📌
The Golden Rule of Schema Changes on Live Tables Always follow this sequence: (1) Backup → (2) Test on a dev copy → (3) Inspect the data → (4) Execute → (5) Verify → (6) Notify dependent teams. Even "safe" operations like adding a nullable column can affect query plans and execution performance in high-volume systems.

DROP & TRUNCATE — Removing Tables and Data

Outcome 5 · DROP and TRUNCATERemoving Tables, Views, and Databases — and Clearing All Rows

T-SQL · DROP TABLE — With and Without Safety Guards
-- Basic DROP — fails with error if table does not exist
DROP TABLE Student;

-- Safe DROP — checks for existence first (SQL Server 2016+)
DROP TABLE IF EXISTS Student;

-- Drop multiple tables in one statement (must be in correct order:
-- child tables before parent tables, to avoid FK violations)
DROP TABLE IF EXISTS Enrollment;   -- child first
DROP TABLE IF EXISTS Section;
DROP TABLE IF EXISTS Course;
DROP TABLE IF EXISTS Student;
DROP TABLE IF EXISTS Instructor;
DROP TABLE IF EXISTS Department;   -- parent last

-- ⚠️  DROP TABLE removes the table structure AND all data permanently.
--     It cannot be undone without a database backup.
--     If a child table has a FK referencing the target, DROP fails.
--     Drop the child table (or its FK constraint) first.
T-SQL · DROP Other Object Types
-- Drop a view
DROP VIEW IF EXISTS vw_ActiveStudents;

-- Drop an index
DROP INDEX IF EXISTS IX_Student_Email  ON Student;

-- Drop a stored procedure
DROP PROCEDURE IF EXISTS usp_GetStudentGPA;

-- Drop a database (must not be the current database; switch first)
USE master;
DROP DATABASE IF EXISTS RegistrationDB;

TRUNCATE TABLE vs. DELETE — Critical Differences

Both remove rows, but they behave very differently in terms of performance, logging, identity reset, and transaction safety.

FeatureTRUNCATE TABLEDELETE (no WHERE)
Speed✅ Very fast — minimal logging⚠️ Slower — logs every row deletion
Transaction LogLogs only page deallocationsLogs every individual row
IDENTITY reset✅ Resets IDENTITY counter to seed valueDoes not reset — counter continues
WHERE clause❌ None — removes all rows✅ Supports WHERE for selective removal
Triggers fired❌ Does not fire DELETE triggers✅ Fires DELETE triggers
FK constraints❌ Fails if any FK references the table⚠️ Fails per-row based on FK action
Rollback✅ Can be rolled back inside a transaction✅ Can be rolled back inside a transaction
DDL or DML?DDLDML
T-SQL · TRUNCATE vs DELETE
-- TRUNCATE: removes all rows instantly, resets IDENTITY, minimal logging
--           Use for clearing staging/temp tables during ETL processes
TRUNCATE TABLE StagingImport;

-- After TRUNCATE: IDENTITY restarts at seed value (e.g., 1)
INSERT INTO StagingImport (Name) VALUES ('Test');
SELECT SCOPE_IDENTITY();  -- returns 1 (reset)

-- DELETE: removes all rows with full per-row logging
--         Use when you need triggers, WHERE filtering, or IDENTITY preservation
DELETE FROM Student;

-- After DELETE: IDENTITY continues from where it left off
INSERT INTO Student (LastName, FirstName, Email)
VALUES ('Smith', 'John', 'j.smith@school.edu');
SELECT SCOPE_IDENTITY();  -- returns 501 (continues)

-- ⚠️  TRUNCATE fails if another table has an active FK referencing this table:
--       "Cannot truncate table because it is being referenced by a FOREIGN KEY"
--       Solution: disable the FK, truncate, re-enable — or use DELETE instead

Tip — Teardown and Rebuild Script Pattern For development and testing, keep a paired script: a DROP block (reverse order) followed immediately by the CREATE block (dependency order). Running this single script resets the schema to a clean state without manually tracking order each time.

Teardown and Rebuild Script Template

T-SQL · Teardown → Rebuild Pattern
-- ═══ SECTION 1: TEARDOWN (drop in reverse dependency order) ════════════
DROP TABLE IF EXISTS Enrollment;
DROP TABLE IF EXISTS Section;
DROP TABLE IF EXISTS Student;
DROP TABLE IF EXISTS Course;
DROP TABLE IF EXISTS Instructor;

CREATE TABLE Student     ( ... );
CREATE TABLE Section     ( ... );

Verifying Your Schema After Creation

T-SQL · Schema Verification Queries
-- List all tables in the current database
SELECT TABLE_NAME, TABLE_TYPE
FROM   INFORMATION_SCHEMA.TABLES
WHERE  TABLE_TYPE = 'BASE TABLE'
ORDER BY TABLE_NAME;

-- List all columns for a specific table
SELECT
    COLUMN_NAME,
    DATA_TYPE,
    CHARACTER_MAXIMUM_LENGTH    AS MaxLen,
    NUMERIC_PRECISION           AS Precision,
    NUMERIC_SCALE               AS Scale,
    IS_NULLABLE
FROM   INFORMATION_SCHEMA.COLUMNS
WHERE  TABLE_NAME = 'Student'
ORDER BY ORDINAL_POSITION;

-- List all FK relationships in the database
SELECT
    fk.name                       AS ForeignKey,
    tp.name                       AS ParentTable,
    cp.name                       AS ParentColumn,
    tr.name                       AS ReferencedTable,
    cr.name                       AS ReferencedColumn,
    fk.delete_referential_action_desc AS OnDelete,
    fk.update_referential_action_desc AS OnUpdate
FROM  sys.foreign_keys                 fk
JOIN  sys.foreign_key_columns          fkc  ON fk.object_id = fkc.constraint_object_id
JOIN  sys.tables                       tp   ON fk.parent_object_id    = tp.object_id
JOIN  sys.columns                      cp   ON fkc.parent_object_id  = cp.object_id
                                             AND fkc.parent_column_id = cp.column_id
JOIN  sys.tables                       tr   ON fk.referenced_object_id    = tr.object_id
JOIN  sys.columns                      cr   ON fkc.referenced_object_id  = cr.object_id
                                             AND fkc.referenced_column_id = cr.column_id
ORDER BY tp.name, fk.name;

-- Check if a specific table exists
IF OBJECT_ID('Student', 'U') IS NOT NULL
    PRINT 'Table Student exists';
ELSE
    PRINT 'Table Student does NOT exist';

TermDefinition
DDLData Definition Language — the SQL subset for creating, modifying, and removing database structures
CREATE TABLEDDL command that builds a new table with specified columns and constraints
ALTER TABLEDDL command that modifies an existing table by adding, changing, or removing columns and constraints
DROP TABLEDDL command that permanently removes a table and all its data

NOT NULLColumn constraint preventing NULL values; implements mandatory attributes from the data model
UNIQUEConstraint ensuring all non-NULL values in a column (or combination) are distinct; implements alternate keys
CHECKConstraint validating column values against a logical expression before storing; NULL passes CHECK
DEFAULTConstraint automatically supplying a value when an INSERT omits the column
Referential ActionThe behavior specified in a FK constraint (NO ACTION, CASCADE, SET NULL, SET DEFAULT) triggered by parent changes
ON DELETE CASCADEReferential action that automatically deletes child rows when their parent row is deleted
