The Transformation Process

Outcome 1
🔄

Outcome 1 · Mapping RulesStep-by-Step: Entity → Table Transformation

Every element of the E-R diagram maps to one or more elements in the database design. Follow these rules systematically for each entity in the model.

The Eight Mapping Rules

  1. Each strong entity → one table The entity name becomes the table name. Use singular form and avoid reserved SQL words.
  2. Each simple attribute → one column Assign an exact data type, size/precision, and NULL/NOT NULL status. Derived attributes are NOT stored — compute them in a view or application.
  3. Each composite attribute → decompose into atomic columns Example: Address → Street, City, State, ZipCode. Never store a composite value in a single column (violates 1NF).
  4. Each multivalued attribute → separate table with FK back to parent The new table's PK is usually a surrogate key. A FK references the parent entity's PK. This resolves the 1NF violation.
  5. The entity identifier → PRIMARY KEY constraint Decide: use the natural identifier directly, or introduce a surrogate key (auto-integer). See Section 5 for the decision guide.
  6. Each relationship → FOREIGN KEY constraint (for 1:1 and 1:N) or junction table (for M:N) The FK goes in the table on the "many" side. For M:N, create a new junction table containing both parent PKs as a composite PK.
  7. Required attributes → NOT NULL; optional attributes → NULL This directly implements minimum cardinality from the E-R diagram. A mandatory attribute can never be left empty.
  8. Domain constraints → CHECK constraints; alternate keys → UNIQUE constraints Every business rule about allowed values should be enforced by a database constraint, not only by application code.

Data Types & Sizes

Outcome 1
📊

Outcome 1 · Data TypesChoosing the Right Data Type for Each Column (SQL Server T-SQL)

Selecting the correct data type is one of the most consequential decisions in database design. The wrong type wastes storage, reduces performance, or silently corrupts data. Always use the most restrictive type that still fits all valid values.

🔢 Numeric Types
  • TINYINT 0–255 · 1 byte
  • SMALLINT ±32K · 2 bytes
  • INT ±2.1B · 4 bytes · default PK
  • BIGINT ±9.2 quintillion · 8 bytes
  • DECIMAL(p,s) Exact: p digits, s decimal
  • NUMERIC(p,s) Same as DECIMAL
  • FLOAT Approximate · scientific use
  • MONEY 4 decimal places · financial
🔤 Character / Text
  • CHAR(n) Fixed length · n ≤ 8000
  • VARCHAR(n) Variable · up to n chars
  • VARCHAR(MAX) Up to 2GB text
  • NCHAR(n) Fixed · Unicode (2B/char)
  • NVARCHAR(n) Variable · Unicode
  • TEXT Legacy · use VARCHAR(MAX)
📅 Date & Time
  • DATE Date only · 3 bytes
  • TIME Time only · 3–5 bytes
  • DATETIME Date + time · 8 bytes
  • DATETIME2 Higher precision · 6–8 bytes
  • SMALLDATETIME 1900–2079 · 4 bytes
  • DATETIMEOFFSET With timezone · 10 bytes
✅ Boolean & Special
  • BIT 0/1/NULL · boolean flag
  • UNIQUEIDENTIFIER GUID · globally unique
  • XML XML document storage
  • JSON Stored as NVARCHAR
💾 Binary / Large Object
  • BINARY(n) Fixed binary · up to 8000
  • VARBINARY(n) Variable binary
  • VARBINARY(MAX) Up to 2GB · images/files
  • IMAGE Legacy · use VARBINARY(MAX)

Data Type Decision Guide — Common Design Scenarios

Column ScenarioRecommended TypeReasoning
Surrogate primary key (auto-increment) INT IDENTITY(1,1) 4 bytes, fast joins, handles up to 2.1 billion rows before overflow
Large-table surrogate key BIGINT IDENTITY(1,1) 8 bytes; use when INT may overflow in high-volume systems
Natural key — course code, abbreviation CHAR(n) Fixed-length known codes (e.g., "COP4708") store efficiently and compare quickly
Names, descriptions (variable length) VARCHAR(n) Only uses space actually needed; size n should reflect the max realistic value
Currency / financial amounts DECIMAL(10,2) Exact decimal; avoids floating-point rounding errors in financial calculations
Quantities, counts, ages TINYINT or SMALLINT Smaller footprint for small ranges; use INT when range is uncertain
Date of an event (no time needed) DATE 3 bytes vs. 8 for DATETIME; cleaner for birthday, hire date, etc.
Timestamp with time (order placed) DATETIME2 Higher precision than DATETIME; preferred for modern applications
Yes/No flag (IsActive, IsVerified) BIT NOT NULL DEFAULT 0 1 bit storage; explicit default prevents accidental NULLs
Phone number, ZIP code VARCHAR(20) / CHAR(10) Store as strings, not numbers — leading zeros matter; no arithmetic needed
⚠️
Common Type Mistakes to Avoid
  • Storing phone numbers as INT — leading zeros are lost; no arithmetic is needed
  • Using FLOAT for money — floating-point imprecision can cause penny-off errors in totals
  • Using VARCHAR(MAX) everywhere — oversized fields consume extra memory in query buffers
  • Using CHAR for variable-length names — wastes space by padding short values with blanks
  • Storing dates as VARCHAR — prevents date arithmetic, sorting, and range filtering

Constraints — Enforcing Business Rules

Outcome 1
🔒

Outcome 1 · ConstraintsThe Six Constraint Types and When to Use Each

Definition — Constraint

A constraint is a rule enforced by the database engine that restricts the values allowed in a column or table. Constraints implement business rules at the data layer — guaranteeing validity regardless of which application inserts or modifies the data.

PRIMARY KEY

Uniquely identifies every row. Values must be unique and never NULL. Only one per table. Automatically creates a clustered index.

CONSTRAINT PK_Customer PRIMARY KEY (CustomerID)
FOREIGN KEY

Links a column to the PK of another table. Enforces referential integrity — prevents orphaned rows. Can define ON DELETE / ON UPDATE actions.

CONSTRAINT FK_Order_Cust FOREIGN KEY (CustomerID) REFERENCES Customer(CustomerID)
NOT NULL

Prevents a column from holding a NULL value. Implements "required" (mandatory) attributes from the E-R model. Applied at column definition.

LastName VARCHAR(60) NOT NULL -- NULL values rejected at INSERT/UPDATE
UNIQUE

Ensures all values in a column (or combination) are distinct. Implements alternate keys. Unlike PK, a UNIQUE column can allow one NULL (in SQL Server).

CONSTRAINT UQ_Customer_Email UNIQUE (Email)
CHECK

Validates column values against a logical expression. Any INSERT or UPDATE that fails the check is rejected with an error. Implements domain constraints.

CONSTRAINT CK_GPA CHECK (GPA BETWEEN 0.00 AND 4.00) CONSTRAINT CK_State CHECK (LEN(State) = 2)
DEFAULT

Supplies an automatic value when no value is provided during INSERT. Not a validation constraint — it fills in missing data. Does not prevent NULL if explicitly passed.

EnrollDate DATE NOT NULL DEFAULT GETDATE() IsActive BIT NOT NULL DEFAULT 1

Naming Conventions for Constraints

Always name constraints explicitly — do not rely on system-generated names. Named constraints are far easier to drop, alter, or reference in error messages.

Primary Keys & Surrogate Keys

Outcome 2
🔑

Outcome 2 · KeysChoosing the Right Primary Key Strategy

Definition — Surrogate Key

A surrogate key is an artificial, system-generated primary key that has no business meaning. It is typically an auto-incrementing integer (IDENTITY in SQL Server) that the database assigns automatically. It exists solely to identify rows — never interpreted by users.

Surrogate Key vs. Natural Key — Side-by-Side

PropertySurrogate KeyNatural Key
Meaning No business meaning — internal ID only Has real-world meaning (SSN, ISBN, course code)
Stability ✅ Never changes — database controls it ⚠️ Can change (legal name, organizational code updates)
Size ✅ 4 bytes (INT) — compact for joins ⚠️ Often larger (VARCHAR, CHAR) — wider index, slower joins
Uniqueness Guarantee ✅ Database guarantees via IDENTITY ⚠️ Must validate against external source
Privacy / Security ✅ Safe to expose (meaningless number) ⚠️ SSN, DOB etc. are sensitive if exposed in URLs
Readability ⚠️ Not human-readable (CustomerID = 10482) ✅ Meaningful to users (CourseID = "COP4708")
FK reference size ✅ All child tables store a small integer ⚠️ Child tables store full string — multiplies storage

When to Use a Surrogate Key — Decision Framework

Surrogate Key Decision Guide
Q1: Does the entity have a natural attribute that is unique, stable, non-null, and minimal?
✅ YES

Consider using it as a natural key. Examples: CourseID "COP4708", ISBN, VIN. Still add a UNIQUE constraint as an alternate key.

❌ NO

Use a surrogate key. Add the best natural identifier as a UNIQUE constraint so it's still searchable. Proceed to Q2.

Q2: Could the natural key ever change during the entity's lifetime?
✅ NO — Stable

Natural key is safe to use as PK. Changes would be rare and controlled.

⚠️ YES — Could change

Use a surrogate key. If the natural identifier changes, updating FK values in all child tables is painful and error-prone.

Q3: Will this key appear as a FK in many child tables?
Few child tables

Either strategy is workable. Choose based on Q1 and Q2.

⚠️ Many child tables

Strongly prefer surrogate (integer). Storing a 4-byte INT in 10 child tables costs far less than a 20-byte string.

SQL Server IDENTITY — How Surrogate Keys Are Implemented

T-SQL · Surrogate Key with IDENTITY
-- IDENTITY(seed, increment) -- seed = starting value (usually 1) -- increment = step between each new value (usually 1) CREATE TABLE Customer ( CustomerID INT IDENTITY(1,1) NOT NULL, LastName VARCHAR(60) NOT NULL, FirstName VARCHAR(50) NOT NULL, Email VARCHAR(100) NOT NULL, CONSTRAINT PK_Customer PRIMARY KEY (CustomerID), CONSTRAINT UQ_Customer_Email UNIQUE (Email) ); -- SQL Server auto-assigns CustomerID: 1, 2, 3... -- You cannot INSERT a value into an IDENTITY column directly -- To see the last inserted ID: SELECT SCOPE_IDENTITY() AS NewCustomerID;
🔑
Best Practice — Always Keep the Natural Identifier Even when using a surrogate PK, always keep the natural business identifier as an alternate key enforced by a UNIQUE constraint. Users will search by email, course code, or SSN — not by CustomerID 10482. The surrogate key handles internal linking; the natural key handles user-facing lookup.

Referential Integrity Constraints

Outcome 3
🔗

Outcome 3 · Referential IntegrityForeign Keys, Orphaned Rows, and Referential Actions

Definition — Referential Integrity

Referential integrity is the guarantee that every foreign key value in a child table matches a primary key value in its parent table — or is NULL. No row in a child table can reference a parent row that does not exist.

The Parent–Child Relationship

Every FK constraint involves two tables with clear roles:

👑
Parent Table (Referenced) Contains the Primary Key being referenced. The "one" side of a 1:N relationship. Must exist before child rows can reference it. Example: Customer in the Customer → Order relationship.
🧒
Child Table (Referencing) Contains the Foreign Key that points to the parent. The "many" side. A child row cannot exist without a valid matching parent row. Example: Order — every order must reference a valid CustomerID.

What Referential Integrity Prevents

The FK constraint automatically rejects the INSERT of CustomerID=9999 with a foreign key violation error — before any bad data reaches the table.

Referential Actions — What Happens to Child Rows When the Parent Changes

When defining a FK constraint, you must decide what happens to child rows if the parent row is deleted or its PK is updated:

ActionON DELETE BehaviorBest Used When
NO ACTION (default) Blocks the DELETE if matching child rows exist. Parent cannot be deleted while children reference it. Most situations — forces manual cleanup of child rows first. Safest default.
RESTRICT Same effect as NO ACTION in SQL Server (prevents deletion of referenced parent) Standard relational integrity protection
CASCADE Automatically deletes all child rows when the parent row is deleted When children have no meaning without the parent (e.g., order lines deleted with the order). Use cautiously.
SET NULL Sets the FK column to NULL in all child rows when the parent is deleted When the relationship is optional and the child can exist without a parent
SET DEFAULT Sets the FK column to its DEFAULT value when the parent is deleted Rare; requires a valid default FK value that references another parent row

FK with Referential Action — SQL Server Example

T-SQL · Foreign Key with Referential Actions
CREATE TABLE [Order] ( OrderID INT IDENTITY(1,1) NOT NULL, CustomerID INT NOT NULL, OrderDate DATETIME2 NOT NULL DEFAULT GETDATE(), Status VARCHAR(20) NOT NULL DEFAULT 'Pending', CONSTRAINT PK_Order PRIMARY KEY (OrderID), CONSTRAINT CK_Order_Status CHECK (Status IN ('Pending', 'Shipped', 'Delivered', 'Cancelled')), -- NO ACTION: cannot delete a Customer who has orders CONSTRAINT FK_Order_Customer FOREIGN KEY (CustomerID) REFERENCES Customer (CustomerID) ON DELETE NO ACTION ON UPDATE NO ACTION ); CREATE TABLE OrderLine ( OrderID INT NOT NULL, LineNumber TINYINT NOT NULL, ProductID INT NOT NULL, Quantity SMALLINT NOT NULL CONSTRAINT CK_OL_Qty CHECK (Quantity > 0), UnitPrice DECIMAL(10,2) NOT NULL CONSTRAINT CK_OL_Price CHECK (UnitPrice >= 0), CONSTRAINT PK_OrderLine PRIMARY KEY (OrderID, LineNumber), -- CASCADE: delete order lines when the order is deleted CONSTRAINT FK_OL_Order FOREIGN KEY (OrderID) REFERENCES [Order] (OrderID) ON DELETE CASCADE, CONSTRAINT FK_OL_Product FOREIGN KEY (ProductID) REFERENCES Product (ProductID) ON DELETE NO ACTION );

An ID-dependent entity cannot be uniquely identified by its own attributes alone — it requires the primary key of its parent (owner) entity as part of its own composite primary key. This is the formal name for what the E-R model calls a weak entity.

1 : 1 One-to-One Relationship

In a true 1:1 relationship, each row in Table A corresponds to at most one row in Table B. This pattern is used to split a wide table (for security, optional data, or performance), or when two real-world entities naturally form a 1:1 pair.

📐
Mapping Rule Place the FK in the table on the optional or subordinate side. The FK column should also have a UNIQUE constraint — this enforces that only one child row can reference each parent row, making it a true 1:1.

1 : N One-to-Many Relationship — The Most Common Pattern

One-to-many is the foundation of relational database design. One parent row is related to many child rows. This relationship is implemented entirely through a Foreign Key in the child table — no new table is required.

📐
Mapping Rule Add a FK column to the child ("many" side) table. The FK value holds the PK of the related parent row. No junction table needed. Cardinality controls whether the FK is NOT NULL (mandatory) or NULL (optional).

N : M Many-to-Many — Requires a Junction Table

Many-to-many relationships cannot be implemented directly in a relational database. They must always be resolved into two 1:N relationships through a junction table (also called a bridge table, associative table, or intersection table).

📐
Mapping Rule Create a new table. Its PK is a composite of the two parent PKs (acting as FKs). Add any attributes that describe the relationship itself (e.g., enrollment date, grade) as columns in the junction table. The junction table is a full citizen of the schema — it can participate in further relationships.

Documenting the Design: The Data Dictionary

Outcome 1
📖

Outcome 1 · DocumentationWhat a Data Dictionary Is and Why It Is Required

Definition — Data Dictionary

A data dictionary is the formal documentation of every table, column, data type, size, constraint, and business rule in the database design. It translates the design decisions into a written specification that developers, DBAs, and business stakeholders can all reference.

The data dictionary is the blueprint document for the database — just as an architect's blueprint specifies every dimension before construction begins. It is created during database design and maintained throughout the system's lifetime.

What the Data Dictionary Contains

  • Table name — exact name as it will appear in the database
  • Table description — what business subject this table represents
  • Column name — exact, final name for each attribute
  • Data type — SQL Server type (INT, VARCHAR, DECIMAL, DATE, etc.)
  • Size / precision / scale — length or numeric precision
  • Null allowed? — YES (optional) or NO (required)
  • Primary Key indicator — is this column part of the PK?
  • Foreign Key indicator — which table/column does this reference?
  • Default value — if applicable
  • Additional constraints — UNIQUE, CHECK rule, referential action
  • Business description — plain-English explanation of the column's purpose

Junction TableA new table created to resolve an M:N relationship; its PK is composite — the PKs of both parent tables
ID-Dependent EntityA weak entity whose PK includes the PK of its owner (parent) entity as a foreign key component
NOT NULLColumn constraint enforcing that a value must always be provided; implements mandatory (required) attributes
UNIQUEConstraint ensuring all values in a column or combination are distinct; implements alternate keys and 1:1 relationships
CHECKConstraint that validates column values against a logical expression; implements domain constraints from the E-R model
DEFAULTConstraint that automatically supplies a value when an INSERT does not provide one
