Cursors and Iterating Through Data

📚 Module Overview

Cursors allow you to process database result sets row-by-row, but they come with significant performance tradeoffs. This module teaches you when cursors are necessary, when they should be avoided, and how to use them effectively in T-SQL and other database systems. You'll learn the fundamentals of cursor programming, explore alternatives, and understand the performance implications of your choices.

The Fundamental Concept

📖 Definition

A cursor is a database object that enables traversal over the rows of a result set, one row at a time. Think of it as a pointer that moves through query results, allowing you to perform operations on each row individually.

How SQL Typically Works (Set-Based)

Standard SQL operations work on entire sets of data at once:

-- Update all employees at once (set-based operation) UPDATE Employees SET Salary = Salary * 1.10 WHERE Department = 'Sales'; -- Deletes all matching rows in one operation DELETE FROM Orders WHERE OrderDate < DATEADD(YEAR, -2, GETDATE());

These operations are fast, efficient, and optimized by the database engine.

How Cursors Work (Row-by-Row)

Cursors process one row at a time:

-- Declare cursor for a SELECT statement DECLARE employee_cursor CURSOR FOR SELECT EmployeeID, Salary FROM Employees WHERE Department = 'Sales'; -- Open the cursor OPEN employee_cursor; -- Fetch first row FETCH NEXT FROM employee_cursor INTO @EmpID, @EmpSalary; -- Loop through all rows WHILE @@FETCH_STATUS = 0 BEGIN -- Do something with each row UPDATE Employees SET Salary = @EmpSalary * 1.10 WHERE EmployeeID = @EmpID; -- Fetch next row FETCH NEXT FROM employee_cursor INTO @EmpID, @EmpSalary; END; -- Clean up CLOSE employee_cursor; DEALLOCATE employee_cursor;

This processes rows individually, which is slower but gives you more control.

The Set-Based vs Row-Based Paradigm

✓ Set-Based Operations

  • Process all rows at once
  • Optimized by query engine
  • Fast and efficient
  • Declarative (what, not how)
  • Uses indexes effectively
VS

⚠ Cursors (Row-Based)

  • Process one row at a time
  • Limited optimization
  • Generally slower
  • Procedural (how to do it)
  • More overhead per row

Why Do Cursors Exist?

✅ Valid Use Cases for Cursors:

  • Complex row-by-row logic: When each row requires different treatment based on complex business rules
  • External API calls: When you need to call an external system for each record
  • Administrative tasks: One-time maintenance operations (rebuilding indexes, updating statistics)
  • Reporting: When building complex reports that need row-level calculations
  • Data migration: Complex transformations during ETL processes
  • Sending emails/notifications: Individual messages for each customer

🚫 When NOT to Use Cursors:

  • Simple updates that can be done with a single UPDATE statement
  • Calculations that can be done with aggregate functions (SUM, AVG, COUNT)
  • Data transformation that can be done with JOINs
  • Any operation that can be accomplished with set-based SQL
  • High-volume transactional processing (OLTP)

The Golden Rule

⚡ Performance Principle:

"Cursors should be your last resort, not your first choice."

Always try to solve the problem with set-based SQL first. Only use cursors when set-based operations are truly impossible or impractical.

The Five Steps of Cursor Usage

1 DECLARE - Define the Cursor

Create the cursor and associate it with a SELECT statement:

DECLARE cursor_name CURSOR [options] FOR select_statement;

Example:

DECLARE @EmployeeID INT; DECLARE @EmployeeName NVARCHAR(100); DECLARE employee_cursor CURSOR FOR SELECT EmployeeID, FirstName + ' ' + LastName FROM Employees WHERE Department = 'IT' ORDER BY LastName;
2 OPEN - Populate the Cursor

Execute the SELECT statement and populate the result set:

OPEN cursor_name;

Example:

OPEN employee_cursor;

What happens: The query executes, and the result set is loaded into memory (or managed according to cursor type).

3 FETCH - Retrieve Rows

Retrieve data from the cursor into variables:

FETCH [direction] FROM cursor_name INTO @variable1, @variable2, ...;

Fetch directions (for scrollable cursors):

  • NEXT - Next row (default)
  • PRIOR - Previous row
  • FIRST - First row
  • LAST - Last row
  • ABSOLUTE n - Nth row
  • RELATIVE n - N rows from current position

Example:

-- Fetch first row FETCH NEXT FROM employee_cursor INTO @EmployeeID, @EmployeeName; -- Check if fetch was successful IF @@FETCH_STATUS = 0 BEGIN PRINT 'Employee: ' + @EmployeeName; END;
4 CLOSE - Release the Result Set

Close the cursor and release the current result set:

CLOSE cursor_name;

Example:

CLOSE employee_cursor;

What happens: The result set is released, but the cursor definition still exists. You can OPEN it again if needed.

5 DEALLOCATE - Remove the Cursor

Remove the cursor definition and free all resources:

DEALLOCATE cursor_name;

Example:

DEALLOCATE employee_cursor;

⚠️ Important: Always DEALLOCATE cursors to free memory. Cursors that aren't deallocated can cause memory leaks!

@@FETCH_STATUS

Understanding Fetch Status Codes

The @@FETCH_STATUS function returns the status of the last FETCH statement:

Return Value Meaning
0 FETCH was successful
-1 FETCH failed or row is beyond the result set
-2 Row fetched is missing (only for keyset cursors)

Complete Basic Example

-- Complete cursor example: Print all employee names -- Step 1: Declare variables DECLARE @EmployeeID INT; DECLARE @EmployeeName NVARCHAR(100); -- Step 2: Declare cursor DECLARE employee_cursor CURSOR FOR SELECT EmployeeID, FirstName + ' ' + LastName FROM Employees ORDER BY LastName; -- Step 3: Open cursor OPEN employee_cursor; -- Step 4: Fetch first row FETCH NEXT FROM employee_cursor INTO @EmployeeID, @EmployeeName; -- Step 5: Loop through all rows WHILE @@FETCH_STATUS = 0 BEGIN -- Process current row PRINT CAST(@EmployeeID AS VARCHAR) + ': ' + @EmployeeName; -- Fetch next row FETCH NEXT FROM employee_cursor INTO @EmployeeID, @EmployeeName; END; -- Step 6: Close cursor CLOSE employee_cursor; -- Step 7: Deallocate cursor DEALLOCATE employee_cursor;

Cursor Declaration Syntax

DECLARE cursor_name CURSOR [LOCAL | GLOBAL] [FORWARD_ONLY | SCROLL] [STATIC | KEYSET | DYNAMIC | FAST_FORWARD] [READ_ONLY | SCROLL_LOCKS | OPTIMISTIC] [TYPE_WARNING] FOR select_statement;

Scope Options

LOCAL vs GLOBAL

LOCAL (default):

  • Cursor is local to the batch, stored procedure, or trigger
  • Automatically deallocated when the scope ends
  • Recommended for most scenarios

GLOBAL:

  • Cursor is available to any stored procedure or batch in the connection
  • Must be explicitly deallocated
  • Use when cursor needs to be shared across procedures
-- Local cursor (recommended) DECLARE emp_cursor CURSOR LOCAL FOR SELECT EmployeeID FROM Employees; -- Global cursor DECLARE emp_cursor CURSOR GLOBAL FOR SELECT EmployeeID FROM Employees;

Scrollability Options

FORWARD_ONLY vs SCROLL

FORWARD_ONLY (default):

  • Can only move forward through the result set (FETCH NEXT)
  • Faster and uses less memory
  • Best for sequential processing
DECLARE emp_cursor CURSOR FORWARD_ONLY FOR SELECT EmployeeID FROM Employees; OPEN emp_cursor; FETCH NEXT FROM emp_cursor INTO @ID; -- ✓ Works FETCH PRIOR FROM emp_cursor INTO @ID; -- ✗ Error!

SCROLL:

  • Allows forward and backward movement
  • Supports NEXT, PRIOR, FIRST, LAST, ABSOLUTE, RELATIVE
  • Uses more memory
  • Use when you need random access to rows
DECLARE emp_cursor CURSOR SCROLL FOR SELECT EmployeeID FROM Employees; OPEN emp_cursor; FETCH NEXT FROM emp_cursor INTO @ID; -- ✓ Move forward FETCH PRIOR FROM emp_cursor INTO @ID; -- ✓ Move backward FETCH FIRST FROM emp_cursor INTO @ID; -- ✓ Jump to first FETCH LAST FROM emp_cursor INTO @ID; -- ✓ Jump to last FETCH ABSOLUTE 5 FROM emp_cursor INTO @ID; -- ✓ Jump to row 5

Cursor Types

1. STATIC Cursor

Creates a snapshot of the data at the time the cursor opens

  • Data is copied to tempdb
  • Changes to underlying tables are NOT reflected in the cursor
  • Always scrollable
  • Higher memory usage
  • Good for reporting where data consistency is critical
DECLARE emp_cursor CURSOR STATIC FOR SELECT EmployeeID, Salary FROM Employees; OPEN emp_cursor; -- Someone updates salary in another session -- Cursor still sees OLD salary values FETCH NEXT FROM emp_cursor INTO @ID, @Salary;

2. KEYSET Cursor

Keys are fixed, but values can change

  • Set of keys is fixed when cursor opens
  • Values are fetched when you retrieve each row
  • Sees updates to existing rows
  • Does NOT see new rows inserted after open
  • Deleted rows show as @@FETCH_STATUS = -2
  • Always scrollable
DECLARE emp_cursor CURSOR KEYSET FOR SELECT EmployeeID, Salary FROM Employees; OPEN emp_cursor; -- Someone updates salary: Cursor WILL see the new value -- Someone inserts new employee: Cursor will NOT see it -- Someone deletes employee: @@FETCH_STATUS = -2

3. DYNAMIC Cursor

Reflects all changes to the data

  • Always reflects current state of data
  • Sees inserts, updates, and deletes
  • Most overhead and slowest
  • Row order can change
  • Always scrollable
DECLARE emp_cursor CURSOR DYNAMIC FOR SELECT EmployeeID, Salary FROM Employees ORDER BY Salary; OPEN emp_cursor; -- All changes are visible in real-time -- New rows appear, updates are seen, deletes remove rows

4. FAST_FORWARD (Recommended for Performance)

Optimized forward-only, read-only cursor

  • Fastest cursor type
  • Forward-only, read-only
  • Performance optimizations enabled
  • Recommended for most cursor scenarios
DECLARE emp_cursor CURSOR FAST_FORWARD FOR SELECT EmployeeID, Salary FROM Employees; -- This is equivalent to: -- FORWARD_ONLY READ_ONLY with performance optimizations

✅ Best Practice: Use FAST_FORWARD unless you specifically need scrollability or to see data changes.

Concurrency Options

READ_ONLY, SCROLL_LOCKS, and OPTIMISTIC

READ_ONLY (recommended):

  • Cannot update data through the cursor
  • Fastest option
  • No locking overhead

SCROLL_LOCKS:

  • Locks each row as it's fetched
  • Guarantees updates succeed
  • Can cause blocking

OPTIMISTIC:

  • No locks on fetch
  • Updates may fail if row changed
  • Uses timestamps or values to detect changes

Complete Stored Procedure Example

/* Stored Procedure: Calculate employee bonuses based on performance Uses cursor to process each employee individually */ CREATE PROCEDURE CalculateEmployeeBonuses @DepartmentID INT, @BonusMultiplier DECIMAL(5,2) = 1.0 AS BEGIN SET NOCOUNT ON; -- Declare variables for cursor data DECLARE @EmployeeID INT; DECLARE @EmployeeName NVARCHAR(100); DECLARE @Salary DECIMAL(10,2); DECLARE @PerformanceRating DECIMAL(3,2); DECLARE @CalculatedBonus DECIMAL(10,2); DECLARE @ProcessedCount INT = 0; DECLARE @TotalBonuses DECIMAL(15,2) = 0; -- Declare the cursor DECLARE employee_cursor CURSOR LOCAL FAST_FORWARD FOR SELECT e.EmployeeID, e.FirstName + ' ' + e.LastName, e.Salary, ISNULL(pr.Rating, 1.0) FROM Employees e LEFT JOIN PerformanceReviews pr ON e.EmployeeID = pr.EmployeeID AND pr.ReviewYear = YEAR(GETDATE()) WHERE e.DepartmentID = @DepartmentID AND e.IsActive = 1 ORDER BY e.LastName; -- Error handling BEGIN TRY -- Open cursor OPEN employee_cursor; -- Fetch first row FETCH NEXT FROM employee_cursor INTO @EmployeeID, @EmployeeName, @Salary, @PerformanceRating; -- Process each employee WHILE @@FETCH_STATUS = 0 BEGIN -- Complex bonus calculation logic -- Base: 10% of salary -- Modified by performance rating and bonus multiplier SET @CalculatedBonus = (@Salary * 0.10) * @PerformanceRating * @BonusMultiplier; -- Apply business rules IF @PerformanceRating >= 1.5 BEGIN -- Top performers get additional 20% SET @CalculatedBonus = @CalculatedBonus * 1.2; END -- Cap bonuses at 25% of salary IF @CalculatedBonus > (@Salary * 0.25) BEGIN SET @CalculatedBonus = @Salary * 0.25; END -- Update employee bonus UPDATE Employees SET BonusAmount = @CalculatedBonus, LastBonusCalculation = GETDATE() WHERE EmployeeID = @EmployeeID; -- Log the bonus calculation INSERT INTO BonusLog (EmployeeID, BonusAmount, CalculationDate) VALUES (@EmployeeID, @CalculatedBonus, GETDATE()); -- Track totals SET @ProcessedCount = @ProcessedCount + 1; SET @TotalBonuses = @TotalBonuses + @CalculatedBonus; -- Optional: Print progress PRINT 'Processed: ' + @EmployeeName + ' - Bonus: $' + CAST(@CalculatedBonus AS VARCHAR); -- Fetch next row FETCH NEXT FROM employee_cursor INTO @EmployeeID, @EmployeeName, @Salary, @PerformanceRating; END; -- Clean up cursor CLOSE employee_cursor; DEALLOCATE employee_cursor; -- Return summary SELECT @ProcessedCount AS EmployeesProcessed, @TotalBonuses AS TotalBonusAmount; END TRY BEGIN CATCH -- Ensure cursor is cleaned up even on error IF CURSOR_STATUS('local', 'employee_cursor') >= 0 BEGIN CLOSE employee_cursor; DEALLOCATE employee_cursor; END; -- Re-throw error THROW; END CATCH; END; GO -- Execute the stored procedure EXEC CalculateEmployeeBonuses @DepartmentID = 5, @BonusMultiplier = 1.15;

Cursor with Transaction Control

CREATE PROCEDURE ProcessOrderBatch @BatchSize INT = 100 AS BEGIN SET NOCOUNT ON; DECLARE @OrderID INT; DECLARE @ProcessedCount INT = 0; -- Cursor for unprocessed orders DECLARE order_cursor CURSOR LOCAL FAST_FORWARD FOR SELECT TOP (@BatchSize) OrderID FROM Orders WHERE ProcessingStatus = 'Pending' ORDER BY OrderDate; OPEN order_cursor; FETCH NEXT FROM order_cursor INTO @OrderID; WHILE @@FETCH_STATUS = 0 BEGIN -- Process each order in its own transaction BEGIN TRANSACTION; BEGIN TRY -- Complex processing logic here EXEC ProcessSingleOrder @OrderID; -- Mark as processed UPDATE Orders SET ProcessingStatus = 'Completed', ProcessedDate = GETDATE() WHERE OrderID = @OrderID; -- Commit this order COMMIT TRANSACTION; SET @ProcessedCount = @ProcessedCount + 1; END TRY BEGIN CATCH -- Rollback just this order, continue with others ROLLBACK TRANSACTION; -- Log the error INSERT INTO OrderProcessingErrors (OrderID, ErrorMessage, ErrorDate) VALUES (@OrderID, ERROR_MESSAGE(), GETDATE()); END CATCH; FETCH NEXT FROM order_cursor INTO @OrderID; END; CLOSE order_cursor; DEALLOCATE order_cursor; SELECT @ProcessedCount AS OrdersProcessed; END;

Nested Cursors Example

⚠️ Warning: Nested cursors can be extremely slow. Always look for set-based alternatives first!

/* Example: Generate invoice details for each customer Outer cursor: Customers Inner cursor: Customer's orders */ CREATE PROCEDURE GenerateCustomerInvoices AS BEGIN DECLARE @CustomerID INT; DECLARE @CustomerName NVARCHAR(100); DECLARE @OrderID INT; DECLARE @OrderTotal DECIMAL(10,2); DECLARE @CustomerTotal DECIMAL(10,2); -- Outer cursor: Customers DECLARE customer_cursor CURSOR LOCAL FAST_FORWARD FOR SELECT CustomerID, CustomerName FROM Customers WHERE IsActive = 1; OPEN customer_cursor; FETCH NEXT FROM customer_cursor INTO @CustomerID, @CustomerName; WHILE @@FETCH_STATUS = 0 BEGIN SET @CustomerTotal = 0; PRINT 'Customer: ' + @CustomerName; -- Inner cursor: Orders for this customer DECLARE order_cursor CURSOR LOCAL FAST_FORWARD FOR SELECT OrderID, OrderTotal FROM Orders WHERE CustomerID = @CustomerID AND InvoiceGenerated = 0; OPEN order_cursor; FETCH NEXT FROM order_cursor INTO @OrderID, @OrderTotal; WHILE @@FETCH_STATUS = 0 BEGIN -- Process each order PRINT ' Order ' + CAST(@OrderID AS VARCHAR) + ': $' + CAST(@OrderTotal AS VARCHAR); SET @CustomerTotal = @CustomerTotal + @OrderTotal; FETCH NEXT FROM order_cursor INTO @OrderID, @OrderTotal; END; CLOSE order_cursor; DEALLOCATE order_cursor; PRINT ' Total: $' + CAST(@CustomerTotal AS VARCHAR); PRINT ''; FETCH NEXT FROM customer_cursor INTO @CustomerID, @CustomerName; END; CLOSE customer_cursor; DEALLOCATE customer_cursor; END;

💡 Better Alternative (Set-Based):

-- This does the same thing MUCH faster without nested cursors SELECT c.CustomerID, c.CustomerName, o.OrderID, o.OrderTotal, SUM(o.OrderTotal) OVER (PARTITION BY c.CustomerID) AS CustomerTotal FROM Customers c INNER JOIN Orders o ON c.CustomerID = o.CustomerID WHERE c.IsActive = 1 AND o.InvoiceGenerated = 0 ORDER BY c.CustomerName, o.OrderID;

Why Cursors Are Slow

Performance Issues with Cursors

  1. Row-by-row processing: Each fetch requires a round trip, no bulk operations
  2. No query optimization: Optimizer can't optimize procedural code
  3. Locks held longer: If in a transaction, locks are held during entire loop
  4. Memory overhead: Cursor structures consume memory
  5. Tempdb usage: Some cursor types use tempdb extensively
  6. Network traffic: More round trips between SQL Server and client

Performance Comparison: Cursor vs Set-Based

-- Test scenario: Update 10,000 employee salaries -- ❌ SLOW: Using cursor (typically 2-5 seconds) DECLARE @EmpID INT, @CurrentSalary DECIMAL(10,2); DECLARE emp_cursor CURSOR FOR SELECT EmployeeID, Salary FROM Employees; OPEN emp_cursor; FETCH NEXT FROM emp_cursor INTO @EmpID, @CurrentSalary; WHILE @@FETCH_STATUS = 0 BEGIN UPDATE Employees SET Salary = @CurrentSalary * 1.05 WHERE EmployeeID = @EmpID; FETCH NEXT FROM emp_cursor INTO @EmpID, @CurrentSalary; END; CLOSE emp_cursor; DEALLOCATE emp_cursor; -- ✅ FAST: Set-based operation (typically < 100 milliseconds) UPDATE Employees SET Salary = Salary * 1.05;

⚡ Performance Impact: The cursor version can be 20-100x slower than the set-based version!

Set-Based Alternatives to Common Cursor Patterns

Pattern 1: Running Totals

❌ Using Cursor (Slow)

DECLARE @OrderID INT, @Amount DECIMAL(10,2); DECLARE @RunningTotal DECIMAL(10,2) = 0; DECLARE order_cursor CURSOR FOR SELECT OrderID, Amount FROM Orders ORDER BY OrderDate; OPEN order_cursor; FETCH NEXT FROM order_cursor INTO @OrderID, @Amount; WHILE @@FETCH_STATUS = 0 BEGIN SET @RunningTotal = @RunningTotal + @Amount; UPDATE Orders SET RunningTotal = @RunningTotal WHERE OrderID = @OrderID; FETCH NEXT FROM order_cursor INTO @OrderID, @Amount; END; CLOSE order_cursor; DEALLOCATE order_cursor;

✅ Using Window Functions (Fast)

-- Update all at once using window function WITH OrdersWithRunningTotal AS ( SELECT OrderID, SUM(Amount) OVER ( ORDER BY OrderDate ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS RunningTotal FROM Orders ) UPDATE o SET o.RunningTotal = ort.RunningTotal FROM Orders o INNER JOIN OrdersWithRunningTotal ort ON o.OrderID = ort.OrderID;

Pattern 2: Conditional Updates Based on Other Tables

❌ Using Cursor

DECLARE @ProductID INT; DECLARE @AvgRating DECIMAL(3,2); DECLARE prod_cursor CURSOR FOR SELECT ProductID FROM Products; OPEN prod_cursor; FETCH NEXT FROM prod_cursor INTO @ProductID; WHILE @@FETCH_STATUS = 0 BEGIN SELECT @AvgRating = AVG(Rating) FROM ProductReviews WHERE ProductID = @ProductID; UPDATE Products SET AverageRating = @AvgRating WHERE ProductID = @ProductID; FETCH NEXT FROM prod_cursor INTO @ProductID; END; CLOSE prod_cursor; DEALLOCATE prod_cursor;

✅ Using JOIN

-- Single UPDATE with JOIN UPDATE p SET p.AverageRating = ratings.AvgRating FROM Products p INNER JOIN ( SELECT ProductID, AVG(Rating) AS AvgRating FROM ProductReviews GROUP BY ProductID ) ratings ON p.ProductID = ratings.ProductID;

Pattern 3: Processing in Batches

❌ Using Cursor for All Rows

-- Process 1 million rows one at a time DECLARE log_cursor CURSOR FOR SELECT LogID FROM AccessLogs WHERE Archived = 0; -- Very slow for large datasets

✅ Using Batch Processing

-- Process in batches of 10,000 DECLARE @BatchSize INT = 10000; DECLARE @RowsAffected INT = @BatchSize; WHILE @RowsAffected = @BatchSize BEGIN UPDATE TOP (@BatchSize) AccessLogs SET Archived = 1 WHERE Archived = 0 AND LogDate < DATEADD(YEAR, -1, GETDATE()); SET @RowsAffected = @@ROWCOUNT; -- Brief pause to avoid blocking WAITFOR DELAY '00:00:01'; END;

When Cursors Are Actually Better

✅ Legitimate Cursor Use Cases:

1. Calling External APIs/Systems:

-- Need to call external API for each customer DECLARE customer_cursor CURSOR FOR SELECT CustomerID, Email FROM Customers WHERE EmailPreference = 'Weekly'; WHILE @@FETCH_STATUS = 0 BEGIN -- Call external email service API EXEC sp_SendEmailViaAPI @Email, @NewsletterContent; END;

2. Complex Business Logic Per Row:

-- Different calculation for each product type WHILE @@FETCH_STATUS = 0 BEGIN IF @ProductType = 'Electronics' EXEC CalculateElectronicsPrice @ProductID; ELSE IF @ProductType = 'Clothing' EXEC CalculateClothingPrice @ProductID; ELSE IF @ProductType = 'Food' EXEC CalculateFoodPrice @ProductID; -- etc... END;

3. Administrative/Maintenance Tasks:

-- Rebuild indexes on all tables (one-time maintenance) DECLARE table_cursor CURSOR FOR SELECT name FROM sys.tables WHERE type = 'U'; WHILE @@FETCH_STATUS = 0 BEGIN EXEC ('ALTER INDEX ALL ON ' + @TableName + ' REBUILD'); END;

Example 1: Data Cleanup with Logging

/* Clean up duplicate customer records Log each merge operation */ CREATE PROCEDURE CleanupDuplicateCustomers AS BEGIN SET NOCOUNT ON; DECLARE @KeepCustomerID INT; DECLARE @DuplicateCustomerID INT; DECLARE @Email NVARCHAR(100); DECLARE @MergeCount INT = 0; -- Find duplicate emails, keep oldest account DECLARE duplicate_cursor CURSOR LOCAL FAST_FORWARD FOR SELECT MIN(CustomerID) AS KeepID, MAX(CustomerID) AS DuplicateID, Email FROM Customers GROUP BY Email HAVING COUNT(*) > 1; BEGIN TRY OPEN duplicate_cursor; FETCH NEXT FROM duplicate_cursor INTO @KeepCustomerID, @DuplicateCustomerID, @Email; WHILE @@FETCH_STATUS = 0 BEGIN BEGIN TRANSACTION; -- Move orders to kept customer UPDATE Orders SET CustomerID = @KeepCustomerID WHERE CustomerID = @DuplicateCustomerID; -- Log the merge INSERT INTO CustomerMergeLog (KeptCustomerID, MergedCustomerID, Email, MergeDate) VALUES (@KeepCustomerID, @DuplicateCustomerID, @Email, GETDATE()); -- Delete duplicate DELETE FROM Customers WHERE CustomerID = @DuplicateCustomerID; COMMIT TRANSACTION; SET @MergeCount = @MergeCount + 1; FETCH NEXT FROM duplicate_cursor INTO @KeepCustomerID, @DuplicateCustomerID, @Email; END; CLOSE duplicate_cursor; DEALLOCATE duplicate_cursor; PRINT 'Merged ' + CAST(@MergeCount AS VARCHAR) + ' duplicate customers'; END TRY BEGIN CATCH IF @@TRANCOUNT > 0 ROLLBACK TRANSACTION; IF CURSOR_STATUS('local', 'duplicate_cursor') >= 0 BEGIN CLOSE duplicate_cursor; DEALLOCATE duplicate_cursor; END; THROW; END CATCH; END;

Example 2: Dynamic SQL Generation

/* Generate and execute dynamic SQL for each table Example: Update statistics on all user tables */ CREATE PROCEDURE UpdateAllTableStatistics AS BEGIN DECLARE @TableSchema NVARCHAR(128); DECLARE @TableName NVARCHAR(128); DECLARE @SQL NVARCHAR(500); DECLARE @StartTime DATETIME; DECLARE @TableCount INT = 0; DECLARE table_cursor CURSOR LOCAL FAST_FORWARD FOR SELECT s.name AS SchemaName, t.name AS TableName FROM sys.tables t INNER JOIN sys.schemas s ON t.schema_id = s.schema_id WHERE t.type = 'U' -- User tables only ORDER BY s.name, t.name; PRINT 'Starting statistics update at ' + CONVERT(VARCHAR, GETDATE(), 120); OPEN table_cursor; FETCH NEXT FROM table_cursor INTO @TableSchema, @TableName; WHILE @@FETCH_STATUS = 0 BEGIN SET @StartTime = GETDATE(); -- Build dynamic SQL SET @SQL = 'UPDATE STATISTICS ' + QUOTENAME(@TableSchema) + '.' + QUOTENAME(@TableName) + ' WITH FULLSCAN;'; BEGIN TRY -- Execute dynamic SQL EXEC sp_executesql @SQL; SET @TableCount = @TableCount + 1; PRINT 'Updated: ' + @TableSchema + '.' + @TableName + ' (' + CAST(DATEDIFF(SECOND, @StartTime, GETDATE()) AS VARCHAR) + ' seconds)'; END TRY BEGIN CATCH PRINT 'ERROR on ' + @TableSchema + '.' + @TableName + ': ' + ERROR_MESSAGE(); END CATCH; FETCH NEXT FROM table_cursor INTO @TableSchema, @TableName; END; CLOSE table_cursor; DEALLOCATE table_cursor; PRINT 'Completed ' + CAST(@TableCount AS VARCHAR) + ' tables at ' + CONVERT(VARCHAR, GETDATE(), 120); END;

Example 3: Scrollable Cursor for Reporting

/* Generate a report with context from previous rows Uses SCROLL cursor to access previous values */ DECLARE @CurrentSales DECIMAL(15,2); DECLARE @PreviousSales DECIMAL(15,2); DECLARE @Month VARCHAR(7); DECLARE @PercentChange DECIMAL(5,2); -- Scrollable cursor to move backward DECLARE sales_cursor CURSOR SCROLL FOR SELECT FORMAT(SalesMonth, 'yyyy-MM') AS Month, SUM(SalesAmount) AS TotalSales FROM Sales GROUP BY SalesMonth ORDER BY SalesMonth; OPEN sales_cursor; -- Get first month FETCH FIRST FROM sales_cursor INTO @Month, @CurrentSales; PRINT @Month + ': $' + CAST(@CurrentSales AS VARCHAR) + ' (baseline)'; -- Move to second month FETCH NEXT FROM sales_cursor INTO @Month, @CurrentSales; WHILE @@FETCH_STATUS = 0 BEGIN -- Go back to get previous month's sales FETCH PRIOR FROM sales_cursor INTO @Month, @PreviousSales; -- Return to current month FETCH NEXT FROM sales_cursor INTO @Month, @CurrentSales; -- Calculate percent change SET @PercentChange = ((@CurrentSales - @PreviousSales) / @PreviousSales) * 100; PRINT @Month + ': $' + CAST(@CurrentSales AS VARCHAR) + ' (' + CAST(@PercentChange AS VARCHAR) + '% change)'; -- Move to next month FETCH NEXT FROM sales_cursor INTO @Month, @CurrentSales; END; CLOSE sales_cursor; DEALLOCATE sales_cursor;

💡 Better Alternative Using LAG():

-- Same result, much faster with window function SELECT FORMAT(SalesMonth, 'yyyy-MM') AS Month, SUM(SalesAmount) AS CurrentSales, LAG(SUM(SalesAmount)) OVER (ORDER BY SalesMonth) AS PreviousSales, ((SUM(SalesAmount) - LAG(SUM(SalesAmount)) OVER (ORDER BY SalesMonth)) / LAG(SUM(SalesAmount)) OVER (ORDER BY SalesMonth)) * 100 AS PercentChange FROM Sales GROUP BY SalesMonth ORDER BY SalesMonth;

Syntax Examples by Database

SQL Server (T-SQL)

DECLARE @EmployeeName NVARCHAR(100); DECLARE emp_cursor CURSOR FAST_FORWARD FOR SELECT EmployeeName FROM Employees; OPEN emp_cursor; FETCH NEXT FROM emp_cursor INTO @EmployeeName; WHILE @@FETCH_STATUS = 0 BEGIN PRINT @EmployeeName; FETCH NEXT FROM emp_cursor INTO @EmployeeName; END; CLOSE emp_cursor; DEALLOCATE emp_cursor;

Oracle (PL/SQL)

DECLARE v_employee_name VARCHAR2(100); -- Explicit cursor CURSOR emp_cursor IS SELECT employee_name FROM employees; BEGIN OPEN emp_cursor; LOOP FETCH emp_cursor INTO v_employee_name; EXIT WHEN emp_cursor%NOTFOUND; DBMS_OUTPUT.PUT_LINE(v_employee_name); END LOOP; CLOSE emp_cursor; END; -- Cursor FOR loop (easier, automatic open/close/fetch) BEGIN FOR emp_rec IN (SELECT employee_name FROM employees) LOOP DBMS_OUTPUT.PUT_LINE(emp_rec.employee_name); END LOOP; END;

MySQL

DELIMITER $$ CREATE PROCEDURE ProcessEmployees() BEGIN DECLARE done INT DEFAULT 0; DECLARE emp_name VARCHAR(100); DECLARE emp_cursor CURSOR FOR SELECT employee_name FROM employees; -- Handler for end of cursor DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = 1; OPEN emp_cursor; read_loop: LOOP FETCH emp_cursor INTO emp_name; IF done THEN LEAVE read_loop; END IF; SELECT emp_name; END LOOP; CLOSE emp_cursor; END$$ DELIMITER ;

PostgreSQL

-- Explicit cursor DO $$ DECLARE emp_cursor CURSOR FOR SELECT employee_name FROM employees; emp_record RECORD; BEGIN OPEN emp_cursor; LOOP FETCH emp_cursor INTO emp_record; EXIT WHEN NOT FOUND; RAISE NOTICE '%', emp_record.employee_name; END LOOP; CLOSE emp_cursor; END $$; -- Simpler FOR loop (recommended) DO $$ DECLARE emp_record RECORD; BEGIN FOR emp_record IN SELECT employee_name FROM employees LOOP RAISE NOTICE '%', emp_record.employee_name; END LOOP; END $$;

Key Differences Summary

📊 Important Distinctions:

  • Oracle: Most flexible with cursor parameters and REF CURSORs. Cursor FOR loops are very convenient.
  • SQL Server: Requires explicit DEALLOCATE. Best performance with FAST_FORWARD cursors.
  • MySQL: Requires CONTINUE HANDLER for cursor completion. Limited cursor types.
  • PostgreSQL: Similar to Oracle with implicit cursor loops. Good middle ground.

Best Practices

✅ 1. Always Use FAST_FORWARD When Possible

-- Best performance for forward-only processing DECLARE cursor_name CURSOR FAST_FORWARD FOR SELECT columns FROM table;

✅ 2. Use LOCAL Scope

-- Automatically deallocated when procedure ends DECLARE cursor_name CURSOR LOCAL FOR SELECT columns FROM table;

✅ 3. Always Use TRY-CATCH with Cleanup

BEGIN TRY OPEN cursor_name; -- Process rows... CLOSE cursor_name; DEALLOCATE cursor_name; END TRY BEGIN CATCH IF CURSOR_STATUS('local', 'cursor_name') >= 0 BEGIN CLOSE cursor_name; DEALLOCATE cursor_name; END; THROW; END CATCH;

✅ 4. Use SET NOCOUNT ON in Stored Procedures

CREATE PROCEDURE MyProcedure AS BEGIN SET NOCOUNT ON; -- Reduces network traffic -- Cursor code here... END;

✅ 5. Limit Result Set in Cursor Declaration

-- Filter in cursor SELECT, not in loop DECLARE cursor_name CURSOR FOR SELECT columns FROM table WHERE conditions; -- ✓ Filter here -- Not this: WHILE @@FETCH_STATUS = 0 BEGIN IF @condition -- ✗ Don't filter in loop BEGIN ... END; END;

✅ 6. Process in Small Transactions for Large Datasets

WHILE @@FETCH_STATUS = 0 BEGIN BEGIN TRANSACTION; -- Process one row ... COMMIT TRANSACTION; FETCH NEXT FROM cursor_name INTO @variables; END;

Common Mistakes

❌ 1. Forgetting to DEALLOCATE

OPEN cursor_name; -- Process rows... CLOSE cursor_name; -- Missing: DEALLOCATE cursor_name; -- Result: Memory leak!

❌ 2. Not Handling Errors

OPEN cursor_name; WHILE @@FETCH_STATUS = 0 BEGIN -- What if this fails? UPDATE table SET column = @value; -- Cursor never gets closed/deallocated! END;

❌ 3. Using Cursors for Simple Set Operations

-- DON'T DO THIS: DECLARE @Total DECIMAL(10,2) = 0; WHILE @@FETCH_STATUS = 0 BEGIN SET @Total = @Total + @Amount; END; -- DO THIS INSTEAD: SELECT @Total = SUM(Amount) FROM table;

❌ 4. Updating the Same Table Being Cursored

-- Can cause blocking and unexpected results DECLARE cursor_name CURSOR FOR SELECT ID, Value FROM MyTable; WHILE @@FETCH_STATUS = 0 BEGIN UPDATE MyTable -- Same table! SET Value = @NewValue WHERE ID = @ID; END;

❌ 5. Nested Cursors Without Justification

-- Extremely slow - O(n²) performance WHILE @@FETCH_STATUS = 0 -- Outer cursor BEGIN WHILE @@FETCH_STATUS = 0 -- Inner cursor BEGIN -- Almost always can be done with JOINs END; END;

Decision Tree: Should I Use a Cursor?

START: Do I need to process rows individually? │ ├─ NO → Use set-based SQL ✅ │ └─ YES → Can it be done with: │ ├─ Window functions (ROW_NUMBER, LAG, LEAD, etc.)? → Use those instead ✅ ├─ CTEs or derived tables? → Use those instead ✅ ├─ Simple UPDATE/DELETE with WHERE clause? → Use that instead ✅ │ └─ None of the above work? │ ├─ Need to call external APIs/stored procs? → Cursor OK ✅ ├─ Complex per-row business logic? → Cursor OK ✅ ├─ Administrative one-time task? → Cursor OK ✅ │ └─ Still not sure? → Try set-based first, measure performance

Performance Checklist

Before Using a Cursor in Production:

  • ☐ Tried to solve with set-based SQL
  • ☐ Tested with expected data volume
  • ☐ Used FAST_FORWARD type
  • ☐ Limited result set in cursor SELECT
  • ☐ Implemented proper error handling
  • ☐ Ensured cursor is always deallocated
  • ☐ Considered batch processing instead
  • ☐ Documented why cursor is necessary
  • ☐ Compared performance vs alternatives
