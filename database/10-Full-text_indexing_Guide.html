🔎 Full-Text Indexing

COP4709 — Database Management

📖 What Is Full-Text Indexing?

A full-text index is a special kind of database index built to support fast, linguistic searches across large bodies of text — words, phrases, and language patterns — rather than exact value comparisons. Standard indexes support queries like "find rows where status = 'Open'" by matching exact values. Full-text indexes support queries like "find all articles that discuss machine learning near neural networks".

Where a regular LIKE '%keyword%' search must scan every row and every character, a full-text index pre-processes text into a searchable word map — enabling fast lookups by word, phrase, prefix, proximity, or linguistic root, across columns containing thousands of words per row.

This guide uses SQL Server's built-in Full-Text Search engine for examples, covering setup, query syntax, use cases, and the trade-offs involved in adopting full-text indexing.

1. How Full-Text Indexing Works — Under the Hood

The Problem With LIKE for Text Search

❌ LIKE Search

-- Finds "learning" but NOT "learned" -- or "learn" or "learns" -- Must scan EVERY row, EVERY character SELECT title, body FROM articles WHERE body LIKE '%learning%';

Cost: Full table scan every time. No index can help a leading wildcard search. Slow on large text columns. No linguistic awareness.

VS

✅ Full-Text Search

-- Finds "learning", "learned", "learns" -- "learner", "learnable" — all forms -- Uses pre-built word index SELECT title, body FROM articles WHERE CONTAINS(body, '"learn*"');

Cost: Index lookup — fast even on millions of rows. Supports stemming, proximity, ranking, and phrase matching.

How the Full-Text Engine Processes Text

When you create a full-text index on a column, SQL Server's Full-Text Engine runs a background process that transforms the text into an inverted word index:

StepProcessExample
1. Word Breaker Tokenises text into individual words based on language rules "Machine learning is powerful" → Machine, learning, is, powerful
2. Stopword Filter Removes common words with no search value (stopwords / noise words) is, the, a, and, of → discarded
3. Stemmer Reduces words to their linguistic root form for inflectional matching learning → root learn; matches learned, learns, learner
4. Thesaurus (optional) Expands terms to synonyms using a configurable XML thesaurus file car → also matches automobile, vehicle
5. Inverted Index Build Stores each unique word with the list of document IDs and positions where it appears machine → [doc3:pos1, doc7:pos4, doc22:pos12]
The Inverted Index: The core of full-text search is the inverted index — a data structure mapping each unique word to the documents containing it (opposite of a forward index that maps documents to their words). When you search for "neural network", the engine looks up both words in the inverted index, then intersects the document lists. This is how search engines like Google work at their core.

Full-Text Index vs Standard B-Tree Index

AspectStandard B-Tree IndexFull-Text Index
Purpose Exact and range comparisons on column values Word, phrase, and linguistic searches within text
Structure Balanced tree of column values → row pointers Inverted word index → document IDs and positions
Maintenance Synchronous — updated immediately on write Asynchronous — crawled and updated on a schedule or manually
Language awareness None — character-by-character comparison Yes — stemming, stopwords, thesaurus, word breaking per language
Supports LIKE '%word%' No — leading wildcard defeats any B-Tree index Yes — equivalent via CONTAINS or FREETEXT
Relevance ranking No Yes — CONTAINSTABLE/FREETEXTTABLE return a rank score
2. Setting Up Full-Text Search in SQL Server
SQL Server Full-Text Search requires: (1) the Full-Text Search feature is installed, (2) a Full-Text Catalog exists to hold the indexes, and (3) a Full-Text Index is created on the target table. The table must have a unique single-column index (usually the primary key) for the full-text engine to track rows.

Step 1 — Verify Full-Text Search Is Installed

-- Check whether the Full-Text Search feature is installed and running SELECT FULLTEXTSERVICEPROPERTY('IsFullTextInstalled') AS is_installed; -- Returns 1 if installed, 0 if not -- If not installed: add via SQL Server Setup > Feature Selection > Full-Text and Semantic Extractions -- Check whether full-text is enabled on the current database SELECT DATABASEPROPERTY(DB_NAME(), 'IsFullTextEnabled') AS ft_enabled_on_this_db;

Step 2 — Create the Sample Table

-- A knowledge base articles table — typical full-text candidate CREATE TABLE kb_articles ( article_id INT IDENTITY PRIMARY KEY, title NVARCHAR(300) NOT NULL, body NVARCHAR(MAX) NOT NULL, author NVARCHAR(100), category VARCHAR(50), published_at DATETIME NOT NULL, language_id INT DEFAULT 1033 -- 1033 = US English ); -- Insert sample data INSERT INTO kb_articles (title, body, author, category, published_at) VALUES ('Introduction to Machine Learning', 'Machine learning is a subset of artificial intelligence that enables systems to learn and improve from experience without being explicitly programmed. It focuses on developing computer programs that can access data and use it to learn for themselves.', 'Dr. Ada Lovelace', 'AI', '2025-01-10'), ('Neural Networks and Deep Learning', 'Deep learning is a machine learning technique that teaches computers to do what comes naturally to humans: learn by example. Deep learning is a key technology behind driverless cars and voice recognition systems.', 'Prof. Alan Turing', 'AI', '2025-01-15'), ('Database Performance Tuning', 'Database performance tuning involves optimizing query execution plans, adding appropriate indexes, and analysing wait statistics to identify bottlenecks. Regular maintenance such as index rebuilds and statistics updates are essential.', 'Jane Doe', 'Databases', '2025-02-01'), ('SQL Server Security Best Practices', 'Securing a SQL Server instance requires a layered approach: network isolation, strong authentication, least-privilege authorization, transparent data encryption, and comprehensive audit logging.', 'John Smith', 'Security', '2025-02-05'), ('Natural Language Processing Overview', 'Natural language processing (NLP) is a branch of artificial intelligence that helps computers understand, interpret, and manipulate human language. NLP draws from many disciplines including linguistics and machine learning.', 'Dr. Ada Lovelace', 'AI', '2025-02-10');

Step 3 — Create a Full-Text Catalog

-- A full-text catalog is a logical container for full-text indexes -- One catalog can hold indexes from multiple tables CREATE FULLTEXT CATALOG KnowledgeBaseCatalog WITH ACCENT_SENSITIVITY = OFF -- 'resume' and 'résumé' treated the same AS DEFAULT; -- use this catalog when none is specified -- Verify the catalog was created SELECT name, path, is_default, is_accent_sensitivity_on FROM sys.fulltext_catalogs;

Step 4 — Create the Full-Text Index

-- Create the full-text index on the title and body columns -- KEY INDEX: the unique index the FT engine uses to identify rows -- LANGUAGE: controls which word breaker and stemmer are used CREATE FULLTEXT INDEX ON kb_articles ( title LANGUAGE 1033, -- US English word breaking for title body LANGUAGE 1033, -- US English word breaking for body author LANGUAGE 1033 -- also index the author name ) KEY INDEX pk_kb_articles -- must reference a unique index ON KnowledgeBaseCatalog -- which catalog to store it in WITH ( CHANGE_TRACKING = AUTO -- automatically crawl changes ); -- Check index status and when it was last populated SELECT OBJECT_NAME(fti.object_id) AS table_name, ftc.name AS catalog_name, fti.change_tracking_state_desc AS change_tracking, ftc.item_count AS indexed_items FROM sys.fulltext_indexes AS fti JOIN sys.fulltext_catalogs AS ftc ON fti.fulltext_catalog_id = ftc.fulltext_catalog_id;

Manual Population (Rebuilding the Index)

-- Force a full re-population (re-crawl and re-index everything) -- Use after bulk loads or when AUTO change tracking is off ALTER FULLTEXT INDEX ON kb_articles START FULL POPULATION; -- Incremental population — only processes rows changed since last crawl -- Requires a timestamp column on the table ALTER FULLTEXT INDEX ON kb_articles START INCREMENTAL POPULATION; -- Update only the tracked changed rows (fast, minimal overhead) ALTER FULLTEXT INDEX ON kb_articles START UPDATE POPULATION;
⚠️ Full-Text Indexes Are Not Synchronously Updated. Unlike B-Tree indexes that update immediately on INSERT/UPDATE/DELETE, full-text indexes are populated by a background crawl process. With CHANGE_TRACKING = AUTO, changes propagate within seconds to minutes. With CHANGE_TRACKING = MANUAL, you must explicitly trigger a population. This means there can be a window where newly inserted data is not yet searchable.
3. CONTAINS — Precise Full-Text Searching
CONTAINS is the primary predicate for full-text searching. It is used in the WHERE clause and returns a boolean (match / no match). It supports simple words, phrases, prefix terms, proximity, inflectional forms, and thesaurus lookups.

Simple Word and Phrase Search

-- Single word: finds all articles containing the word "machine" SELECT article_id, title, category FROM kb_articles WHERE CONTAINS(body, 'machine'); -- Exact phrase: both words must appear adjacent in this order SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, '"machine learning"'); -- Boolean AND: both words must appear somewhere in the column SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, 'machine AND learning'); -- Boolean OR: either word may appear SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, 'machine OR neural'); -- Boolean NOT: word must appear but second must NOT SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, 'machine AND NOT "deep learning"');

Prefix Search — Words Starting With

-- Prefix term: finds "learn", "learning", "learned", "learner", etc. -- The asterisk (*) is a prefix wildcard in full-text syntax SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, '"learn*"'); -- Multiple prefixes with AND/OR SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, '"optim*" OR "perform*"'); -- Matches: optimize, optimizing, optimization, performance, performing...

Inflectional Forms (FORMSOF INFLECTIONAL)

Matches all grammatical inflections of a word — singular/plural, verb tenses — without needing to know each form explicitly.

-- FORMSOF INFLECTIONAL: matches "index", "indexes", "indexed", "indexing" SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, 'FORMSOF(INFLECTIONAL, index)'); -- FORMSOF THESAURUS: matches synonyms from the configured thesaurus -- e.g. "automobile" might also match "car", "vehicle" SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, 'FORMSOF(THESAURUS, optimize)');

Proximity Search (NEAR)

Finds documents where two terms appear near each other — not just anywhere in the document. Closer matches rank higher than distant ones.

-- Simple NEAR: "machine" and "learning" must appear close to each other SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, 'machine NEAR learning'); -- Custom distance NEAR: within 5 words of each other, in order SELECT article_id, title FROM kb_articles WHERE CONTAINS(body, 'NEAR((machine, learning), 5, TRUE)'); -- term1 term2 max_distance order_matters -- Search across MULTIPLE columns — use column list or ALL columns SELECT article_id, title, author FROM kb_articles WHERE CONTAINS((title, body), '"artificial intelligence"'); -- Search ALL full-text indexed columns at once SELECT article_id, title FROM kb_articles WHERE CONTAINS(*, 'Lovelace'); -- searches title, body, AND author
4. FREETEXT — Natural Language Searching
FREETEXT is a simpler, higher-level predicate that interprets the search string as natural language rather than a structured query. SQL Server breaks the search string into words, generates inflectional forms, expands synonyms, and searches for any meaningful combination — no special syntax required from the user.

FREETEXT vs CONTAINS

FeatureCONTAINSFREETEXT

Thesaurus expansionOpt-in with FORMSOFAutomatic
PrecisionHigh — exact controlLower — broader matching
RecallLower — misses variantsHigher — catches more results

-- FREETEXT: "Has anyone written about how systems learn from data?" -- SQL Server extracts: systems, learn, data (drops stopwords) -- Generates inflections: learning, learned, learned, learner... -- Returns any article matching any of these concepts SELECT article_id, title, category FROM kb_articles WHERE FREETEXT(body, 'systems that learn from data automatically'); -- Same query with FREETEXT across multiple columns SELECT article_id, title FROM kb_articles WHERE FREETEXT((title, body), 'artificial intelligence and neural networks'); -- Combine FREETEXT with regular WHERE conditions SELECT article_id, title, category, published_at FROM kb_articles WHERE FREETEXT(body, 'database performance optimization') AND category = 'Databases' AND published_at >= '2025-01-01';
5. Ranked Results — CONTAINSTABLE & FREETEXTTABLE
CONTAINSTABLE and FREETEXTTABLE work like their predicate counterparts but return a table with a relevance rank score (0–1000) for each match. Higher rank = better match. These are used with a JOIN to retrieve ranked results — essential for "most relevant first" search results pages.
-- CONTAINSTABLE: returns a table of (KEY, RANK) pairs -- Join back to the base table to get the full row data SELECT a.article_id, a.title, a.category, ft.RANK AS relevance_score FROM kb_articles AS a JOIN CONTAINSTABLE( kb_articles, -- table body, -- column(s) to search 'machine AND learning' -- search condition ) AS ft ON a.article_id = ft.[KEY] ORDER BY ft.RANK DESC; -- most relevant first -- FREETEXTTABLE: same table-valued output, natural language input -- Limit results with TOP N in the table-valued function SELECT TOP 5 a.article_id, a.title, a.author, ft.RANK AS relevance_score FROM kb_articles AS a JOIN FREETEXTTABLE( kb_articles, (title, body), -- search across both columns 'artificial intelligence machine learning', 10 -- LANGUAGE optional; number limits candidates ) AS ft ON a.article_id = ft.[KEY] ORDER BY ft.RANK DESC;
▶ RESULT — FREETEXTTABLE ranked results

Using Rank for "More Like This": Rank scores allow you to order results by relevance rather than insertion order — just like a search engine. A score close to 1000 means the document is a very strong match; lower scores indicate partial relevance. You can also use rank as a filter with WHERE ft.RANK > 500 to return only high-confidence matches.
6. Stopwords, Stoplists & Configuration

Stopwords and Stoplists

Stopwords are common words that carry no search value and are excluded from the full-text index to save space and improve performance. SQL Server ships with built-in stoplists per language, and you can customise them.

-- View the system stopwords for US English (language 1033) SELECT stopword, language FROM sys.fulltext_system_stopwords WHERE language_id = 1033 ORDER BY stopword; -- Returns words like: a, about, after, all, also, an, and, any, are, as, at, be... -- Create a custom stoplist (copy system list and add your own words) CREATE FULLTEXT STOPLIST CustomStopList FROM SYSTEM STOPLIST; -- start from the built-in list -- Add domain-specific words that appear everywhere but add no search value ALTER FULLTEXT STOPLIST CustomStopList ADD 'example' LANGUAGE 'English'; ALTER FULLTEXT STOPLIST CustomStopList ADD 'note' LANGUAGE 'English'; -- Associate the custom stoplist with the full-text index ALTER FULLTEXT INDEX ON kb_articles SET STOPLIST = CustomStopList;

Configuring the Thesaurus

SQL Server's thesaurus is an XML file per language that maps terms to their synonyms. When FORMSOF(THESAURUS, term) is used, or when FREETEXT auto-expands terms, the thesaurus is consulted.

-- Thesaurus files are located at: -- C:\Program Files\Microsoft SQL Server\MSSQL16.MSSQLSERVER\MSSQL\FTData\ -- File for English: tsenu.xml (ts + enu = thesaurus + English US) -- Example tsenu.xml entry — define synonyms for "database": <!-- XML structure of the thesaurus file --> <XML ID="Microsoft Search Thesaurus"> <thesaurus xmlns="..."> <expansion> <sub>database</sub> <sub>datastore</sub> <sub>data store</sub> <sub>RDBMS</sub> </expansion> </thesaurus> </XML> -- After editing the XML file, reload the thesaurus for English: EXEC sys.sp_fulltext_load_thesaurus_file 1033; -- 1033 = English US

Managing Full-Text Indexes

-- View all full-text indexed columns and their settings SELECT OBJECT_NAME(ftc.object_id) AS table_name, c.name AS column_name, c.system_type_id, ftc.language_id FROM sys.fulltext_index_columns AS ftc JOIN sys.columns AS c ON ftc.object_id = c.object_id AND ftc.column_id = c.column_id; -- Disable (without dropping) the full-text index ALTER FULLTEXT INDEX ON kb_articles DISABLE; -- Re-enable and trigger a full population ALTER FULLTEXT INDEX ON kb_articles ENABLE; -- Drop the full-text index entirely DROP FULLTEXT INDEX ON kb_articles; -- Drop the catalog (all indexes in it must be dropped first) DROP FULLTEXT CATALOG KnowledgeBaseCatalog;
7. When To Use Full-Text Indexing
Full-text indexing is the right choice when your queries need to search within the content of long text fields, and exact-value B-tree indexes cannot help.

Strong Use Cases

📄 Knowledge Bases & Documentation

Support portals, internal wikis, and technical documentation where users search for topics, keywords, and phrases across article bodies containing hundreds to thousands of words.

FREETEXT search box CONTAINSTABLE ranked

🛒 Product Search & Catalogues

E-commerce product descriptions, specifications, and names — users type natural language into a search box and expect relevant results regardless of exact wording.

FREETEXTTABLE ranked

📰 Content Management Systems

News sites, blogs, and publishing platforms where editors and readers search across article bodies, headlines, and metadata for related content.

CONTAINS phrase

🏥 Medical / Legal Records

Free-text clinical notes, case descriptions, and legal documents where searches need to find relevant cases by concept rather than exact phrase — "patient presented with chest pain".

FREETEXT + NEAR

📧 Email & Message Archives

Message bodies, email subjects, and communication logs. Users search for past conversations using approximate recollections — not exact text they wrote months ago.

FORMSOF INFLECTIONAL

🔍 Resume / Job Matching

Matching job descriptions to candidate CVs by skill keywords, job titles, and experience descriptions. Proximity search identifies candidates who mention skills in context, not just incidentally.

NEAR operator

SignalWhy Full-Text Helps

Your text columns are VARCHAR(MAX) or NVARCHAR(MAX) Standard indexes cannot index MAX columns effectively; full-text was designed for this

8. When NOT To Use Full-Text Indexing
Full-text indexing has real overhead and is not always the right tool. Using it for the wrong problem adds complexity and maintenance cost without benefit.

Cases Where Full-Text Is the Wrong Choice

❌ Short Code or ID Fields

Searching for exact values in short fields like product codes (SKU-1042), customer IDs, or phone numbers. A standard unique or non-clustered index is faster and simpler.

Use instead: WHERE sku = 'SKU-1042' with a B-Tree index

❌ Exact String Lookups

When you need an exact character-for-character match — username, email address, account number. Full-text applies word breaking and stopwords, which can interfere with exact matching.

Use instead: Unique non-clustered index + WHERE email = @email

❌ Numeric & Date Ranges

Range queries on numbers and dates (price BETWEEN 10 AND 50, date >= '2025-01-01') are handled perfectly by B-Tree range scans. Full-text cannot index numeric or date columns.

Use instead: Standard index + range WHERE clause

❌ Small Tables

On a table with a few hundred rows, a full table scan with LIKE is nearly instantaneous and simpler to maintain. Full-text overhead — catalog, crawler, index files — is not justified.

Use instead: LIKE search, which is fine at small scale

❌ Near-Real-Time Search Requirements

If your application requires a newly inserted record to be searchable within milliseconds, full-text's asynchronous population can be a problem. Even with AUTO tracking, propagation is not instant.

Use instead: Elasticsearch or Azure Cognitive Search for sub-second indexing

❌ Pattern-Based Searches

Regex-style patterns ("starts with AB followed by any 3 digits"), structured codes, or format validation. Full-text indexes words, not patterns — it cannot replicate LIKE 'AB[0-9][0-9][0-9]'.

Use instead: LIKE with trailing wildcard, or computed column + index

Full-Text vs Dedicated Search Engines

SQL Server's built-in full-text search is suitable for many use cases, but for very large datasets or advanced search requirements, a dedicated search platform may be a better fit:

RequirementSQL Server Full-TextElasticsearch / Azure AI Search
Setup simplicity ✅ In-database, no extra infrastructure ❌ Separate service, more operational complexity
Billions of documents ⚠️ Works but may strain the DB server ✅ Designed for petabyte-scale distributed search
Fuzzy / typo-tolerant search ❌ Not supported natively ✅ Built-in fuzziness parameter
Faceted filtering (e.g. filter by category + price range + rating) ⚠️ Combine with SQL WHERE — workable ✅ Native aggregation buckets and facets

9. Advantages & Disadvantages

✅ Advantages

  • Dramatically faster text search — inverted index lookup vs full table scan; 10×–1000× faster on large data
  • Linguistic intelligence — stemming, inflectional forms, and stopwords mean users get relevant results without needing exact keywords
  • Phrase and proximity matching — find documents where terms appear near each other; impossible with LIKE
  • Relevance ranking — CONTAINSTABLE and FREETEXTTABLE return a rank score for result ordering
  • Built into SQL Server — no separate search service needed; queries stay in T-SQL; uses existing connection security
  • Multi-language support — different word breakers and stemmers per language; one database can hold multilingual content
  • NVARCHAR(MAX) support — can index large text columns that B-Tree indexes cannot handle efficiently
  • Thesaurus customisation — domain-specific synonym expansion (e.g. "heart attack" = "myocardial infarction")
  • Eliminates LIKE '%...%' scans — fixes one of the most common query performance anti-patterns

❌ Disadvantages

  • Asynchronous updates — newly inserted or updated data is not immediately searchable; there is a propagation delay
  • Additional storage — the full-text catalog and index files require significant disk space (can approach the size of the indexed text itself)
  • Maintenance overhead — catalogs must be populated; fragmentation occurs over time; rebuilds are resource-intensive
  • Not exact-match reliable — stopwords, stemming, and word breaking can cause unexpected misses on exact terms (e.g. a stopword in a phrase may be dropped)
  • Limited data type support — cannot full-text index numeric, date, or binary columns directly
  • CPU/memory pressure — full-text queries run in-process and compete with OLTP workloads for server resources
  • No fuzzy/typo matching — a misspelled search term returns no results; dedicated search engines handle typos natively
  • Requires unique key on table — tables without a single-column unique index cannot have a full-text index
  • Debugging complexity — understanding why a document did or didn't match requires understanding the full text pipeline (word breaking, stopwords, stemming)

📌 Implementation Checklist:
  1. Verify Full-Text Search feature is installed (FULLTEXTSERVICEPROPERTY)
  2. Create a Full-Text Catalog (CREATE FULLTEXT CATALOG)
  3. Ensure target table has a single-column unique index (usually the PK)
  4. Create Full-Text Index with CHANGE_TRACKING = AUTO for production
  5. Trigger initial population (START FULL POPULATION)
  6. Use CONTAINS for precise queries; FREETEXT for user search boxes
  7. Use FREETEXTTABLE / CONTAINSTABLE when relevance ranking is needed
  8. Monitor catalog size and schedule population jobs during off-peak hours
  9. Customise stoplist and thesaurus for domain-specific vocabulary
