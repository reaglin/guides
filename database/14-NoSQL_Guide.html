🔍 What Is NoSQL?

NoSQL ("Not Only SQL") is a broad category of database systems that store and retrieve data using models other than the traditional relational table. NoSQL databases were developed to address limitations of relational systems when dealing with massive scale, flexible or evolving schemas, unstructured data, or high-velocity writes.

NoSQL does not mean "no SQL" — many NoSQL systems support SQL-like query languages. It means the system is not constrained to the relational model. NoSQL databases trade some of the strict consistency guarantees of relational databases for gains in scalability, flexibility, and performance in specific workloads.

This guide uses MongoDB — the world's most popular document database — as the primary example, with comparisons to relational SQL throughout.

1. Types of NoSQL Databases
NoSQL is not a single technology — it is a family of different data models. Each type is optimised for a specific kind of data or access pattern. Choosing the right NoSQL type (or deciding to use a relational database instead) depends entirely on what your data looks like and how you need to access it.

📄 Document Stores

Store data as self-contained documents (typically JSON or BSON). Each document can have a different structure — no fixed schema. Documents are grouped into collections (analogous to tables). The most flexible and widely-used NoSQL type.

MongoDB ← this guide CouchDB Amazon DocumentDB Firestore

Best for: User profiles, product catalogues, content management, e-commerce, mobile apps

🔑 Key-Value Stores

The simplest NoSQL model. Every item is stored as a key and its associated value. Blazing fast for simple lookups — the value is opaque to the database (it may be a string, number, JSON, binary, etc.).

Redis Amazon DynamoDB Memcached Riak

Best for: Session caching, shopping carts, real-time leaderboards, pub/sub messaging, rate limiting

📊 Column-Family (Wide-Column) Stores

Data is stored in rows, but columns are grouped into column families and each row can have a different set of columns. Designed for massive distributed write throughput and time-series data across many nodes.

Apache Cassandra HBase Google Bigtable Amazon Keyspaces

Best for: IoT sensor data, time-series metrics, event logging, write-heavy analytics at scale

🕸️ Graph Databases

Store data as nodes (entities) and edges (relationships between nodes). Relationships are first-class citizens — traversing connections is extremely fast compared to JOIN chains in relational databases.

Neo4j Amazon Neptune ArangoDB TigerGraph

Best for: Social networks, recommendation engines, fraud detection, knowledge graphs, network topology

NoSQL vs Relational — At a Glance

CharacteristicRelational (SQL)NoSQL (Document)
Data Model Tables with rows and columns; fixed schema Collections of flexible JSON documents
Schema Schema-on-write — defined before data is inserted Schema-on-read — structure inferred at query time
Relationships Foreign keys, JOINs across tables Embedding data within documents; or manual references
Scaling Primarily vertical (bigger server) Primarily horizontal (more servers / sharding)
Consistency ACID transactions; strong consistency Eventually consistent by default; ACID available in MongoDB 4+
Best Fit Structured business data, financial records, complex queries Variable-structure data, rapid development, high-volume writes
NoSQL is not a replacement for relational databases — it is an alternative optimised for different problems. Many modern applications use both: a relational database for transactional data (orders, accounts) and a document database for flexible content (user profiles, logs, product details).
2. MongoDB Core Concepts — Documents, Collections & BSON
MongoDB stores data as BSON documents (Binary JSON) organised into collections inside a database. This hierarchy maps loosely onto the relational model but with key differences.

The MongoDB Hierarchy

MongoDB ConceptRelational EquivalentDescription
Database Database A named container for collections
Collection Table A group of documents — no enforced schema by default
Document Row A single JSON/BSON object — the unit of storage
Field Column A key-value pair within a document
_id Primary Key Auto-generated unique identifier (ObjectId by default)
Embedded document Foreign key + JOIN A nested document inside another document
Array field One-to-many table A list of values or sub-documents inside a document

A MongoDB Document

Documents look like JSON objects. Unlike a table row, a document can contain nested objects and arrays — entire related data sets can live inside a single document.

// A product document in the "products" collection { "_id": ObjectId("64b7c3e2f4a5d10023abc001"), "sku": "LAPTOP-PRO-15", "name": "ProBook 15 Laptop", "category": "Electronics", "price": 1299.99, "in_stock": true, "qty_available": 47, // Embedded sub-document — no separate "specs" table needed "specs": { "cpu": "Intel Core i7-13700H", "ram_gb": 16, "ssd_gb": 512, "display": "15.6 inch 1920x1080" }, // Array of strings — no separate tags table needed "tags": [ "laptop", "windows", "portable", "business" ], // Array of sub-documents — no separate reviews table needed "reviews": [ { "user": "jdoe42", "rating": 5, "comment": "Excellent build quality and performance.", "date": ISODate("2025-02-01") }, { "user": "techfan99", "rating": 4, "comment": "Great laptop, battery could be better.", "date": ISODate("2025-02-08") } ], "created_at": ISODate("2025-01-15T09:30:00Z"), "updated_at": ISODate("2025-02-10T14:22:00Z") }

3. Create — Inserting Documents
insertOne insertMany MongoDB creates a collection automatically the first time you insert into it — no CREATE TABLE statement required. Each inserted document receives an _id field automatically if one is not provided.

insertOne — Add a Single Document

// Insert one book document into the "books" collection db.books.insertOne({ "title": "The Pragmatic Programmer", "authors": ["David Thomas", "Andrew Hunt"], "isbn": "978-0135957059", "year": 2019, "price": 49.95, "category": "Technology", "in_stock": true, "pages": 352, "tags": ["programming", "software", "best-practices"] })
{ acknowledged: true, insertedId: ObjectId('64b7c3e2f4a5d10023abc002') }

insertMany — Add Multiple Documents at Once

// Insert several books in one operation db.books.insertMany([ { "title": "Clean Code", "authors": ["Robert C. Martin"], "isbn": "978-0132350884", "year": 2008, "price": 44.99, "category": "Technology", "in_stock": true, "pages": 431 }, { "title": "Design Patterns", "authors": ["Gang of Four"], "isbn": "978-0201633610", "year": 1994, "price": 54.99, "category": "Technology", "in_stock": false, "pages": 395 }, { "title": "Dune", "authors": ["Frank Herbert"], "isbn": "978-0441013593", "year": 1965, "price": 12.99, "category": "Fiction", "in_stock": true, "pages": 896 } ])
Schema Flexibility in Action: Notice that the books above have slightly different field sets — that is perfectly valid in MongoDB. One book might have a "tags" field, another might not. There is no error for missing fields, and no ALTER TABLE required to add a new field.
4. Read — Querying Documents
find findOne MongoDB queries use a filter document — a JSON object specifying which fields must match. An empty filter {} returns all documents. Projection controls which fields are returned.

Basic find — Return All or Filtered Documents

// Return ALL documents in the collection db.books.find() // Return all books in the Fiction category db.books.find({ "category": "Fiction" }) // findOne — return only the first matching document db.books.findOne({ "title": "Dune" }) // Multiple conditions — implicit AND (both must be true) db.books.find({ "category": "Technology", "in_stock": true })

Projection — Selecting Which Fields to Return

The second argument to find() is a projection document. Use 1 to include a field and 0 to exclude it. You cannot mix includes and excludes except for _id.

// Return only title and price — exclude _id db.books.find( { "category": "Technology" }, { "title": 1, "price": 1, "_id": 0 } // include title+price, exclude _id )
[ { title: 'The Pragmatic Programmer', price: 49.95 }, { title: 'Clean Code', price: 44.99 }, { title: 'Design Patterns', price: 54.99 } ]

Comparison Operators

OperatorMeaningSQL Equivalent
$eqEqual to= value
$neNot equal<> value
$gtGreater than> value
$gteGreater than or equal>= value
$ltLess than< value
$lteLess than or equal<= value
$inValue in a listIN (a, b, c)
$ninValue NOT in a listNOT IN (a, b, c)
// Books priced between $10 and $50 db.books.find({ "price": { $gte: 10, $lte: 50 } }) // Books published after 2000 db.books.find({ "year": { $gt: 2000 } }) // Books in a specific category list db.books.find({ "category": { $in: ["Fiction", "Biography"] } })

Logical Operators

// $and — both conditions must be true (explicit AND) db.books.find({ $and: [ { "category": "Technology" }, { "price": { $lt: 50 } } ] }) // $or — at least one condition must be true db.books.find({ $or: [ { "category": "Fiction" }, { "price": { $lt: 15 } } ] }) // $not — negate a condition db.books.find({ "in_stock": { $not: { $eq: true } } })

Querying Arrays and Embedded Documents

// Find books where the tags array CONTAINS "programming" db.books.find({ "tags": "programming" }) // $all — array must contain ALL of these values db.books.find({ "tags": { $all: ["programming", "software"] } }) // Dot notation — query inside an embedded document // Find books where specs.ram_gb is 16 or more db.products.find({ "specs.ram_gb": { $gte: 16 } }) // $elemMatch — match array element meeting multiple conditions // Books with at least one review rated 5 stars db.products.find({ "reviews": { $elemMatch: { "rating": 5 } } })

Sorting, Limiting, and Skipping

// Sort by price ascending (1) or descending (-1) db.books.find().sort({ "price": 1 }) // cheapest first db.books.find().sort({ "price": -1 }) // most expensive first // Limit — return at most 5 results db.books.find().sort({ "price": -1 }).limit(5) // Skip + Limit — pagination (page 2, 10 items per page) db.books.find() .sort({ "title": 1 }) .skip(10) // skip first 10 .limit(10) // return next 10 // Count matching documents db.books.countDocuments({ "category": "Technology" })
5. Update — Modifying Documents
updateOne updateMany replaceOne MongoDB updates use update operators to describe what should change. The $set operator is the most common — it updates or adds specific fields without touching the rest of the document.

Key Update Operators

OperatorEffectExample
$setSet a field to a value (add if not exists){ $set: { "price": 39.99 } }
$unsetRemove a field from the document{ $unset: { "old_code": "" } }
$incIncrement a numeric field by N{ $inc: { "qty": -1 } }
$mulMultiply a numeric field by N{ $mul: { "price": 0.9 } }
$renameRename a field{ $rename: { "colour": "color" } }
$pushAppend a value to an array{ $push: { "tags": "sale" } }
$pullRemove matching values from an array{ $pull: { "tags": "discontinued" } }
$addToSetAdd to array only if value not already present{ $addToSet: { "tags": "featured" } }

updateOne — Modify the First Matching Document

// Update the price of "Clean Code" db.books.updateOne( { "title": "Clean Code" }, // filter: which document { $set: { "price": 39.99, "on_sale": true } } // update: what to change ) // Decrement stock quantity when a book is sold db.books.updateOne( { "isbn": "978-0135957059" }, { $inc: { "qty_available": -1 } } ) // Add a tag to the tags array db.books.updateOne( { "title": "Dune" }, { $addToSet: { "tags": "sci-fi" } } // only adds if not already there ) // Push a new review sub-document into the reviews array db.products.updateOne( { "sku": "LAPTOP-PRO-15" }, { $push: { "reviews": { "user": "alice_m", "rating": 5, "comment": "Perfect for coding. Highly recommend.", "date": new Date() } } } )

updateMany — Modify All Matching Documents

// Apply a 10% price reduction to all out-of-stock books db.books.updateMany( { "in_stock": false }, { $mul: { "price": 0.9 } } // multiply all prices by 0.9 (10% off) ) // Add a "last_reviewed" field to ALL documents that don't have it db.books.updateMany( { "last_reviewed": { $exists: false } }, { $set: { "last_reviewed": new Date() } } ) // upsert: update if exists, insert if not found db.books.updateOne( { "isbn": "978-9999999999" }, { $set: { "title": "New Release", "price": 29.99 } }, { "upsert": true } // creates a new document if no match found )
6. Delete — Removing Documents
deleteOne deleteMany Deletion in MongoDB is permanent and immediate — there is no ROLLBACK unless you are inside an explicit transaction. Always verify your filter with a find() before running deleteMany().
// Delete one specific book by ISBN db.books.deleteOne({ "isbn": "978-0201633610" }) // Delete all books that have been out of stock for over a year // First VERIFY with find(): db.books.find({ "in_stock": false, "last_updated": { $lt: new Date("2024-01-01") } }) // Then DELETE after confirming the results look right: db.books.deleteMany({ "in_stock": false, "last_updated": { $lt: new Date("2024-01-01") } }) // Drop an entire collection (USE WITH EXTREME CAUTION) db.books.drop() // Drop the entire database db.dropDatabase()
⚠️ deleteMany({}) deletes every document in the collection — the equivalent of DELETE FROM table without a WHERE clause. Always test your filter with find() first. In production, consider using a "deleted": true soft-delete field instead of permanent deletion.
7. The Aggregation Pipeline — Analytics in MongoDB
aggregate The aggregation pipeline processes documents through a sequence of stages. Each stage transforms the documents and passes the result to the next stage. This is MongoDB's answer to SQL's GROUP BY, JOIN, HAVING, and complex SELECT expressions.

Core Pipeline Stages

StageSQL EquivalentWhat It Does
$matchWHEREFilter documents — put early to reduce data volume
$groupGROUP BY + aggregatesGroup documents and compute aggregate values
$projectSELECT columnsInclude, exclude, or compute fields
$sortORDER BYSort the result documents
$limitTOP N / LIMITReturn only the first N documents
$skipOFFSETSkip the first N documents
$unwindUNNEST / LATERAL JOINExpand an array — one document per array element
$lookupLEFT JOINJoin documents from another collection
$addFieldsComputed column in SELECTAdd new computed fields without removing others
$countCOUNT(*)Count documents passing through this stage

Example 1 — Revenue by Category (GROUP BY equivalent)

// SQL equivalent: // SELECT category, COUNT(*) AS book_count, // AVG(price) AS avg_price, SUM(price) AS total_value // FROM books // WHERE in_stock = true // GROUP BY category // ORDER BY total_value DESC db.books.aggregate([ // Stage 1: filter to in-stock books only { $match: { "in_stock": true } }, // Stage 2: group by category and compute aggregates { $group: { "_id": "$category", // group key "book_count": { $sum: 1 }, // COUNT(*) "avg_price": { $avg: "$price" }, // AVG(price) "total_value":{ $sum: "$price" }, // SUM(price) "min_price": { $min: "$price" }, // MIN(price) "max_price": { $max: "$price" } // MAX(price) } }, // Stage 3: clean up the output fields { $project: { "category": "$_id", "book_count": 1, "avg_price": { $round: ["$avg_price", 2] }, "total_value":{ $round: ["$total_value", 2] }, "_id": 0 // hide the internal _id field } }, // Stage 4: sort by total_value descending { $sort: { "total_value": -1 } } ])
[ { category: 'Technology', book_count: 2, avg_price: 47.47, total_value: 94.94 }, { category: 'Fiction', book_count: 1, avg_price: 12.99, total_value: 12.99 } ]

Example 2 — $unwind: Analysing Array Elements

$unwind expands an array field so each element becomes its own document. Use it to analyse individual items within arrays.

// Count how many books have each tag // (tags is an array — $unwind creates one doc per tag) db.books.aggregate([ { $unwind: "$tags" }, // expand tags array { $group: { "_id": "$tags", "tag_count": { $sum: 1 } } }, { $sort: { "tag_count": -1 } }, { $limit: 10 } ])

Example 3 — $lookup: Joining Two Collections

// $lookup is the LEFT JOIN equivalent // Assume: "orders" collection has a customer_id field // "customers" collection has _id and name // Get each order with the full customer document attached db.orders.aggregate([ { $lookup: { "from": "customers", // collection to join "localField": "customer_id", // field in orders "foreignField": "_id", // field in customers "as": "customer" // name for the joined array } }, // $unwind to flatten the joined array (LEFT JOIN result) { $unwind: { "path": "$customer", "preserveNullAndEmptyArrays": true } }, // Shape the output { $project: { "order_date": 1, "total": 1, "customer_name": "$customer.name", "customer_email":"$customer.email" } } ])
8. Data Modelling — Embedding vs Referencing
The most important design decision in MongoDB is whether to embed related data inside a document or reference it from a separate collection. Unlike relational databases where normalisation drives schema design, MongoDB design is driven by how the data is accessed.

Embedding — Related Data Inside One Document

// EMBEDDED: a customer and their addresses in one document // Best when: addresses are always retrieved with the customer, // a customer has a small, bounded number of addresses { "_id": ObjectId("..."), "name": "Jane Smith", "email": "jane@example.com", "addresses": [ { "type": "home", "street": "123 Maple St", "city": "Tampa", "state": "FL", "zip": "33601" }, { "type": "work", "street": "400 University Dr", "city": "Tampa", "state": "FL", "zip": "33620" } ] }

Referencing — Linked Documents Across Collections

// REFERENCED: orders stored in a separate collection, // linked to customers by customer_id. // Best when: orders are queried independently of customers, // or a customer could have thousands of orders // Customer document: { "_id": ObjectId("64b7c3e2f4a5d10023abc010"), "name": "Jane Smith", "email": "jane@example.com" } // Order document (separate collection): { "_id": ObjectId("64b7c3e2f4a5d10023abc099"), "customer_id": ObjectId("64b7c3e2f4a5d10023abc010"), // reference "order_date": ISODate("2025-03-15"), "status": "Completed", "total": 94.94 }

✅ Use Embedding When:

  • Related data is always accessed together (one-to-one or one-to-few)
  • The embedded data has a bounded size (won't grow to thousands of items)
  • The embedded data is not shared between multiple parent documents
  • You want single-document reads — no join required
  • Examples: addresses, order line items, product specs, review list (small)

✅ Use Referencing When:

  • Related data is queried independently or accessed separately
  • The relationship is one-to-many with unbounded growth (many orders per customer)
  • The related data is shared by multiple documents (same author for many books)
  • Embedded data would make documents exceed 16MB (MongoDB document size limit)
  • Examples: orders, events, log entries, comments on popular content
9. Indexes in MongoDB
createIndex MongoDB indexes work on the same B-tree principle as relational indexes. Without an index, every query performs a collection scan — reading every document. With the right index, MongoDB performs a fast index scan. Every collection has a default index on _id.
// Single-field index — speeds up queries filtering on category db.books.createIndex({ "category": 1 }) // 1 = ascending, -1 = descending // Compound index — covers queries filtering on both fields db.books.createIndex({ "category": 1, "price": -1 }) // Unique index — enforces uniqueness (like a UNIQUE CONSTRAINT) db.books.createIndex({ "isbn": 1 }, { "unique": true }) // Text index — enables full-text search on string fields db.books.createIndex({ "title": "text", "description": "text" }) // Use the text index with $text search db.books.find({ $text: { $search: "pragmatic programming" } }) // TTL index — automatically deletes documents after a time period // Useful for session data, logs, temporary records db.sessions.createIndex( { "created_at": 1 }, { "expireAfterSeconds": 3600 } // auto-delete after 1 hour ) // View all indexes on a collection db.books.getIndexes() // Drop an index by name db.books.dropIndex("category_1_price_-1") // Explain a query — shows if an index is being used db.books.find({ "category": "Technology" }).explain("executionStats")
💡 Checking for a COLLSCAN in explain(): When explain() reports a COLLSCAN (collection scan) on a large collection, you need an index. Look for IXSCAN to confirm an index was used. The totalDocsExamined vs nReturned ratio tells you how efficient the query is.

11. When to Choose NoSQL vs Relational
There is no universally superior database type — the right choice depends on your data characteristics, access patterns, consistency requirements, and team familiarity. Many production systems use both.

Choose MongoDB (Document Store) When...

  • Data structure varies between records — products with different attributes, user profiles with optional fields, multi-tenant data with per-customer schemas
  • Data is naturally hierarchical or nested — orders with line items, blog posts with comments, medical records with sections
  • Schema evolves rapidly — early-stage applications where the data model changes frequently
  • High write volume at scale — MongoDB shards horizontally across many nodes
  • Data is best read as a complete unit — fetch one document and have everything you need, no JOINs
  • Geospatial queries — MongoDB has native 2D sphere index support for location-based queries

Choose a Relational Database (SQL Server, PostgreSQL) When...

  • Data is highly structured and consistent — financial transactions, accounting, payroll, inventory
  • Complex relationships and multi-table JOINs are central to how the data is used
  • ACID transactions across multiple tables are required — banking, order processing
  • Data integrity constraints are critical — foreign keys, check constraints, not-null enforcement
  • Complex reporting and ad-hoc analysis — SQL's GROUP BY and window functions are more expressive for analytics
  • Team expertise is in SQL — relational databases have more mature tooling and wider knowledge bases

Decision Summary Table

FactorFavour SQLFavour MongoDB
Schema Fixed, well-defined, stable Variable, evolving, flexible
Data shape Flat, many tables, normalised Hierarchical, nested, self-contained
Relationships Complex many-to-many, referential integrity Simple references or embedded data
Transactions Multi-table ACID required Single-document operations; multi-doc transactions available
Scale Vertical scaling; hundreds of GB Horizontal sharding; terabytes to petabytes
Query complexity Complex analytics, multi-join reports Document retrieval, simple aggregations
Development speed Schema changes require migration planning Add fields instantly, no migrations required
💡 Polyglot Persistence: Large-scale applications routinely use multiple database types together. A typical architecture might use PostgreSQL for orders and accounts (ACID, relational), MongoDB for product catalogues and user content (flexible schema), Redis for session caching (key-value, sub-millisecond), and Elasticsearch for full-text product search. Each database is chosen for its strengths in the specific use case.
