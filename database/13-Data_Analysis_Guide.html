COP4709 — Database Management

🔍 Databases as the Foundation of Data Analysis

Data analysis is the process of inspecting, cleaning, transforming, and modelling data to discover useful information, draw conclusions, and support decision making. Databases are the primary system of record for organisational data — they store it, structure it, and make it queryable at scale.

While dedicated analytics platforms (Power BI, Tableau, Python pandas) are commonly used for visualisation and advanced modelling, the core data extraction and preparation work almost always originates with SQL against a relational database. Understanding how to use a database analytically is a fundamental skill for any role that deals with data-driven decisions.

This guide focuses on the SQL-based analytical work that feeds regular business reporting: the types of data involved, how to prepare it for analysis, and the patterns of queries that produce the cyclic reports organisations run every day.

1. Where Databases Are Used for Data Analysis

Analytical vs Transactional Use

Databases serve two fundamentally different roles. Understanding the distinction is important because the same database that powers a business application often also serves as the source for its reports.

🔄 Transactional (OLTP)

Purpose: Record and process live business events
Queries: Many short INSERT/UPDATE/SELECT
Data: Current state, high concurrency
Examples: Order entry, banking, healthcare records, e-commerce checkout
Optimised for: Write speed, data integrity, concurrency

📊 Analytical (OLAP)

Purpose: Aggregate and summarise data for insight
Queries: Complex SELECT over large row counts
Data: Historical snapshots, often denormalised
Examples: Revenue reports, trend analysis, dashboards, forecasting
Optimised for: Read speed, large aggregations, flexibility

In practice, most organisations run analytical queries directly against their OLTP database — especially during off-peak hours or using read replicas. Larger organisations separate the two into a dedicated data warehouse. Either way, the SQL skills for analysis are the same.

2. Types of Data Used in Analysis
Before analysing data, you need to understand what kind of data you are working with. Different data types support different analytical operations. Applying the wrong analysis to a data type (for example, averaging a categorical field) produces meaningless results.

Quantitative vs Qualitative Data

📐 Quantitative (Numeric)

Data that represents a measurable quantity. Mathematical operations (SUM, AVG, MIN, MAX) are meaningful.

  • Continuous — any value in a range (price: $42.75, temperature: 98.6°F)
  • Discrete — whole number counts (order qty: 3, employees: 147)

SUM AVG MIN/MAX STDDEV

🏷️ Qualitative (Categorical)

Data that represents categories, labels, or descriptions. COUNT operations are meaningful; arithmetic is not.

  • Nominal — no natural order (colour, region, product category)
  • Ordinal — ordered categories (rating: Low/Medium/High, survey: 1–5)

COUNT COUNT DISTINCT GROUP BY

Structured, Semi-Structured, and Unstructured Data

TypeDescriptionDatabase StorageAnalytical Use

Unstructured Free-form content with no defined schema BLOBs, file paths, text columns External NLP/text analysis tools; SQL handles metadata

Time-Series Data

One of the most important data types for business analysis. A time series is a sequence of values recorded at regular time intervals. Most business metrics are time-series in nature — revenue by day, users by month, inventory by week.

📅 Event Timestamps

A datetime column recording when something happened: order_date, login_time, created_at. Used to group events into time buckets.

📆 Snapshot Records

A record taken at a regular interval: daily stock price, end-of-day inventory count, monthly account balance. One row per entity per period.

⏱️ Duration Data

Records a start and end time: session_start / session_end, ticket open/close. Analysed for duration, frequency, and concurrency.

Dimensions and Measures

In analytical thinking, every column in a dataset is either a dimension or a measure. This framing guides how you write GROUP BY queries.

📦 Dimensions (Grouping Attributes)

Categorical columns you slice and dice data by. They answer "which?" and "what kind?"

  • Region, country, city
  • Product name, category, brand
  • Customer segment, age group
  • Date, month, quarter, year
  • Department, team, employee

📏 Measures (Numeric Values)

Numeric columns you aggregate. They answer "how much?" and "how many?"

  • Revenue, cost, profit, margin
  • Quantity sold, units returned
  • Session duration, page views
  • Headcount, absences
  • Score, rating, temperature
The core analytical query pattern: SELECT dimension, AGGREGATE(measure) FROM table GROUP BY dimension — choose which dimension to slice by, choose which measure to aggregate. This pattern generates the majority of business reports.

Common Data Types in Analytical Tables

-- Example: a typical sales fact table with common analytical data types CREATE TABLE fact_sales ( sale_id INT PRIMARY KEY, -- Date/time dimensions — used for time-series grouping sale_date DATE NOT NULL, sale_timestamp DATETIME NOT NULL, -- Categorical dimensions — used in GROUP BY region VARCHAR(50), product_category VARCHAR(100), sales_channel VARCHAR(30), -- 'Online', 'In-Store', 'Wholesale' customer_segment VARCHAR(30), -- 'Retail', 'Corporate', 'Government' -- Ordinal dimension (ordered categories) satisfaction_score TINYINT, -- 1-5 rating -- Numeric measures — aggregated with SUM, AVG, MIN, MAX quantity INT NOT NULL, unit_price DECIMAL(8,2) NOT NULL, discount_pct DECIMAL(5,2), revenue DECIMAL(10,2), -- pre-calculated: qty * unit_price * (1 - discount) cost DECIMAL(10,2), profit DECIMAL(10,2) -- revenue - cost );
3. Preparing & Cleaning Data for Analysis
Raw data from operational systems is rarely clean enough to analyse directly. Studies consistently report that data preparation takes 60–80% of the time in a typical analysis project. Skipping this step produces misleading results. The rule of thumb: garbage in, garbage out.

Common Data Quality Problems

❌ NULL / Missing Values

A column has no recorded value. Aggregations that ignore NULLs silently change the denominator. Must decide: exclude, substitute zero, or substitute a default.

❌ Duplicate Rows

The same event recorded twice — from re-imports, system errors, or ETL bugs. Causes counts and sums to be overstated. Must deduplicate before aggregating.

❌ Inconsistent Formats

Same value represented differently: 'USA', 'US', 'United States', 'U.S.A.' — all refer to the same country but will create four separate groups in a GROUP BY.

❌ Wrong Data Types

Dates stored as strings ('2025-01-15' in a VARCHAR column). Prevents date arithmetic and range filtering. Must convert before use.

❌ Outliers

Extreme values that distort averages: a $1,000,000 test order inflating the mean order value. Must identify and decide: exclude, cap, or investigate.

❌ Out-of-Range Values

Values that fail a logical sanity check: a negative quantity, a birth date in the future, a discount of 150%. Indicate data entry errors or system bugs.

Handling NULL Values

-- Scenario: discount_pct is NULL when no discount was applied -- NULL means "unknown" — not zero. Different treatments have different meanings. -- ❌ AVG(discount_pct) silently excludes NULLs from the count -- giving the average discount among orders THAT HAD a discount SELECT AVG(discount_pct) AS avg_discount FROM fact_sales; -- ✅ COALESCE replaces NULL with 0, giving the average across ALL orders -- (treating no discount as a 0% discount — often what we want) SELECT AVG(COALESCE(discount_pct, 0)) AS avg_discount_all_orders FROM fact_sales; -- Check for NULL prevalence before deciding how to handle it SELECT COUNT(*) AS total_rows, COUNT(discount_pct) AS non_null_discount, COUNT(*) - COUNT(discount_pct) AS null_count, ROUND( (COUNT(*) - COUNT(discount_pct)) * 100.0 / COUNT(*), 1) AS null_pct FROM fact_sales;

Removing Duplicates

-- Identify exact duplicate rows SELECT sale_date, customer_id, product_id, quantity, revenue, COUNT(*) AS occurrences FROM fact_sales GROUP BY sale_date, customer_id, product_id, quantity, revenue HAVING COUNT(*) > 1 ORDER BY occurrences DESC; -- Deduplicate using ROW_NUMBER — keep the first occurrence WITH ranked_sales AS ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY sale_date, customer_id, product_id, quantity ORDER BY sale_id ) AS row_num FROM fact_sales ) SELECT * FROM ranked_sales WHERE row_num = 1; -- only the first row per duplicate group

Standardising Inconsistent Category Values

-- Problem: same country stored with different spellings/formats SELECT region, COUNT(*) AS row_count FROM fact_sales GROUP BY region ORDER BY region; -- Returns: 'US', 'USA', 'United States', 'united states', 'U.S.' — 5 separate groups -- Fix with CASE to normalise on the fly SELECT CASE WHEN UPPER(TRIM(region)) IN ('US', 'USA', 'UNITED STATES', 'U.S.', 'U.S.A.') THEN 'United States' WHEN UPPER(TRIM(region)) IN ('UK', 'UNITED KINGDOM', 'GREAT BRITAIN', 'GB') THEN 'United Kingdom' WHEN UPPER(TRIM(region)) IN ('CA', 'CAN', 'CANADA') THEN 'Canada' ELSE TRIM(region) END AS region_standardised, COUNT(*) AS row_count, SUM(revenue) AS total_revenue FROM fact_sales GROUP BY CASE WHEN UPPER(TRIM(region)) IN ('US', 'USA', 'UNITED STATES', 'U.S.', 'U.S.A.') THEN 'United States' WHEN UPPER(TRIM(region)) IN ('UK', 'UNITED KINGDOM', 'GREAT BRITAIN', 'GB') THEN 'United Kingdom' WHEN UPPER(TRIM(region)) IN ('CA', 'CAN', 'CANADA') THEN 'Canada' ELSE TRIM(region) END ORDER BY total_revenue DESC;

Converting and Formatting Dates

-- Problem: dates stored as strings in a VARCHAR column SELECT sale_date_raw, -- '20250115' or '15/01/2025' or 'Jan 15 2025' CONVERT(DATE, sale_date_raw, 112) AS as_date_yyyymmdd, CONVERT(DATE, sale_date_raw, 103) AS as_date_ddmmyyyy, TRY_CONVERT(DATE, sale_date_raw) AS safe_convert -- NULL on failure vs error FROM raw_imports; -- Useful date parts to extract for grouping SELECT sale_date, YEAR(sale_date) AS year, MONTH(sale_date) AS month_num, DATENAME(MONTH, sale_date) AS month_name, DATEPART(QUARTER, sale_date) AS quarter, DATEPART(WEEK, sale_date) AS week_number, DATENAME(WEEKDAY, sale_date) AS day_of_week, EOMONTH(sale_date) AS last_day_of_month FROM fact_sales;

Detecting and Handling Outliers

-- Find statistical outliers using mean and standard deviation -- Values more than 3 standard deviations from the mean are candidate outliers WITH stats AS ( SELECT AVG(revenue) AS mean_rev, STDEV(revenue) AS stdev_rev FROM fact_sales WHERE revenue > 0 ) SELECT fs.sale_id, fs.sale_date, fs.revenue, ROUND((fs.revenue - s.mean_rev) / s.stdev_rev, 2) AS z_score FROM fact_sales AS fs CROSS JOIN stats AS s WHERE ABS(fs.revenue - s.mean_rev) > (3 * s.stdev_rev) ORDER BY ABS(fs.revenue - s.mean_rev) DESC;
💡 Data Profiling — Know Your Data Before You Analyse It: Before writing analytical queries, run a quick profile of your dataset. Check row counts, NULL rates, min/max values, and distinct value counts for key columns. This prevents surprises that invalidate your analysis results.
-- Quick data profile — run this before any analysis SELECT COUNT(*) AS total_rows, COUNT(DISTINCT customer_id) AS unique_customers, COUNT(DISTINCT product_category) AS unique_categories, MIN(sale_date) AS earliest_date, MAX(sale_date) AS latest_date, MIN(revenue) AS min_revenue, MAX(revenue) AS max_revenue, ROUND(AVG(revenue), 2) AS avg_revenue, COUNT(*) - COUNT(revenue) AS null_revenue_count, SUM(CASE WHEN revenue <= 0 THEN 1 ELSE 0 END) AS zero_or_negative_revenue FROM fact_sales;
4. Descriptive Analysis — Summarising What Happened
Descriptive analysis answers the question: What happened? It uses aggregation to compress large volumes of raw events into meaningful summary numbers. It is the most common type of analysis in business reporting — the foundation of every dashboard and weekly report.

The Standard Summary Report

Most daily and weekly reports are built on a small set of aggregation patterns. Here is a complete monthly sales summary using all standard descriptive measures:

-- Monthly sales performance summary report SELECT YEAR(sale_date) AS year, MONTH(sale_date) AS month, DATENAME(MONTH, sale_date) AS month_name, -- Volume metrics COUNT(*) AS total_transactions, COUNT(DISTINCT customer_id) AS unique_customers, SUM(quantity) AS units_sold, -- Revenue metrics ROUND(SUM(revenue), 2) AS total_revenue, ROUND(SUM(cost), 2) AS total_cost, ROUND(SUM(profit), 2) AS total_profit, -- Average metrics ROUND(AVG(revenue), 2) AS avg_transaction_value, ROUND(AVG(quantity), 1) AS avg_units_per_transaction, -- Margin (profit as a % of revenue) ROUND( SUM(profit) * 100.0 / NULLIF(SUM(revenue), 0), 1) AS profit_margin_pct FROM fact_sales WHERE sale_date >= '2025-01-01' GROUP BY YEAR(sale_date), MONTH(sale_date), DATENAME(MONTH, sale_date) ORDER BY year, month;

-- Revenue breakdown by product category for a given month SELECT product_category, COUNT(*) AS transactions, SUM(quantity) AS units_sold, ROUND(SUM(revenue), 2) AS revenue, ROUND( SUM(revenue) * 100.0 / SUM(SUM(revenue)) OVER(), -- window function: % of total 1) AS revenue_share_pct FROM fact_sales WHERE sale_date >= '2025-03-01' AND sale_date < '2025-04-01' GROUP BY product_category ORDER BY revenue DESC;

Conditional Counting with CASE WHEN

One of the most powerful descriptive analysis patterns — pivot a column of values into separate count columns in a single query.

-- Order status breakdown — count each status as its own column SELECT YEAR(sale_date) AS year, MONTH(sale_date) AS month, COUNT(*) AS total_orders, SUM(CASE WHEN status = 'Completed' THEN 1 ELSE 0 END) AS completed, SUM(CASE WHEN status = 'Cancelled' THEN 1 ELSE 0 END) AS cancelled, SUM(CASE WHEN status = 'Returned' THEN 1 ELSE 0 END) AS returned, ROUND( SUM(CASE WHEN status = 'Cancelled' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) AS cancellation_rate_pct FROM fact_sales GROUP BY YEAR(sale_date), MONTH(sale_date) ORDER BY year, month;
5. Trend & Time-Series Analysis — Understanding Change Over Time
Trend analysis answers: Is this metric going up, down, or staying flat? Most business decisions require understanding not just the current state, but whether the current state is better or worse than before — and by how much. Period-over-period comparison is the most commonly requested analysis in any reporting context.

Period-Over-Period Comparison (Month vs Prior Month)

-- Month-over-month revenue with absolute and percentage change -- LAG() retrieves the value from the previous row (window function) WITH monthly AS ( SELECT YEAR(sale_date) AS yr, MONTH(sale_date) AS mo, ROUND(SUM(revenue), 2) AS revenue FROM fact_sales GROUP BY YEAR(sale_date), MONTH(sale_date) ) SELECT yr, mo, revenue AS this_month_revenue, LAG(revenue) OVER (ORDER BY yr, mo) AS prior_month_revenue, revenue - LAG(revenue) OVER (ORDER BY yr, mo) AS mom_change, ROUND( (revenue - LAG(revenue) OVER (ORDER BY yr, mo)) * 100.0 / NULLIF(LAG(revenue) OVER (ORDER BY yr, mo), 0), 1) AS mom_change_pct FROM monthly ORDER BY yr, mo;

Year-Over-Year Comparison (Same Month, Prior Year)

-- YoY: compare each month to the same month 12 months ago -- LAG() with offset of 12 skips back one full year of monthly rows WITH monthly AS ( SELECT YEAR(sale_date) AS yr, MONTH(sale_date) AS mo, ROUND(SUM(revenue), 2) AS revenue FROM fact_sales GROUP BY YEAR(sale_date), MONTH(sale_date) ) SELECT yr, mo, revenue AS this_year_revenue, LAG(revenue, 12) OVER (ORDER BY yr, mo) AS prior_year_revenue, ROUND( (revenue - LAG(revenue, 12) OVER (ORDER BY yr, mo)) * 100.0 / NULLIF(LAG(revenue, 12) OVER (ORDER BY yr, mo), 0), 1) AS yoy_growth_pct FROM monthly ORDER BY yr, mo;

Running Totals (Cumulative Sum)

A running total accumulates values over time — useful for tracking year-to-date (YTD) progress against a target.

-- Daily cumulative (running total) revenue for the current year -- SUM() OVER with ROWS UNBOUNDED PRECEDING accumulates from the start SELECT sale_date, ROUND(SUM(revenue), 2) AS daily_revenue, ROUND( SUM(SUM(revenue)) OVER ( ORDER BY sale_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ), 2) AS ytd_revenue FROM fact_sales WHERE YEAR(sale_date) = 2025 GROUP BY sale_date ORDER BY sale_date;

Rolling Average (Moving Average)

A moving average smooths out day-to-day volatility to reveal the underlying trend. A 7-day moving average is standard for daily data.

-- 7-day rolling average of daily revenue -- Smooths weekly seasonality (weekends vs weekdays patterns) SELECT sale_date, ROUND(SUM(revenue), 2) AS daily_revenue, ROUND( AVG(SUM(revenue)) OVER ( ORDER BY sale_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ), 2) AS rolling_7day_avg FROM fact_sales WHERE YEAR(sale_date) = 2025 GROUP BY sale_date ORDER BY sale_date;
💡 Common Time-Series Patterns to Watch For:
  • Seasonality — repeating patterns tied to the calendar (holiday spikes, summer slowdowns, month-end surges)
  • Trends — sustained directional movement over multiple periods (growing or declining baseline)
  • Spikes / Anomalies — single-period deviations that break the pattern (system outages, promotions, data errors)
  • Day-of-week effects — many metrics behave differently on weekends; a 7-day rolling average smooths this out
6. Segmentation & Ranking Analysis
Segmentation analysis divides a population into meaningful sub-groups to reveal which segments drive the most value. Ranking analysis orders entities by a measure to identify top performers, bottom performers, and where to focus attention.

Top-N Analysis

Finding the top or bottom N records by a measure is one of the most requested report types — top products, top customers, worst-performing regions.

-- Top 10 products by revenue this quarter SELECT TOP 10 product_id, product_category, COUNT(*) AS transactions, SUM(quantity) AS units_sold, ROUND(SUM(revenue), 2) AS total_revenue, ROUND(AVG(revenue), 2) AS avg_transaction_value FROM fact_sales WHERE sale_date >= '2025-01-01' AND sale_date < '2025-04-01' GROUP BY product_id, product_category ORDER BY total_revenue DESC;

Ranking Within Groups (RANK and DENSE_RANK)

-- Rank products within each category by revenue -- Useful for category-level reporting: "top product per category" WITH product_revenue AS ( SELECT product_category, product_id, ROUND(SUM(revenue), 2) AS revenue FROM fact_sales GROUP BY product_category, product_id ) SELECT product_category, product_id, revenue, RANK() OVER ( PARTITION BY product_category ORDER BY revenue DESC ) AS rank_in_category FROM product_revenue ORDER BY product_category, rank_in_category;

Customer Value Segmentation

Group customers into value tiers based on their purchase behaviour. This informs which customers should receive loyalty programmes, re-engagement campaigns, or premium support.

-- Customer value segmentation based on total spend WITH customer_totals AS ( SELECT customer_id, COUNT(*) AS order_count, ROUND(SUM(revenue), 2) AS lifetime_value FROM fact_sales GROUP BY customer_id ) SELECT customer_id, order_count, lifetime_value, CASE WHEN lifetime_value >= 5000 THEN 'Platinum' WHEN lifetime_value >= 1000 THEN 'Gold' WHEN lifetime_value >= 250 THEN 'Silver' ELSE 'Bronze' END AS customer_tier FROM customer_totals ORDER BY lifetime_value DESC; -- Then summarise the tier distribution WITH customer_totals AS ( SELECT customer_id, SUM(revenue) AS lifetime_value FROM fact_sales GROUP BY customer_id ), tiered AS ( SELECT CASE WHEN lifetime_value >= 5000 THEN 'Platinum' WHEN lifetime_value >= 1000 THEN 'Gold' WHEN lifetime_value >= 250 THEN 'Silver' ELSE 'Bronze' END AS tier, lifetime_value FROM customer_totals ) SELECT tier, COUNT(*) AS customer_count, ROUND(SUM(lifetime_value), 2) AS total_revenue_from_tier, ROUND( SUM(lifetime_value) * 100.0 / SUM(SUM(lifetime_value)) OVER(), 1) AS pct_of_total_revenue FROM tiered GROUP BY tier ORDER BY total_revenue_from_tier DESC;

Percentile / Quartile Segmentation (NTILE)

-- Divide customers into 4 equal-sized groups (quartiles) by spend -- Q4 = top 25% spenders; Q1 = bottom 25% WITH customer_spend AS ( SELECT customer_id, SUM(revenue) AS total_spend FROM fact_sales GROUP BY customer_id ) SELECT customer_id, total_spend, NTILE(4) OVER (ORDER BY total_spend) AS spend_quartile, NTILE(10) OVER (ORDER BY total_spend) AS spend_decile FROM customer_spend ORDER BY total_spend DESC;

Computing Rate KPIs in SQL

-- A set of common rate KPIs computed in a single query SELECT -- Conversion rate: what % of visits resulted in a purchase? ROUND( COUNT(DISTINCT o.customer_id) * 100.0 / NULLIF(COUNT(DISTINCT v.visitor_id), 0), 2) AS conversion_rate_pct, -- Average Order Value (AOV) ROUND(SUM(o.revenue) / NULLIF(COUNT(o.order_id), 0), 2) AS avg_order_value, -- Return rate: what % of items sold were returned? ROUND( SUM(CASE WHEN o.status = 'Returned' THEN 1 ELSE 0 END) * 100.0 / NULLIF(COUNT(*), 0), 2) AS return_rate_pct, -- Revenue per visitor ROUND( SUM(o.revenue) / NULLIF(COUNT(DISTINCT v.visitor_id), 0), 2) AS revenue_per_visitor FROM website_visits AS v LEFT JOIN orders AS o ON v.visitor_id = o.visitor_id AND CAST(v.visit_date AS DATE) = CAST(o.order_date AS DATE) WHERE v.visit_date >= '2025-03-01' AND v.visit_date < '2025-04-01';
⚠️ Always use NULLIF() when dividing to prevent division-by-zero errors. NULLIF(denominator, 0) returns NULL when the denominator is zero, causing the result to be NULL rather than throwing an error — which is the correct behaviour for a rate with no base data.

8. Cyclic Reporting — Regular Reports That Drive Decisions
Cyclic reports are reports that run on a regular schedule — daily, weekly, monthly, quarterly — and feed into recurring business decisions. They are the backbone of operational and management reporting. The queries for cyclic reports are typically standardised and parameterised by date so the same query produces the right data for any period.

Designing a Parameterised Cyclic Report Query

A well-designed cyclic report query uses date parameters so the same query works for any reporting period without modification.

-- Parameterised weekly operations report -- Substitute @start and @end with any Monday–Sunday range DECLARE @report_start DATE = '2025-03-17'; -- Monday DECLARE @report_end DATE = '2025-03-23'; -- Sunday SELECT DATENAME(WEEKDAY, sale_date) AS day_of_week, sale_date, COUNT(*) AS orders, COUNT(DISTINCT customer_id) AS unique_customers, SUM(quantity) AS units_sold, ROUND(SUM(revenue), 2) AS revenue, ROUND(AVG(revenue), 2) AS avg_order_value FROM fact_sales WHERE sale_date BETWEEN @report_start AND @report_end GROUP BY sale_date, DATENAME(WEEKDAY, sale_date) ORDER BY sale_date;

Stored Procedure for a Scheduled Monthly Report

Wrapping the report query in a stored procedure makes it callable by a scheduler (SQL Server Agent job) with automatic date calculation — no manual parameter entry required.

-- Monthly summary report procedure — automatically calculates -- prior month start/end from the current date when called CREATE PROCEDURE usp_MonthlySalesReport @year INT = NULL, @month INT = NULL AS BEGIN SET NOCOUNT ON; -- Default to prior month if no parameters passed IF @year IS NULL SET @year = YEAR(DATEADD(MONTH, -1, GETDATE())); IF @month IS NULL SET @month = MONTH(DATEADD(MONTH, -1, GETDATE())); DECLARE @start DATE = DATEFROMPARTS(@year, @month, 1); DECLARE @end DATE = EOMONTH(@start); -- Report result set SELECT @year AS report_year, @month AS report_month, DATENAME(MONTH, @start) AS month_name, product_category, COUNT(*) AS transactions, COUNT(DISTINCT customer_id) AS unique_customers, SUM(quantity) AS units_sold, ROUND(SUM(revenue), 2) AS revenue, ROUND(SUM(profit), 2) AS profit, ROUND( SUM(profit) * 100.0 / NULLIF(SUM(revenue), 0), 1) AS margin_pct FROM fact_sales WHERE sale_date BETWEEN @start AND @end GROUP BY product_category ORDER BY revenue DESC; END; -- Execute for a specific month: EXEC usp_MonthlySalesReport @year = 2025, @month = 3; -- Execute for the prior month (default behaviour — scheduled job calls this): EXEC usp_MonthlySalesReport;

Year-to-Date (YTD) Summary

-- YTD performance: cumulative totals for the current year up to today -- Compare against the same period last year (SPLY) for context DECLARE @this_year INT = YEAR(GETDATE()); DECLARE @prior_year INT = YEAR(GETDATE()) - 1; DECLARE @day_of_year INT = DATEPART(DAYOFYEAR, GETDATE()); SELECT -- Current year YTD SUM(CASE WHEN YEAR(sale_date) = @this_year THEN revenue ELSE 0 END) AS ytd_revenue, -- Prior year for the same elapsed days SUM(CASE WHEN YEAR(sale_date) = @prior_year AND DATEPART(DAYOFYEAR, sale_date) <= @day_of_year THEN revenue ELSE 0 END) AS prior_ytd_revenue, -- YoY growth rate ROUND( (SUM(CASE WHEN YEAR(sale_date) = @this_year THEN revenue ELSE 0 END) - SUM(CASE WHEN YEAR(sale_date) = @prior_year AND DATEPART(DAYOFYEAR, sale_date) <= @day_of_year THEN revenue ELSE 0 END) ) * 100.0 / NULLIF( SUM(CASE WHEN YEAR(sale_date) = @prior_year AND DATEPART(DAYOFYEAR, sale_date) <= @day_of_year THEN revenue ELSE 0 END), 0), 1) AS ytd_yoy_growth_pct FROM fact_sales WHERE YEAR(sale_date) IN (@this_year, @prior_year);
9. Summary — Analysis Patterns Quick Reference

Core SQL Patterns for Data Analysis

Analysis TypeQuestion AnsweredKey SQL Technique
Descriptive Summary What happened in this period? GROUP BY + SUM / COUNT / AVG
Category Breakdown Which dimension drives the most value? GROUP BY dimension ORDER BY measure DESC
Percentage Share What proportion does each group represent? SUM(x) * 100.0 / SUM(SUM(x)) OVER()
Conditional Count How many rows meet specific criteria? SUM(CASE WHEN condition THEN 1 ELSE 0 END)
Period-over-Period How did this period compare to last period? LAG(measure) OVER (ORDER BY period)
Running Total What is the cumulative value to date? SUM(SUM(x)) OVER (ORDER BY date ROWS UNBOUNDED PRECEDING)
Moving Average What is the smoothed trend? AVG(SUM(x)) OVER (ORDER BY date ROWS BETWEEN N PRECEDING AND CURRENT ROW)
Top-N Ranking Who are the top performers? TOP N ... ORDER BY measure DESC
Rank Within Groups Who ranks first in each category? RANK() OVER (PARTITION BY group ORDER BY measure DESC)
Segmentation Which tier or group does each entity fall into? CASE WHEN measure >= threshold THEN 'Tier'
Rate / KPI What is the efficiency or ratio? measure_a * 100.0 / NULLIF(measure_b, 0)
Deduplication How do I remove duplicate records? ROW_NUMBER() OVER (PARTITION BY key ORDER BY id)
NULL Handling How do I treat missing values? COALESCE(column, default_value)

Analytical Thinking Framework

📊 What Happened?

Descriptive — Summarise the past. Aggregations, counts, totals. Answers: "We had 4,504 orders in March generating $203K revenue."

📈 Is It Getting Better?

Comparative — Period-over-period trends. LAG, running totals, growth rates. Answers: "March revenue grew 18.2% vs February."

🎯 Where to Focus?

Segmentation — Rank and group. Top-N, CASE tiers, RANK within groups. Answers: "42% of revenue comes from just 1.2% of customers."

📌 Building a Regular Reporting Practice:
  1. Define the question first — know what decision the report will inform before writing any SQL
  2. Profile the data — check for NULLs, duplicates, and outliers before trusting your aggregations
  3. Standardise category values — inconsistent labels create phantom groups in GROUP BY
  4. Parameterise by date — reports that hardcode dates require manual editing every period
  5. Add context — a raw number means nothing; a number with a prior-period comparison tells a story
  6. Wrap in a stored procedure — schedule it so the report runs automatically every cycle
  7. Document your definitions — "revenue" means different things to different teams; define it in a comment in the query
🔗 Connecting to Visualisation Tools: The SQL queries in this guide produce structured result sets that can be directly consumed by reporting tools. Excel via ODBC/PowerQuery, Power BI, Tableau, Looker, and similar tools connect to a database and run these queries on a schedule — the SQL does the heavy lifting of preparation and aggregation, and the visualisation tool handles charts and formatting. Understanding the SQL layer makes you more effective with any downstream tool.
