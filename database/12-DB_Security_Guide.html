🔒 Database Security

COP4709 — Database Management

🛡️ Why Database Security Matters

Databases are among the most high-value targets in any organization's technology infrastructure. They store customer records, financial data, healthcare information, credentials, intellectual property — assets that are both critically sensitive and legally regulated. A single breach can result in regulatory fines, litigation, reputational damage, and operational shutdown.

Database security is not a single feature or tool — it is a layered discipline combining access control, encryption, auditing, network hardening, patch management, and threat detection. Each layer reduces the likelihood or impact of a breach; no single layer is sufficient alone.

This guide covers: the database security threat landscape, best practices and implementation, common attack methods with real examples, database forensics after an incident, and a complete security hardening checklist.

1. Database Security Overview — The Threat Landscape

The Database Security Layers (Defence in Depth)

Effective database security requires protection at every layer. An attacker who penetrates one layer must still be stopped by the next.

LayerWhat It ProtectsExample Controls
Network Access to the database server itself Firewalls, VPNs, private subnets, port restrictions
Operating System The host running the database engine Patch management, service accounts, file permissions
Authentication Who can connect to the database Strong passwords, MFA, certificate auth, Windows auth
Authorization What authenticated users can do Roles, GRANT/REVOKE, row-level security, schemas
Data The data itself — at rest and in transit TDE, column encryption, TLS/SSL, masking
Audit & Monitoring Detecting and recording events Audit logs, alerting, anomaly detection, SIEM
Application The code that queries the database Parameterized queries, input validation, least privilege

Common Sources of Database Breaches

💉 Injection Attacks

Malicious SQL injected through application inputs to manipulate or extract data. The #1 database attack vector for decades.

🔑 Privilege Abuse

Legitimate users or compromised accounts accessing data beyond their role. Includes insider threats and credential theft.

🔓 Misconfiguration

Default passwords, public-facing databases, excessive privileges, unpatched vulnerabilities. The most preventable category.

🕵️ Unpatched Vulnerabilities

Known CVEs in the database engine exploited because patches were not applied. Many breaches exploit vulnerabilities months old.

👤 Weak Authentication

Default credentials, shared accounts, no MFA, weak passwords. Attackers use credential stuffing and brute force.

📡 Unencrypted Channels

Data transmitted in plaintext can be intercepted. Man-in-the-middle attacks on unencrypted database connections.

Regulatory Frameworks Affecting Database Security

FrameworkApplies ToKey Database Requirements

HIPAA US healthcare data (PHI) Access controls, encryption, audit trails, breach notification within 60 days

2. Authentication, Authorization & Least Privilege
Authentication answers "Who are you?" — verifying identity before allowing connection. Authorization answers "What can you do?" — defining the permissions of an authenticated user. Failing either is a critical security gap.

Authentication Best Practices in SQL Server

-- ❌ BAD: SQL authentication with a weak, guessable password CREATE LOGIN AppUser WITH PASSWORD = 'password123'; -- trivially crackable -- ✅ BETTER: Strong password policy enforced by SQL Server CREATE LOGIN AppUser WITH PASSWORD = 'xK9#mQr7!vLp2@Wz', MUST_CHANGE = ON, -- force change on first login CHECK_POLICY = ON, -- enforce Windows password policy CHECK_EXPIRATION= ON; -- enforce password expiration -- ✅ BEST for internal systems: Windows/Active Directory authentication -- No passwords stored in SQL Server; uses Kerberos/NTLM CREATE LOGIN [DOMAIN\AppServiceAccount] FROM WINDOWS WITH DEFAULT_DATABASE = SalesDB;

The Principle of Least Privilege

Every login, application account, and user should have only the minimum permissions required to perform their function — nothing more. Over-privileged accounts are among the most exploited attack surfaces in database environments.

❌ Over-Privileged (Common Mistake)

  • Application connects as sa or a sysadmin
  • Developers have db_owner on production
  • Reporting account has INSERT and DELETE
  • All users share a single service account
  • No separation between read and write roles

✅ Least Privilege (Correct Approach)

  • App account has only EXECUTE on stored procedures
  • Reporting account has only SELECT on views
  • ETL account has INSERT/UPDATE on specific tables only
  • Each service has its own dedicated account
  • DBA accounts are separate from application accounts

Creating Roles and Granting Permissions

-- Step 1: Create a database user mapped to a login CREATE USER AppReadUser FOR LOGIN AppReadLogin; CREATE USER AppWriteUser FOR LOGIN AppWriteLogin; -- Step 2: Create role-based permission groups CREATE ROLE db_app_reader; CREATE ROLE db_app_writer; -- Step 3: Grant permissions to roles (not directly to users) GRANT SELECT ON SCHEMA::dbo TO db_app_reader; GRANT INSERT, UPDATE ON dbo.orders TO db_app_writer; GRANT INSERT ON dbo.order_items TO db_app_writer; -- Explicitly DENY dangerous permissions even if inherited from a higher role DENY DELETE ON dbo.orders TO db_app_writer; DENY DROP TABLE TO db_app_writer; -- Step 4: Add users to roles ALTER ROLE db_app_reader ADD MEMBER AppReadUser; ALTER ROLE db_app_writer ADD MEMBER AppWriteUser;

Row-Level Security (RLS)

RLS restricts which rows a user can see within a table — automatically and transparently. A sales representative sees only their own region's data from the same table query that a manager uses to see everything.

-- RLS: each sales rep can only query their own region's orders -- Step 1: Create the security predicate function CREATE FUNCTION dbo.fn_SalesRegionFilter (@region_id INT) RETURNS TABLE WITH SCHEMABINDING AS RETURN SELECT 1 AS result WHERE @region_id = ( SELECT region_id FROM dbo.sales_reps WHERE username = USER_NAME() ) OR IS_MEMBER('db_app_manager') = 1; -- managers see all GO -- Step 2: Create and enable the security policy CREATE SECURITY POLICY SalesRegionPolicy ADD FILTER PREDICATE dbo.fn_SalesRegionFilter(region_id) ON dbo.orders WITH (STATE = ON);

Dynamic Data Masking

Data masking hides sensitive column values from unauthorized users at query time — without changing the stored data. Useful for allowing support staff to work with data without seeing raw PII.

-- Define masking rules on sensitive columns at table creation CREATE TABLE customers ( customer_id INT PRIMARY KEY, full_name VARCHAR(100), email VARCHAR(150) MASKED WITH (FUNCTION = 'email()'), phone VARCHAR(20) MASKED WITH (FUNCTION = 'partial(0,"XXX-XXX-",4)'), credit_card VARCHAR(20) MASKED WITH (FUNCTION = 'partial(0,"XXXX-XXXX-XXXX-",4)'), ssn CHAR(9) MASKED WITH (FUNCTION = 'default()') ); -- Unmask permission granted only to authorized roles GRANT UNMASK ON dbo.customers TO db_app_manager;

3. Encryption — Protecting Data at Rest and in Transit
Encryption protects data even when other controls fail. If an attacker steals a database file or intercepts network traffic, encryption ensures they obtain only unintelligible ciphertext rather than readable data.

Transparent Data Encryption (TDE)

TDE encrypts the entire database at the file level — data files, log files, and backups are all encrypted. It is transparent to applications; no query changes are needed. Protects against theft of physical media or backup files.

-- Step 1: Create the database master key USE master; CREATE MASTER KEY ENCRYPTION BY PASSWORD = 'Str0ng!MasterKeyP@ss'; -- Step 2: Create a certificate to protect the TDE key CREATE CERTIFICATE TDE_Certificate WITH SUBJECT = 'TDE Certificate for SalesDB'; -- Step 3: Create the database encryption key in the target database USE SalesDB; CREATE DATABASE ENCRYPTION KEY WITH ALGORITHM = AES_256 ENCRYPTION BY SERVER CERTIFICATE TDE_Certificate; -- Step 4: Enable TDE on the database ALTER DATABASE SalesDB SET ENCRYPTION ON; -- Verify TDE status SELECT db.name AS database_name, dek.encryption_state_desc AS encryption_state, dek.percent_complete AS progress_pct, dek.key_algorithm AS algorithm FROM sys.databases AS db JOIN sys.dm_database_encryption_keys AS dek ON db.database_id = dek.database_id WHERE db.name = 'SalesDB';
⚠️ Back up the TDE certificate immediately after creation. If the certificate is lost, the database cannot be decrypted — even by the database owner. Store the backup in a secure, separate location from the database itself.

Column-Level Encryption

For highly sensitive fields (SSN, credit cards, healthcare IDs), column-level encryption provides finer control than TDE — specific columns are encrypted and only decryptable by users with the correct key.

-- Always Encrypted: encryption happens in the application driver -- SQL Server never sees the plaintext — even DBAs cannot read it -- Define column encryption using Always Encrypted in SSMS -- or via T-SQL with the following pattern: CREATE TABLE patient_records ( patient_id INT PRIMARY KEY, patient_name VARCHAR(100), -- Encrypted with deterministic encryption (allows equality search) ssn CHAR(9) ENCRYPTED WITH ( COLUMN_ENCRYPTION_KEY = CEK_Patient, ENCRYPTION_TYPE = DETERMINISTIC, ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256' ), -- Randomized encryption (more secure; no equality searches) diagnosis VARCHAR(500) ENCRYPTED WITH ( COLUMN_ENCRYPTION_KEY = CEK_Patient, ENCRYPTION_TYPE = RANDOMIZED, ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256' ) );

Encrypting Connections (TLS/SSL)

-- Force all connections to use encrypted TLS channels -- Set in SQL Server Configuration Manager, or via T-SQL: EXEC sp_configure 'force encryption', 1; RECONFIGURE; -- In connection strings, always specify encryption: -- SQL Server: Encrypt=True;TrustServerCertificate=False -- Verify TLS is being used on active connections: SELECT session_id, DB_NAME(database_id) AS database_name, encrypt_option AS is_encrypted, auth_scheme AS auth_method FROM sys.dm_exec_connections WHERE session_id = @@SPID;

Hashing Passwords — Never Store Plaintext

-- ❌ NEVER store passwords in plaintext or with reversible encryption INSERT INTO app_users (username, password) VALUES ('jsmith', 'mypassword123'); -- catastrophic if DB is breached -- ✅ Hash with a modern algorithm + salt in your application layer -- SQL Server HASHBYTES for internal/non-credential use cases: SELECT HASHBYTES('SHA2_256', 'mypassword123' + 'random_salt_value') AS hashed_password; -- For application passwords: use bcrypt, Argon2, or PBKDF2 -- in the application layer — NOT raw SHA in SQL Server. -- These algorithms are designed to be slow, defeating brute force.
4. Auditing & Activity Monitoring
Auditing creates an immutable record of who accessed what data, when, and from where. It serves two purposes: detection (identifying suspicious activity in real time) and forensics (reconstructing events after an incident). Many compliance frameworks mandate it.

SQL Server Audit

-- Step 1: Create a server audit — defines where logs go CREATE SERVER AUDIT DatabaseSecurityAudit TO FILE ( FILEPATH = 'C:\SQLAuditLogs\', -- secure, restricted path MAXSIZE = 100 MB, MAX_ROLLOVER_FILES = 30, RESERVE_DISK_SPACE = OFF ) WITH ( ON_FAILURE = FAIL_OPERATION -- if audit fails, fail the operation too ); ALTER SERVER AUDIT DatabaseSecurityAudit WITH (STATE = ON); -- Step 2: Create a database audit specification — what to capture CREATE DATABASE AUDIT SPECIFICATION SalesDB_DataAccess FOR SERVER AUDIT DatabaseSecurityAudit ADD (SELECT ON dbo.customers BY PUBLIC), ADD (INSERT ON dbo.customers BY PUBLIC), ADD (UPDATE ON dbo.customers BY PUBLIC), ADD (DELETE ON dbo.customers BY PUBLIC), ADD (EXECUTE ON SCHEMA::dbo BY PUBLIC), ADD (DATABASE_OBJECT_PERMISSION_CHANGE_GROUP), ADD (DATABASE_PRINCIPAL_CHANGE_GROUP) -- user/role changes WITH (STATE = ON); -- Step 3: Query the audit log SELECT event_time, server_principal_name AS login_name, database_name, object_name AS table_name, action_id AS action, succeeded, client_ip, application_name, statement FROM sys.fn_get_audit_file( 'C:\SQLAuditLogs\*.sqlaudit', DEFAULT, DEFAULT) ORDER BY event_time DESC;

Monitoring for Suspicious Activity

-- Detect currently blocked sessions (possible attack or abuse) SELECT r.session_id, r.blocking_session_id, r.wait_type, r.wait_time / 1000 AS wait_seconds, DB_NAME(r.database_id) AS database_name, s.login_name, s.host_name, s.program_name, qt.text AS query_text FROM sys.dm_exec_requests AS r JOIN sys.dm_exec_sessions AS s ON r.session_id = s.session_id CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) AS qt WHERE r.blocking_session_id > 0; -- Detect logins from unexpected hosts or applications SELECT login_name, host_name, program_name, COUNT(*) AS connection_count, MIN(connect_time) AS first_seen FROM sys.dm_exec_sessions WHERE is_user_process = 1 GROUP BY login_name, host_name, program_name ORDER BY connection_count DESC;

SQL Server Login Auditing

-- Capture failed logins — important for detecting brute force attacks -- Set in SSMS: Server Properties > Security > Login Auditing -- Or via registry/configuration — audit both failed AND successful logins EXEC xp_instance_regwrite N'HKEY_LOCAL_MACHINE', N'Software\Microsoft\MSSQLServer\MSSQLServer', N'AuditLevel', REG_DWORD, 3; -- 0=None, 1=Success, 2=Failure, 3=Both -- Failed login events appear in the SQL Server Error Log and Windows Event Log -- Alert on multiple failed logins from same host within a short window
5. Attack Methods — How Databases Are Compromised
⚠️ Educational Purpose: The attack examples in this section are presented to help you understand how attacks work so you can defend against them. Understanding attack mechanics is foundational to security engineering. The same knowledge enables both attack and defence — context and intent determine which.

💉 SQL Injection (SQLi) CRITICAL

SQL injection is the most prevalent database attack. An attacker inserts SQL code into an input field that the application incorporates into a query without sanitization — causing the database to execute the attacker's commands instead of (or in addition to) the intended query.

Classic Authentication Bypass

-- Application login query (vulnerable): -- "SELECT * FROM users WHERE username='" + input + "' AND password='" + pass + "'" -- Attacker enters as username: ' OR '1'='1' -- -- Resulting query after injection: SELECT * FROM users WHERE username = '' OR '1'='1' --' AND password = '' -- '1'='1' is always TRUE; -- comments out the password check -- Result: the attacker is authenticated as the first user in the table

UNION-Based Data Extraction

-- Attacker probes the number of columns, then appends a UNION -- to extract data from a completely different table -- Input field: product ID → injected value: 1 UNION SELECT username, password, email, NULL FROM users-- -- Resulting query: SELECT product_name, description, price, image_url FROM products WHERE product_id = 1 UNION SELECT username, password, email, NULL FROM users -- -- The application displays the user credentials alongside product data

Blind SQL Injection — Boolean-Based

-- When no data is returned directly, attackers ask TRUE/FALSE questions -- and infer data from whether the page behaves differently -- Does the first character of the admin password start with 'a'? 1 AND SUBSTRING((SELECT password FROM users WHERE username='admin'),1,1) = 'a' -- Attacker automates this with tools like sqlmap, iterating character by character -- Each request reveals one bit of information; thousands of requests = full password

Defending Against SQL Injection

-- ❌ VULNERABLE: string concatenation builds the query DECLARE @sql NVARCHAR(500); SET @sql = 'SELECT * FROM users WHERE username = ''' + @username + ''''; EXEC (@sql); -- attacker controls the query structure -- ✅ SAFE: parameterized query — input is ALWAYS treated as data, never code SELECT user_id, username, email FROM users WHERE username = @username; -- @username is a parameter, not concatenated -- ✅ SAFE: stored procedure with typed parameters CREATE PROCEDURE usp_GetUser @username NVARCHAR(100) AS BEGIN SET NOCOUNT ON; SELECT user_id, username, email FROM users WHERE username = @username; -- SQL Server treats this as data END;

⬆️ Privilege Escalation HIGH

An attacker with limited database access finds a way to gain higher privileges — often by exploiting misconfigured permissions, vulnerabilities in stored procedures, or chained SQL injection into an elevated context.

-- Scenario: attacker has only SELECT permissions -- but finds a stored procedure that runs with elevated rights -- Vulnerable stored procedure (runs with EXECUTE AS OWNER = dbo) CREATE PROCEDURE usp_GetProductInfo @product_name NVARCHAR(100) WITH EXECUTE AS OWNER -- runs as dbo — dangerous if vulnerable AS BEGIN DECLARE @sql NVARCHAR(500) SET @sql = 'SELECT * FROM products WHERE name = ''' + @product_name + '''' -- ← injection point! EXEC(@sql) END -- Attacker injects as @product_name to execute as dbo: '; CREATE LOGIN hacker WITH PASSWORD='h4ck!'; EXEC sp_addsrvrolemember 'hacker','sysadmin'--

🔑 Credential Attacks HIGH

Brute force, credential stuffing (using breached password lists), and dictionary attacks against database login endpoints. Default credentials (sa with blank password) are trivially exploited and are checked first by automated scanners.

-- Common default credentials that attackers try immediately: -- SQL Server: sa / (blank) or sa / sa or sa / password -- MySQL: root / (blank) or root / root -- Oracle: system / manager or sys / change_on_install -- PostgreSQL: postgres / postgres -- Check for the sa account enabled with a blank or weak password: SELECT name, is_disabled, LOGINPROPERTY(name, 'IsExpired') AS password_expired, LOGINPROPERTY(name, 'IsMustChange') AS must_change FROM sys.sql_logins WHERE name = 'sa'; -- Best practice: disable the sa account entirely ALTER LOGIN sa DISABLE;

🔍 Reconnaissance — Enumerating Database Structure MEDIUM

Before exfiltrating data, attackers map the database structure — discovering table names, column names, data types, and stored procedures — using system catalog views that are often world-readable.

-- Attackers use system tables to discover database structure -- List all user tables: SELECT TABLE_NAME FROM INFORMATION_SCHEMA.TABLES WHERE TABLE_TYPE = 'BASE TABLE'; -- List all columns in a specific table: SELECT COLUMN_NAME, DATA_TYPE, CHARACTER_MAXIMUM_LENGTH FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_NAME = 'customers'; -- Find stored procedures (may contain business logic or sensitive operations): SELECT name, type_desc FROM sys.objects WHERE type = 'P';
-- DEFENCE: Restrict access to system views; limit schema visibility -- Revoke VIEW DEFINITION from non-admin roles: DENY VIEW DEFINITION TO db_app_reader; DENY VIEW DEFINITION TO db_app_writer; -- Grant access only to specific objects users legitimately need: GRANT VIEW DEFINITION ON dbo.orders TO db_app_reader; GRANT VIEW DEFINITION ON dbo.products TO db_app_reader;

💾 Backup Theft & Data Exfiltration CRITICAL

Attackers who gain filesystem access or cloud storage access can steal database backup files, which are regular files containing a complete copy of all data. Without TDE, a backup file opened on any SQL Server instance exposes all data immediately.

Defences Against Backup Theft:
  • Enable TDE so backup files are encrypted and unreadable without the certificate
  • Add ENCRYPTION to backup commands as an additional layer
  • Store backups in access-controlled locations separate from the database server
  • Audit all access to backup storage paths
-- Encrypted backup — useless to an attacker without the certificate BACKUP DATABASE SalesDB TO DISK = '\\BackupServer\SQLBackups\SalesDB.bak' WITH COMPRESSION, ENCRYPTION ( ALGORITHM = AES_256, SERVER CERTIFICATE = TDE_Certificate ), STATS = 10;

💻 xp_cmdshell OS Command Execution CRITICAL

xp_cmdshell is a SQL Server extended stored procedure that executes operating system commands. If enabled, an attacker with sysadmin privileges can run arbitrary commands on the host OS — installing malware, creating OS accounts, or pivoting to other systems.

-- Attacker who gains sysadmin access enables and uses xp_cmdshell: EXEC sp_configure 'show advanced options', 1; RECONFIGURE; EXEC sp_configure 'xp_cmdshell', 1; -- enables OS access RECONFIGURE; -- Now attacker can run OS commands as the SQL Server service account: EXEC xp_cmdshell 'net user hacker P@ssw0rd /add && net localgroup administrators hacker /add';
-- DEFENCE: Ensure xp_cmdshell is disabled (it is by default in modern SQL Server) EXEC sp_configure 'show advanced options', 1; RECONFIGURE; EXEC sp_configure 'xp_cmdshell', 0; -- disable OS command execution RECONFIGURE; -- Verify it is disabled: SELECT name, value, value_in_use FROM sys.configurations WHERE name = 'xp_cmdshell';
6. Database Forensics — Investigating After an Incident
Database forensics is the process of collecting, preserving, and analyzing database evidence following a suspected security incident. The goal is to determine what happened, who did it, when it happened, and what data was affected — while maintaining a chain of custody that may be required for legal proceedings.

The Forensic Principles

🔒 Preserve First

Before investigating, capture a forensic snapshot. Never run queries on a live system that might overwrite evidence. Copy log files, take memory dumps, snapshot the database.

📋 Chain of Custody

Document every action taken during investigation with timestamps, who performed it, and on what system. This is legally required if prosecution is pursued.

🔍 Reconstruct, Don't Assume

Build a timeline from evidence only. Avoid assumptions about who did what — follow the data. Multiple sources (logs, plans, backups) provide cross-validation.

Step 1 — Capture Current State Immediately

-- Capture active sessions at the moment of discovery -- Run this FIRST before anything changes SELECT s.session_id, s.login_name, s.host_name, s.client_interface_name, s.program_name, s.login_time, s.last_request_start_time, r.status AS request_status, r.command, qt.text AS current_query, c.client_net_address, c.local_net_address, c.local_tcp_port FROM sys.dm_exec_sessions AS s LEFT JOIN sys.dm_exec_requests AS r ON s.session_id = r.session_id LEFT JOIN sys.dm_exec_connections AS c ON s.session_id = c.session_id OUTER APPLY sys.dm_exec_sql_text(r.sql_handle) AS qt WHERE s.is_user_process = 1 ORDER BY s.login_time;

Step 2 — Examine the Transaction Log

The SQL Server transaction log records every modification to the database. Using fn_dblog (undocumented but widely used) or a third-party tool like ApexSQL Log, you can reconstruct exactly what data was changed, when, and by which session.

-- Read the active transaction log (undocumented fn_dblog) -- Useful for seeing recent DML operations SELECT [Current LSN], [Transaction ID], Operation, Context, [Transaction Name], [Begin Time], [End Time], [Lock Information] FROM fn_dblog(NULL, NULL) WHERE Operation IN ( 'LOP_INSERT_ROWS', 'LOP_DELETE_ROWS', 'LOP_MODIFY_ROW' ) ORDER BY [Begin Time] DESC; -- Find all DML operations on a specific table within a time window SELECT [Begin Time], [Transaction ID], Operation, [Transaction Name], AllocUnitName FROM fn_dblog(NULL, NULL) WHERE AllocUnitName LIKE 'dbo.customers%' AND [Begin Time] > '2025/02/11 00:00:00' ORDER BY [Begin Time];

Step 3 — Reconstruct the Timeline from Audit Logs

-- Build a forensic timeline from SQL Server Audit files -- combining login events, DDL, and DML in chronological order SELECT af.event_time, af.server_principal_name AS actor, af.client_ip, af.application_name, af.action_id AS action, af.object_name AS target_object, af.statement, af.succeeded FROM sys.fn_get_audit_file( 'C:\SQLAuditLogs\*.sqlaudit', DEFAULT, DEFAULT) AS af WHERE af.event_time BETWEEN '2025-02-10 20:00:00' AND '2025-02-11 04:00:00' ORDER BY af.event_time;

Step 4 — Identify What Data Was Accessed or Exfiltrated

-- From the audit log: find SELECT statements on sensitive tables -- Large result sets sent to external hosts are strong exfiltration signals SELECT event_time, server_principal_name AS login, client_ip, object_name AS table_accessed, statement FROM sys.fn_get_audit_file( 'C:\SQLAuditLogs\*.sqlaudit', DEFAULT, DEFAULT) WHERE action_id = 'SL' -- SELECT events AND object_name IN ('customers', 'payment_cards', 'patient_records') ORDER BY event_time; -- Review SQL Server error log for unusual activity patterns EXEC sp_readerrorlog 0, -- current log file (0) 1, -- SQL Server log type 'Login'; -- filter for login events

Step 5 — Check for Persistence Mechanisms Left by Attacker

-- Check for recently created logins or users (attacker backdoors) SELECT name, type_desc, create_date, modify_date, is_disabled FROM sys.server_principals WHERE create_date > '2025-02-10' -- date of suspected compromise ORDER BY create_date DESC; -- Check for recently modified stored procedures (code injection/backdoors) SELECT name, type_desc, create_date, modify_date, OBJECT_DEFINITION(OBJECT_ID(name)) AS procedure_code FROM sys.objects WHERE type IN ('P', 'TR', 'FN') -- procedures, triggers, functions AND modify_date > '2025-02-10' ORDER BY modify_date DESC; -- Check for new sysadmin-level members (privilege escalation evidence) SELECT l.name AS login_name, r.name AS role_name FROM sys.server_role_members AS m JOIN sys.server_principals AS r ON m.role_principal_id = r.principal_id JOIN sys.server_principals AS l ON m.member_principal_id = l.principal_id WHERE r.name IN ('sysadmin', 'securityadmin', 'serveradmin') ORDER BY r.name, l.name;

7. Network & Infrastructure Hardening

Network Isolation — Databases Should Never Be Public-Facing

A database should never be directly accessible from the public internet. It should sit in a private network segment (DMZ or internal zone), accessible only from application servers. Any open database port exposed to the internet is an immediate critical vulnerability.
-- SQL Server: Restrict which IPs can connect using Windows Firewall -- (configure at OS level — not T-SQL, but essential security config) -- Default SQL Server ports to restrict: -- TCP 1433 — Default SQL Server instance -- TCP 1434 — SQL Server Browser service (enumerate instances) -- UDP 1434 — SQL Server Browser (discovery) -- Only allow traffic from known application server IPs: -- netsh advfirewall firewall add rule -- name="SQL Server - AppServers Only" -- dir=in action=allow protocol=TCP localport=1433 -- remoteip=10.10.1.50,10.10.1.51 -- Disable SQL Server Browser if only one instance is used -- Change default port to a non-standard port (reduces automated scanning hits) -- Configured in SQL Server Configuration Manager > Network Configuration -- Note: security by obscurity — not a substitute for real controls

Surface Area Reduction

-- View current surface area configuration SELECT name, value, value_in_use, description FROM sys.configurations WHERE name IN ( 'xp_cmdshell', -- OS commands: disable 'clr enabled', -- CLR integration: disable if unused 'remote admin connections', -- DAC: restrict to localhost 'Ole Automation Procedures', -- OLE automation: disable 'remote access', -- linked server RPC: disable if unused 'Ad Hoc Distributed Queries' -- OPENROWSET: disable if unused ) ORDER BY name; -- Disable dangerous features not in use EXEC sp_configure 'show advanced options', 1; RECONFIGURE; EXEC sp_configure 'xp_cmdshell', 0; RECONFIGURE; EXEC sp_configure 'Ole Automation Procedures', 0; RECONFIGURE; EXEC sp_configure 'Ad Hoc Distributed Queries', 0; RECONFIGURE;

SQL Server Service Account Hardening

SQL Server Service Account Best Practices:
  • Run SQL Server under a dedicated, least-privilege domain service account — not LocalSystem or Administrator
  • The service account should have no interactive login rights
  • Grant only the permissions SQL Server actually needs on the filesystem and network
  • Use Managed Service Accounts (MSA) or Group Managed Service Accounts (gMSA) for automatic password management
  • Never use the same service account for multiple SQL Server instances
8. 🔐 Complete Database Security Hardening Checklist
Use this checklist as a systematic audit tool. Every unchecked item represents a potential attack surface. Priority should be given to items marked with ⚠️ as these are the most commonly exploited weaknesses.

🔑 Authentication & Account Management

⚠️ Disable or rename the sa account; set a complex password even if disabled
⚠️ Audit all logins with sysadmin, securityadmin, or serveradmin roles — minimize membership
Use Windows Authentication (Kerberos) instead of SQL Authentication where possible
Enable CHECK_POLICY and CHECK_EXPIRATION on all SQL Authentication logins
Enforce password complexity: minimum 12 characters, mixed case, numbers, and symbols
Remove or disable all default, sample, or test accounts
Implement Multi-Factor Authentication (MFA) for DBA and privileged accounts
Separate application service accounts from DBA accounts — never share credentials
Implement an account lockout policy after repeated failed logins (lockout threshold: 5–10 attempts)
Audit and rotate service account passwords at least annually

🛡️ Authorization & Least Privilege

⚠️ No application account should connect as sa, dbo, or any fixed server role
⚠️ Grant permissions to roles, never directly to individual user accounts
Application accounts should have only EXECUTE on stored procedures — not direct table access
Reporting accounts should have only SELECT on approved views or schemas
Explicitly DENY DROP, TRUNCATE, and ALTER for non-admin roles
Implement Row-Level Security (RLS) for multi-tenant data or regional data separation
Apply Dynamic Data Masking on PII columns accessible to support or reporting roles
Review and revoke PUBLIC role permissions — remove any non-essential grants
Conduct a permission audit at least quarterly using sys.database_permissions
Document and review all users with IMPERSONATE or EXECUTE AS rights

🔐 Encryption

⚠️ Enable Transparent Data Encryption (TDE) on all production databases
⚠️ Back up the TDE certificate immediately and store it securely offline
Force encrypted connections: Force Encryption = Yes in SQL Server Configuration Manager
Use TLS 1.2 or higher — disable SSL, TLS 1.0, and TLS 1.1
Encrypt backup files using ENCRYPTION clause in BACKUP DATABASE
Use Always Encrypted for columns containing SSN, payment card data, or healthcare IDs
Never store passwords in plaintext — use bcrypt/Argon2 in the application layer
Rotate encryption keys and certificates according to your key management policy

🌐 Network & Infrastructure

⚠️ Database server must not be directly accessible from the public internet
⚠️ Restrict TCP 1433 to known application server IPs only via firewall rules
Disable the SQL Server Browser service if not required (port UDP 1434)
Change SQL Server from default port 1433 to a non-standard port
Run SQL Server under a dedicated least-privilege domain service account
Keep SQL Server patched — apply Cumulative Updates (CUs) within 90 days of release
Keep the host OS fully patched and up to date
Run SQL Server in a separate network segment from web-facing servers

⚙️ Surface Area Reduction

⚠️ Disable xp_cmdshell — verify with sys.configurations
Disable Ole Automation Procedures if not in use
Disable Ad Hoc Distributed Queries (OPENROWSET/OPENDATASOURCE) if not in use
Disable CLR integration unless your application specifically requires it
Remove unused linked server definitions
Disable remote admin connections except on the local server
Remove or restrict access to dangerous extended stored procedures

📋 Auditing & Monitoring

⚠️ Enable SQL Server Audit capturing all logins, failed logins, DDL, and sensitive DML
⚠️ Set Login Auditing to "Both failed and successful logins"
Enable Query Store on all production databases for historical plan and performance data
Centralize audit logs to a SIEM or protected log server (not on the database server)
Set up alerts for: multiple failed logins, logins from unexpected IPs, DDL changes, sysadmin membership changes
Retain audit logs for at minimum 90 days online; 1 year in archive (PCI DSS: 1 year minimum)
Protect audit log files with restricted filesystem permissions
Review audit logs regularly — automated anomaly detection tools are preferred at scale

💻 Application-Level Security

⚠️ All database queries must use parameterized queries or prepared statements — no string concatenation
⚠️ All application database access must go through stored procedures with least-privilege execution accounts
Validate and sanitize all user input at the application layer before it reaches the database
Implement a Web Application Firewall (WAF) with SQL injection signatures for internet-facing applications
Store database connection strings in secure configuration stores (Azure Key Vault, environment variables) — never in source code
Use separate database accounts for read vs read-write vs admin operations
Implement error handling that does not expose database schema, error messages, or stack traces to users
Conduct regular penetration testing including SQL injection testing on all input vectors

💾 Backup & Recovery Security

⚠️ All backups must be encrypted — use TDE + backup encryption clause
Store backups in a physically separate location from the primary database
Restrict access to backup storage — only backup service accounts and designated DBAs
Test backup restoration regularly — an untested backup may be corrupted or incomplete
Audit all access to backup storage paths and files
Maintain an offline/air-gapped copy of backups for ransomware resilience
Store TDE certificates and encryption keys separately from database backups
📌 Prioritization Guidance: If you are hardening a database from scratch, address items in this order:
  1. Network isolation — ensure the database is not public-facing
  2. Disable xp_cmdshell and sa — immediate critical risk reduction
  3. Enable TDE and connection encryption — protect data at rest and in transit
  4. Parameterized queries — eliminate SQL injection risk at the application layer
  5. Enable audit logging — you can't investigate what you didn't record
  6. Implement least privilege — limit blast radius of any compromise
  7. Patch management — close known vulnerabilities
