COP4709 — Database Management  |  SQL Server Edition

🎯 What Is Database Performance?

Database performance is the measure of how quickly and efficiently a database system responds to queries, processes transactions, and manages data under real workload conditions. Poor performance is one of the most common — and most costly — problems in production systems.

Performance is not a single metric. It encompasses query execution time, resource utilization (CPU, memory, disk I/O), throughput (queries per second), concurrency (simultaneous users), and wait times. Tuning one dimension can improve or worsen another.

This guide uses Microsoft SQL Server (T-SQL) as the primary platform for tools and examples, though the concepts apply broadly across relational database systems.

1. Measuring Performance — Key Metrics and Tools

Core Performance Metrics

I/O
Logical & Physical Reads — most expensive resource
CPU
Time spent executing logic — sorting, hashing, computing
Wait
Time a query spends waiting on locks, disk, network
The Golden Rule of SQL Server Tuning: Reduce I/O first. Logical reads (reading data pages from memory) and physical reads (reading from disk) are the single biggest drivers of slow query performance. Every index and query change should be evaluated by its impact on read counts, not just elapsed time.

SET STATISTICS — Measuring a Query's Cost

The quickest way to measure an individual query's resource consumption is to enable statistics output before running it.

-- Enable statistics before your query SET STATISTICS IO ON; -- reports logical/physical reads per table SET STATISTICS TIME ON; -- reports CPU time and elapsed time (ms) SELECT c.customer_name, COUNT(o.order_id) AS order_count, SUM(o.total) AS lifetime_value FROM customers AS c JOIN orders AS o ON c.customer_id = o.customer_id GROUP BY c.customer_name ORDER BY lifetime_value DESC; SET STATISTICS IO OFF; SET STATISTICS TIME OFF;

Reading the Output:
  • Logical Reads — pages read from the buffer cache (memory). The primary tuning target.
  • Physical Reads — pages read from disk because they weren't in memory. Non-zero physical reads on repeated runs indicates a data size or memory pressure problem.
  • Scan Count — how many times the table or index was accessed. High counts often indicate a missing index or inefficient join.

Dynamic Management Views (DMVs)

SQL Server's DMVs are system views that expose real-time internal state — query execution stats, index usage, wait types, and more. They are the primary diagnostic tool for production performance work.

-- Find the most expensive queries currently in the plan cache -- ranked by total logical reads SELECT TOP 10 qs.total_logical_reads AS total_logical_reads, qs.execution_count AS execution_count, qs.total_logical_reads / qs.execution_count AS avg_logical_reads, qs.total_worker_time / qs.execution_count AS avg_cpu_time_us, SUBSTRING(qt.text, (qs.statement_start_offset / 2) + 1, ((CASE qs.statement_end_offset WHEN -1 THEN DATALENGTH(qt.text) ELSE qs.statement_end_offset END - qs.statement_start_offset) / 2) + 1) AS query_text FROM sys.dm_exec_query_stats AS qs CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS qt ORDER BY total_logical_reads DESC;
-- Top wait types — what is SQL Server spending time waiting for? -- Run this when the system feels slow to identify the bottleneck SELECT TOP 15 wait_type, waiting_tasks_count, wait_time_ms, CAST(wait_time_ms * 100.0 / SUM(wait_time_ms) OVER() AS DECIMAL(5,2)) AS pct_total_wait FROM sys.dm_os_wait_stats WHERE wait_type NOT IN ( 'SLEEP_TASK', 'BROKER_TO_FLUSH', -- exclude benign system waits 'SQLTRACE_BUFFER_FLUSH', 'CLR_AUTO_EVENT' ) ORDER BY wait_time_ms DESC;

2. Execution Plans — Reading How SQL Server Runs Your Query
An execution plan is SQL Server's step-by-step blueprint for executing a query. The query optimizer evaluates hundreds of possible execution strategies and selects the one it estimates to be cheapest. Understanding execution plans is the foundation of query tuning.

Viewing Execution Plans in T-SQL

-- Estimated plan: does NOT execute the query; shows optimizer's plan -- Use in SSMS: Ctrl+L or Query > Display Estimated Execution Plan SET SHOWPLAN_ALL ON; SELECT p.product_name, SUM(od.quantity) AS units_sold FROM order_details AS od JOIN products AS p ON od.product_id = p.product_id GROUP BY p.product_name; SET SHOWPLAN_ALL OFF; -- Actual plan: EXECUTES the query and includes real vs estimated rows -- Use in SSMS: Ctrl+M or Query > Include Actual Execution Plan SET STATISTICS PROFILE ON; SELECT p.product_name, SUM(od.quantity) AS units_sold FROM order_details AS od JOIN products AS p ON od.product_id = p.product_id GROUP BY p.product_name; SET STATISTICS PROFILE OFF;

Key Plan Operators to Recognise

Operator What It Means Good or Bad?
Index Seek Navigates directly to matching rows using the B-tree index ✅ Excellent — low I/O, targeted
Index Scan Reads all rows in an index ⚠️ OK for small tables; costly at scale
Table Scan / Clustered Index Scan Reads every row in the table (no useful index found) ❌ Expensive — investigate missing index
Key Lookup Goes back to the base table to fetch columns not in the index ⚠️ Per-row overhead — consider covering index
Nested Loops Join For each row in outer set, looks up matching row in inner set ✅ Best for small row sets / indexed lookups
Hash Match Join Builds hash table in memory to join large unordered sets ⚠️ Good for large tables; requires memory
Merge Join Joins two pre-sorted inputs simultaneously ✅ Very efficient when data is sorted/indexed
Sort Sorts data — triggered by ORDER BY, GROUP BY, or Merge Join ⚠️ Expensive; can often be eliminated with indexes
Spool Temporarily stores intermediate results (often in tempdb) ❌ Indicates the optimizer needed a workaround

Warning Signs in Execution Plans

Look for these red flags in any execution plan:
  • Fat arrows — the width of flow arrows represents estimated row counts. Sudden width changes indicate cardinality estimation errors.
  • Yellow warning triangles — SQL Server detected a problem (missing statistics, implicit conversion, etc.).
  • Estimated vs Actual rows differ by 10×+ — the optimizer is working with bad statistics; run UPDATE STATISTICS.
  • Table Scans on large tables — a missing or unused index is likely the cause.
  • RID Lookup / Key Lookup operators — the index found the rows but had to fetch additional columns; consider a covering index.

Query Store — Tracking Plan Changes Over Time

Query Store (SQL Server 2016+) persists execution plans and performance history, making it possible to identify when a plan changed and caused a regression.

-- Enable Query Store on a database ALTER DATABASE SalesDB SET QUERY_STORE = ON WITH ( OPERATION_MODE = READ_WRITE, CLEANUP_POLICY = (STALE_QUERY_THRESHOLD_DAYS = 30), DATA_FLUSH_INTERVAL_SECONDS = 900 ); -- Find top 10 most regressed queries (plans that got worse) SELECT TOP 10 qt.query_sql_text AS query_text, q.query_id, rs.avg_logical_io_reads AS avg_logical_reads, rs.avg_duration / 1000 AS avg_duration_ms FROM sys.query_store_query AS q JOIN sys.query_store_query_text AS qt ON q.query_text_id = qt.query_text_id JOIN sys.query_store_plan AS p ON q.query_id = p.query_id JOIN sys.query_store_runtime_stats AS rs ON p.plan_id = rs.plan_id ORDER BY rs.avg_logical_io_reads DESC;
3. Indexes — Types, Structure, and When to Use Them
An index is a separate data structure that SQL Server maintains alongside a table to enable fast data retrieval without scanning every row. Think of it like the index at the back of a textbook — you look up the term to find the page, rather than reading every page to find the term.

Clustered Index

📌 Clustered Index

1 per table Defines physical storage order

The table is the clustered index. Rows are physically stored in order of the clustered index key. A table without a clustered index is called a heap — row order is unpredictable and range queries are slow.

-- Creating a clustered index (primary key creates one by default) CREATE TABLE orders ( order_id INT IDENTITY NOT NULL, customer_id INT NOT NULL, order_date DATE NOT NULL, total DECIMAL(8,2), -- PRIMARY KEY creates a clustered index by default CONSTRAINT pk_orders PRIMARY KEY CLUSTERED (order_id) ); -- Or create explicitly on an existing table CREATE CLUSTERED INDEX cx_orders_date ON orders (order_date); -- table is now ordered by date

Non-Clustered Index

📌 Non-Clustered Index

Up to 999 per table Separate B-tree structure

A separate structure that contains the index key columns plus a pointer (row locator) back to the base table row. Lookups first search the index B-tree, then follow the pointer to the table to retrieve any additional columns not in the index.

-- Basic non-clustered index on a single column CREATE NONCLUSTERED INDEX ix_orders_customer ON orders (customer_id); -- Composite index — column order matters! -- Useful for queries filtering/sorting on both columns CREATE NONCLUSTERED INDEX ix_orders_customer_date ON orders (customer_id, order_date); -- INCLUDE — covering index: adds non-key columns to the leaf level -- Eliminates the Key Lookup to fetch those columns from the base table CREATE NONCLUSTERED INDEX ix_orders_customer_covering ON orders (customer_id) INCLUDE (order_date, total, status); -- no extra lookup needed

Covering Index — Eliminating Key Lookups

A covering index includes all columns a query needs, so SQL Server never needs to go back to the base table. This dramatically reduces I/O.

-- This query needs: customer_id (filter), order_date, total, status SELECT order_date, total, status FROM orders WHERE customer_id = 1042; -- Without covering index: Index Seek on customer_id + Key Lookup for other cols -- With covering index below: Index Seek only, no Key Lookup CREATE NONCLUSTERED INDEX ix_orders_covering ON orders (customer_id) INCLUDE (order_date, total, status);

Filtered Index

📌 Filtered Index

Partial table index Smaller, faster, targeted

An index that only includes rows matching a WHERE clause. When queries frequently filter on a subset of data (e.g., active records, open orders), a filtered index is smaller and more efficient than a full index.

-- Only index rows where status is 'Open' — these are the hot rows -- All closed/completed orders are excluded from this index entirely CREATE NONCLUSTERED INDEX ix_orders_open ON orders (customer_id, order_date) INCLUDE (total) WHERE status = 'Open';

Columnstore Index

📌 Columnstore Index

Analytical workloads 10–100× compression Batch mode execution

Stores data column-by-column rather than row-by-row. Extremely efficient for analytical queries that aggregate a few columns across millions of rows (data warehousing / OLAP). Available as clustered or non-clustered.

-- Non-clustered columnstore: add analytics capability to an OLTP table -- Queries reading large ranges will automatically use batch mode CREATE NONCLUSTERED COLUMNSTORE INDEX nccs_orders_analytics ON orders (order_date, category, total, qty, status); -- Clustered columnstore: converts entire table to columnar storage -- Best for dedicated reporting/data warehouse tables CREATE CLUSTERED COLUMNSTORE INDEX ccs_fact_sales ON fact_sales;

Unique Index

-- Enforces uniqueness AND provides a fast lookup path -- UNIQUE CONSTRAINT creates a unique index automatically CREATE UNIQUE NONCLUSTERED INDEX uix_customers_email ON customers (email);

Index Types Summary

Index Type Limit Best For Storage
Clustered 1 per table Primary key, range scans, ordered retrieval IS the table
Non-Clustered 999 per table Column filters, join keys, secondary lookups Separate B-tree
Covering Part of non-clustered Eliminating Key Lookups Larger leaf level
Filtered Part of non-clustered Sparse columns, partial-table queries Smaller than full index
Columnstore 1 clustered; multiple non-clustered Analytics, data warehousing, aggregations Highly compressed columns

4. Index Advantages, Disadvantages & Management

✅ Advantages of Indexes

  • Dramatically faster SELECT queries — Index Seek vs full Table Scan can be orders of magnitude faster on large tables
  • Faster JOIN operations — join columns with indexes enable Nested Loop and Merge joins instead of expensive Hash Matches
  • Faster ORDER BY / GROUP BY — pre-sorted index can eliminate Sort operators entirely
  • Enforce uniqueness — unique indexes prevent duplicate values as a database-level constraint
  • Covering indexes eliminate I/O — all needed data retrieved from index without touching the base table
  • Filtered indexes reduce overhead — smaller than full indexes, faster to scan and maintain

❌ Disadvantages of Indexes

  • Write overhead on every INSERT — every new row must update all indexes on the table
  • Write overhead on UPDATE — if an indexed column is updated, the index must be maintained
  • Write overhead on DELETE — deleted rows must be removed from all indexes
  • Disk storage cost — each non-clustered index is a separate data structure using additional disk space
  • Fragmentation over time — INSERT/UPDATE/DELETE cause index fragmentation that degrades read performance
  • More indexes = slower bulk loads — loading millions of rows is significantly slower with many indexes
  • Optimizer confusion — too many overlapping indexes can lead the optimizer to make poor choices

Finding Missing Indexes

SQL Server's optimizer tracks when it wishes an index existed. These recommendations are available via DMVs:

-- SQL Server's missing index recommendations -- avg_user_impact = estimated % improvement if index were created SELECT TOP 20 migs.avg_user_impact AS est_pct_improvement, migs.user_seeks + migs.user_scans AS total_uses, mid.statement AS table_name, mid.equality_columns AS equality_cols, mid.inequality_columns AS inequality_cols, mid.included_columns AS include_cols FROM sys.dm_db_missing_index_groups AS mig JOIN sys.dm_db_missing_index_group_stats AS migs ON mig.index_group_handle = migs.group_handle JOIN sys.dm_db_missing_index_details AS mid ON mig.index_handle = mid.index_handle ORDER BY migs.avg_user_impact DESC;
⚠️ Don't blindly create every missing index recommendation. SQL Server recommends indexes greedily — it sees each query in isolation. Creating all recommendations can result in dozens of overlapping indexes that slow writes significantly. Evaluate each suggestion against the query's frequency and cost.

Finding Unused Indexes

-- Indexes that are never used for seeks or scans -- but still incur maintenance cost on every write -- (reset when SQL Server service restarts) SELECT OBJECT_NAME(i.object_id) AS table_name, i.name AS index_name, i.type_desc AS index_type, ius.user_seeks AS seeks, ius.user_scans AS scans, ius.user_updates AS write_overhead FROM sys.indexes AS i LEFT JOIN sys.dm_db_index_usage_stats AS ius ON i.object_id = ius.object_id AND i.index_id = ius.index_id AND ius.database_id = DB_ID() WHERE i.type_desc = 'NONCLUSTERED' AND (ISNULL(ius.user_seeks, 0) + ISNULL(ius.user_scans, 0)) = 0 AND ISNULL(ius.user_updates, 0) > 0 ORDER BY ius.user_updates DESC;

Index Fragmentation and Maintenance

Indexes fragment over time as pages split from INSERTs and UPDATEs. Fragmented indexes cause more I/O because SQL Server reads more pages to return the same data.

-- Check fragmentation levels for all indexes in a database SELECT OBJECT_NAME(ips.object_id) AS table_name, i.name AS index_name, ips.avg_fragmentation_in_percent AS fragmentation_pct, ips.page_count AS page_count FROM sys.dm_db_index_physical_stats( DB_ID(), NULL, NULL, NULL, 'LIMITED') AS ips JOIN sys.indexes AS i ON ips.object_id = i.object_id AND ips.index_id = i.index_id WHERE ips.page_count > 100 -- ignore tiny indexes ORDER BY fragmentation_pct DESC; -- Maintenance strategy based on fragmentation level: -- < 5% fragmentation: do nothing -- 5–30% fragmentation: REORGANIZE (online, less I/O) -- > 30% fragmentation: REBUILD (more thorough, can go offline) ALTER INDEX ix_orders_customer ON orders REORGANIZE; -- light defrag ALTER INDEX ix_orders_customer ON orders REBUILD; -- full rebuild ALTER INDEX ALL ON orders REBUILD; -- rebuild all at once
5. Query Optimization — Writing Better SQL
Many performance problems are caused by how queries are written, not by missing indexes. The same logical result can be retrieved with very different query structures — and those structures can have wildly different execution costs.

SARGability — Writing Index-Friendly Conditions

A condition is SARGable (Search ARGument able) when SQL Server can use an index seek to evaluate it. Non-SARGable conditions force index scans or table scans.

❌ Non-SARGable (Forces Scan)

-- Function on the column prevents index seek WHERE YEAR(order_date) = 2025 -- Arithmetic on the column WHERE total * 1.1 > 100 -- Leading wildcard kills index WHERE name LIKE '%Smith' -- Implicit type conversion WHERE customer_id = '1042' -- int vs varchar

✅ SARGable (Enables Index Seek)

-- Range condition on the column itself WHERE order_date >= '2025-01-01' AND order_date < '2026-01-01' -- Move arithmetic to the other side WHERE total > 100 / 1.1 -- Trailing wildcard is fine WHERE name LIKE 'Smith%' -- Use matching data types WHERE customer_id = 1042 -- int = int

SELECT * — Why It Hurts Performance

-- ❌ SELECT * retrieves every column — defeats covering indexes, -- wastes I/O, network bandwidth, and memory SELECT * FROM orders WHERE customer_id = 1042; -- ✅ Request only the columns you actually need SELECT order_id, order_date, total FROM orders WHERE customer_id = 1042;

EXISTS vs IN vs JOIN for Existence Checks

-- ❌ Correlated subquery with IN — evaluates for every outer row SELECT customer_id, customer_name FROM customers WHERE customer_id IN ( SELECT customer_id FROM orders WHERE status = 'Open' ); -- ✅ EXISTS stops searching after the first match — more efficient SELECT c.customer_id, c.customer_name FROM customers AS c WHERE EXISTS ( SELECT 1 FROM orders AS o WHERE o.customer_id = c.customer_id AND o.status = 'Open' );

Avoiding Implicit Conversions

When a WHERE clause compares columns of different data types, SQL Server must convert one to match the other. This conversion is applied to every row and prevents index seeks.

-- Problem: customer_id is INT but query passes a string literal -- SQL Server converts EVERY customer_id row to VARCHAR to compare SELECT * FROM customers WHERE customer_id = '1042'; -- Fix: match the data type of the column exactly SELECT * FROM customers WHERE customer_id = 1042; -- Detect implicit conversions in the plan: -- Look for CONVERT_IMPLICIT in the execution plan XML -- or yellow warning triangles in the graphical plan

Statistics — What the Optimizer Uses to Estimate Costs

SQL Server statistics are histograms describing the distribution of values in a column. The optimizer uses them to estimate how many rows a condition will return. Stale statistics lead to bad plans.

-- Check when statistics were last updated SELECT OBJECT_NAME(s.object_id) AS table_name, s.name AS stat_name, sp.last_updated AS last_updated, sp.rows AS table_rows, sp.rows_sampled AS rows_sampled FROM sys.stats AS s CROSS APPLY sys.dm_db_stats_properties(s.object_id, s.stats_id) AS sp WHERE OBJECT_NAME(s.object_id) = 'orders' ORDER BY sp.last_updated; -- Manually update statistics on a table UPDATE STATISTICS orders; -- all stats on the table UPDATE STATISTICS orders ix_orders_customer; -- specific index stats only -- Or with FULLSCAN for the most accurate histogram UPDATE STATISTICS orders WITH FULLSCAN;
6. Normalization and Database Performance
Normalization reduces data redundancy by organizing tables so that each piece of information is stored exactly once. It is excellent for data integrity and write performance, but can require more JOIN operations for reads — which has performance implications at scale.

Normalization in Action — OLTP Schema (3NF)

A fully normalized schema is ideal for Online Transaction Processing (OLTP) — systems with high INSERT/UPDATE/DELETE rates like banking or e-commerce order entry.

-- Fully normalized 3NF design -- Each concept is stored once; no redundant data CREATE TABLE customers ( customer_id INT PRIMARY KEY, customer_name VARCHAR(100), email VARCHAR(150) ); CREATE TABLE products ( product_id INT PRIMARY KEY, product_name VARCHAR(200), category_id INT, unit_price DECIMAL(8,2) ); CREATE TABLE orders ( order_id INT PRIMARY KEY, customer_id INT REFERENCES customers(customer_id), order_date DATE ); CREATE TABLE order_items ( item_id INT PRIMARY KEY, order_id INT REFERENCES orders(order_id), product_id INT REFERENCES products(product_id), quantity INT, unit_price DECIMAL(8,2) -- price at time of sale );

✅ Normalization Advantages

  • Fast INSERTs, UPDATEs, DELETEs — minimal data to write
  • No update anomalies — change a customer name in one row
  • Smaller table sizes — better buffer pool efficiency
  • Easier to maintain data consistency with constraints
  • Less locking contention on writes

❌ Normalization Disadvantages

  • Complex queries require multiple JOINs
  • Each JOIN adds overhead — CPU, memory, I/O
  • More indexes needed across more tables
  • Reporting queries are slower on highly normalized data
  • Hard to read and write queries for deeply nested data

The Cost of JOINs on Normalized Data

-- Reporting query on normalized 3NF schema requires 4 JOINs -- Each JOIN is additional cost: hash builds, loop iterations, or merges SELECT c.customer_name, p.product_name, cat.category_name, oi.quantity, oi.quantity * oi.unit_price AS line_total FROM order_items AS oi JOIN orders AS o ON oi.order_id = o.order_id JOIN customers AS c ON o.customer_id = c.customer_id JOIN products AS p ON oi.product_id = p.product_id JOIN categories AS cat ON p.category_id = cat.category_id WHERE o.order_date >= '2025-01-01';
7. Denormalization — Trading Integrity for Read Speed
Denormalization deliberately introduces redundancy into a schema to reduce the number of JOINs required by frequently run queries. It trades write complexity and potential data anomalies for faster reads. It is a deliberate, measured design decision — not an excuse for poor design.

Denormalization Technique 1 — Storing Derived/Redundant Columns

Add a column that can technically be computed via a JOIN, to avoid the JOIN entirely on hot queries.

-- Normalized: customer_name is in customers table -- Every order query needs to JOIN customers just to show the name -- Denormalized: add customer_name directly to orders -- Hot queries now retrieve the name without any JOIN ALTER TABLE orders ADD customer_name VARCHAR(100); -- redundant copy -- TRADE-OFF: if a customer changes their name, you must now -- update BOTH customers.customer_name AND orders.customer_name

Denormalization Technique 2 — Pre-Aggregated Summary Tables

Maintain a table of pre-computed aggregates, updated by triggers or scheduled jobs. Reporting queries hit the summary instead of aggregating millions of rows on demand.

CREATE TABLE daily_sales_summary ( summary_date DATE NOT NULL, category_id INT NOT NULL, order_count INT NOT NULL, total_revenue DECIMAL(12,2) NOT NULL, total_qty_sold INT NOT NULL, PRIMARY KEY (summary_date, category_id) ); -- Dashboard query now reads ONE small summary table instead of -- aggregating millions of order_items rows on every page load SELECT summary_date, cat.category_name, dss.order_count, dss.total_revenue FROM daily_sales_summary AS dss JOIN categories AS cat ON dss.category_id = cat.category_id WHERE summary_date >= '2025-01-01' ORDER BY summary_date;

Denormalization Technique 3 — Flattened Reporting Tables (Star Schema)

Used in data warehouses. Dimension attributes are stored directly in the fact table or a single wide dimension table, eliminating multi-hop JOINs entirely.

-- Data warehouse fact table — category_name stored directly -- No JOIN to categories table required for standard reports CREATE TABLE fact_sales ( sale_id INT PRIMARY KEY, sale_date DATE, customer_id INT, customer_name VARCHAR(100), -- denormalized from dim_customer product_id INT, product_name VARCHAR(200), -- denormalized from dim_product category_name VARCHAR(100), -- denormalized from dim_category quantity INT, unit_price DECIMAL(8,2), revenue DECIMAL(12,2) -- pre-computed quantity * unit_price );

✅ Denormalization Advantages

  • Dramatically faster read queries — fewer JOINs
  • Simpler SELECT statements — easier for analysts
  • Better fit for reporting / analytical workloads
  • Pre-aggregation moves cost to write time (once) vs read time (many)
  • Enables columnstore indexes and batch mode most effectively

❌ Denormalization Disadvantages

  • Data redundancy — the same value stored in multiple places
  • Update anomalies — must update all copies when data changes
  • Increased storage requirements
  • More complex write logic (triggers, ETL processes)
  • Risk of data inconsistency if updates are missed
  • Harder to enforce referential integrity

Normalization vs Denormalization — When to Use Which

Scenario Recommended Approach Reason
E-commerce orders, banking transactions Normalize (3NF) Write-heavy OLTP; data integrity is critical
Executive dashboards, monthly reports Denormalize / Summary tables Read-heavy; pre-aggregate for instant response
Data warehouse / historical analytics Star / Snowflake schema Bulk analytical reads; columnstore optimized
Mixed OLTP + some reporting Normalize + indexed views SQL Server maintains the aggregation automatically
High-concurrency read API Selective denormalization Eliminate JOINs on the hottest query path

Indexed Views — The Best of Both Worlds

An indexed view (materialized view in other databases) is a view whose result set is physically stored and automatically maintained by SQL Server. You keep the normalized tables but get denormalized query performance.

-- Step 1: Create a schema-bound view over normalized tables CREATE VIEW vw_order_summary WITH SCHEMABINDING AS SELECT o.customer_id, p.category_id, COUNT_BIG(*) AS order_count, SUM(oi.quantity) AS total_qty, SUM(oi.quantity * oi.unit_price) AS total_revenue FROM dbo.order_items AS oi JOIN dbo.orders AS o ON oi.order_id = o.order_id JOIN dbo.products AS p ON oi.product_id = p.product_id GROUP BY o.customer_id, p.category_id; -- Step 2: Create a unique clustered index — this materializes the view -- SQL Server now stores and auto-maintains this result set CREATE UNIQUE CLUSTERED INDEX cx_vw_order_summary ON vw_order_summary (customer_id, category_id); -- Step 3: Queries hitting the base tables can automatically -- use the indexed view — zero application changes needed SELECT customer_id, SUM(total_revenue) AS lifetime_revenue FROM vw_order_summary GROUP BY customer_id;
8. Additional Performance Techniques

Partitioning — Dividing Large Tables

Table partitioning physically divides a large table into smaller segments based on a column value (usually a date). Queries targeting a date range only read the relevant partition(s) — dramatically reducing I/O for time-series data.

-- Step 1: Define a partition function (how data is divided) CREATE PARTITION FUNCTION pf_orders_by_year (DATE) AS RANGE RIGHT FOR VALUES ( '2023-01-01', '2024-01-01', '2025-01-01' ); -- Partition 1: order_date < 2023 -- Partition 2: 2023-01-01 <= order_date < 2024-01-01 -- Partition 3: 2024-01-01 <= order_date < 2025-01-01 -- Partition 4: order_date >= 2025-01-01 -- Step 2: Define a partition scheme (which filegroup each partition lives on) CREATE PARTITION SCHEME ps_orders_by_year AS PARTITION pf_orders_by_year ALL TO ([PRIMARY]); -- Step 3: Create the table using the partition scheme CREATE TABLE orders_partitioned ( order_id INT NOT NULL, order_date DATE NOT NULL, customer_id INT, total DECIMAL(8,2) ) ON ps_orders_by_year (order_date); -- partition column
Partition Elimination: When a WHERE clause includes the partition column, SQL Server skips non-matching partitions entirely. A query for order_date >= '2025-01-01' only reads partition 4 — even if partitions 1–3 contain billions of rows.

Temp Tables and Table Variables — Staging Complex Logic

-- Temp table (#temp): stored in tempdb, has statistics, can be indexed -- Use for intermediate results that are queried more than once SELECT customer_id, SUM(total) AS lifetime_value INTO #customer_totals FROM orders WHERE status = 'Completed' GROUP BY customer_id; -- Add an index to the temp table before joining to it CREATE NONCLUSTERED INDEX ix_ct_customer ON #customer_totals (customer_id); -- Now join — optimizer has statistics and an index to work with SELECT c.customer_name, ct.lifetime_value FROM customers AS c JOIN #customer_totals AS ct ON c.customer_id = ct.customer_id ORDER BY ct.lifetime_value DESC; DROP TABLE #customer_totals; -- always clean up

CTEs vs Subqueries vs Temp Tables

Approach Materialized? Has Statistics? Best Used For
Subquery No — inlined each time Uses outer table stats Simple one-level nesting
CTE No — expanded inline Uses outer table stats Readability; recursive queries
Temp Table (#temp) Yes — stored in tempdb Yes — auto-created Multi-step logic; large intermediate results
Table Variable (@tbl) Yes — stored in tempdb Minimal (1 row estimate) Small sets; short-lived scope

Stored Procedures — Plan Caching and Reuse

-- Stored procedures compile and cache their execution plan -- Subsequent calls reuse the plan — no recompilation overhead CREATE PROCEDURE usp_CustomerOrders @customer_id INT, @start_date DATE, @end_date DATE AS BEGIN SET NOCOUNT ON; -- suppress row-count messages: small but real overhead SELECT o.order_id, o.order_date, o.total, o.status FROM orders AS o WHERE o.customer_id = @customer_id AND o.order_date BETWEEN @start_date AND @end_date ORDER BY o.order_date DESC; END;
9. Performance Tuning Process & Best Practices Checklist

The Performance Tuning Workflow

Step Action SQL Server Tool
1. Identify Find the slow query or bottleneck sys.dm_exec_query_stats, Query Store, Activity Monitor
2. Baseline Measure current cost (reads, CPU, elapsed time) SET STATISTICS IO / TIME ON
3. Analyse Understand why it's slow Execution Plan, DMVs, wait stats
4. Hypothesise Identify the most likely fix Missing index DMVs, plan operators, SARGability check
5. Implement Apply the change (index, rewrite, schema) CREATE INDEX, ALTER TABLE, refactored query

7. Monitor Confirm the fix holds under real workload Query Store regressions, scheduled DMV alerts

✅ Performance Best Practices Checklist

Indexing:
  • Every table should have a clustered index (usually the primary key)
  • Index columns used in WHERE, JOIN ON, and ORDER BY clauses
  • Use INCLUDE to create covering indexes for hot queries
  • Remove unused indexes — they cost write performance with no benefit
  • Rebuild indexes with >30% fragmentation; reorganize at 5–30%
  • Keep statistics up to date — schedule weekly or after bulk loads
Query Writing:
  • Always use column-specific SELECT lists — never SELECT * in production
  • Write SARGable conditions — no functions on the left side of WHERE
  • Use matching data types to avoid implicit conversions
  • Pre-filter with WHERE before GROUP BY to reduce rows being aggregated
  • Use EXISTS rather than IN with subqueries for existence checks
  • Add SET NOCOUNT ON to all stored procedures
Schema Design:
  • Use 3NF for OLTP systems — write-heavy workloads benefit from normalization
  • Denormalize selectively for read-heavy reporting paths
  • Consider indexed views to materialize frequent aggregations automatically
  • Partition large tables on date columns for time-series query elimination
  • Separate OLTP and reporting workloads where possible (read replicas / DW)
  • Choose appropriate data types — smaller types fit more rows per page

❌ Common Performance Anti-Patterns

  • Functions in WHERE clauses — WHERE YEAR(date) = 2025 always causes a scan
  • SELECT * in production queries — retrieves unnecessary columns, defeats covering indexes
  • Cursor-based row processing — process sets, not rows; a single set-based query replaces thousands of cursor iterations
  • Implicit type conversions — comparing INT column to VARCHAR literal forces a full scan
  • Over-indexing — every additional index slows INSERTs, UPDATEs, and DELETEs
  • Stale statistics — the optimizer makes bad plans when histograms don't reflect current data
  • Ignoring fragmentation — heavily fragmented indexes cause excessive I/O on reads
  • Denormalizing prematurely — add complexity only when a measured performance problem demands it
