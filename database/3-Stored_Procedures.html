Introduction

When working with relational databases, managing data insertion across tables with foreign key relationships is a common challenge. SQL Server's stored procedures provide a powerful way to encapsulate this logic, ensuring data integrity and simplifying application code.

The Foreign Key Challenge

Consider a typical scenario: inserting an order and its line items. The order must be inserted first to generate its primary key (OrderID), which is then used as a foreign key in the order line items table. Stored procedures can handle this entire workflow atomically and efficiently.

Why Identity Columns?

  • Automatic Generation: SQL Server automatically generates unique values
  • Performance: Optimized for high-volume inserts
  • Simplicity: No need for application-level ID generation
  • Consistency: Guaranteed uniqueness within the table
  • Concurrency: Handles multiple simultaneous inserts safely
💡 Key Concept: Identity columns automatically generate sequential numeric values, but you need special functions to retrieve the generated value after an INSERT statement.

Identity Columns Overview

Defining Identity Columns

-- Basic identity column definition
CREATE TABLE Orders (
    OrderID INT IDENTITY(1,1) PRIMARY KEY,
    OrderDate DATETIME NOT NULL,
    CustomerID INT NOT NULL
);

-- Identity with custom seed and increment
CREATE TABLE Products (
    ProductID INT IDENTITY(1000,1) PRIMARY KEY,  -- Starts at 1000
    ProductName NVARCHAR(100) NOT NULL
);

-- Identity with increment by 10
CREATE TABLE BatchJobs (
    BatchID INT IDENTITY(1,10) PRIMARY KEY,  -- 1, 11, 21, 31...
    BatchName NVARCHAR(100) NOT NULL
);

Identity Syntax

Syntax Description Example
IDENTITY(seed, increment) Seed is starting value, increment is step size IDENTITY(1,1) = 1,2,3,4...
IDENTITY(1000,1) Start at 1000, increment by 1 1000,1001,1002...
IDENTITY(1,10) Start at 1, increment by 10 1,11,21,31...
⚠️ Important: Identity values are not reused even if a transaction is rolled back. Gaps in identity sequences are normal and expected.

Retrieving Identity Values

SQL Server provides several methods to retrieve the most recently generated identity value. Choosing the right method is crucial for correctness in concurrent environments.

1. SCOPE_IDENTITY() - Recommended

-- Returns the last identity value inserted in the CURRENT scope
INSERT INTO Orders (OrderDate, CustomerID)
VALUES (GETDATE(), 100);

DECLARE @NewOrderID INT = SCOPE_IDENTITY();
SELECT @NewOrderID AS NewOrderID;

2. @@IDENTITY - Use with Caution

-- Returns the last identity value in the SESSION (any scope)
-- Can return incorrect value if triggers insert into other tables
INSERT INTO Orders (OrderDate, CustomerID)
VALUES (GETDATE(), 100);

DECLARE @NewOrderID INT = @@IDENTITY;
SELECT @NewOrderID AS NewOrderID;

3. IDENT_CURRENT() - Table-Specific

-- Returns the last identity value for a specific table across all sessions
-- NOT safe for concurrent inserts
SELECT IDENT_CURRENT('Orders') AS LastOrderID;

4. OUTPUT Clause - Most Flexible

-- Returns the inserted values directly
DECLARE @InsertedOrders TABLE (OrderID INT);

INSERT INTO Orders (OrderDate, CustomerID)
OUTPUT inserted.OrderID INTO @InsertedOrders
VALUES (GETDATE(), 100);

SELECT OrderID FROM @InsertedOrders;

Comparison of Methods

Method Scope Trigger-Safe Concurrent-Safe Best Use Case
SCOPE_IDENTITY() Current scope ✅ Yes ✅ Yes Single row inserts in stored procedures
@@IDENTITY Session (all scopes) ❌ No ✅ Yes Legacy code (not recommended)
IDENT_CURRENT() Table (all sessions) N/A ❌ No Monitoring/reporting only
OUTPUT Clause Statement ✅ Yes ✅ Yes Multiple row inserts, returning multiple columns
✅ Best Practice: Use SCOPE_IDENTITY() for single-row inserts and the OUTPUT clause for multiple-row inserts. Avoid @@IDENTITY and never use IDENT_CURRENT() for retrieving just-inserted IDs.

Example Schema

Let's create a realistic schema to demonstrate foreign key management. We'll use an e-commerce example with customers, orders, and order items.

Complete Schema Definition

-- 1. Customers table (no dependencies)
CREATE TABLE Customers (
    CustomerID INT IDENTITY(1,1) PRIMARY KEY,
    FirstName NVARCHAR(50) NOT NULL,
    LastName NVARCHAR(50) NOT NULL,
    Email NVARCHAR(100) NOT NULL,
    CreatedDate DATETIME DEFAULT GETDATE()
);

-- 2. Products table (no dependencies)
CREATE TABLE Products (
    ProductID INT IDENTITY(1,1) PRIMARY KEY,
    ProductName NVARCHAR(100) NOT NULL,
    Description NVARCHAR(500),
    UnitPrice DECIMAL(10,2) NOT NULL,
    StockQuantity INT NOT NULL
);

-- 3. Orders table (depends on Customers)
CREATE TABLE Orders (
    OrderID INT IDENTITY(1,1) PRIMARY KEY,
    CustomerID INT NOT NULL,
    OrderDate DATETIME NOT NULL DEFAULT GETDATE(),
    TotalAmount DECIMAL(10,2) NOT NULL,
    Status NVARCHAR(20) NOT NULL,
    CONSTRAINT FK_Orders_Customers FOREIGN KEY (CustomerID) 
        REFERENCES Customers(CustomerID)
);

-- 4. OrderItems table (depends on Orders and Products)
CREATE TABLE OrderItems (
    OrderItemID INT IDENTITY(1,1) PRIMARY KEY,
    OrderID INT NOT NULL,
    ProductID INT NOT NULL,
    Quantity INT NOT NULL,
    UnitPrice DECIMAL(10,2) NOT NULL,
    LineTotal AS (Quantity * UnitPrice) PERSISTED,
    CONSTRAINT FK_OrderItems_Orders FOREIGN KEY (OrderID) 
        REFERENCES Orders(OrderID),
    CONSTRAINT FK_OrderItems_Products FOREIGN KEY (ProductID) 
        REFERENCES Products(ProductID)
);

-- 5. Addresses table (depends on Customers)
CREATE TABLE Addresses (
    AddressID INT IDENTITY(1,1) PRIMARY KEY,
    CustomerID INT NOT NULL,
    AddressType NVARCHAR(20) NOT NULL, -- 'Billing', 'Shipping'
    StreetAddress NVARCHAR(200) NOT NULL,
    City NVARCHAR(100) NOT NULL,
    State NVARCHAR(50) NOT NULL,
    ZipCode NVARCHAR(20) NOT NULL,
    CONSTRAINT FK_Addresses_Customers FOREIGN KEY (CustomerID) 
        REFERENCES Customers(CustomerID)
);

Basic Stored Procedures

1. Simple Insert with Identity Return

Example: Insert Customer

CREATE PROCEDURE usp_InsertCustomer
    @FirstName NVARCHAR(50),
    @LastName NVARCHAR(50),
    @Email NVARCHAR(100),
    @CustomerID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    -- Insert customer
    INSERT INTO Customers (FirstName, LastName, Email)
    VALUES (@FirstName, @LastName, @Email);
    
    -- Get the generated identity value
    SET @CustomerID = SCOPE_IDENTITY();
    
    -- Return success indicator
    RETURN 0;
END;
GO

-- Usage example
DECLARE @NewCustomerID INT;
EXEC usp_InsertCustomer 
    @FirstName = 'John',
    @LastName = 'Smith',
    @Email = 'john.smith@email.com',
    @CustomerID = @NewCustomerID OUTPUT;
    
SELECT @NewCustomerID AS NewCustomerID;

2. Insert with One-Level Foreign Key

Example: Insert Order for Existing Customer

CREATE PROCEDURE usp_InsertOrder
    @CustomerID INT,
    @TotalAmount DECIMAL(10,2),
    @Status NVARCHAR(20),
    @OrderID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    -- Validate customer exists
    IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
    BEGIN
        RAISERROR('Customer does not exist', 16, 1);
        RETURN -1;
    END
    
    -- Insert order
    INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
    VALUES (@CustomerID, GETDATE(), @TotalAmount, @Status);
    
    -- Get the generated OrderID
    SET @OrderID = SCOPE_IDENTITY();
    
    RETURN 0;
END;
GO

-- Usage example
DECLARE @NewOrderID INT;
EXEC usp_InsertOrder 
    @CustomerID = 1,
    @TotalAmount = 299.99,
    @Status = 'Pending',
    @OrderID = @NewOrderID OUTPUT;
    
SELECT @NewOrderID AS NewOrderID;

3. Insert with Multi-Level Foreign Keys

Example: Insert Order Item

CREATE PROCEDURE usp_InsertOrderItem
    @OrderID INT,
    @ProductID INT,
    @Quantity INT,
    @OrderItemID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @UnitPrice DECIMAL(10,2);
    
    -- Validate order exists
    IF NOT EXISTS (SELECT 1 FROM Orders WHERE OrderID = @OrderID)
    BEGIN
        RAISERROR('Order does not exist', 16, 1);
        RETURN -1;
    END
    
    -- Get product price and validate product exists
    SELECT @UnitPrice = UnitPrice
    FROM Products
    WHERE ProductID = @ProductID;
    
    IF @UnitPrice IS NULL
    BEGIN
        RAISERROR('Product does not exist', 16, 1);
        RETURN -1;
    END
    
    -- Insert order item
    INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
    VALUES (@OrderID, @ProductID, @Quantity, @UnitPrice);
    
    -- Get the generated OrderItemID
    SET @OrderItemID = SCOPE_IDENTITY();
    
    RETURN 0;
END;
GO

-- Usage example
DECLARE @NewOrderItemID INT;
EXEC usp_InsertOrderItem 
    @OrderID = 1,
    @ProductID = 5,
    @Quantity = 2,
    @OrderItemID = @NewOrderItemID OUTPUT;
    
SELECT @NewOrderItemID AS NewOrderItemID;

Advanced Techniques

Complete Order Creation in One Procedure

This procedure demonstrates creating a complete order with customer and addresses in a single transaction.

Example: Create Complete Customer Order

CREATE PROCEDURE usp_CreateCompleteOrder
    -- Customer info
    @FirstName NVARCHAR(50),
    @LastName NVARCHAR(50),
    @Email NVARCHAR(100),
    -- Address info
    @BillingAddress NVARCHAR(200),
    @BillingCity NVARCHAR(100),
    @BillingState NVARCHAR(50),
    @BillingZip NVARCHAR(20),
    -- Order info
    @TotalAmount DECIMAL(10,2),
    @Status NVARCHAR(20),
    -- Output parameters
    @CustomerID INT OUTPUT,
    @OrderID INT OUTPUT,
    @AddressID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- 1. Insert Customer
        INSERT INTO Customers (FirstName, LastName, Email)
        VALUES (@FirstName, @LastName, @Email);
        
        SET @CustomerID = SCOPE_IDENTITY();
        
        -- 2. Insert Address (uses CustomerID from step 1)
        INSERT INTO Addresses (
            CustomerID, AddressType, StreetAddress, 
            City, State, ZipCode
        )
        VALUES (
            @CustomerID, 'Billing', @BillingAddress,
            @BillingCity, @BillingState, @BillingZip
        );
        
        SET @AddressID = SCOPE_IDENTITY();
        
        -- 3. Insert Order (uses CustomerID from step 1)
        INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
        VALUES (@CustomerID, GETDATE(), @TotalAmount, @Status);
        
        SET @OrderID = SCOPE_IDENTITY();
        
        COMMIT TRANSACTION;
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        -- Return error information
        DECLARE @ErrorMessage NVARCHAR(4000) = ERROR_MESSAGE();
        RAISERROR(@ErrorMessage, 16, 1);
        RETURN -1;
    END CATCH
END;
GO

-- Usage example
DECLARE @CustID INT, @OrdID INT, @AddrID INT;

EXEC usp_CreateCompleteOrder
    @FirstName = 'Jane',
    @LastName = 'Doe',
    @Email = 'jane.doe@email.com',
    @BillingAddress = '123 Main St',
    @BillingCity = 'Seattle',
    @BillingState = 'WA',
    @BillingZip = '98101',
    @TotalAmount = 449.99,
    @Status = 'Pending',
    @CustomerID = @CustID OUTPUT,
    @OrderID = @OrdID OUTPUT,
    @AddressID = @AddrID OUTPUT;

SELECT 
    @CustID AS CustomerID,
    @OrdID AS OrderID,
    @AddrID AS AddressID;

Using OUTPUT Clause for Multiple Inserts

Example: Insert Multiple Order Items at Once

CREATE PROCEDURE usp_InsertMultipleOrderItems
    @OrderID INT
AS
BEGIN
    SET NOCOUNT ON;
    
    -- Table variable to hold inserted IDs
    DECLARE @InsertedItems TABLE (
        OrderItemID INT,
        ProductID INT,
        Quantity INT,
        LineTotal DECIMAL(10,2)
    );
    
    -- Insert multiple items using OUTPUT clause
    INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
    OUTPUT 
        inserted.OrderItemID,
        inserted.ProductID,
        inserted.Quantity,
        inserted.LineTotal
    INTO @InsertedItems
    SELECT 
        @OrderID,
        ProductID,
        2 AS Quantity, -- Example: 2 of each product
        UnitPrice
    FROM Products
    WHERE ProductID IN (1, 2, 3);
    
    -- Return the inserted items
    SELECT * FROM @InsertedItems;
END;
GO

Transaction Management

Transactions ensure that all related inserts succeed or fail together, maintaining data integrity across foreign key relationships.

Transaction Best Practices

✅ Transaction Guidelines:
  • Always use explicit transactions when inserting related data
  • Keep transactions as short as possible
  • Handle errors with TRY...CATCH blocks
  • Always check @@TRANCOUNT before ROLLBACK
  • Use appropriate isolation levels
  • Set XACT_ABORT ON for automatic rollback on errors

Comprehensive Transaction Example

Example: Order with Line Items (Full Transaction)

CREATE PROCEDURE usp_CreateOrderWithItems
    @CustomerID INT,
    @Status NVARCHAR(20),
    @OrderID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT ON; -- Auto-rollback on errors
    
    DECLARE @TotalAmount DECIMAL(10,2) = 0;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Validate customer
        IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
        BEGIN
            RAISERROR('Customer ID %d does not exist', 16, 1, @CustomerID);
            RETURN -1;
        END
        
        -- Step 1: Create order (with temporary total)
        INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
        VALUES (@CustomerID, GETDATE(), 0, @Status);
        
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Step 2: Add order items (example: products 1, 2, 3)
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
        SELECT 
            @OrderID,
            p.ProductID,
            CASE p.ProductID 
                WHEN 1 THEN 2
                WHEN 2 THEN 1
                WHEN 3 THEN 3
            END AS Quantity,
            p.UnitPrice
        FROM Products p
        WHERE p.ProductID IN (1, 2, 3);
        
        -- Step 3: Calculate and update order total
        SELECT @TotalAmount = SUM(LineTotal)
        FROM OrderItems
        WHERE OrderID = @OrderID;
        
        UPDATE Orders
        SET TotalAmount = @TotalAmount
        WHERE OrderID = @OrderID;
        
        COMMIT TRANSACTION;
        
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        -- Rollback if transaction is active
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        -- Log error details
        DECLARE @ErrorMessage NVARCHAR(4000);
        DECLARE @ErrorSeverity INT;
        DECLARE @ErrorState INT;
        
        SELECT 
            @ErrorMessage = ERROR_MESSAGE(),
            @ErrorSeverity = ERROR_SEVERITY(),
            @ErrorState = ERROR_STATE();
        
        -- Re-throw the error
        RAISERROR(@ErrorMessage, @ErrorSeverity, @ErrorState);
        
        RETURN -1;
    END CATCH
END;
GO

Nested Transaction Handling

CREATE PROCEDURE usp_NestedTransactionExample
    @CustomerID INT
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @TransactionStarted BIT = 0;
    
    BEGIN TRY
        -- Start transaction only if not already in one
        IF @@TRANCOUNT = 0
        BEGIN
            BEGIN TRANSACTION;
            SET @TransactionStarted = 1;
        END
        
        -- Your insert logic here
        INSERT INTO Customers (FirstName, LastName, Email)
        VALUES ('Test', 'User', 'test@email.com');
        
        -- Commit only if this procedure started the transaction
        IF @TransactionStarted = 1
            COMMIT TRANSACTION;
            
    END TRY
    BEGIN CATCH
        -- Rollback only if this procedure started the transaction
        IF @TransactionStarted = 1 AND @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
            
        THROW; -- Re-throw error to caller
    END CATCH
END;
GO

Error Handling

Robust error handling is crucial for stored procedures that manage complex foreign key relationships.

Comprehensive Error Handling Pattern

Example: Complete Error Handling

CREATE PROCEDURE usp_InsertOrderWithErrorHandling
    @CustomerID INT,
    @ProductID INT,
    @Quantity INT,
    @OrderID INT OUTPUT,
    @ErrorMessage NVARCHAR(4000) OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT OFF; -- We'll handle errors explicitly
    
    DECLARE @UnitPrice DECIMAL(10,2);
    DECLARE @TotalAmount DECIMAL(10,2);
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Validation 1: Customer exists
        IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
        BEGIN
            SET @ErrorMessage = 'Customer ID ' + CAST(@CustomerID AS NVARCHAR) + ' not found';
            RAISERROR(@ErrorMessage, 16, 1);
        END
        
        -- Validation 2: Product exists and get price
        SELECT @UnitPrice = UnitPrice
        FROM Products
        WHERE ProductID = @ProductID;
        
        IF @UnitPrice IS NULL
        BEGIN
            SET @ErrorMessage = 'Product ID ' + CAST(@ProductID AS NVARCHAR) + ' not found';
            RAISERROR(@ErrorMessage, 16, 1);
        END
        
        -- Validation 3: Quantity is positive
        IF @Quantity <= 0
        BEGIN
            SET @ErrorMessage = 'Quantity must be greater than zero';
            RAISERROR(@ErrorMessage, 16, 1);
        END
        
        -- Validation 4: Check stock availability
        IF EXISTS (
            SELECT 1 FROM Products 
            WHERE ProductID = @ProductID 
            AND StockQuantity < @Quantity
        )
        BEGIN
            SET @ErrorMessage = 'Insufficient stock for Product ID ' + 
                              CAST(@ProductID AS NVARCHAR);
            RAISERROR(@ErrorMessage, 16, 1);
        END
        
        -- Calculate total
        SET @TotalAmount = @UnitPrice * @Quantity;
        
        -- Insert order
        INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
        VALUES (@CustomerID, GETDATE(), @TotalAmount, 'Pending');
        
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Insert order item
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
        VALUES (@OrderID, @ProductID, @Quantity, @UnitPrice);
        
        -- Update stock
        UPDATE Products
        SET StockQuantity = StockQuantity - @Quantity
        WHERE ProductID = @ProductID;
        
        COMMIT TRANSACTION;
        
        SET @ErrorMessage = NULL; -- Success
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        -- Capture comprehensive error information
        SET @ErrorMessage = 
            'Error ' + CAST(ERROR_NUMBER() AS NVARCHAR) + ': ' +
            ERROR_MESSAGE() + 
            ' (Line ' + CAST(ERROR_LINE() AS NVARCHAR) + ')';
        
        SET @OrderID = NULL;
        RETURN -1;
    END CATCH
END;
GO

-- Usage with error handling
DECLARE @NewOrderID INT;
DECLARE @Error NVARCHAR(4000);
DECLARE @ReturnCode INT;

EXEC @ReturnCode = usp_InsertOrderWithErrorHandling
    @CustomerID = 999, -- Non-existent customer
    @ProductID = 1,
    @Quantity = 2,
    @OrderID = @NewOrderID OUTPUT,
    @ErrorMessage = @Error OUTPUT;

IF @ReturnCode = 0
    SELECT 'Success' AS Result, @NewOrderID AS OrderID;
ELSE
    SELECT 'Error' AS Result, @Error AS ErrorMessage;

OUTPUT Parameters

OUTPUT parameters provide a clean way to return identity values and status information from stored procedures.

Multiple OUTPUT Parameters

Example: Return Multiple Generated IDs

CREATE PROCEDURE usp_CreateCustomerWithAddresses
    -- Input parameters
    @FirstName NVARCHAR(50),
    @LastName NVARCHAR(50),
    @Email NVARCHAR(100),
    @BillingAddress NVARCHAR(200),
    @BillingCity NVARCHAR(100),
    @BillingState NVARCHAR(50),

    @BillingAddressID INT OUTPUT,
    @ShippingAddressID INT OUTPUT,
    @RecordsCreated INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    SET @RecordsCreated = 0;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Insert customer
        INSERT INTO Customers (FirstName, LastName, Email)
        VALUES (@FirstName, @LastName, @Email);
        
        SET @CustomerID = SCOPE_IDENTITY();
        SET @RecordsCreated = @RecordsCreated + 1;
        
        -- Insert billing address
        INSERT INTO Addresses (
            CustomerID, AddressType, StreetAddress, City, State, ZipCode
        )
        VALUES (
            @CustomerID, 'Billing', @BillingAddress, 
            @BillingCity, @BillingState, @BillingZip
        );
        
        SET @BillingAddressID = SCOPE_IDENTITY();
        SET @RecordsCreated = @RecordsCreated + 1;
        
        -- Insert shipping address
        INSERT INTO Addresses (
            CustomerID, AddressType, StreetAddress, City, State, ZipCode
        )
        VALUES (
            @CustomerID, 'Shipping', @ShippingAddress,
            @ShippingCity, @ShippingState, @ShippingZip
        );
        
        SET @ShippingAddressID = SCOPE_IDENTITY();
        SET @RecordsCreated = @RecordsCreated + 1;
        
        COMMIT TRANSACTION;
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        -- Set outputs to NULL on error
        SET @CustomerID = NULL;
        SET @BillingAddressID = NULL;
        SET @ShippingAddressID = NULL;
        SET @RecordsCreated = 0;
        
        THROW;
    END CATCH
END;
GO

Table-Valued Parameters

Table-valued parameters (TVPs) allow you to pass multiple rows of data to a stored procedure in a single call, perfect for batch inserts with foreign key relationships.

Creating User-Defined Table Types

-- Define table type for order items
CREATE TYPE OrderItemType AS TABLE (
    ProductID INT NOT NULL,
    Quantity INT NOT NULL
);
GO

-- Define table type for addresses
CREATE TYPE AddressType AS TABLE (
    AddressType NVARCHAR(20) NOT NULL,
    StreetAddress NVARCHAR(200) NOT NULL,
    City NVARCHAR(100) NOT NULL,
    State NVARCHAR(50) NOT NULL,
    ZipCode NVARCHAR(20) NOT NULL
);
GO

Using TVPs in Stored Procedures

Example: Create Order with Multiple Items

CREATE PROCEDURE usp_CreateOrderWithItemsTVP
    @CustomerID INT,
    @Status NVARCHAR(20),
    @OrderItems OrderItemType READONLY, -- TVP parameter
    @OrderID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @TotalAmount DECIMAL(10,2);
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Validate customer exists
        IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
        BEGIN
            RAISERROR('Customer does not exist', 16, 1);
        END
        
        -- Validate all products exist
        IF EXISTS (
            SELECT 1
            FROM @OrderItems oi
            LEFT JOIN Products p ON oi.ProductID = p.ProductID
            WHERE p.ProductID IS NULL
        )
        BEGIN
            RAISERROR('One or more products do not exist', 16, 1);
        END
        
        -- Calculate total amount
        SELECT @TotalAmount = SUM(oi.Quantity * p.UnitPrice)
        FROM @OrderItems oi
        INNER JOIN Products p ON oi.ProductID = p.ProductID;
        
        -- Create order
        INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
        VALUES (@CustomerID, GETDATE(), @TotalAmount, @Status);
        
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Insert all order items
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
        SELECT 
            @OrderID,
            oi.ProductID,
            oi.Quantity,
            p.UnitPrice
        FROM @OrderItems oi
        INNER JOIN Products p ON oi.ProductID = p.ProductID;
        
        COMMIT TRANSACTION;
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;
GO

VALUES 
    (1, 2),
    (3, 1),

-- Execute procedure with TVP
EXEC usp_CreateOrderWithItemsTVP
    @CustomerID = 1,
    @Status = 'Pending',
    @OrderItems = @Items,
    @OrderID = @NewOrderID OUTPUT;

SELECT @NewOrderID AS NewOrderID;
💡 TVP Benefits: Table-valued parameters reduce network round-trips, improve performance, and make code more maintainable compared to passing delimited strings or making multiple procedure calls.

Advantages of Stored Procedures

✅ Advantages

  • Data Integrity: Encapsulate complex foreign key logic, ensuring consistency
  • Performance: Compiled execution plans, reduced network traffic
  • Security: Parameterized queries prevent SQL injection
  • Maintainability: Centralized business logic, easier to update
  • Reusability: Call from multiple applications
  • Atomicity: Built-in transaction support
  • Abstraction: Hide database complexity from application
  • Version Control: Database changes through migrations
  • Testing: Can unit test database logic

❌ Considerations

  • Vendor Lock-in: T-SQL is SQL Server specific
  • Version Control: Requires database migration tools
  • Development: Requires SQL expertise
  • Testing: Need database for integration tests
  • Debugging: Limited debugging tools compared to application code
  • Deployment: Database deployments can be complex
  • Flexibility: Less flexible than ORM dynamic queries

When to Use Stored Procedures

Scenario Use Stored Procedure? Reason
Complex multi-table inserts ✅ Yes Ensures atomicity and referential integrity
Bulk data operations ✅ Yes Better performance, reduced network traffic
Business logic validation ✅ Yes Centralized, consistent enforcement
Simple CRUD on single table ⚠️ Maybe ORM might be simpler for basic operations
Dynamic filtering/searching ⚠️ Maybe ORM dynamic queries can be more flexible
Frequently changing business rules ❌ Consider alternatives Application code may be easier to modify

Best Practices

1. Naming Conventions

✅ Recommended Naming:
  • Stored Procedures: usp_VerbNoun (e.g., usp_InsertOrder, usp_GetCustomerOrders)
  • Parameters: @PascalCase (e.g., @CustomerID, @OrderDate)
  • Table Types: NounType (e.g., OrderItemType, AddressType)
  • Local Variables: @PascalCase with descriptive names

2. Performance Optimization

✅ Performance Tips:
  • Always use SET NOCOUNT ON at procedure start
  • Use SCOPE_IDENTITY() instead of @@IDENTITY
  • Minimize transaction scope - only wrap necessary operations
  • Use table-valued parameters for batch inserts
  • Consider using OUTPUT clause instead of separate SELECT statements
  • Index foreign key columns for better join performance
  • Use appropriate isolation levels (READ COMMITTED is usually sufficient)

3. Error Handling Standards

✅ Error Handling Guidelines:
  • Always use TRY...CATCH blocks for error handling
  • Check @@TRANCOUNT before ROLLBACK
  • Use meaningful error messages with RAISERROR or THROW
  • Return status codes (0 for success, negative for errors)
  • Consider OUTPUT parameters for error messages
  • Log errors to an error table for diagnostics
  • Validate inputs before beginning transactions

4. Security Best Practices

✅ Security Recommendations:
  • Grant EXECUTE permissions on procedures, not direct table access
  • Use parameterized procedures to prevent SQL injection
  • Validate all input parameters
  • Use separate service accounts with minimal permissions
  • Avoid dynamic SQL when possible; if needed, use sp_executesql
  • Don't expose sensitive data in error messages
  • Audit procedure executions for sensitive operations

6. Testing Recommendations

✅ Testing Best Practices:
  • Test with valid data to verify successful operations
  • Test with invalid data to verify error handling
  • Test transaction rollback scenarios
  • Test concurrent execution scenarios
  • Verify foreign key constraints are enforced
  • Test with NULL values where applicable
  • Verify identity values are returned correctly
  • Performance test with realistic data volumes

7. Common Pitfalls to Avoid

⚠️ Avoid These Mistakes:
  • Using @@IDENTITY instead of SCOPE_IDENTITY()
  • Not checking @@TRANCOUNT before ROLLBACK
  • Leaving transactions open too long
  • Not validating foreign key relationships before INSERT
  • Forgetting SET NOCOUNT ON (causes extra result sets)
  • Not using OUTPUT parameters to return generated IDs
  • Inadequate error handling and logging
  • Not considering concurrency scenarios
  • Hardcoding values that should be parameters
