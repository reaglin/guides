Basic Relational Terminology

Outcome 1
📐

Outcome 1 · FoundationThe Core Vocabulary of the Relational Model

E.F. Codd introduced the relational model in 1970 using precise mathematical language based on set theory and first-order predicate logic. Understanding the formal vocabulary is critical to understanding how databases are structured, queried, and normalized.

The Six Core Terms — Defined

Relation

A two-dimensional table with a unique name. Each relation represents a single, distinct subject (e.g., Student, Course). Named from the mathematical concept of a set of tuples.

Tuple

A single row within a relation. Represents one instance of the subject. A tuple is an ordered set of attribute values—one value per attribute in the relation.

Attribute

A named column in a relation. Each attribute represents a single property of the subject. An attribute has a name and is associated with a domain.

Domain

The complete set of permissible values for an attribute. Defines the allowed data type and range. Example: the domain of GradeLevel might be {Freshman, Sophomore, Junior, Senior}.

Degree

The number of attributes (columns) in a relation. A relation with 5 attributes has a degree of 5. Also called arity. A higher degree means more data per tuple.

Cardinality

The number of tuples (rows) in a relation at a given moment. A relation with 200 student records has a cardinality of 200. Cardinality changes as rows are added or removed.

Characteristics of Relations

Outcome 2
📏

Outcome 2 · RulesThe Seven Defining Properties of a Well-Formed Relation

Not every table in a database qualifies as a relation in the strict mathematical sense. A relation must satisfy all seven of the following properties. Understanding these characteristics is essential to designing correct, normalized databases.

  • 1
    Every relation has a unique name No two relations in the same database schema may share the same name. The name identifies the subject the relation describes (e.g., Student, Enrollment, Course).
  • 2
    Every attribute within a relation has a unique name Column names must be distinct within the same relation. Two tables can have a column named ID, but within a single table, ID can only appear once.
  • 3
    All values in a column belong to the same domain Every cell in a given column must contain a value drawn from that attribute's domain. You cannot store a name string in a numeric GPA column, even once.
  • 4
    The order of tuples is insignificant A relation is a mathematical set — sets are unordered. The rows in a table have no inherent top-to-bottom order. Any ordering seen in a query result is a display choice, not a property of the relation itself.
  • 5
    The order of attributes is insignificant Similarly, columns have no inherent left-to-right ordering in the relational model. Attributes are always referenced by name, never by position.
  • 6
    Every tuple is unique — no duplicate tuples are allowed Because a relation is a set, and sets do not allow duplicates, no two tuples may be identical across all attribute values. This is why every relation requires a key — to guarantee uniqueness.
  • 7
    All attribute values are atomic (indivisible) Each cell must contain a single, indivisible value — no lists, no repeating groups, no embedded arrays. This property is the definition of First Normal Form (1NF). Example: a PhoneNumber column should contain one number, not "555-1234, 555-5678".

Violating Atomicity — A Common Mistake

⚠️
Practical Note: SQL vs. Pure Relational Theory Real SQL databases (like SQL Server) do permit some deviations from pure relational theory — such as duplicate rows in a table without a primary key, or ordering data with ORDER BY. SQL is based on the relational model but is not a perfect mathematical implementation. Always define primary keys to enforce tuple uniqueness in practice.

Functional Dependencies

Outcome 4
🔍

Outcome 4 · ConceptDeterminants, Dependents & the FD Notation

Definition — Functional Dependency

Attribute B is functionally dependent on attribute A if, for every valid value of A, there is exactly one associated value of B. Written formally: A → B (read: "A determines B" or "B is functionally dependent on A").

Determinant & Dependent Defined

🔶
Determinant The attribute(s) on the left side of the functional dependency arrow (A in A → B). Knowing the value of the determinant uniquely determines the value of the dependent. Also called the source or left-hand side.
🔷
Dependent Attribute The attribute(s) on the right side of the arrow (B in A → B). Its value is determined by — dependent on — the determinant. Also called the dependent or right-hand side.

How to Verify a Functional Dependency

Ask yourself: "If I know the value of A, can I always look up exactly one value of B?"

  • YES → A → B is a valid functional dependency
  • NO / Sometimes multiple → A does not functionally determine B
⚠️
Common Misconception A functional dependency is a semantic, business-rule constraint — it cannot be determined just by looking at the data at one point in time. Even if all current records happen to have a 1:1 relationship, the FD only holds if it must always be true by definition. For example, two students can have the same first name, so FirstName → StudentID is NOT a valid FD.
🌿

Outcome 4 · TypesFull, Partial & Transitive Functional Dependencies

The type of functional dependency in a relation is the foundation of normalization. Identifying which type exists tells you what normal form violations are present and how to correct them.

✅ Full Functional Dependency

Attribute B is fully functionally dependent on A when B is dependent on the entire composite key and cannot be determined by any subset of it. This is the desired state for well-normalized tables.

{StudentID, CourseID} → Grade
Grade cannot be determined by StudentID alone or CourseID alone.
⚠️ Partial Functional Dependency

Attribute B is only dependent on part of a composite key, not all of it. This violates Second Normal Form (2NF) and causes data redundancy. The solution is to remove the partially dependent attribute to its own table.

{StudentID, CourseID} → CourseName
CourseName depends only on CourseID — a partial dependency.
🔄 Transitive Dependency

Attribute B depends on attribute C, which itself depends on the primary key A. So B is indirectly dependent on A through C. This violates Third Normal Form (3NF) and must be resolved by splitting the table.

StudentID → MajorID → DepartmentHead
DepartmentHead depends on MajorID, not directly on StudentID.

Keys: Primary, Candidate & Composite

Outcome 5
🗝️

Outcome 5 · ConceptThe Complete Key Taxonomy

Keys are attributes (or combinations of attributes) that play special roles in identifying tuples and establishing relationships between relations. Each type of key serves a specific purpose in database design.

🔑 Candidate Key

Any attribute or minimal set of attributes that could uniquely identify every tuple in a relation. There may be multiple candidate keys in one relation. All candidate keys are minimal — removing any attribute would break uniqueness.

Employee: SSN, EmployeeID, and Email may all qualify as candidate keys.
⭐ Primary Key

The one candidate key chosen by the designer to officially and permanently identify tuples. It must be: unique (no two rows share the same PK), not null, stable (rarely or never changes), and minimal.

Employee: EmployeeID is chosen as PK from the candidate keys above.
🔒 Composite Key

A primary (or candidate) key made up of two or more attributes working together. No single attribute alone is sufficient for uniqueness. Common in junction/bridge tables representing many-to-many relationships.

Enrollment: {StudentID + CourseID} together are unique, but neither alone is sufficient.
🔐 Alternate Key

Any candidate key that was not selected as the primary key. These are still valid unique identifiers — often enforced with a UNIQUE constraint in SQL.

Employee: SSN and Email become alternate keys once EmployeeID is selected as PK.
🤖 Surrogate Key

An artificial, system-generated key with no real-world meaning. Typically an auto-incrementing integer (IDENTITY in SQL Server). Used when no natural key is suitable — stable, guaranteed unique, and small.

CustomerID INT IDENTITY(1,1) — the database generates 1, 2, 3… automatically.
🌿 Natural Key

A key derived from real-world data that already exists and has inherent meaning. Can be problematic if values change or if they are long and difficult to use as a foreign key reference.

SSN, ISBN (book), VIN (vehicle), Email address — all carry real-world meaning.

The Relationship Between Key Types

Key Type Uniquely Identifies? Can Be Null? How Many Per Relation? SQL Implementation
Candidate Key ✅ Yes ❌ No One or more All enforced via UNIQUE or PRIMARY KEY
Primary Key ✅ Yes ❌ No (enforced) Exactly one PRIMARY KEY constraint
Composite Key ✅ Yes (combined) ❌ No One (is the PK or candidate) PRIMARY KEY (colA, colB)
Alternate Key ✅ Yes ❌ No (usually) Zero or more UNIQUE constraint
Surrogate Key ✅ Yes (artificial) ❌ No At most one (is the PK) IDENTITY(1,1)

💻

Outcome 5 · SQLImplementing Keys in T-SQL (SQL Server)

Understanding key concepts is only half the job — you must also know how to implement them correctly in SQL. The following examples show how each key type is defined in a real database.

Example 1 — Simple Surrogate Primary Key

T-SQL · Primary Key · Surrogate
-- Surrogate Primary Key using IDENTITY (auto-increment) -- StudentID has no real-world meaning — purely a generated integer CREATE TABLE Student ( StudentID INT IDENTITY(1,1) NOT NULL, LastName VARCHAR(50) NOT NULL, FirstName VARCHAR(50) NOT NULL, Email VARCHAR(100) NOT NULL, GPA DECIMAL(3,2) -- Surrogate primary key: single integer, auto-generated CONSTRAINT PK_Student PRIMARY KEY (StudentID), -- Enforce alternate keys with UNIQUE constraints CONSTRAINT UQ_Student_Email UNIQUE (Email) );

Example 2 — Composite Primary Key (Junction Table)

T-SQL · Composite Key · Many-to-Many
-- Composite Primary Key: StudentID + CourseID together are unique -- Neither column alone is sufficient to identify a single enrollment CREATE TABLE Enrollment ( StudentID INT NOT NULL, CourseID CHAR(10) NOT NULL, Grade CHAR(2) NULL, EnrollDate DATE NOT NULL -- Composite PK: the combination of both columns is the identifier CONSTRAINT PK_Enrollment PRIMARY KEY (StudentID, CourseID), -- Foreign Keys enforce referential integrity to parent tables CONSTRAINT FK_Enrollment_Student FOREIGN KEY (StudentID) REFERENCES Student (StudentID), CONSTRAINT FK_Enrollment_Course FOREIGN KEY (CourseID) REFERENCES Course (CourseID) );

Example 3 — Natural Key

T-SQL · Natural Key · Course Example
-- Natural Primary Key: CourseID is a meaningful code (e.g., 'COP4708') -- It carries real-world meaning but is stable and unlikely to change CREATE TABLE Course ( CourseID CHAR(10) NOT NULL, -- Natural key: e.g., 'COP4708' CourseName VARCHAR(100) NOT NULL, Credits TINYINT NOT NULL CONSTRAINT CK_Credits CHECK (Credits BETWEEN 1 AND 6), DeptID CHAR(5) NOT NULL, CONSTRAINT PK_Course PRIMARY KEY (CourseID), CONSTRAINT FK_Course_Dept FOREIGN KEY (DeptID) REFERENCES Department (DeptID) );

Choosing Between a Surrogate and Natural Key

Consideration Surrogate Key Natural Key
Stability ✅ Never changes — system-assigned ⚠️ May change (e.g., student name, email)
Meaningfulness ❌ No business meaning (just a number) ✅ Has real-world significance (e.g., course code)
Size as Foreign Key ✅ Small integer — efficient joins ⚠️ May be large (SSN, email) — larger index
Guaranteed Uniqueness ✅ Database enforces it automatically ⚠️ Must validate uniqueness against real-world source
Best Used When No stable natural identifier exists; joining performance matters A stable, unique, meaningful code already exists (ISBN, VIN, course code)
💡
Best Practice In enterprise database design, surrogate keys (auto-incrementing integers) are the default choice for primary keys because they are lightweight, stable, and always unique. Natural keys are retained as alternate keys with UNIQUE constraints so business users can still look up records by meaningful values.

Outcome 5 · PracticeHow to Identify Candidate Keys in a Relation

Identifying candidate keys requires systematic analysis of the data and business rules — not just guessing. Follow this process for any relation.

Step-by-Step Process

  1. List all attributes in the relation
  2. Identify attributes (or combinations) that could be unique across all tuples — now and always, by the rules of the business
  3. Test for minimality — can you remove any attribute from the combination and still maintain uniqueness? If yes, the larger set is not minimal and is therefore not a candidate key
  4. List all valid minimal unique sets — these are your candidate keys
  5. Choose one as the primary key based on stability, size, and simplicity
