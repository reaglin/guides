📐 What Is Aggregation?

Aggregation in SQL means performing a calculation across a set of rows to produce a single summary value. Instead of returning every row, aggregate functions collapse them into meaningful totals, counts, averages, and more.

For example: rather than listing every sale ever made, you can ask SQL to return the total revenue per month, the average order value, or the highest-selling product — all in a single query.

Key supporting tools: AS gives your output columns readable names, GROUP BY controls what rows get aggregated together, and HAVING filters the results after aggregation.

1. The Aggregation Functions at a Glance
COUNT()

Counts rows or non-NULL values

SUM()

Totals numeric values

AVG()

Calculates the mean of numeric values

MIN()

Returns the smallest value

MAX()

Returns the largest value

COUNT(DISTINCT)

Counts unique non-NULL values

General Syntax Pattern:

SELECT FUNCTION(column) AS alias_name FROM table [WHERE ...] [GROUP BY ...] [HAVING ...] [ORDER BY ...];

2. COUNT() — Counting Rows and Values
COUNT() returns the number of rows that match the query. It has three important forms that behave differently.

COUNT(*) — Count Every Row

Counts all rows including those with NULL values in any column.

-- How many orders have been placed in total? SELECT COUNT(*) AS total_orders FROM orders;
▶ RESULT
total_orders
10

COUNT(column) — Count Non-NULL Values

Counts only rows where the specified column is not NULL. Useful for spotting missing data.

-- How many orders have a recorded status? SELECT COUNT(status) AS orders_with_status FROM orders;
▶ RESULT
orders_with_status
10

COUNT(DISTINCT column) — Count Unique Values

Counts only distinct (unique) non-NULL values in the column.

-- How many unique customers have placed an order? SELECT COUNT(DISTINCT customer_id) AS unique_customers FROM orders;
▶ RESULT
unique_customers
5

COUNT() with WHERE

-- How many completed orders are there? SELECT COUNT(*) AS completed_orders FROM orders WHERE status = 'Completed';
▶ RESULT
completed_orders
8
⚠️ COUNT(*) vs COUNT(column): COUNT(*) always counts every row. COUNT(column) skips NULLs. If a column has NULLs in some rows, these two will return different numbers — a common source of confusion in reporting queries.
3. SUM() — Totalling Numeric Values
SUM() adds up all non-NULL values in a numeric column. It returns NULL if there are no rows (not zero), and ignores NULL values in the column.

Basic SUM

-- What is the total revenue across all orders? SELECT SUM(total) AS total_revenue FROM orders;
▶ RESULT
total_revenue
412.93

SUM with WHERE

-- What is the total revenue from completed orders only? SELECT SUM(total) AS completed_revenue FROM orders WHERE status = 'Completed';
▶ RESULT
completed_revenue
341.93

SUM on Multiple Columns

-- Total revenue and total items sold in one query SELECT SUM(total) AS total_revenue, SUM(qty) AS total_books_sold FROM orders WHERE status = 'Completed';
▶ RESULT
total_revenuetotal_books_sold
341.9320
💡 Tip — Arithmetic Inside SUM: You can do math inside the function. For example, SUM(qty * unit_price) calculates a line total before summing — useful when you store quantity and price separately.
4. AVG() — Calculating Averages
AVG() returns the arithmetic mean of all non-NULL values in a column. It is equivalent to SUM(col) / COUNT(col), but it ignores NULLs in the count — which is not always the same as dividing by the total number of rows.

Basic AVG

-- What is the average order value? SELECT AVG(total) AS avg_order_value FROM orders;
▶ RESULT
avg_order_value
41.293000

Rounding AVG Output

AVG often returns many decimal places. Use ROUND() for cleaner output.

-- Round to 2 decimal places for currency display SELECT ROUND(AVG(total), 2) AS avg_order_value FROM orders;
▶ RESULT
avg_order_value
41.29

AVG with a Filter

-- What is the average order value in the Fiction category? SELECT ROUND(AVG(total), 2) AS avg_fiction_order FROM orders WHERE category = 'Fiction';
▶ RESULT
avg_fiction_order
25.12
⚠️ NULL Trap with AVG: If a column has NULLs, AVG() ignores them. This means the average is computed over fewer rows than the total row count. If a missing value should be treated as zero, use AVG(COALESCE(column, 0)) to substitute NULLs before averaging.
5. MIN() and MAX() — Finding Extremes
MIN() and MAX() return the smallest and largest values in a column. They work on numbers, dates, and strings (alphabetically for strings). Both ignore NULL values.

MIN and MAX on Numbers

-- What were the cheapest and most expensive orders? SELECT MIN(total) AS smallest_order, MAX(total) AS largest_order FROM orders;
▶ RESULT
smallest_orderlargest_order
15.0091.00

MIN and MAX on Dates

MIN on a date returns the earliest date; MAX returns the most recent.

-- When was our first and most recent order? SELECT MIN(order_date) AS first_order_date, MAX(order_date) AS latest_order_date FROM orders;
▶ RESULT
first_order_datelatest_order_date
2025-01-052025-03-22

MIN and MAX on Strings

On text columns, MIN and MAX use alphabetical ordering.

-- First and last category alphabetically SELECT MIN(category) AS first_alpha_category, MAX(category) AS last_alpha_category FROM orders;
▶ RESULT
first_alpha_categorylast_alpha_category
BiographyScience

Combining All Five Functions

-- Full summary of completed order activity SELECT COUNT(*) AS order_count, SUM(total) AS total_revenue, ROUND(AVG(total), 2) AS avg_order_value, MIN(total) AS min_order_value, MAX(total) AS max_order_value FROM orders WHERE status = 'Completed';
▶ RESULT
order_counttotal_revenueavg_order_valuemin_order_valuemax_order_value
8341.9342.7415.0091.00
6. AS — Naming Your Output Columns
AS creates a column alias — a custom label for a column in your query results. Without it, aggregate expressions produce ugly auto-generated headers like COUNT(*) or ROUND(AVG(total),2), which are hard to read and can break application code that references column names.

Without AS vs With AS

❌ Without AS

SELECT COUNT(*), SUM(total), AVG(total) FROM orders;

Output headers: COUNT(*), SUM(total), AVG(total) — cryptic and unusable as references.

✅ With AS

SELECT COUNT(*) AS order_count, SUM(total) AS total_revenue, AVG(total) AS avg_order_value FROM orders;

Output headers: order_count, total_revenue, avg_order_value — readable, meaningful, referenceable.

AS Rules and Syntax Notes

-- AS is optional — a space alone also creates an alias (but AS is clearer) SELECT COUNT(*) order_count FROM orders; -- Works but harder to read SELECT COUNT(*) AS order_count FROM orders; -- Preferred: AS is explicit -- Use quotes for aliases with spaces or special characters SELECT COUNT(*) AS "Total Orders Placed" FROM orders; -- AS also works on regular columns and table names SELECT o.order_id AS id, o.total AS order_total, o.order_date AS placed_on FROM orders AS o;

Using AS with Calculated Columns

AS is especially valuable when your SELECT contains math or function calls.

-- Revenue report with formatted/computed output columns SELECT category, COUNT(*) AS num_orders, SUM(total) AS gross_revenue, ROUND(AVG(total), 2) AS avg_order_value, ROUND(SUM(total) * 0.10, 2) AS estimated_tax FROM orders GROUP BY category ORDER BY gross_revenue DESC;
▶ RESULT
categorynum_ordersgross_revenueavg_order_valueestimated_tax
Science3183.5061.1718.35
History2109.9554.9811.00
Fiction4100.4925.1210.05
Biography118.9918.991.90
⚠️ You Cannot Use an Alias in WHERE: Because WHERE is evaluated before SELECT, you cannot reference an alias you defined in SELECT inside a WHERE clause. Use HAVING instead, or repeat the expression. This is a very common beginner mistake.
-- ❌ This fails — alias not yet defined at WHERE stage SELECT SUM(total) AS total_revenue FROM orders WHERE total_revenue > 100; -- ERROR: unknown column -- ✅ Use HAVING instead (filters after aggregation) SELECT category, SUM(total) AS total_revenue FROM orders GROUP BY category HAVING SUM(total) > 100;
7. GROUP BY — Aggregating by Category
GROUP BY splits rows into groups based on one or more columns, then applies an aggregate function to each group separately. Without GROUP BY, aggregate functions collapse the entire table into one row.

GROUP BY a Single Column

-- How many orders and total revenue per category? SELECT category, COUNT(*) AS order_count, SUM(total) AS total_revenue FROM orders GROUP BY category ORDER BY total_revenue DESC;
▶ RESULT
categoryorder_counttotal_revenue
Science3183.50
History2109.95
Fiction4100.49
Biography118.99

GROUP BY Multiple Columns

Adding more columns to GROUP BY creates finer groupings — one row per unique combination.

-- Orders and revenue broken down by category AND status SELECT category, status, COUNT(*) AS order_count, SUM(total) AS total_revenue FROM orders GROUP BY category, status ORDER BY category, status;
▶ RESULT
categorystatusorder_counttotal_revenue
BiographyCompleted118.99
FictionCompleted367.49
FictionPending133.00
HistoryCompleted2109.95
ScienceCancelled138.00
ScienceCompleted2145.50

The Golden Rule of GROUP BY

Every column in SELECT must either:
  1. Be listed in GROUP BY, OR
  2. Be inside an aggregate function like COUNT(), SUM(), etc.
-- ❌ Fails — order_date is neither grouped nor aggregated SELECT category, order_date, -- Which date would SQL pick? Undefined! SUM(total) AS total_revenue FROM orders GROUP BY category; -- ✅ Fixed — aggregate the date or add it to GROUP BY SELECT category, MIN(order_date) AS first_order_date, SUM(total) AS total_revenue FROM orders GROUP BY category;

GROUP BY with a Date Function

A common real-world pattern: group by month or year to create time-series reports.

-- Monthly revenue report SELECT YEAR(order_date) AS order_year, MONTH(order_date) AS order_month, COUNT(*) AS order_count, SUM(total) AS monthly_revenue FROM orders GROUP BY YEAR(order_date), MONTH(order_date) ORDER BY order_year, order_month;
▶ RESULT

8. HAVING — Filtering Aggregated Results
HAVING filters groups after aggregation has been performed — it is the GROUP BY equivalent of WHERE. Use WHERE to filter individual rows before grouping; use HAVING to filter the resulting groups after grouping.

WHERE vs HAVING — Side by Side

WHERE — Filters rows BEFORE grouping

-- Only consider completed orders, -- then group and aggregate SELECT category, SUM(total) AS revenue FROM orders WHERE status = 'Completed' GROUP BY category;

HAVING — Filters groups AFTER grouping

-- Group all orders, then keep only -- categories with revenue over $100 SELECT category, SUM(total) AS revenue FROM orders GROUP BY category HAVING SUM(total) > 100;

Basic HAVING Example

-- Which categories have generated more than $100 in revenue? SELECT category, COUNT(*) AS order_count, SUM(total) AS total_revenue FROM orders GROUP BY category HAVING SUM(total) > 100 ORDER BY total_revenue DESC;
▶ RESULT
categoryorder_counttotal_revenue
Science3183.50
History2109.95
Fiction4100.49

Using WHERE and HAVING Together

-- Among completed orders only, which categories -- have more than 1 order AND average value above $30? SELECT category, COUNT(*) AS order_count, ROUND(AVG(total), 2) AS avg_order_value FROM orders WHERE status = 'Completed' -- row-level filter (before grouping) GROUP BY category HAVING COUNT(*) > 1 -- group-level filter (after grouping) AND AVG(total) > 30 ORDER BY avg_order_value DESC;
▶ RESULT
categoryorder_countavg_order_value
Science272.75
History254.98

HAVING with COUNT — Finding Active Groups

-- Which customers have placed more than one order? SELECT customer_id, COUNT(*) AS order_count, SUM(total) AS lifetime_value FROM orders GROUP BY customer_id HAVING COUNT(*) > 1 ORDER BY order_count DESC;
▶ RESULT

COUNT(DISTINCT) — Unique Value Counts Within Groups

GROUP_CONCAT / STRING_AGG — Aggregating Text

These vendor-specific functions concatenate string values across rows in a group into a single list. Great for producing summary columns.

-- MySQL / MariaDB: GROUP_CONCAT SELECT customer_id, GROUP_CONCAT(category ORDER BY order_date SEPARATOR ', ') AS categories_purchased FROM orders GROUP BY customer_id; -- PostgreSQL / SQL Server 2017+: STRING_AGG SELECT customer_id, STRING_AGG(category, ', ') AS categories_purchased FROM orders GROUP BY customer_id;
▶ RESULT (MySQL)

VARIANCE and STDDEV — Statistical Functions

Available in most SQL databases for statistical analysis of numeric distributions.

-- How consistent are order values per category? -- Lower stddev = more uniform pricing SELECT category, ROUND(AVG(total), 2) AS avg_order_value, ROUND(STDDEV(total), 2) AS std_deviation, ROUND(VARIANCE(total), 2) AS variance FROM orders GROUP BY category ORDER BY std_deviation DESC;
Function Name Variations by Database:
  • MySQL / PostgreSQL: STDDEV(), VARIANCE()
  • SQL Server: STDEV(), VAR()
  • Oracle: STDDEV(), VARIANCE()
  • Most also offer population vs sample variants: STDDEV_POP() / STDDEV_SAMP()

WITH ROLLUP — Subtotals and Grand Totals

WITH ROLLUP (MySQL/SQL Server) or ROLLUP() (PostgreSQL/Oracle) adds subtotal and grand total rows automatically.

-- Revenue by category with a grand total row at the bottom SELECT COALESCE(category, 'GRAND TOTAL') AS category, COUNT(*) AS order_count, SUM(total) AS total_revenue FROM orders GROUP BY category WITH ROLLUP ORDER BY total_revenue DESC;
▶ RESULT

10. Query Execution Order — How SQL Processes Aggregation
Understanding the logical order SQL evaluates a query explains why aliases defined in SELECT can't be used in WHERE, and why HAVING exists as a separate clause.

The 6 Clauses in Execution Order

Step Clause What Happens
1 FROM Identify the source table(s); perform any JOINs
2 WHERE Filter individual rows — aliases don't exist yet
3 GROUP BY Divide the remaining rows into groups
4 HAVING Filter groups using aggregate conditions
5 SELECT Calculate expressions, apply aggregates, assign aliases
6 ORDER BY Sort the final result — aliases CAN be used here

Full Aggregation Query Template

-- Complete template showing all clauses in write order -- (which differs from execution order above) SELECT column_or_expression AS alias, AGGREGATE(column) AS aggregate_alias FROM table_name AS t WHERE row_level_condition -- filters rows BEFORE grouping GROUP BY column_or_expression -- defines the groups HAVING group_level_condition -- filters groups AFTER aggregation ORDER BY aggregate_alias DESC -- aliases OK here LIMIT 10; -- optional row limit
11. Real-World Aggregation Examples

📦 E-Commerce: Top Customers Report

-- Rank the top 5 customers by lifetime value, -- only counting completed orders SELECT customer_id, COUNT(*) AS total_orders, SUM(total) AS lifetime_value, ROUND(AVG(total), 2) AS avg_order_value, MAX(order_date) AS last_order_date FROM orders WHERE status = 'Completed' GROUP BY customer_id ORDER BY lifetime_value DESC LIMIT 5;

🏥 Healthcare: Appointment Summary by Doctor

-- For a healthcare database: summarise each doctor's workload SELECT d.doctor_name, d.specialisation, COUNT(a.appointment_id) AS total_appointments, COUNT(DISTINCT a.patient_id) AS unique_patients, SUM(CASE WHEN a.status = 'No-Show' THEN 1 ELSE 0 END) AS no_shows, ROUND( SUM(CASE WHEN a.status = 'No-Show' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) AS no_show_rate_pct FROM appointments AS a JOIN doctors AS d ON a.doctor_id = d.doctor_id WHERE a.appointment_date >= '2025-01-01' GROUP BY d.doctor_name, d.specialisation HAVING COUNT(a.appointment_id) >= 10 ORDER BY total_appointments DESC;
💡 Pattern — SUM(CASE WHEN ...): This trick lets you count rows that meet a specific condition within an aggregate query without needing a separate subquery. It is one of the most useful aggregation patterns in real-world SQL.

🎓 University: Grade Distribution Report

-- For a university database: grade distribution per course SELECT c.course_code, c.course_name, COUNT(e.student_id) AS enrolled_students, ROUND(AVG(e.final_grade), 2) AS class_average, MIN(e.final_grade) AS lowest_grade, MAX(e.final_grade) AS highest_grade, SUM(CASE WHEN e.final_grade >= 90 THEN 1 ELSE 0 END) AS a_grades, SUM(CASE WHEN e.final_grade < 60 THEN 1 ELSE 0 END) AS failing_grades FROM enrollments AS e JOIN courses AS c ON e.course_id = c.course_id WHERE e.semester_code = 'SPRING2025' GROUP BY c.course_code, c.course_name HAVING COUNT(e.student_id) >= 5 ORDER BY class_average DESC;
12. Best Practices & Common Mistakes

✅ Best Practices

  • Always use AS on aggregate columns — auto-generated names break application code and are unreadable in reports.
  • Use WHERE before GROUP BY to pre-filter — filtering rows early reduces the data the database has to aggregate.
  • Align your SQL keywords — well-formatted queries are easier to debug and maintain.
  • Test aggregates on small data first — verify your GROUP BY logic before running on millions of rows.
  • Use ROUND() on AVG and division — raw averages often produce unnecessary decimal precision.
  • Name aliases after what they represent — monthly_revenue not col1 or sum1.

❌ Common Mistakes

  • Using WHERE instead of HAVING to filter on an aggregate — WHERE runs before GROUP BY so aggregate values don't exist yet.
  • Selecting non-aggregated columns that aren't in GROUP BY — this produces undefined results (or an error in strict mode).
  • Forgetting that COUNT(column) ignores NULLs — use COUNT(*) when you want to count all rows regardless.
  • Referencing aliases in WHERE — the alias doesn't exist at that stage of execution.
  • Assuming SUM returns 0 when there are no rows — it returns NULL; use COALESCE(SUM(col), 0) for zero-safe output.
  • Grouping by a high-cardinality column like order_id — each group will have only one row and aggregation is pointless.
