Introduction to Transactions

A transaction is a logical unit of work that contains one or more database operations. Transactions ensure data integrity by treating multiple operations as a single, indivisible unit that either completes entirely or fails entirely.

What is a Transaction?

A transaction groups multiple database operations into a single atomic unit with the following characteristics:

  • Single Unit: All operations succeed or all fail together
  • Consistent State: Database moves from one valid state to another
  • Isolated Execution: Concurrent transactions don't interfere with each other
  • Permanent Changes: Once committed, changes survive system failures

Why Transactions Matter

Real-World Example: Bank Transfer

-- Without transaction: DANGEROUS!
UPDATE Accounts SET Balance = Balance - 500 WHERE AccountID = 1;
-- System crashes here - money disappears!
UPDATE Accounts SET Balance = Balance + 500 WHERE AccountID = 2;

-- With transaction: SAFE!
BEGIN TRANSACTION;
    UPDATE Accounts SET Balance = Balance - 500 WHERE AccountID = 1;
    UPDATE Accounts SET Balance = Balance + 500 WHERE AccountID = 2;
COMMIT TRANSACTION;
-- Both updates succeed or both fail

Transaction Lifecycle

  1. BEGIN: Transaction starts
  2. EXECUTE: Perform database operations
  3. COMMIT: Make changes permanent (success)
  4. ROLLBACK: Undo all changes (failure)
💡 Key Concept: Transactions are essential for maintaining data integrity in multi-step operations, especially in scenarios involving money, inventory, or any critical business data.

ACID Properties

ACID is an acronym that describes the four fundamental properties that guarantee reliable transaction processing in database systems.

A - Atomicity

Definition: All operations in a transaction succeed or all fail. There is no partial completion.

Analogy: Like an atom, a transaction cannot be split into smaller parts.

Implementation: Transaction log records all changes; rollback undoes incomplete transactions.

Example:

BEGIN TRANSACTION;
    INSERT INTO Orders (CustomerID, TotalAmount) VALUES (1, 100);
    INSERT INTO OrderItems (OrderID, ProductID, Qty) VALUES (SCOPE_IDENTITY(), 5, 2);
    UPDATE Inventory SET Quantity = Quantity - 2 WHERE ProductID = 5;
COMMIT TRANSACTION;
-- All three operations complete or none do

C - Consistency

Definition: Transactions take the database from one valid state to another valid state, preserving all constraints and rules.

Analogy: Like a balanced scale - the database remains balanced before and after.

Implementation: Constraint checking, triggers, and business logic enforcement.

Example:

BEGIN TRANSACTION;
    -- Check constraint: Balance cannot be negative
    UPDATE Accounts SET Balance = Balance - 1000 WHERE AccountID = 1;
    -- If balance would go negative, constraint violation causes rollback
    -- Database remains consistent - no negative balances
COMMIT TRANSACTION;

I - Isolation

Definition: Concurrent transactions execute independently without interfering with each other.

Analogy: Like separate lanes on a highway - transactions don't collide.

Implementation: Locking mechanisms, multi-version concurrency control (MVCC).

Isolation Levels: READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE

Example:

-- Transaction 1
BEGIN TRANSACTION;
    UPDATE Products SET Price = 100 WHERE ProductID = 1;
    -- Other transactions don't see Price = 100 yet
COMMIT TRANSACTION;
-- Now other transactions see the new price

D - Durability

Definition: Once a transaction is committed, changes are permanent and survive system failures.

Analogy: Like writing in permanent marker - it can't be erased by accidents.

Implementation: Write-ahead logging (WAL), transaction logs written to disk before commit.

Example:

BEGIN TRANSACTION;
    INSERT INTO Orders (CustomerID, TotalAmount) VALUES (1, 500);
COMMIT TRANSACTION;
-- Even if server crashes now, order is saved
-- Transaction log ensures recovery on restart

ACID Trade-offs

Property Benefit Cost Can Be Relaxed?
Atomicity All-or-nothing guarantee Transaction log overhead ❌ No - fundamental
Consistency Data integrity enforced Constraint checking overhead ⚠️ Dangerous - not recommended
Isolation Prevents interference Locking/blocking, reduced concurrency ✅ Yes - isolation levels
Durability Survives crashes Disk I/O, slower commits ⚠️ Only in specific scenarios

CAP Theorem and ACID

In distributed systems, the CAP theorem states you can only guarantee two of three properties:

  • C (Consistency): All nodes see the same data
  • A (Availability): System remains operational
  • P (Partition Tolerance): System works despite network failures
Database Type CAP Choice ACID Level
Traditional RDBMS CA (Consistency + Availability) Full ACID
MongoDB, HBase CP (Consistency + Partition Tolerance) Strong ACID
Cassandra, DynamoDB AP (Availability + Partition Tolerance) Eventual Consistency, Limited ACID
💡 Key Insight: Traditional relational databases (SQL Server, PostgreSQL, Oracle) provide the strongest ACID guarantees, making them ideal for financial, medical, and other critical applications where data integrity is paramount.

T-SQL Transaction Basics

SQL Server provides comprehensive transaction support through T-SQL commands and automatic transaction management.

Explicit Transactions

-- Basic explicit transaction
BEGIN TRANSACTION;
    UPDATE Products SET Price = Price * 1.1 WHERE CategoryID = 1;
    INSERT INTO PriceHistory (ProductID, OldPrice, NewPrice, ChangeDate)
    SELECT ProductID, Price / 1.1, Price, GETDATE()
    FROM Products WHERE CategoryID = 1;
COMMIT TRANSACTION;

-- Transaction with error handling
BEGIN TRANSACTION;
    BEGIN TRY
        UPDATE Accounts SET Balance = Balance - 1000 WHERE AccountID = 1;
        UPDATE Accounts SET Balance = Balance + 1000 WHERE AccountID = 2;
        COMMIT TRANSACTION;
    END TRY
    BEGIN CATCH
        ROLLBACK TRANSACTION;
        THROW;
    END CATCH

Implicit Transactions

-- Enable implicit transactions
SET IMPLICIT_TRANSACTIONS ON;

-- This automatically starts a transaction
UPDATE Products SET Price = 100 WHERE ProductID = 1;

-- Must explicitly commit or rollback
COMMIT;

-- Turn off implicit transactions
SET IMPLICIT_TRANSACTIONS OFF;

Autocommit Mode (Default)

-- Each statement is automatically a transaction
UPDATE Products SET Price = 100 WHERE ProductID = 1;
-- Automatically committed if successful

INSERT INTO Orders (CustomerID, TotalAmount) VALUES (1, 500);
-- Each statement is its own transaction

Transaction Control Statements

Statement Purpose Example

ROLLBACK Undo all changes ROLLBACK TRANSACTION or ROLLBACK

Savepoints

Using Savepoints for Partial Rollback

BEGIN TRANSACTION;
    
    INSERT INTO Orders (CustomerID, TotalAmount) 
    VALUES (1, 100);
    
    SAVE TRANSACTION AfterOrderInsert;
    
    INSERT INTO OrderItems (OrderID, ProductID, Quantity)
    VALUES (SCOPE_IDENTITY(), 5, 2);
    
    -- Error occurs here
    IF @@ERROR <> 0
    BEGIN
        -- Rollback only to savepoint, keep order insert
        ROLLBACK TRANSACTION AfterOrderInsert;
        -- Continue with alternative action
    END
    
COMMIT TRANSACTION;

Checking Transaction State

-- Check if in a transaction
SELECT @@TRANCOUNT AS ActiveTransactions;
-- Returns 0 if no transaction, >0 for nested transaction depth

-- Common pattern
IF @@TRANCOUNT > 0
    ROLLBACK TRANSACTION;

-- Get transaction isolation level
DBCC USEROPTIONS;

-- View active transactions
SELECT *
FROM sys.dm_tran_active_transactions;

-- View session transactions
SELECT *
FROM sys.dm_tran_session_transactions;
⚠️ Important: Always check @@TRANCOUNT before executing ROLLBACK to avoid errors. Never assume you're in a transaction.

Transaction Isolation Levels

Isolation levels control how transactions interact with each other and what data they can see. Higher isolation provides more protection but reduces concurrency.

Isolation Level Comparison

Isolation Level Dirty Read Non-Repeatable Read Phantom Read Concurrency
READ UNCOMMITTED ✅ Allowed ✅ Allowed ✅ Allowed ⭐⭐⭐⭐⭐ Highest
READ COMMITTED ❌ Prevented ✅ Allowed ✅ Allowed ⭐⭐⭐⭐ High
REPEATABLE READ ❌ Prevented ❌ Prevented ✅ Allowed ⭐⭐⭐ Moderate
SERIALIZABLE ❌ Prevented ❌ Prevented ❌ Prevented ⭐⭐ Low
SNAPSHOT ❌ Prevented ❌ Prevented ❌ Prevented ⭐⭐⭐⭐⭐ Highest

READ UNCOMMITTED (Level 0)

Description: Lowest isolation level. Can read uncommitted changes from other transactions (dirty reads).

Use Case: Reporting queries where approximate data is acceptable and performance is critical.

Advantages: No locks placed, no blocking, maximum concurrency.

Disadvantages: Can read data that gets rolled back, data accuracy not guaranteed.

SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

SELECT COUNT(*), SUM(TotalAmount)
FROM Orders;
-- Can see uncommitted changes from other transactions

READ COMMITTED (Level 1) - Default

Description: Can only read committed data. Prevents dirty reads but allows non-repeatable reads.

Use Case: Default for most applications. Good balance of consistency and concurrency.

Advantages: Prevents reading uncommitted data, reasonable concurrency.

Disadvantages: Data can change between reads within same transaction.

SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

BEGIN TRANSACTION;
    SELECT Balance FROM Accounts WHERE AccountID = 1; -- Returns 1000
    -- Another transaction commits: UPDATE Accounts SET Balance = 1500
    SELECT Balance FROM Accounts WHERE AccountID = 1; -- Returns 1500!
COMMIT TRANSACTION;

REPEATABLE READ (Level 2)

Description: Guarantees same data on repeated reads. Locks rows that are read, preventing changes.

Use Case: When you need consistent reads throughout a transaction.

Advantages: Data stability within transaction, prevents non-repeatable reads.

Disadvantages: More blocking, can still have phantom reads (new rows).

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

BEGIN TRANSACTION;
    SELECT * FROM Products WHERE CategoryID = 1;
    -- Rows are locked, other transactions can't modify them
    -- But new rows can be inserted (phantom reads)
    SELECT * FROM Products WHERE CategoryID = 1;
    -- Same rows, but count might differ if new rows inserted
COMMIT TRANSACTION;

SERIALIZABLE (Level 3)

Description: Highest isolation level. Complete isolation from other transactions. Prevents phantom reads.

Use Case: Critical operations requiring absolute consistency (rare).

Advantages: Complete isolation, no anomalies possible.

Disadvantages: Significant blocking, lowest concurrency, potential deadlocks.

SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

BEGIN TRANSACTION;
    SELECT COUNT(*) FROM Orders WHERE OrderDate = '2024-02-09';
    -- Range lock prevents inserts, updates, deletes
    SELECT COUNT(*) FROM Orders WHERE OrderDate = '2024-02-09';
    -- Guaranteed same count - no phantoms
COMMIT TRANSACTION;

SNAPSHOT Isolation

Description: Uses row versioning (MVCC). Reads a consistent snapshot of data as it existed at transaction start.

Use Case: High concurrency scenarios with consistent reads needed.

Advantages: Readers don't block writers, writers don't block readers, high concurrency.

Disadvantages: Tempdb overhead for row versions, update conflicts possible.

-- Enable snapshot isolation (database setting)
ALTER DATABASE MyDatabase 
SET ALLOW_SNAPSHOT_ISOLATION ON;

SET TRANSACTION ISOLATION LEVEL SNAPSHOT;

BEGIN TRANSACTION;
    SELECT * FROM Products WHERE CategoryID = 1;
    -- Sees snapshot as of transaction start
    -- Other transactions can modify data without blocking
    SELECT * FROM Products WHERE CategoryID = 1;
    -- Still sees original snapshot, not modifications
COMMIT TRANSACTION;

Setting Isolation Levels

-- Session-level setting
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- Query-level hint
SELECT * FROM Orders WITH (READUNCOMMITTED);
SELECT * FROM Orders WITH (NOLOCK); -- Same as READ UNCOMMITTED

-- Transaction-level (recommended)
BEGIN TRANSACTION;
    SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
    -- queries here use REPEATABLE READ
COMMIT TRANSACTION;
💡 Recommendation: Use READ COMMITTED (default) for most applications. Consider SNAPSHOT for high-concurrency read-heavy workloads. Avoid READ UNCOMMITTED unless data accuracy is not critical.

Locking & Concurrency

SQL Server uses locks to ensure data integrity and isolation between concurrent transactions.

Lock Types

Lock Type Symbol Purpose Typical Use
Shared (S) S Read locks, multiple allowed SELECT statements
Exclusive (X) X Write locks, blocks all others INSERT, UPDATE, DELETE
Update (U) U Intent to update, prevents deadlocks UPDATE operations
Intent (IS, IX, IU) I* Lock hierarchy indicator Automatic by SQL Server
Schema (Sch-S, Sch-M) Sch Prevent schema changes DDL operations

Lock Granularity

Granularity What's Locked Concurrency Overhead
Row (RID) Single row ⭐⭐⭐⭐⭐ Highest High memory
Key Index key ⭐⭐⭐⭐ High Moderate memory
Page 8KB page (rows) ⭐⭐⭐ Moderate Low memory
Extent 8 pages (64KB) ⭐⭐ Low Very low memory
Table Entire table ⭐ Very Low Minimal memory
Database Entire database ❌ None Minimal memory

Lock Compatibility Matrix

Existing → Request ↓ Shared (S) Exclusive (X) Update (U)
Shared (S) ✅ Compatible ❌ Blocks ✅ Compatible
Exclusive (X) ❌ Blocks ❌ Blocks ❌ Blocks
Update (U) ✅ Compatible ❌ Blocks ❌ Blocks

Viewing Locks

-- View current locks
SELECT 
    resource_type,
    resource_description,
    request_mode,
    request_status
FROM sys.dm_tran_locks
WHERE request_session_id = @@SPID;

-- Find blocked processes
SELECT 
    blocking.session_id AS BlockingSessionID,
    blocked.session_id AS BlockedSessionID,
    blocked.wait_type,
    blocked.wait_time / 1000.0 AS WaitTimeSeconds
FROM sys.dm_exec_requests blocked
INNER JOIN sys.dm_exec_requests blocking 
    ON blocked.blocking_session_id = blocking.session_id;

-- Kill blocking session (use with caution!)
KILL 52; -- session_id of blocker

Lock Hints

-- Table-level lock hints
SELECT * FROM Orders WITH (NOLOCK); -- READ UNCOMMITTED
SELECT * FROM Orders WITH (HOLDLOCK); -- Hold locks until transaction ends
SELECT * FROM Orders WITH (UPDLOCK); -- Update lock instead of shared
SELECT * FROM Orders WITH (XLOCK); -- Exclusive lock
SELECT * FROM Orders WITH (TABLOCK); -- Table lock
SELECT * FROM Orders WITH (ROWLOCK); -- Force row-level locks

Deadlocks

Example: Deadlock Scenario

-- Transaction 1
BEGIN TRANSACTION;
    UPDATE Accounts SET Balance = Balance - 100 WHERE AccountID = 1;
    WAITFOR DELAY '00:00:05';
    UPDATE Accounts SET Balance = Balance + 100 WHERE AccountID = 2;
COMMIT TRANSACTION;

-- Transaction 2 (runs simultaneously)
BEGIN TRANSACTION;
    UPDATE Accounts SET Balance = Balance - 50 WHERE AccountID = 2;
    WAITFOR DELAY '00:00:05';
    UPDATE Accounts SET Balance = Balance + 50 WHERE AccountID = 1;
COMMIT TRANSACTION;

-- Result: One transaction becomes deadlock victim and rolls back

Preventing Deadlocks

✅ Deadlock Prevention Strategies:
  • Access tables in consistent order across all transactions
  • Keep transactions short and fast
  • Use lower isolation levels when appropriate
  • Use ROWLOCK hints to reduce lock escalation
  • Create appropriate indexes to reduce scan locks
  • Consider using SNAPSHOT isolation
  • Handle deadlock errors with retry logic

Deadlock Detection

-- Enable deadlock trace flag
DBCC TRACEON(1222, -1); -- Log deadlocks to error log

-- View deadlock history
SELECT 
    CAST(target_data AS XML) AS DeadlockGraph
FROM sys.dm_xe_session_targets st
JOIN sys.dm_xe_sessions s 
    ON s.address = st.event_session_address
WHERE s.name = 'system_health'
    AND st.target_name = 'ring_buffer';

Transactions in Stored Procedures

Stored procedures are ideal for encapsulating transactional logic, providing consistency, reusability, and proper error handling.

Basic Transaction Pattern

Example: Simple Transaction Procedure

CREATE PROCEDURE usp_TransferFunds
    @FromAccount INT,
    @ToAccount INT,
    @Amount DECIMAL(10,2)
AS
BEGIN
    SET NOCOUNT ON;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Debit source account
        UPDATE Accounts
        SET Balance = Balance - @Amount
        WHERE AccountID = @FromAccount;
        
        -- Credit destination account
        UPDATE Accounts
        SET Balance = Balance + @Amount
        WHERE AccountID = @ToAccount;
        
        -- Log transaction
        INSERT INTO TransactionLog (FromAccount, ToAccount, Amount, TransDate)
        VALUES (@FromAccount, @ToAccount, @Amount, GETDATE());
        
        COMMIT TRANSACTION;
        RETURN 0;
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        THROW;
    END CATCH
END;
GO

Validation Before Transaction

Example: Validate Then Execute

CREATE PROCEDURE usp_CreateOrderWithValidation
    @CustomerID INT,
    @ProductID INT,
    @Quantity INT,
    @OrderID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    -- Validate BEFORE starting transaction
    IF NOT EXISTS (SELECT 1 FROM Customers WHERE CustomerID = @CustomerID)
    BEGIN
        RAISERROR('Customer does not exist', 16, 1);
        RETURN -1;
    END
    
    IF NOT EXISTS (SELECT 1 FROM Products WHERE ProductID = @ProductID)
    BEGIN
        RAISERROR('Product does not exist', 16, 1);
        RETURN -1;
    END
    
    IF @Quantity <= 0
    BEGIN
        RAISERROR('Quantity must be positive', 16, 1);
        RETURN -1;
    END
    
    -- NOW start transaction after validation passes
    BEGIN TRY
        BEGIN TRANSACTION;
        
        DECLARE @UnitPrice DECIMAL(10,2);
        
        -- Get price and check stock in one query
        SELECT @UnitPrice = UnitPrice
        FROM Products
        WHERE ProductID = @ProductID
            AND StockQuantity >= @Quantity;
        
        IF @UnitPrice IS NULL
        BEGIN
            RAISERROR('Insufficient stock', 16, 1);
            RETURN -1;
        END
        
        -- Create order
        INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
        VALUES (@CustomerID, GETDATE(), @UnitPrice * @Quantity, 'Pending');
        
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Add order item
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
        VALUES (@OrderID, @ProductID, @Quantity, @UnitPrice);
        
        -- Update inventory
        UPDATE Products
        SET StockQuantity = StockQuantity - @Quantity
        WHERE ProductID = @ProductID;
        
        COMMIT TRANSACTION;
        RETURN 0;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        THROW;
    END CATCH
END;
GO

Using XACT_ABORT

Example: Automatic Rollback with XACT_ABORT

CREATE PROCEDURE usp_ProcessOrderWithXactAbort
    @CustomerID INT,
    @OrderItems XML
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT ON; -- Auto-rollback on any error
    
    BEGIN TRANSACTION;
    
    -- Create order
    INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
    VALUES (@CustomerID, GETDATE(), 0, 'Pending');
    
    DECLARE @OrderID INT = SCOPE_IDENTITY();
    
    -- Insert items from XML
    INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
    SELECT 
        @OrderID,
        item.value('ProductID[1]', 'INT'),
        item.value('Quantity[1]', 'INT'),
        p.UnitPrice
    FROM @OrderItems.nodes('/Items/Item') AS Items(item)
    INNER JOIN Products p ON p.ProductID = item.value('ProductID[1]', 'INT');
    
    -- Update order total
    UPDATE Orders
    SET TotalAmount = (
        SELECT SUM(LineTotal)
        FROM OrderItems
        WHERE OrderID = @OrderID
    )
    WHERE OrderID = @OrderID;
    
    COMMIT TRANSACTION;
    -- With XACT_ABORT ON, any error automatically rolls back
    -- No need for TRY...CATCH for simple cases
END;
GO

Transaction with Multiple Calls

Example: Coordinating Multiple Procedures

CREATE PROCEDURE usp_ProcessCompleteOrder
    @CustomerID INT,
    @ShippingAddress NVARCHAR(200)
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @OrderID INT;
    DECLARE @ShipmentID INT;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Step 1: Create order
        EXEC usp_CreateOrder 
            @CustomerID = @CustomerID,
            @OrderID = @OrderID OUTPUT;
        
        -- Step 2: Add items
        EXEC usp_AddOrderItems @OrderID = @OrderID;
        
        -- Step 3: Calculate shipping
        EXEC usp_CalculateShipping 
            @OrderID = @OrderID,
            @ShippingAddress = @ShippingAddress;
        
        -- Step 4: Process payment
        EXEC usp_ProcessPayment @OrderID = @OrderID;
        
        -- Step 5: Create shipment
        EXEC usp_CreateShipment 
            @OrderID = @OrderID,
            @ShipmentID = @ShipmentID OUTPUT;
        
        COMMIT TRANSACTION;
        
        -- Return results
        SELECT @OrderID AS OrderID, @ShipmentID AS ShipmentID;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        DECLARE @ErrorMsg NVARCHAR(4000) = ERROR_MESSAGE();
        RAISERROR(@ErrorMsg, 16, 1);
    END CATCH
END;
GO
💡 Best Practice: Perform all validation before starting the transaction to keep transaction duration short and reduce lock contention.

Nested Transactions

SQL Server supports nested BEGIN TRANSACTION statements, but they don't work as most developers expect.

How "Nested" Transactions Really Work

⚠️ Important Truth: SQL Server doesn't truly support nested transactions. Multiple BEGIN TRANSACTION statements only increment @@TRANCOUNT. Only the outermost COMMIT actually commits. A ROLLBACK at any level rolls back the entire transaction.
BEGIN TRANSACTION; -- @@TRANCOUNT = 1
    INSERT INTO Table1 ...
    
    BEGIN TRANSACTION; -- @@TRANCOUNT = 2 (not a new transaction!)
        INSERT INTO Table2 ...
    COMMIT TRANSACTION; -- @@TRANCOUNT = 1 (nothing actually committed)
    
    INSERT INTO Table3 ...
COMMIT TRANSACTION; -- @@TRANCOUNT = 0 (NOW everything commits)

Proper Nested Transaction Pattern

Example: Safe Nesting with @@TRANCOUNT Check

CREATE PROCEDURE usp_InnerProcedure
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @TranStarted BIT = 0;
    
    BEGIN TRY
        -- Only start transaction if not already in one
        IF @@TRANCOUNT = 0
        BEGIN
            BEGIN TRANSACTION;
            SET @TranStarted = 1;
        END
        
        -- Do work
        INSERT INTO LogTable (Message) VALUES ('Inner procedure executed');
        
        -- Only commit if we started the transaction
        IF @TranStarted = 1
            COMMIT TRANSACTION;
            
    END TRY
    BEGIN CATCH
        -- Only rollback if we started the transaction
        IF @TranStarted = 1 AND @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        
        THROW; -- Re-throw to caller
    END CATCH
END;
GO

CREATE PROCEDURE usp_OuterProcedure
AS
BEGIN
    SET NOCOUNT ON;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        INSERT INTO LogTable (Message) VALUES ('Outer procedure start');
        
        -- Call inner procedure (which checks @@TRANCOUNT)
        EXEC usp_InnerProcedure;
        
        INSERT INTO LogTable (Message) VALUES ('Outer procedure end');
        
        COMMIT TRANSACTION;
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;
GO

Savepoints for Partial Rollback

Example: Using Savepoints Instead of Nesting

CREATE PROCEDURE usp_ProcessOrderWithSavepoints
    @CustomerID INT
AS
BEGIN
    SET NOCOUNT ON;
    
    BEGIN TRANSACTION;
    
    BEGIN TRY
        -- Step 1: Create order
        INSERT INTO Orders (CustomerID, OrderDate) 
        VALUES (@CustomerID, GETDATE());
        
        DECLARE @OrderID INT = SCOPE_IDENTITY();
        
        -- Create savepoint before risky operation
        SAVE TRANSACTION BeforeInventoryUpdate;
        
        BEGIN TRY
            -- Step 2: Update inventory (might fail)
            UPDATE Products 
            SET StockQuantity = StockQuantity - 10
            WHERE ProductID = 1;
            
            IF @@ROWCOUNT = 0
                RAISERROR('Product not found', 16, 1);
                
        END TRY
        BEGIN CATCH
            -- Rollback only to savepoint
            ROLLBACK TRANSACTION BeforeInventoryUpdate;
            
            -- Log error but continue
            INSERT INTO ErrorLog (OrderID, ErrorMsg)
            VALUES (@OrderID, ERROR_MESSAGE());
            
            -- Mark order as pending review
            UPDATE Orders 
            SET Status = 'Pending Review'
            WHERE OrderID = @OrderID;
        END CATCH
        
        -- Commit main transaction (order is saved)
        COMMIT TRANSACTION;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;
GO
✅ Nested Transaction Best Practices:
  • Always check @@TRANCOUNT before BEGIN/COMMIT/ROLLBACK
  • Use a flag to track if procedure started the transaction
  • Only the procedure that started transaction should commit it
  • Use savepoints for partial rollback scenarios
  • Always re-throw errors to calling procedure
  • Document transaction expectations in procedure header

Distributed Transactions

Distributed transactions span multiple databases or servers, requiring coordination between multiple resource managers.

Distributed Transaction Coordinators

Technology Platform Use Case
MS-DTC Windows, SQL Server Cross-database, cross-server transactions
Two-Phase Commit (2PC) Most RDBMS Traditional distributed transactions
Saga Pattern Microservices Long-running, compensating transactions

Using Distributed Transactions in T-SQL

-- Distributed transaction across linked servers
BEGIN DISTRIBUTED TRANSACTION;
    
    -- Update local database
    UPDATE LocalDB.dbo.Orders 
    SET Status = 'Completed'
    WHERE OrderID = 1;
    
    -- Update remote database via linked server
    UPDATE RemoteServer.RemoteDB.dbo.Inventory
    SET Quantity = Quantity - 1
    WHERE ProductID = 5;
    
COMMIT TRANSACTION;
-- MS-DTC coordinates commit across both servers

Distributed Transaction Challenges

⚠️ Challenges with Distributed Transactions:
  • Performance: Slower than local transactions
  • Complexity: More points of failure
  • Availability: All participants must be available
  • Network: Network latency and failures
  • Locking: Locks held across multiple systems
  • Configuration: Requires MS-DTC or similar setup

Alternative: Saga Pattern

Example: Saga with Compensating Transactions

-- Instead of distributed transaction, use saga pattern
CREATE PROCEDURE usp_ProcessOrderSaga
    @CustomerID INT,
    @ProductID INT,
    @Quantity INT
AS
BEGIN
    DECLARE @OrderID INT;
    DECLARE @InventoryReserved BIT = 0;
    DECLARE @PaymentProcessed BIT = 0;
    
    BEGIN TRY
        -- Step 1: Create order (local transaction)
        INSERT INTO Orders (CustomerID, Status)
        VALUES (@CustomerID, 'Pending');
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Step 2: Reserve inventory (could be remote call)
        EXEC usp_ReserveInventory @ProductID, @Quantity;
        SET @InventoryReserved = 1;
        
        -- Step 3: Process payment (could be external API)
        EXEC usp_ProcessPayment @OrderID;
        SET @PaymentProcessed = 1;
        
        -- Step 4: Confirm order
        UPDATE Orders SET Status = 'Confirmed' 
        WHERE OrderID = @OrderID;
        
    END TRY
    BEGIN CATCH
        -- Compensating transactions (undo what was done)
        IF @PaymentProcessed = 1
            EXEC usp_RefundPayment @OrderID;
            
        IF @InventoryReserved = 1
            EXEC usp_ReleaseInventory @ProductID, @Quantity;
            
        IF @OrderID IS NOT NULL
            UPDATE Orders SET Status = 'Cancelled' 
            WHERE OrderID = @OrderID;
        
        THROW;
    END CATCH
END;
GO
💡 Modern Approach: For microservices and distributed systems, the Saga pattern with compensating transactions is often preferred over traditional distributed transactions due to better scalability and fault tolerance.

Common Transaction Use Cases

Transactions are essential in many real-world scenarios where data integrity is critical.

1. Financial Transactions

Banking: Account Transfer

CREATE PROCEDURE usp_BankTransfer
    @FromAccount VARCHAR(20),
    @ToAccount VARCHAR(20),
    @Amount DECIMAL(18,2),
    @Reference VARCHAR(100)
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT ON;
    
    BEGIN TRANSACTION;
    
    -- Lock accounts for update
    DECLARE @FromBalance DECIMAL(18,2);
    SELECT @FromBalance = Balance 
    FROM Accounts WITH (UPDLOCK, ROWLOCK)
    WHERE AccountNumber = @FromAccount;
    
    -- Check sufficient funds
    IF @FromBalance < @Amount
    BEGIN
        ROLLBACK TRANSACTION;
        RAISERROR('Insufficient funds', 16, 1);
        RETURN;
    END
    
    -- Debit source
    UPDATE Accounts 
    SET Balance = Balance - @Amount,
        LastTransaction = GETDATE()
    WHERE AccountNumber = @FromAccount;
    
    -- Credit destination
    UPDATE Accounts 
    SET Balance = Balance + @Amount,
        LastTransaction = GETDATE()
    WHERE AccountNumber = @ToAccount;
    
    -- Record transaction
    INSERT INTO Transactions (FromAccount, ToAccount, Amount, Reference, TransDate)
    VALUES (@FromAccount, @ToAccount, @Amount, @Reference, GETDATE());
    
    COMMIT TRANSACTION;
END;
GO

2. E-Commerce: Order Processing

Complete Order with Inventory and Payment

CREATE PROCEDURE usp_ProcessEcommerceOrder
    @CustomerID INT,
    @CartID INT,
    @PaymentMethod VARCHAR(20),
    @OrderID INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        -- Create order header
        INSERT INTO Orders (CustomerID, OrderDate, Status)
        VALUES (@CustomerID, GETDATE(), 'Processing');
        
        SET @OrderID = SCOPE_IDENTITY();
        
        -- Transfer cart items to order
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
        SELECT @OrderID, ProductID, Quantity, UnitPrice
        FROM CartItems
        WHERE CartID = @CartID;
        
        -- Check inventory and reserve stock
        UPDATE p
        SET p.StockQuantity = p.StockQuantity - oi.Quantity,
            p.ReservedQuantity = p.ReservedQuantity + oi.Quantity
        FROM Products p
        INNER JOIN OrderItems oi ON p.ProductID = oi.ProductID
        WHERE oi.OrderID = @OrderID
            AND p.StockQuantity >= oi.Quantity;
        
        -- Verify all items were reserved
        IF @@ROWCOUNT < (SELECT COUNT(*) FROM OrderItems WHERE OrderID = @OrderID)
        BEGIN
            RAISERROR('One or more items out of stock', 16, 1);
        END
        
        -- Calculate totals
        UPDATE Orders
        SET TotalAmount = (
            SELECT SUM(Quantity * UnitPrice)
            FROM OrderItems
            WHERE OrderID = @OrderID
        )
        WHERE OrderID = @OrderID;
        
        -- Record payment
        INSERT INTO Payments (OrderID, PaymentMethod, Amount, Status)
        SELECT @OrderID, @PaymentMethod, TotalAmount, 'Pending'
        FROM Orders
        WHERE OrderID = @OrderID;
        
        -- Clear cart
        DELETE FROM CartItems WHERE CartID = @CartID;
        
        -- Update order status
        UPDATE Orders 
        SET Status = 'Confirmed'
        WHERE OrderID = @OrderID;
        
        COMMIT TRANSACTION;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;
GO

3. Inventory Management

Stock Adjustment with Audit Trail

CREATE PROCEDURE usp_AdjustInventory
    @ProductID INT,
    @AdjustmentQty INT,
    @Reason VARCHAR(200),
    @UserID INT
AS
BEGIN
    SET NOCOUNT ON;
    
    BEGIN TRY
        BEGIN TRANSACTION;
        
        DECLARE @OldQuantity INT;
        DECLARE @NewQuantity INT;
        
        -- Get current quantity
        SELECT @OldQuantity = StockQuantity
        FROM Products WITH (UPDLOCK)
        WHERE ProductID = @ProductID;
        
        IF @OldQuantity IS NULL
        BEGIN
            RAISERROR('Product not found', 16, 1);
        END
        
        SET @NewQuantity = @OldQuantity + @AdjustmentQty;
        
        -- Prevent negative inventory
        IF @NewQuantity < 0
        BEGIN
            RAISERROR('Adjustment would result in negative inventory', 16, 1);
        END
        
        -- Update inventory
        UPDATE Products
        SET StockQuantity = @NewQuantity,
            LastUpdated = GETDATE()
        WHERE ProductID = @ProductID;
        
        -- Create audit record
        INSERT INTO InventoryAudit (
            ProductID, 
            OldQuantity, 
            NewQuantity, 
            AdjustmentQty,
            Reason, 
            UserID, 
            AdjustmentDate
        )
        VALUES (
            @ProductID,
            @OldQuantity,
            @NewQuantity,
            @AdjustmentQty,
            @Reason,
            @UserID,
            GETDATE()
        );
        
        COMMIT TRANSACTION;
        
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;
GO

Error Handling in Transactions

Proper error handling is crucial for transaction management to ensure data integrity.

TRY...CATCH with Transactions

BEGIN TRY
    BEGIN TRANSACTION;
    
    -- Your operations here
    UPDATE Table1 ...
    INSERT INTO Table2 ...
    
    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    -- ALWAYS check @@TRANCOUNT before ROLLBACK
    IF @@TRANCOUNT > 0
        ROLLBACK TRANSACTION;
    
    -- Log error details
    DECLARE @ErrorMessage NVARCHAR(4000) = ERROR_MESSAGE();
    DECLARE @ErrorSeverity INT = ERROR_SEVERITY();
    DECLARE @ErrorState INT = ERROR_STATE();
    DECLARE @ErrorNumber INT = ERROR_NUMBER();
    DECLARE @ErrorLine INT = ERROR_LINE();
    
    -- Re-throw the error
    RAISERROR(@ErrorMessage, @ErrorSeverity, @ErrorState);
END CATCH

Using THROW (SQL Server 2012+)

BEGIN TRY
    BEGIN TRANSACTION;
    
    -- Operations
    
    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    IF @@TRANCOUNT > 0
        ROLLBACK TRANSACTION;
    
    THROW; -- Re-throws original error with all details
END CATCH

Handling Specific Errors

BEGIN TRY
    BEGIN TRANSACTION;
    
    INSERT INTO Users (Username, Email)
    VALUES ('john', 'john@email.com');
    
    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    IF @@TRANCOUNT > 0
        ROLLBACK TRANSACTION;
    
    -- Handle specific error codes
    IF ERROR_NUMBER() = 2627 -- Unique constraint violation
    BEGIN
        RAISERROR('Username or email already exists', 16, 1);
    END
    ELSE IF ERROR_NUMBER() = 547 -- FK constraint violation
    BEGIN
        RAISERROR('Invalid reference to related table', 16, 1);
    END
    ELSE IF ERROR_NUMBER() = 1205 -- Deadlock victim
    BEGIN
        -- Implement retry logic
        RAISERROR('Deadlock occurred, please retry', 16, 1);
    END
    ELSE
    BEGIN
        THROW; -- Unknown error
    END
END CATCH
⚠️ Critical: NEVER let a transaction remain open after an error. Always ensure transactions are either committed or rolled back in error handlers.

Performance Considerations

Transaction management has significant performance implications. Understanding these is crucial for building scalable systems.

Transaction Duration Impact

Duration Impact Recommendation
< 1 second ✅ Good - Minimal blocking Ideal for most operations
1-5 seconds ⚠️ Acceptable - Some blocking Acceptable for complex operations
5-10 seconds ⚠️ Concerning - Significant blocking Optimize or break into smaller transactions
> 10 seconds ❌ Poor - High blocking, deadlock risk Redesign required

Performance Best Practices

✅ Keep Transactions Short: 
-- BAD: Long transaction with user interaction
BEGIN TRANSACTION;
    UPDATE Orders SET Status = 'Processing';
    WAITFOR DELAY '00:00:30'; -- Waiting for external API!
    UPDATE Orders SET Status = 'Completed';
COMMIT TRANSACTION;

-- GOOD: Short transaction
-- Do external work OUTSIDE transaction
DECLARE @Result BIT = dbo.fn_CallExternalAPI();

BEGIN TRANSACTION;
    UPDATE Orders 
    SET Status = CASE WHEN @Result = 1 THEN 'Completed' ELSE 'Failed' END;
COMMIT TRANSACTION;

Minimize Lock Contention

✅ Access Tables in Consistent Order: 
-- Transaction 1 and Transaction 2 should access tables in SAME order
-- GOOD: Both transactions access Orders then OrderItems
BEGIN TRANSACTION;
    UPDATE Orders ...
    UPDATE OrderItems ...
COMMIT TRANSACTION;

-- BAD: Different order causes deadlocks
-- Transaction 1: Orders → OrderItems
-- Transaction 2: OrderItems → Orders

Use Appropriate Isolation Levels

-- For reports, use lower isolation
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT COUNT(*), SUM(TotalAmount)
FROM Orders;

-- For critical updates, use higher isolation
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN TRANSACTION;
    UPDATE CriticalData ...
COMMIT TRANSACTION;

Batch Large Operations

Example: Process in Batches

-- Instead of one huge transaction
DECLARE @BatchSize INT = 1000;
DECLARE @RowsAffected INT = 1;

WHILE @RowsAffected > 0
BEGIN
    BEGIN TRANSACTION;
    
    -- Process one batch
    UPDATE TOP (@BatchSize) Orders
    SET Status = 'Archived'
    WHERE OrderDate < DATEADD(YEAR, -5, GETDATE())
        AND Status = 'Completed';
    
    SET @RowsAffected = @@ROWCOUNT;
    
    COMMIT TRANSACTION;
    
    -- Brief pause to allow other transactions
    WAITFOR DELAY '00:00:00.100';
END

Monitor Transaction Performance

-- Find long-running transactions
SELECT 
    s.session_id,
    s.login_name,
    t.transaction_id,
    DATEDIFF(SECOND, t.transaction_begin_time, GETDATE()) AS DurationSeconds,
    t.transaction_state,
    s.last_request_start_time
FROM sys.dm_tran_active_transactions t
INNER JOIN sys.dm_tran_session_transactions st 
    ON t.transaction_id = st.transaction_id
INNER JOIN sys.dm_exec_sessions s 
    ON st.session_id = s.session_id
WHERE DATEDIFF(SECOND, t.transaction_begin_time, GETDATE()) > 10
ORDER BY DurationSeconds DESC;

DBCC SQLPERF(LOGSPACE);

Transaction Best Practices Summary

Essential Guidelines

✅ Do:
  • Keep transactions as short as possible
  • Validate data BEFORE starting transaction
  • Always use TRY...CATCH for error handling
  • Check @@TRANCOUNT before ROLLBACK
  • Use explicit transactions for multi-step operations
  • Access tables in consistent order to prevent deadlocks
  • Use appropriate isolation levels for your scenario
  • Log errors comprehensively
  • Use XACT_ABORT ON for simple cases
  • Monitor transaction performance regularly
  • Test rollback scenarios thoroughly
  • Document transaction expectations in procedures
❌ Don't:
  • Leave transactions open after errors
  • Include user interaction within transactions
  • Make external API calls inside transactions
  • Use high isolation levels unless necessary
  • Forget to commit or rollback
  • Hold locks longer than necessary
  • Ignore deadlock possibilities
  • Use distributed transactions unless required
  • Nest transactions without checking @@TRANCOUNT
  • Skip validation to save time
  • Assume transactions will always succeed

Quick Reference Checklist

Scenario Recommendation
Single INSERT/UPDATE/DELETE Use autocommit (no explicit transaction)
Multiple related operations Wrap in explicit transaction with TRY...CATCH
Financial operations Always use explicit transactions with full validation
Read-only reporting Consider READ UNCOMMITTED or SNAPSHOT isolation
High-concurrency updates Use READ COMMITTED or SNAPSHOT, keep very short
Calling nested procedures Check @@TRANCOUNT, use savepoints if needed
Large batch operations Process in batches with separate transactions
Cross-database operations Consider Saga pattern over distributed transactions
