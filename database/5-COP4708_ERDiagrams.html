The Two-Phase Data Modeling Process

Outcome 1
🔄

Outcome 1 · Process OverviewConceptual Phase then Logical Phase

Database design follows a deliberate, two-phase process to separate business understanding from technical implementation. Rushing directly to SQL tables skips the thinking that prevents expensive mistakes.

Phase 1
Conceptual Design
Understand the Business
  • Interview stakeholders and subject-matter experts
  • Identify the real-world entities the business tracks
  • Identify the relationships between entities
  • Determine key attributes for each entity
  • Produce an Entity-Relationship (E-R) Diagram
  • Technology-independent — no SQL, no platform decisions
  • Audience: business analysts, users, management
Transforms
into
Phase 2
Logical Design
Map to a Database
  • Convert E-R diagram entities into tables
  • Convert attributes into columns with data types
  • Establish primary keys and foreign keys
  • Resolve M:N relationships with junction tables
  • Apply normalization rules (1NF → 2NF → 3NF)
  • Produce a database schema ready to implement
  • Audience: database designers, developers, DBAs

Why Two Phases Instead of One?

Design ConcernPhase 1 (Conceptual)Phase 2 (Logical)
Language Business language — entities, relationships Technical language — tables, columns, keys
Audience Business stakeholders (can validate accuracy) Database designers and developers
Tool Entity-Relationship Diagram (ERD) Relational Schema / DDL SQL
Platform Technology-independent Specific DBMS (SQL Server, MySQL, etc.)
Goal Verify what needs to be stored Decide how to store it efficiently
💡
The Cost of Skipping Phase 1 Building tables without a conceptual model is like constructing a building without blueprints. Mistakes discovered during Phase 2 or later — missing relationships, wrong cardinalities, un-modeled entities — cost far more to fix in SQL than they would have cost to correct on paper during Phase 1.

Purpose of the Data Modeling Process

Outcome 2
🎯

Outcome 2 · RationaleWhy We Model Data Before Building

Definition — Data Model

A data model is a visual representation of a set of real-world objects (entities), their properties (attributes), and the relationships among them — created to communicate, validate, and guide the design of a database system.

The Six Core Purposes of Data Modeling

🗺️
1. Communicate Requirements The E-R diagram serves as a shared language between business users and technical developers. It translates business rules ("a customer can place many orders") into a precise visual diagram that both sides can read and validate.
🔍
2. Discover Hidden Requirements The modeling process forces structured questions: "What do we need to track about this entity? What connects these entities?" These questions often surface requirements that stakeholders had not yet articulated.
🏗️
3. Blueprint for Database Construction The E-R diagram directly maps to relational tables, columns, and constraints. A quality model reduces implementation errors and rework.
📐
4. Document Business Rules Cardinalities, required/optional attributes, and entity identifiers in the model capture real business constraints. This documentation outlasts any single project.
⚡
5. Enable Normalization Functional dependencies and relationships identified in the model feed directly into 2NF and 3NF normalization — ensuring the database is free of redundancy and update anomalies.
🔄
6. Support System Evolution As business requirements change, a well-documented model makes it far easier to extend the database safely — adding new entities, attributes, or relationships without breaking existing structure.

Types of Data Models — The Three-Schema Architecture

LevelModel TypeAudienceDetail Level
External / View User Views (subschemas) End users; shows only what each user needs Partial — subset of data
Conceptual E-R Diagram (this module) Business analysts, designers, stakeholders Entities, relationships, key attributes
Internal / Physical Relational Schema / DDL DBAs and developers Full — tables, columns, indexes, types

Entities & Attributes

Outcome 4
📦

Outcome 4 · Building BlocksDefining and Classifying Entities

Definition — Entity

An entity is any real-world person, place, object, event, or concept about which the business needs to collect and store data. In an E-R diagram, an entity becomes a table in the final database.

Strong vs. Weak Entities

🟦
Strong Entity (Regular Entity) Can be uniquely identified by its own attributes alone — it does not depend on any other entity for its existence or identification. Drawn as a solid rectangle.

Example: CUSTOMER — identified by CustomerID, which the entity owns.
🟩
Weak Entity Cannot be uniquely identified by its own attributes alone — it depends on a related strong (owner) entity for both existence and identification. Requires the owner's key plus its own partial key. Drawn as a dashed rectangle.

Example: ORDER_LINE — identified by {OrderID (from ORDER) + LineNumber}.

Entity Box Notation — Information Engineering Style

In the Information Engineering (IE) / Crow's Foot notation used in most modern tools (ERwin, Lucidchart, SQL Server SSMS), entities are drawn as rectangles with the name in the header and attributes listed below. The primary key is separated at the top.

Entity Notation Examples

Strong Entity

CUSTOMER
  • CustomerID PK
  • LastName REQ
  • FirstName REQ
  • Email REQ
  • Phone OPT
  • City OPT

Strong Entity

ORDER
  • OrderID PK
  • CustomerID FK
  • OrderDate REQ
  • ShipDate OPT
  • TotalAmount DERIVED

Weak Entity (depends on ORDER)

ORDER_LINE
  • OrderID PK/FK
  • LineNumber PK
  • ProductID FK
  • Quantity REQ
  • UnitPrice REQ
🔑
Entity Naming Conventions Entity names should be: singular (CUSTOMER, not CUSTOMERS), uppercase by convention, nouns (not verbs), and meaningful to the business. Avoid generic names like THING or DATA.
🏷️

Outcome 4 · AttributesTypes of Attributes and Their Notation

Definition — Attribute

An attribute is a property or characteristic of an entity that the business needs to store. Each attribute holds one piece of information about each instance of the entity.

Six Types of Attributes

Simple (Atomic)

A single, indivisible value. Cannot be broken into smaller meaningful parts. The most common type.

LastName, Price, Quantity
Composite

Can be divided into smaller sub-attributes, each with independent meaning. In the database, store the parts, not the whole.

FullName → {FirstName, LastName}
Address → {Street, City, State, Zip}
Multivalued

Can hold more than one value for a single entity instance. Violates 1NF — must be resolved into a separate table.

PhoneNumbers {555-1234, 555-5678}
EmailAddresses
Derived

Can be calculated from other stored attributes. Generally not stored in the database — computed on demand. Denoted with a leading "/" in diagrams.

/Age ← from BirthDate
/TotalAmount ← SUM of lines
Required

Must have a value for every entity instance — NOT NULL in SQL. Represents a mandatory business rule.

LastName NOT NULL
OrderDate NOT NULL
Optional

May be left blank — NULL allowed in SQL. Represents data the business might not always have.

MiddleName NULL
Phone NULL

Handling Composite and Multivalued Attributes in SQL

Composite Attribute → Decompose into Columns
T-SQL · Composite Attribute
-- DON'T store composite as one column -- FullName VARCHAR(100) ← wrong -- DO decompose into atomic columns CREATE TABLE Customer ( CustomerID INT IDENTITY(1,1), FirstName VARCHAR(50) NOT NULL, LastName VARCHAR(50) NOT NULL, Street VARCHAR(100) NULL, City VARCHAR(60) NULL, State CHAR(2) NULL, ZipCode CHAR(10) NULL, CONSTRAINT PK_Customer PRIMARY KEY (CustomerID) );
Multivalued Attribute → Separate Table
T-SQL · Multivalued Attribute
-- DON'T store multiple phones in one column -- Phone VARCHAR(200) ← violates 1NF -- DO create a separate table (1:M relationship) CREATE TABLE CustomerPhone ( PhoneID INT IDENTITY(1,1), CustomerID INT NOT NULL, PhoneType VARCHAR(20) NOT NULL, PhoneNumber VARCHAR(20) NOT NULL, CONSTRAINT PK_CustPhone PRIMARY KEY (PhoneID), CONSTRAINT FK_CustPhone_Cust FOREIGN KEY (CustomerID) REFERENCES Customer(CustomerID) );

Relationships

Outcome 4
🔗

Outcome 4 · RelationshipsTypes of Relationships and How to Model Them

Definition — Relationship

A relationship is a named, meaningful association between two or more entities. It represents a business rule about how entity instances connect to each other. In the E-R diagram, relationships become foreign key constraints in the logical design.

The Three Relationship Cardinality Types

One-to-One
1 : 1

Each instance of Entity A is associated with at most one instance of Entity B, and vice versa.

Example: EMPLOYEE has one PARKING_SPACE; each space is assigned to one employee.
One-to-Many
1 : M

One instance of Entity A is associated with many instances of Entity B, but each B instance relates to only one A. The most common relationship type.

Example: One CUSTOMER places many ORDERs; each ORDER belongs to one CUSTOMER.
Many-to-Many
M : N

Many instances of A relate to many instances of B. Cannot be directly implemented in a relational database — must be resolved into two 1:M relationships via a junction table.

Example: STUDENTs enroll in many COURSEs; each COURSE has many STUDENTs → ENROLLMENT junction.

Resolving a Many-to-Many Relationship

Every M:N relationship must be decomposed before it can be implemented in SQL. The junction (bridge/associative) table is created, inheriting the primary keys of both parent entities as a composite primary key.

Relationship Participation: Total vs. Partial

TypeMeaningMin CardinalityExample
Total Participation Every instance of the entity must participate in the relationship — no exceptions Minimum = 1 (mandatory) Every ORDER must have a CUSTOMER. An order without a customer cannot exist.
Partial Participation Some instances may not participate in the relationship — optional Minimum = 0 (optional) A CUSTOMER may or may not have placed any orders yet.

Entity Identifiers

Outcome 5
🔑

Outcome 5 · IdentifiersChoosing and Defining Entity Identifiers in the Conceptual Model

Definition — Entity Identifier

An entity identifier is the attribute (or minimal combination of attributes) that uniquely distinguishes every instance of an entity from all other instances. It is the conceptual-model equivalent of a primary key, and becomes the primary key during logical design.

Rules for a Good Entity Identifier

  • Unique — no two instances of the entity can share the same identifier value, now or ever
  • Non-null — every instance must have an identifier value; it cannot be unknown or missing
  • Stable — the value should rarely if ever change over the entity's lifetime
  • Minimal — use the smallest combination of attributes that achieves uniqueness
  • Simple — prefer short values (integers) over complex strings for performance

Identifier Notation in the E-R Diagram

In the Information Engineering (crow's foot) notation, the entity identifier is placed in the top section of the entity rectangle, separated from other attributes by a horizontal line, and underlined or marked with "PK".

Identifier Placement in Entity Notation

Simple Identifier

PRODUCT
  • ProductID PK · Surrogate
  • ProductName REQ
  • UnitPrice REQ
  • StockQty REQ

Single surrogate integer key

Composite Identifier

ENROLLMENT
  • StudentID PK / FK
  • CourseID PK / FK
  • Grade OPT
  • EnrollDate REQ

Two FKs together form composite PK

Natural Identifier

COURSE
  • CourseID PK · Natural
  • CourseName REQ
  • Credits REQ
  • DeptID FK

CourseID like "COP4708" is meaningful & stable

When to Use a Surrogate vs. Natural Identifier

ScenarioRecommended IdentifierReasoning
Person, Customer, Employee Surrogate (auto-int) Names and even SSNs can change; auto-int is guaranteed stable
Course, Product with stable codes Natural (code string) "COP4708" or "SKU-123" is already unique, meaningful, and stable
Junction table (M:N resolution) Composite FK keys The combination of parent PKs is the natural identifier
Event / Transaction (Order, Invoice) Surrogate OR Natural number Invoice numbers can be natural keys if guaranteed unique by the business
⚠️
Common Mistake: Using Names as Identifiers Never use a person's name (first name, last name, or full name) as an entity identifier. Names are not unique — two customers can have the same name. Names also change (marriage, legal name change), violating the stability requirement.

Minimum and Maximum Cardinality

Outcome 6
🔢

Outcome 6 · ConceptDefining Min and Max Cardinality

Every relationship in an E-R diagram has cardinality constraints on both sides. Cardinality answers two questions for each participating entity: "How few instances can participate?" (minimum) and "How many instances can participate?" (maximum).

↘️
Minimum Cardinality The smallest number of instances of Entity B that must be associated with each instance of Entity A. Can be 0 (optional/partial participation) or 1 (mandatory/total participation). Answers: "Must this entity participate?"
↗️
Maximum Cardinality The largest number of instances of Entity B that can be associated with each instance of Entity A. Can be 1 (at most one) or many / N (multiple). Answers: "How many can this entity have?"

The Four Cardinality Combinations

(0, 1)
"Zero or one"
Optional AND maximum of one. Entity may or may not participate, and if it does, it links to at most one instance.
Example: EMPLOYEE may or may not have one ASSIGNED_VEHICLE (not all employees get a vehicle).
(1, 1)
"Exactly one"
Mandatory AND maximum of one. Must participate and can link to only one instance. The strictest constraint.
Example: Every ORDER must belong to exactly one CUSTOMER. An order cannot exist without a customer.
(0, N)
"Zero or many"
Optional AND can link to many. Entity may not participate at all, or it may associate with many instances.
Example: A CUSTOMER may have zero or many ORDERs (a new customer has placed none yet).
(1, N)
"One or many"
Mandatory AND can link to many. Must have at least one association and can have multiple.
Example: An ORDER must have at least one ORDER_LINE (you cannot place an empty order).

How to Read Cardinality — Both Directions

Cardinality is always stated from both sides of the relationship. You must read the relationship in each direction separately using this formula:

Reading Template:

"Each [Entity A] is associated with a minimum of [min] and a maximum of [max] instance(s) of [Entity B]."
Then reverse: "Each [Entity B] is associated with a minimum of [min] and a maximum of [max] instance(s) of [Entity A]."

Worked example — CUSTOMER places ORDER:

  • Each CUSTOMER is associated with a minimum of 0 and a maximum of many (N) ORDERs → "A customer may have placed no orders yet, or many."
  • Each ORDER is associated with a minimum of 1 and a maximum of 1 CUSTOMERs → "Every order must belong to exactly one customer."
🐦

Outcome 6 · NotationCrow's Foot Notation — Reading and Drawing Cardinality

The Crow's Foot (Information Engineering) notation is the most widely used modern ERD notation. It expresses both minimum and maximum cardinality on each side of a relationship using a combination of symbols placed on the line near the entity.

Symbol Legend

Symbol at End of LineMeaningMin or Max?
Two vertical bars || — Exactly ONE Both min = 1 AND max = 1
Circle and bar ○| — Zero or One Min = 0, Max = 1
Bar and crow's foot |< — One or Many Min = 1, Max = many
Circle and crow's foot ○< — Zero or Many Min = 0, Max = many

Phase 2 TransitionTranslating the ERD to a Relational Schema (DDL)

Once the conceptual E-R diagram is validated, each entity maps directly to a table. The DDL below implements the university model from the ERD above in T-SQL.

T-SQL · ERD → Relational Schema
-- ═══════════════════════════════════════════════ -- COP 4708 | ERD → SQL Schema Translation -- University Registration System -- ═══════════════════════════════════════════════ -- 1. DEPARTMENT (no FK dependencies — create first) CREATE TABLE Department ( DeptID CHAR(5) NOT NULL, DeptName VARCHAR(80) NOT NULL, Building VARCHAR(60) NULL, ChairPersonID INT NULL, -- FK added after INSTRUCTOR table CONSTRAINT PK_Department PRIMARY KEY (DeptID) ); -- 2. INSTRUCTOR (depends on DEPARTMENT) CREATE TABLE Instructor ( InstructorID INT IDENTITY(1,1) NOT NULL, DeptID CHAR(5) NOT NULL, LastName VARCHAR(50) NOT NULL, FirstName VARCHAR(50) NOT NULL, Email VARCHAR(100) NOT NULL, Rank VARCHAR(30) NOT NULL, CONSTRAINT PK_Instructor PRIMARY KEY (InstructorID), CONSTRAINT UQ_Instructor_Email UNIQUE (Email), CONSTRAINT FK_Instructor_Dept FOREIGN KEY (DeptID) REFERENCES Department(DeptID) ); -- 3. COURSE (depends on DEPARTMENT) CREATE TABLE Course ( CourseID CHAR(10) NOT NULL, -- natural key e.g. 'COP4708' DeptID CHAR(5) NOT NULL, CourseName VARCHAR(100) NOT NULL, Credits TINYINT NOT NULL CONSTRAINT CK_Credits CHECK (Credits BETWEEN 1 AND 6), Description VARCHAR(500) NULL, CONSTRAINT PK_Course PRIMARY KEY (CourseID), CONSTRAINT FK_Course_Dept FOREIGN KEY (DeptID) REFERENCES Department(DeptID) ); -- 4. SECTION (depends on COURSE and INSTRUCTOR) CREATE TABLE Section ( SectionID INT IDENTITY(1,1) NOT NULL, CourseID CHAR(10) NOT NULL, InstructorID INT NOT NULL, Semester VARCHAR(10) NOT NULL, Year SMALLINT NOT NULL, Room VARCHAR(20) NULL, MaxEnrollment TINYINT NOT NULL, CONSTRAINT PK_Section PRIMARY KEY (SectionID), CONSTRAINT FK_Section_Course FOREIGN KEY (CourseID) REFERENCES Course(CourseID), CONSTRAINT FK_Section_Instructor FOREIGN KEY (InstructorID) REFERENCES Instructor(InstructorID) ); -- 5. STUDENT CREATE TABLE Student ( StudentID INT IDENTITY(1,1) NOT NULL, LastName VARCHAR(50) NOT NULL, FirstName VARCHAR(50) NOT NULL, Email VARCHAR(100) NOT NULL, DeclaredMajor CHAR(10) NULL, -- optional attribute -- GPA is derived — computed in a VIEW, not stored CONSTRAINT PK_Student PRIMARY KEY (StudentID), CONSTRAINT UQ_Student_Email UNIQUE (Email) ); -- 6. ENROLLMENT — weak entity / junction table (M:N resolution) CREATE TABLE Enrollment ( StudentID INT NOT NULL, SectionID INT NOT NULL, EnrollDate DATE NOT NULL DEFAULT GETDATE(), Grade CHAR(2) NULL, -- optional; assigned at end of term -- Composite primary key = identity of this weak entity CONSTRAINT PK_Enrollment PRIMARY KEY (StudentID, SectionID), CONSTRAINT FK_Enrollment_Student FOREIGN KEY (StudentID) REFERENCES Student(StudentID), CONSTRAINT FK_Enrollment_Section FOREIGN KEY (SectionID) REFERENCES Section(SectionID) );
