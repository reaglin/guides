Git & GitHub Integration Module

📚 Module Overview

This module covers the essential skills for using Git version control and GitHub in modern software development. You'll learn how to manage your code using visual tools integrated into popular IDEs, enabling you to work efficiently on individual and team projects without touching the command line.

What is Version Control?

Version control is a system that records changes to files over time, allowing you to:

  • Track every modification made to your code
  • Revert to previous versions if something breaks
  • Collaborate with team members without overwriting each other's work
  • Understand who made what changes and when
  • Maintain multiple versions of your project simultaneously (branches)
💡 Why Git?

Git is the most popular version control system used by over 95% of professional developers. It's distributed, fast, and integrates seamlessly with modern development tools.

Key Git Concepts

Repository (Repo)

A repository is like a project folder that Git tracks. It contains all your files and the complete history of changes.

Commit

A commit is a snapshot of your project at a specific point in time. Think of it as saving a checkpoint in a video game - you can always come back to it.

Branch

A branch is a parallel version of your code. The main branch (often called "main" or "master") is your production code, while feature branches let you work on new features without affecting the main code.

Remote

A remote is a version of your repository hosted on the internet (like GitHub). It allows you to back up your work and collaborate with others.

The Git Workflow

  1. Clone/Create: Start with a repository (either create new or clone existing)
  2. Branch: Create a branch for your new feature or bug fix
  3. Edit: Make changes to your files
  4. Stage: Select which changes to include in your next commit
  5. Commit: Save your staged changes with a descriptive message
  6. Push: Upload your commits to GitHub
  7. Pull Request: Request to merge your changes into the main branch
  8. Merge: Combine your changes with the main branch

Prerequisites

1 Install Git

Download and install Git from https://git-scm.com

💡 Tip: During installation, choose "Use Git from the Windows Command Prompt" and "Use the OpenSSL library" for best compatibility with IDEs.
2 Create a GitHub Account

Sign up for free at https://github.com

GitHub provides unlimited public and private repositories for students and developers.

3 Configure Git Identity

This only needs to be done once. Your IDE will help you set this up the first time you use Git:

  • Name: Your full name (appears in commit history)
  • Email: The email associated with your GitHub account

Initial Setup

1 Sign in to GitHub
  1. Open Visual Studio 2022
  2. Go to File → Account Settings
  3. Click "Add" under "All Accounts"
  4. Select "GitHub" and sign in with your credentials
✅ Success! Visual Studio can now access your GitHub repositories.

Starting a New Project with Git

2 Create New Project
  1. Click "Create a new project"
  2. Choose your project template (e.g., ASP.NET Core Web App)
  3. Configure your project name and location
  4. On the final screen, check "Create a new Git repository"
  5. Choose "GitHub" as the hosting service
  6. Configure repository settings:
    • Account: Your GitHub account
    • Repository name: Choose a descriptive name
    • Description: Brief description of your project
    • Visibility: Public or Private
  7. Click "Create and Push"
🎉 Project Created! Your project is now on GitHub and tracked by Git.

Cloning an Existing Repository

3 Clone from GitHub
  1. Open Visual Studio 2022
  2. Click "Clone a repository"
  3. In the "GitHub" section, you'll see your repositories
  4. Select the repository you want to clone
  5. Choose a local path where you want to save it
  6. Click "Clone"
💡 Alternative: You can also paste the repository URL directly to clone any GitHub repository.

The Git Changes Window

This is your primary interface for Git operations in Visual Studio.

4 Open Git Changes
  1. Go to View → Git Changes (or press Ctrl + 0, Ctrl + G)
  2. The window appears on the right side by default

The Git Changes window shows:

  • Changes: Files you've modified
  • Commit message box: Where you describe your changes
  • Current branch: Which branch you're working on
  • Actions: Commit, Push, Pull, Fetch options

Making Your First Commit

5 Modify Files and Commit
  1. Make changes to your project (edit a file, add a new file, etc.)
  2. Open the Git Changes window
  3. You'll see modified files listed under "Changes"
  4. Review your changes by clicking on a file (shows a diff view)
  5. Enter a commit message in the text box:
    Add user registration form - Created registration.html - Added form validation - Styled with Bootstrap
  6. Click "Commit All" button
⚠️ Important: "Commit All" saves changes locally. To upload to GitHub, you must also "Push".
💡 Commit Message Best Practices:
  • First line: Brief summary (50 characters or less)
  • Blank line
  • Detailed description of what and why (if needed)
  • Use present tense: "Add feature" not "Added feature"

Pushing Changes to GitHub

6 Push Commits

After committing, your changes are saved locally but not yet on GitHub.

  1. In the Git Changes window, look for the upward arrow with a number (indicates commits to push)
  2. Click the "Push" button or arrow
  3. Visual Studio uploads your commits to GitHub
✅ Pushed! Your changes are now safely backed up on GitHub and visible to collaborators.

Working with Branches

7 Create a New Branch

Branches let you work on features without affecting the main code.

  1. In the Git Changes window, click the branch dropdown (shows current branch)
  2. Click "New Branch"
  3. Enter a descriptive branch name:
    • feature/user-authentication
    • bugfix/login-error
    • enhancement/improve-ui
  4. Check "Checkout branch" to switch to it immediately
  5. Click "Create"
📌 Branch Naming Convention:

Use prefixes like feature/, bugfix/, or hotfix/ followed by a descriptive name. This makes it clear what each branch is for.

8 Switch Between Branches
  1. Click the branch dropdown in Git Changes
  2. Select any branch from the list to switch to it
  3. Your files will update to reflect that branch's state
⚠️ Warning: Commit or stash your changes before switching branches, or Visual Studio will warn you about uncommitted changes.
9 Push Your Branch

After creating a new branch and making commits:

  1. Click "Push" in the Git Changes window
  2. If it's a new branch, Visual Studio will ask to publish it to GitHub
  3. Click "Publish Branch"

Merging Branches

10 Merge Using Pull Requests (Recommended)
  1. Push your feature branch to GitHub
  2. In Visual Studio, go to Git → GitHub → Create Pull Request
  3. Or visit your repository on GitHub.com
  4. Click "Compare & pull request" (appears after pushing)
  5. Review the changes being merged
  6. Add a description of what the pull request does
  7. Click "Create pull request"
  8. After review, click "Merge pull request"
  9. Click "Confirm merge"
🎊 Merged! Your feature is now part of the main branch.
11 Merge Locally (Alternative)
  1. Right-click on the target branch (e.g., main) in the Git Changes window
  2. Select "Checkout" to switch to it
  3. Right-click on your feature branch
  4. Select "Merge 'branch-name' into 'main'"
  5. Resolve any conflicts if they appear
  6. Push the merged changes to GitHub

Updating Your Local Repository

12 Pull Latest Changes

When working with others, regularly pull updates from GitHub:

  1. In Git Changes, click the downward arrow (Pull)
  2. Or click Git → Pull from the menu
  3. Visual Studio downloads and merges the latest changes
💡 Best Practice: Always pull before starting new work to ensure you have the latest code.

Handling Conflicts

13 Resolve Merge Conflicts

Conflicts occur when two people edit the same lines of code. Visual Studio helps resolve them:

  1. After a pull or merge with conflicts, Visual Studio shows conflicted files
  2. Click on a conflicted file
  3. Visual Studio opens the Merge Editor with three panes:
    • Incoming: Changes from GitHub
    • Current: Your local changes
    • Result: The final merged version
  4. Choose which changes to keep by clicking checkboxes
  5. Or manually edit the Result pane
  6. Click "Accept Merge"
  7. Commit the resolved conflicts

Viewing History

14 Explore Commit History
  1. Go to View → Git Repository
  2. The Git Repository window shows a visual graph of commits
  3. Click any commit to see:
    • What changed
    • Who made the change
    • When it was made
    • The commit message
  4. Right-click a commit for options like:
    • View Commit Details
    • Reset to this commit (use carefully!)
    • Create branch from commit

  3. Click "Sign in with GitHub"

Cloning a Repository

The Source Control Panel

3 Open Source Control
  1. Click the Source Control icon in the left sidebar (looks like a branch)
  2. Or press Ctrl + Shift + G

The Source Control panel shows:

  • Changes: Modified files
  • Staged Changes: Files ready to commit
  • Commit message box: Where you describe changes
  • Branch indicator: Current branch at the bottom

Making Commits

4 Stage and Commit Changes
  1. Make changes to your files
  2. Open Source Control panel
  3. Modified files appear under "Changes"
  4. Click the + icon next to a file to stage it
  5. Or click + next to "Changes" to stage all files
  6. Enter a commit message in the text box
  7. Click the ✓ (checkmark) button to commit
💡 Quick Stage & Commit: Type your message and press Ctrl + Enter to commit all changes at once.

Pushing and Pulling

5 Sync with GitHub

To Push:

  1. After committing, click the ... (more actions) in Source Control
  2. Select "Push"
  3. Or click the cloud icon with an arrow in the status bar

To Pull:

  1. Click ... in Source Control
  2. Select "Pull"
📌 Sync: The "Sync Changes" option pulls and then pushes in one action.

Working with Branches

6 Create and Switch Branches

Create a Branch:

  1. Click the branch name in the bottom-left corner of the status bar
  2. Select "Create new branch"
  3. Enter a branch name
  4. Press Enter (automatically switches to the new branch)

Switch Branches:

  1. Click the branch name in the status bar
  2. Select the branch you want to switch to from the list

Publish a Branch:

  1. After creating a branch, the cloud icon appears in status bar
  2. Click it to publish the branch to GitHub

Using GitLens Extension (Optional but Recommended)

  1. Click the Extensions icon in the sidebar (or press Ctrl + Shift + X)
  2. Search for "GitLens"
  3. Click "Install"

GitLens Features:

  • Inline blame: See who changed each line of code

Viewing History and Diffs

8 Compare Changes

View file changes:

  1. In Source Control, click on any modified file
  2. VS Code opens a diff view showing old vs. new

View commit history:

  1. Right-click on a file in Explorer
  2. Select "View Timeline"
  3. Or use GitLens to see full repository history

Resolving Conflicts

9 Handle Merge Conflicts
  1. When a conflict occurs, VS Code highlights the conflicted files
  2. Click on a conflicted file to open it
  3. VS Code shows conflict markers with options:
    • Accept Current Change: Keep your version
    • Accept Incoming Change: Use the GitHub version
    • Accept Both Changes: Keep both versions
    • Compare Changes: See side-by-side diff
  4. Click the option that makes sense for your situation
  5. Remove any remaining conflict markers
  6. Stage and commit the resolved file

Workflow 1: Feature Development

1 Start: Pull Latest Changes

Always start by getting the latest code from the main branch.

2 Create Feature Branch

Create a new branch with a descriptive name like feature/shopping-cart

3 Develop and Commit Regularly

Make small, logical commits as you work. Each commit should represent a complete thought.

4 Push to GitHub Daily

Push your branch at least once per day to back up your work.

5 Create Pull Request

When the feature is complete, create a pull request to merge into main.

6 Code Review and Merge

Have a teammate review your code, make requested changes, then merge.

7 Delete Feature Branch

After merging, delete the feature branch to keep your repository clean.

Workflow 2: Hotfix (Urgent Bug Fix)

1 Create Hotfix Branch from Main

Branch directly from main: hotfix/login-crash

2 Fix and Test

Make the minimal changes needed to fix the issue and test thoroughly.

3 Merge Immediately

Create pull request and merge as soon as the fix is verified.

Workflow 3: Team Collaboration

1 Start Each Day with Pull

Pull changes from main to get your teammates' latest work.

2 Communicate About Work

Let teammates know which files you're working on to minimize conflicts.

3 Pull Before Pushing

Always pull before pushing to catch and resolve conflicts locally.

4 Use Pull Requests for Review

Never push directly to main. Always use pull requests for code review.

Daily Git Routine

Morning Routine

  1. Open your project
  2. Pull latest changes

  4. Start working on your branch

Throughout the Day

  1. Make logical changes
  2. Commit frequently (every 30-60 minutes of work)

End of Day

  2. Push to GitHub
  3. Create or update pull request if feature is ready

Commit Best Practices

✅ Good Commits

  • Small and focused: Each commit does one thing
  • Complete: The code works after the commit
  • Clear message: Anyone can understand what changed and why
  • Frequent: Commit often to save your progress

❌ Bad Commits

  • Too large: Changing 20 files at once
  • Broken code: Code doesn't work after the commit
  • Vague messages: "Fixed stuff" or "Updates"
  • Rare: Committing only once per day

Commit Message Format

Add user authentication system - Implemented login and registration forms - Added password hashing with bcrypt - Created user session management - Added "Remember me" functionality
💡 Template:
  • Line 1: Summary in present tense (50 chars max)
  • Line 2: Blank line
  • Lines 3+: Detailed bullet points explaining what and why

Branch Naming Conventions

Use Prefixes

  • feature/ - New features: feature/user-dashboard
  • bugfix/ - Bug fixes: bugfix/payment-error
  • hotfix/ - Urgent production fixes: hotfix/security-patch
  • enhancement/ - Improvements: enhancement/faster-loading
  • refactor/ - Code restructuring: refactor/database-layer

What NOT to Commit

⚠️ Never Commit These Files:

  • Passwords and API keys - Use environment variables instead
  • Generated files - Build outputs, compiled code
  • Dependencies - node_modules, packages (use package managers)
  • Personal settings - IDE configuration files
  • Large binary files - Videos, large datasets
  • Temporary files - Cache, logs, temp folders

Using .gitignore

1 Create .gitignore File

Create a file named .gitignore in your project root to tell Git which files to ignore.

Example .gitignore for web projects:

# Dependencies node_modules/ vendor/ # Environment variables .env .env.local # Build outputs dist/ build/ *.min.js *.min.css # IDE files .vscode/ .idea/ *.suo # Operating system files .DS_Store Thumbs.db # Logs *.log logs/ # Temporary files tmp/ temp/

Merge Conflict Prevention

💡 Strategies to Minimize Conflicts:

  • Pull frequently: Get updates from main daily
  • Keep branches short-lived: Merge features within a few days
  • Communicate with team: Coordinate who works on what files
  • Small commits: Easier to identify conflict sources
  • Consistent formatting: Use auto-formatters to avoid style conflicts

Security Best Practices

🔒 Protect Your Credentials:

  • Use environment variables: Never hardcode passwords or API keys
  • Enable 2FA on GitHub: Secure your account
  • Use SSH keys or tokens: More secure than passwords
  • Review before pushing: Check for accidental secrets
  • Use GitHub Secrets: For CI/CD pipelines

Repository Hygiene

Keep Your Repository Clean

  • Delete merged branches: Reduces clutter
  • Write a good README: Explain what the project does
  • Use meaningful folder structure: Organize code logically
  • Tag releases: Mark important versions
  • Close stale issues: Keep issue tracker relevant

Collaboration Etiquette

✅ Be a Good Team Member:

  • Review pull requests promptly: Don't leave teammates waiting
  • Write constructive reviews: Be helpful, not critical
  • Test before merging: Make sure changes work
  • Update documentation: Keep docs in sync with code
  • Ask questions: Better to clarify than assume

Issue: Can't Push to GitHub

Problem: "Permission Denied" or "Authentication Failed"

Solution:

  1. Verify you're signed in to GitHub in your IDE
  2. Check that you have push access to the repository
  3. If using tokens, ensure the token hasn't expired
  4. Try signing out and back in to refresh credentials

Issue: Merge Conflicts Are Scary

Problem: Don't Know How to Resolve Conflicts

Solution:

  1. Don't panic - conflicts are normal!
  2. Understand what conflicted:
    • Read both versions carefully
    • Ask the other developer if unsure
  3. Use your IDE's merge tool - it shows three panels
  4. Test the resolved code before committing
  5. If completely stuck, create a backup and ask for help

Issue: Accidentally Committed Wrong Files

Problem: Committed .env File or Build Artifacts

Solution:

  1. Before pushing:
    • Open Git Changes/Source Control
    • Right-click the last commit
    • Select "Undo Last Commit" (VS) or "Undo Commit" (VS Code)
    • Remove the files you don't want
    • Add to .gitignore
    • Commit again without those files
  2. After pushing:
    • Contact your instructor or team lead
    • May need to use advanced Git commands
    • Consider if the repository needs to be made private

Issue: Lost Work

Problem: Deleted Files or Lost Changes

Solution:

  1. Check Git history - your work is probably there!
  2. In VS Code/Visual Studio:
    • Right-click the file
    • Select "Timeline" or "View History"
    • Find the version you want
    • Restore it
  3. If you committed but didn't push, look in local commits
  4. If pushed, the code is safe on GitHub - just pull it

Issue: Can't Switch Branches

Problem: "Please commit your changes or stash them"

Solution:

  1. Option 1 - Commit:
    • Commit your current work
    • Then switch branches
  2. Option 2 - Stash:
    • In VS: Right-click in Git Changes → Stash
    • In VS Code: Click ... → Stash → Stash (Include Untracked)
    • Switch branches
    • Come back and "Apply Stash" to restore your work

Issue: Pushed to Wrong Branch

Problem: Committed to Main Instead of Feature Branch

Solution:

⚠️ Prevention is Best: Always check which branch you're on before committing!
  1. Create the correct branch
  2. Ask your instructor for help moving the commits
  3. This involves cherry-picking or reset commands

Issue: Repository is Broken

Problem: Everything Seems Messed Up

Solution:

  1. Don't delete the repository!
  2. Make a copy of your current work folder (just in case)
  3. Clone a fresh copy from GitHub to a new location
  4. Manually copy your recent changes to the fresh clone
  5. Commit and push from the clean copy

Getting Help

📌 When to Ask for Help:

  • You've tried the solutions above and they don't work
  • You're about to do something that might delete code
  • Error messages you don't understand
  • Merge conflicts involving critical files
  • You accidentally pushed sensitive information

Better to ask than to make things worse!
