PHASE 1 Google Account & Play Console Setup

One-time setup before you can publish anything

  • ✓
    Create a Google Play Developer Account
    Register at play.google.com/console — one-time $25 fee
    • Go to play.google.com/console and sign in with a Google account
    • Pay the one-time $25 USD registration fee
    • Fill in developer name (this is what users will see on your app listing)
    • Complete identity verification if prompted (can take 24–48 hours)
    💡 Use a dedicated Google account for your developer identity, not a personal Gmail.
  • ✓
    Complete Play Console Account Setup
    Fill in developer profile, contact details, and payment info
    • Add a developer profile photo/icon
    • Provide a public email address for user contact
    • Set up a payments profile if you plan to charge for the app or use in-app purchases
    • Review and accept the Developer Distribution Agreement
PHASE 2 Prepare Your App in Visual Studio

Configure your MAUI project for release before building

  • ✓
    Set Application ID (Package Name)
    Unique identifier — cannot be changed after publishing
    In your .csproj file, ensure this is set:
    • <ApplicationId>com.yourcompany.appname</ApplicationId>
    • Use reverse-domain format: com.yourname.fitnesstracker
    • Must be unique across all apps on Google Play
    • Lowercase letters, numbers, and dots only — no hyphens
    ⚠️ Once your app is published, this ID is permanent. Choose carefully.
  • ✓
    Set Version Code and Version Name
    versionCode must increment with every upload
    In your .csproj file:
    • <ApplicationVersion>1</ApplicationVersion> — integer, must increase each release
    • <ApplicationDisplayVersion>1.0</ApplicationDisplayVersion> — shown to users
    ⚠️ Once a version code is used (even in a failed upload), it can never be reused. Always increment before a new upload.
  • ✓
    Set App Display Name and Icon
    Name shown on device and icon displayed on home screen
    • In .csproj: <ApplicationTitle>Your App Name</ApplicationTitle>
    • App icon lives at Resources/AppIcon/appicon.svg (or .png)
    • MAUI auto-generates all required icon sizes for Android from this file
    • Recommended: use a square SVG at least 512×512px
    • Google Play also requires a 512×512 icon uploaded separately in the store listing
  • ✓
    Review Android Permissions in AndroidManifest.xml
    Remove any permissions your app doesn't actually use
    • Located at Platforms/Android/AndroidManifest.xml
    • Remove unused permissions — Google Play flags excessive permissions
    • Common permissions: INTERNET, ACCESS_NETWORK_STATE, CAMERA
    • Sensitive permissions (Location, Camera, Contacts) require justification in your store listing
    💡 Only declare permissions that your app genuinely needs. Reviewers and users both scrutinize this.
  • ✓
    Set Target Android Version
    Google Play requires a minimum target SDK level
    In your .csproj:
    • <SupportedOSPlatformVersion>21.0</SupportedOSPlatformVersion> — minimum Android version (21 = Android 5.0)
    • Google Play requires targeting API level 33 or higher for new apps (as of 2024)
    • In AndroidManifest.xml: android:targetSdkVersion="34"
PHASE 3 Create a Keystore & Sign Your App

Your signing key proves you are the authentic publisher — back it up permanently

  • ✓
    Create a Keystore File
    A .keystore file that permanently identifies you as the publisher
    In Visual Studio: Tools → Command Line → Developer Command Prompt, then run:
    • keytool -genkeypair -v -keystore myapp.keystore -alias myapp -keyalg RSA -keysize 2048 -validity 10000
    • You'll be prompted for: keystore password, name, organization, city, country
    • Alternatively use Android Studio's keystore wizard if installed
    ⚠️ CRITICAL: Back up your .keystore file and passwords in multiple secure locations. If you lose it, you can NEVER update your app on Google Play.
  • ✓
    Configure Signing in Your Project
    Link the keystore to your MAUI project for release builds
    Add to your .csproj inside a <PropertyGroup Condition="$(Configuration) == 'Release'"> block:
    • <AndroidKeyStore>true</AndroidKeyStore>
    • <AndroidSigningKeyStore>myapp.keystore</AndroidSigningKeyStore>
    • <AndroidSigningKeyAlias>myapp</AndroidSigningKeyAlias>
    • <AndroidSigningKeyPass>yourpassword</AndroidSigningKeyPass>
    • <AndroidSigningStorePass>yourstorepassword</AndroidSigningStorePass>
    💡 Consider using environment variables or a secrets file instead of hardcoding passwords in .csproj.
  • ✓
    Consider Enrolling in Google Play App Signing
    Google manages your release key — highly recommended for safety
    • With Play App Signing, Google holds the final release key in their secure infrastructure
    • You still sign your AAB with an upload key (your keystore) — Google re-signs for distribution
    • If you lose your upload key, Google can generate a new one — you won't lose your app
    • Enable this in Play Console: Setup → App Signing
    💡 This is now the default for new apps. Strongly recommended.
PHASE 4 Build the Release AAB

Compile a signed Android App Bundle for upload

  • ✓
    Switch to Release Configuration
    Make sure you're not building in Debug mode
    • In Visual Studio toolbar, change the dropdown from Debug to Release
    • Set target framework to net8.0-android (or your current .NET version)
    • Set the target device to Android Emulators or the framework directly (not a physical device)
  • ✓
    Publish the AAB from Visual Studio
    Generate a signed .aab file ready for upload
    • Right-click your project in Solution Explorer → Publish...
    • Or: Build → Publish [Project Name]
    • Select Android as the target platform
    • Choose Android App Bundle (.aab) — NOT APK (Play Store prefers AAB)
    • Enter your keystore path, alias, and passwords when prompted
    • Click Publish — the AAB will be output to your project's bin/Release folder
    💡 Alternatively via CLI: dotnet publish -f net8.0-android -c Release
  • ✓
    Verify the AAB File Was Created
    Confirm the output file exists before proceeding
    • Check the output folder: usually bin\Release\net8.0-android\publish\
    • Look for a file ending in .aab — e.g., com.yourcompany.app.aab
    • File size should be reasonable for your app (typically 10–100MB+)
    • A very small file (under 1MB) may indicate a build problem
PHASE 5 Create Your App in Play Console

Set up the app listing before uploading your build

  • ✓
    Create a New App in Play Console
    Start the app creation process in the dashboard
    • Go to play.google.com/console and click Create app
    • Enter your app name, default language, and whether it's an app or game
    • Declare if it's free or paid
    • Accept the declarations (developer program policies, US export laws)
  • ✓
    Complete the Main Store Listing
    Title, description, screenshots, and graphics
    Required assets:
    • Short description: max 80 characters
    • Full description: max 4000 characters
    • App icon: 512×512 PNG (32-bit, with alpha)
    • Feature graphic: 1024×500 JPG or PNG (banner shown at top of listing)
    • Screenshots: minimum 2 phone screenshots (JPEG or PNG, 16:9 or 9:16 ratio recommended)
    💡 Use the Google Play Console's graphic design tools or Canva to create a feature graphic if you don't have a designer.
  • ✓
    Complete App Content Ratings
    Fill out the content rating questionnaire (IARC)
    • Go to Policy → App Content → Content ratings
    • Complete the IARC questionnaire about your app's content
    • This assigns age ratings (Everyone, Teen, Mature, etc.) for different countries
    • This step is mandatory — you cannot publish without it
  • ✓
    Set App Category and Tags
    Helps users discover your app in search
    • Go to Store presence → Main store listing → App category
    • Choose a primary category (e.g., Health & Fitness, Tools, Education)
    • Add relevant tags to help discoverability
    • Set a contact email address for the listing
    • Add a Privacy Policy URL (required for most apps)
    ⚠️ A Privacy Policy URL is required if your app collects ANY user data (including analytics). Host one at a stable URL.
  • ✓
    Complete Data Safety Section
    Declare what data your app collects and shares
    • Go to Policy → App content → Data safety
    • Declare all data types your app collects (location, health data, contacts, etc.)
    • State whether data is shared with third parties
    • State whether data is encrypted in transit
    • This information is shown prominently on your Play Store listing
    ⚠️ Inaccurate data safety declarations can result in app removal. Be thorough and honest.
PHASE 6 Upload, Test & Release

Upload your AAB through testing tracks before going live

  • ✓
    Create an Internal Testing Release
    Test with up to 100 testers instantly (no review needed)
    • Go to Testing → Internal testing → Create new release
    • Click Upload and select your .aab file
    • Wait for the bundle to process (may take a few minutes)
    • Add release notes (what's new in this version)
    • Click Save and publish
    • Add tester email addresses under the Testers tab
    💡 Internal testing releases go live within minutes and don't go through full Google review — great for early testing.
  • ✓
    Test the App on a Real Device via Testing Link
    Verify the production build works correctly end-to-end
    • Share the internal testing opt-in link with yourself and testers
    • Install on a real Android device from the Play Store (not sideloaded)
    • Test all major features, especially permissions and network calls
    • Check that the app installs, launches, and works without debug tooling
  • ✓
    Promote to Closed Testing (Optional but Recommended)
    Broader beta test with a named group of testers
    • Go to Testing → Closed testing and create a track
    • Add tester groups by email list or Google Groups
    • Undergoes a brief Google review (usually 1–3 days)
    • Good for gathering broader feedback before full production release
  • ✓
    Create a Production Release
    Submit to full Google review for the live Play Store
    • Go to Production → Create new release
    • Upload your AAB (increment version code if needed)
    • Write release notes for users
    • Review the Errors, warnings and messages — resolve all errors before proceeding
    • Choose rollout percentage: start with 10–20% if you want a staged rollout
    • Click Send for review
    💡 First-time app reviews typically take 1–7 days. Updates to existing apps are usually faster (hours to 2 days).
  • ✓
    Monitor Review Status and Respond to Any Issues
    Google may request changes before approving
    • Check Play Console for status updates and email notifications
    • If rejected, read the rejection reason carefully in the Policy issues section
    • Fix the issue, increment the version code, rebuild, and resubmit
    • Common rejection reasons: missing privacy policy, misleading screenshots, permissions not justified
  • ✓
    App is Live — Verify the Play Store Listing
    Search for your app and confirm everything looks correct
    • Search for your app by name on Google Play (from a device not enrolled in testing)
    • Check the listing icon, screenshots, description, and category appear correctly
    • Do a fresh install from the store and test the app once more
    • Share your Play Store URL: https://play.google.com/store/apps/details?id=com.yourpackage.name
    🎉 Congratulations! Your app is published on Google Play.
