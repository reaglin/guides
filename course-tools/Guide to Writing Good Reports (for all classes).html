Academic Writing Guide

Guide to Writing Good Reports

A student reference for producing clear, professional homework submissions

Purpose of this guide: Every assignment submission tells a story — what the problem was, how you approached it, and what you found. A well-structured report demonstrates not just that you got an answer, but that you understood the problem and reasoned through it clearly. This guide walks through each section of a standard report, what belongs in it, and what to avoid.

1
Cover Page
Name · Date · Class · Assignment title
▼

The cover page is a quick identity card for your submission. It must contain — at minimum:

  • Your full name (and student ID if required)
  • Date of submission
  • Course name and number (e.g., COP 3530 – Data Structures)
  • Assignment title (e.g., Homework 2: Binary Search Trees)
  • Instructor name (recommended)

Formatting tips

Keep it clean. The cover page is not a place to demonstrate creativity — it is a place to be instantly legible. Use consistent font sizes and align content logically (centered or left-aligned block). Do not combine a cover page with the start of your Problem Statement on the same page.

✔ Do

Use the assignment name exactly as it appears in the syllabus or assignment sheet.

Include the submission date, not the assignment release date.

✘ Don't

Don't abbreviate your name or skip your course number.

Don't use the cover page for decorative graphics or filler text.

2
Problem Statement
Restate the problem clearly in your own words
▼

The Problem Statement section demonstrates that you understood what was being asked. You should restate the problem — either paraphrased or quoted from the assignment — and make it clear what inputs, constraints, and deliverables are expected.

Think of it this way: A reader who has never seen the assignment sheet should understand exactly what you were asked to do after reading your Problem Statement.

What to include

  • A clear statement of the task or question being solved
  • Any relevant inputs, data sets, or given conditions
  • What form the answer or output should take
  • Constraints or special conditions mentioned in the assignment
✔ Do

Paraphrase the problem in your own words — it shows you read and understood it.

Highlight any ambiguities you noticed and how you resolved them.

✘ Don't

Don't copy-paste the full assignment prompt verbatim without any synthesis.

Don't skip this section because "the instructor already knows what the problem is" — this is for your understanding.

3
Approach & Methodology
Your plan of attack — before showing the solution
▼

This section describes how you chose to approach the problem. It should be written before presenting the solution and should explain your reasoning, not just list the steps you took.

What to include

  • The algorithm, technique, or design pattern you chose and why
  • Any alternatives you considered and why you ruled them out
  • Assumptions you made about the problem
  • A high-level description of your process (flowchart, pseudocode, or prose)
  • Tools, languages, or libraries you used and why
This is where critical thinking lives. A strong Approach section shows the instructor you didn't just stumble onto an answer — you made informed decisions. Even if your chosen approach has limitations, discussing them here demonstrates maturity as a problem-solver.

When code is part of your methodology

Sometimes the assignment asks you to compute or analyze something, and you write code as a tool to do that work — the code is not the answer, it is how you obtained the answer. In these cases, mention in the Methodology section that you wrote a program to perform the computation, describe what it does at a high level, and note the language and any relevant libraries used. The full code goes in the Appendix (not the Solution section), since it is a tool, not a deliverable.

Compare this to assignments where the deliverable is the code itself (such as "implement this data structure"). In that case, the code belongs in the Solution section because it is the answer.

Ask yourself: "Is the code what the instructor is grading, or is it just how I found the answer?" If you would turn in the same result if you had calculated it by hand, the code is methodology — put it in the Appendix.
✔ Do

Explain the why, not just the what. "I used a hash map for O(1) lookups because the data set was large and sorted order wasn't required."

If code was a tool, describe what it computed and why you chose that approach.

✘ Don't

Don't write "I just wrote the code." That is not a methodology.

Don't paste full source code into Methodology — a brief description and pseudocode is enough here.

4
Solution & Results
The answer — presented clearly, NOT as screenshots
▼
⚠️ Critical rule: Your solution should never be a screen capture as a substitute for a written solution. Screen captures belong in the Appendix as supplemental evidence. The Solution section must present your work in a clear, readable, typeset form.

This is the core of your report. Depending on the assignment type, your solution may take different forms — but in all cases it must be self-contained and legible.

Code as the solution vs. code as a tool

This is one of the most important distinctions in technical report writing. Before writing this section, ask yourself: what is the deliverable?

💻 Code IS the Solution

The assignment asks you to implement something — a function, data structure, algorithm, or program. The code itself is what you are turning in. In this case, include the formatted, commented source code directly in the Solution section and explain how it works.

Example: "Implement a binary search function in C."

📊 Code IS a Tool

The assignment asks you to find, compute, or analyze something, and you wrote code to do the work. The code is how you found the answer, not the answer itself. Present only the results in this section — the code goes in the Appendix.

Example: "Analyze this dataset and report the mean, median, and standard deviation."

A useful test: Could you have arrived at the same answer by hand calculation or another method? If yes, the code is a tool — its job ends when it produces the result. Only the result belongs in Section 4.

For programming assignments (code IS the solution)

Present your code using formatted, syntax-highlighted code blocks. Walk through the key functions and explain what they do. Show sample input/output in a clean table or formatted block — not a screenshot. Include any complexity analysis requested.

// Example: Clean, readable code presentation
// Always include comments explaining non-obvious logic

int binarySearch(int arr[], int n, int target) {
    int left  = 0;
    int right = n - 1;

    while (left <= right) {
        int mid = left + (right - left) / 2;   // avoids integer overflow

        if (arr[mid] == target) return mid;     // found
        if (arr[mid] <  target) left  = mid + 1; // search right half
        else                    right = mid - 1; // search left half
    }

    return -1;  // target not found
}

For mathematical or analytical assignments

Show your work step by step. Label each step. A grader should be able to follow your reasoning from the given data to the final answer without guessing. Use tables where results are comparative.

For design or written assignments

Use headings, diagrams (embedded as images or described clearly), and concise prose. If you created a design, describe it explicitly — don't rely on the reader seeing what you saw.

Presenting results

After the solution, summarize your results in plain language. What answer did you get? Does it make sense? Did you verify it? A brief results narrative ties the section together.

✔ Do

Format code with consistent indentation and meaningful variable names.

Present data in tables, not prose lists, when comparing multiple values.

State the answer explicitly: "The algorithm runs in O(n log n) time."

✘ Don't

Don't paste raw, unformatted output or terminal dumps.

Don't use screenshots of your IDE as the solution itself.

Don't leave results unexplained — always say what your answer means.

5
Conclusion — optional
Reflections, limitations, and takeaways
▼

The Conclusion is optional for short homework assignments, but it adds real value when:

  • Your solution has known limitations you want to acknowledge
  • You encountered unexpected difficulties that affected your result
  • You want to reflect on what you learned from the assignment
  • There are extensions or improvements you would explore with more time
A good conclusion is 2–4 sentences to a short paragraph. It does not re-summarize the entire report. It looks forward or inward — lessons learned, caveats, or next steps.

Example of a strong conclusion (programming assignment)

"This implementation correctly solves the stated problem for all tested inputs. The current approach uses O(n) auxiliary space for the stack; a future improvement could explore an iterative solution to reduce space overhead. The assignment reinforced the relationship between recursion and explicit stack-based iteration, which I found clarifying."

✔ Do

Be honest about limitations — it shows maturity, not weakness.

Keep it brief and substantive.

✘ Don't

Don't write "In conclusion, I completed the assignment." That adds nothing.

Don't use the conclusion to apologize for errors — fix them instead.

6
Appendices
Code, screenshots, and all supporting material
▼

The Appendices are where supporting materials live. Everything that would interrupt the flow of your Solution section — but that a grader might want to verify — belongs here.

Common appendix contents

  • Full source code — especially if only excerpts were shown in the Solution section, or if code was written as a tool to compute the solution rather than as the solution itself
  • Screen captures — program output, test runs, IDE windows, terminal sessions
  • Test data — input files, data sets, or sample test cases
  • Reference materials — any cited sources, algorithm references, or external documentation
  • Additional diagrams or UML — supporting design artifacts
Label each appendix clearly: Appendix A: Full Source Code, Appendix B: Sample Output Screenshots, etc. Reference them from the body of the report: "(see Appendix A)".

When code belongs in the Appendix (not Section 4)

If the assignment asked you to compute or analyze something and you wrote a script or program to do it, that code is a methodology tool. Place the full source code in the Appendix with a label like Appendix A: Python script used to compute descriptive statistics. Your Solution section (Section 4) should contain only the results the code produced — presented in a table or formatted clearly — not the code itself.

✔ Do

Include screen captures here as evidence that your program ran correctly.

Put methodology code here when the result — not the code — is the deliverable.

Label every appendix and reference it from the main body.

✘ Don't

Don't substitute appendix screenshots for a written solution in Section 4.

Don't dump unorganized files — label and order everything logically.

Don't put solution code in the Appendix when the implementation is what was asked for.

AssignmentHomework 3: Implementing Binary Search in C
Section 2 — Problem Statement ▼
Problem Statement

This assignment requires implementing the binary search algorithm in C. Given a sorted integer array of arbitrary size and a target value, the program must return the index of the target if found, or -1 if the target is not present. The implementation must run in O(log n) time and must be tested on at least three cases: target found in the middle, at an endpoint, and not present in the array.

Section 3 — Approach & Methodology ▼
Approach & Methodology

Binary search works on the divide-and-conquer principle: by maintaining left and right boundary pointers on a sorted array, each iteration eliminates half the remaining search space. This guarantees O(log n) time complexity, which is the requirement.

I chose the iterative approach over a recursive one for two reasons: (1) it avoids function call overhead and stack growth on large inputs, and (2) the iterative version is simpler to reason about for index boundary conditions, which are the most common source of off-by-one errors in binary search.

The key design decision was the midpoint calculation: using mid = left + (right - left) / 2 instead of (left + right) / 2 to prevent integer overflow when indices are large.

Section 4 — Solution & Results ▼
Implementation
// binary_search.c  –  COP 3530 Homework 3
// Author: Alex Rivera  |  Date: Sept 20, 2024

#include <stdio.h>

int binarySearch(int arr[], int n, int target) {
    int left  = 0;
    int right = n - 1;

    while (left <= right) {
        int mid = left + (right - left) / 2;

        if      (arr[mid] == target) return mid;
        else if (arr[mid] <  target) left  = mid + 1;
        else                         right = mid - 1;
    }
    return -1;   // not found
}

int main() {
    int data[] = {2, 5, 8, 12, 16, 23, 38, 42, 55, 72};
    int n = 10;

    printf("Search 23:  index %d\n", binarySearch(data, n, 23));
    printf("Search 2:   index %d\n", binarySearch(data, n,  2));
    printf("Search 99:  index %d\n", binarySearch(data, n, 99));

    return 0;
}
Test Results
Test Case Target Expected Index Actual Index Pass?
Middle element 23 5 5 ✓
First element 2 0 0 ✓
Not present 99 −1 −1 ✓

All three test cases produced the expected results. The function correctly handles elements at the boundary (index 0) and correctly returns −1 for values outside the array.

Section 5 — Conclusion ▼
Conclusion

The implementation meets all stated requirements and passes all three required test cases. The iterative approach avoided recursion-related edge cases and produces a compact, readable solution. One area for future improvement would be extending the function to work with generic types using function pointers, or adding a version that returns all matching indices for duplicate values in the array.

Section 2 — Problem Statement ▼
Problem Statement

Given a file containing the exam scores of 50 students (provided as scores.txt), compute the following descriptive statistics for the dataset: mean, median, mode, standard deviation, minimum, and maximum. Identify how many students scored in each letter-grade band (A: 90–100, B: 80–89, C: 70–79, D: 60–69, F: below 60). Present the results in a clearly formatted summary.

The deliverable is the statistical summary and grade distribution table — not the program used to compute them.

Section 3 — Approach & Methodology ▼
Approach & Methodology

I wrote a Python script to load the scores from scores.txt, compute all required statistics, and classify each score into its letter-grade band. Python was chosen for its built-in statistics module and list comprehension capabilities, which make this type of data processing concise and readable.

The script reads the file line by line into a list, sorts the values (necessary for median and grade-band counting), then applies the standard formulas for each statistic. The full source code is included in Appendix A.

No manual calculation was performed; all values were computed programmatically and verified by cross-checking the mean against a spot calculation on a subset of the data.

Appendix A — Python Script Used for Analysis ▼
Appendix A

This script was written to compute the statistics reported in Section 4. It is supporting methodology, not the solution itself.

# grade_analysis.py  –  COP 3530 Homework 5
# Author: Jordan Kim  |  Date: Oct 11, 2024
# PURPOSE: Compute descriptive stats and grade bands for scores.txt
# NOTE: This script is methodology. The results it produces are the solution.

import statistics

# ── Load scores ──────────────────────────────────────
with open("scores.txt") as f:
    scores = [int(line.strip()) for line in f if line.strip()]

# ── Descriptive statistics ───────────────────────────
mean   = statistics.mean(scores)
median = statistics.median(scores)
mode   = statistics.mode(scores)
stdev  = statistics.stdev(scores)
lo     = min(scores)
hi     = max(scores)

# ── Grade band counts ────────────────────────────────
bands = {"A": 0, "B": 0, "C": 0, "D": 0, "F": 0}
for s in scores:
    if   s >= 90: bands["A"] += 1
    elif s >= 80: bands["B"] += 1
    elif s >= 70: bands["C"] += 1
    elif s >= 60: bands["D"] += 1
    else:         bands["F"] += 1

# ── Print results (used to verify output) ────────────
n = len(scores)
print(f"Mean:   {mean:.1f}  |  Median: {median}  |  Mode: {mode}")
print(f"StDev:  {stdev:.1f}  |  Min: {lo}  |  Max: {hi}")
for grade, count in bands.items():
    print(f"  {grade}: {count} students ({count/n*100:.0f}%)")

Pre-Submission Checklist

✓
Before You Submit — Checklist
Use this to review your report before turning it in
▼

Cover Page

  • Full name, date, course name and number, and assignment title are present
  • Cover page is on its own page, not merged with Section 2

Problem Statement

  • The problem is described clearly enough that someone unfamiliar with the assignment understands what was asked
  • Inputs, outputs, and constraints are identified

Approach & Methodology

  • Explains why this approach was chosen, not just what was done
  • Any assumptions made about the problem are stated

Solution & Results

  • No screen captures used as a substitute for a written solution
  • If code is the solution (you were asked to implement something): formatted code with explanation is present in this section
  • If code was a tool used to reach the solution: only results and findings appear here — the code is in the Appendix
  • Code is properly formatted and indented
  • Results are presented in a table or structured format, not as a paragraph of numbers
  • The final answer is explicitly stated

Appendices

  • All appendices are labeled (Appendix A, B, …)
  • Appendices are referenced from the body of the report
  • Screen captures (if any) are in the appendix, not the solution section

General

  • Spell-checked and proofread
  • Sections are in the correct order
  • Document is submitted in the required format (PDF, Word, etc.)
  • File is named according to the course naming convention
